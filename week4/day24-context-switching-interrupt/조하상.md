## 1. 컨텍스트 스위칭이란 무엇이며 왜 발생하는가?

**멀티태스킹의 핵심 기법**
- 실행 중인 프로세스나 스레드의 상태를 저장하고, 다음에 실행할 프로세스의 상태를 복구하여 CPU를 할당하는 과정을 말한다.
- **발생 이유:** 하나의 CPU가 여러 개의 프로세스를 번갈아 가며 처리하여, 사용자에게는 여러 프로그램이 동시에 실행되는 것처럼 보이게 하기 위함이다. 주로 멀티태스킹 환경에서의 시분할(Time Sharing), 입출력 대기, 인터럽트 발생 시에 나타난다.

---

## 2. 컨텍스트 스위칭 과정을 구체적으로 설명하면?

**PCB를 이용한 상태 관리**
1. 인터럽트나 시스템 콜에 의해 컨텍스트 스위칭 요청이 들어온다.
2. 현재 실행 중인 프로세스(P1)의 상태(CPU 레지스터, PC, 스택 포인터 등)를 해당 프로세스의 **PCB(Process Control Block)**에 저장한다.
3. OS의 스케줄러가 다음에 실행할 프로세스(P2)를 선택한다.
4. 선택된 프로세스(P2)의 PCB에서 이전에 저장했던 상태 정보를 CPU 레지스터에 복구한다.
5. P2가 마지막으로 실행되었던 지점부터 다시 실행을 시작한다.



---

## 3. 컨텍스트 스위칭의 오버헤드는 무엇이며 어떻게 줄일 수 있는가?

**성능 저하의 요인과 최적화**
- **오버헤드:** 상태 저장 및 복구에 걸리는 시간, 스케줄러 알고리즘 실행 시간, 그리고 **캐시 플러시(Cache Flush)**로 인한 TLB 미스 및 캐시 미스 등이 발생하여 시스템 전체 성능을 떨어뜨린다.
- **줄이는 방법:** - 프로세스 대신 공유 메모리를 사용하는 **스레드(Thread)**를 활용하여 스위칭 비용을 낮춘다.
  - 컨텍스트 스위칭 빈도를 적절히 조절한다(타임 슬라이스 크기 최적화).
  - 프로세스 어피니티(Process Affinity)를 통해 특정 CPU에서 계속 실행되게 함으로써 캐시 효율을 높인다.

## 4. 인터럽트는 왜 필요하며 어떻게 동작하는가?

**비동기적 사건의 즉각적 처리**
- CPU가 프로그램을 실행하고 있을 때, 입출력 완료나 하드웨어 오류 등 외부의 긴급한 사건이 발생했음을 알리기 위해 필요하다.
- **동작 과정:** 인터럽트가 발생하면 CPU는 현재 작업을 중단하고 상태를 저장한다. 이후 **인터럽트 벡터(Interrupt Vector)**를 참조하여 해당 인터럽트를 처리할 **인터럽트 서비스 루틴(ISR)**으로 점프하여 실행한다. 처리가 끝나면 다시 원래 작업으로 돌아온다.



---

## 5. 인터럽트와 폴링의 차이는? 각각 언제 사용하는가?

- **폴링(Polling):** CPU가 주기적으로 장치의 상태를 확인하며 사건 발생 여부를 체크하는 방식이다. 구현은 간단하지만 무의미한 CPU 사이클 낭비가 심하다. 아주 짧고 예측 가능한 작업이나, 하드웨어 구조상 인터럽트 지원이 안 될 때 사용한다.
- **인터럽트(Interrupt):** 장치가 사건 발생 시 직접 CPU에 신호를 보내 알리는 방식이다. CPU가 다른 작업을 하다가 필요할 때만 대응하므로 효율적이다. 대부분의 범용적인 I/O 처리에 사용한다.

---

## 6. 인터럽트 우선순위는 어떻게 결정되는가?

**중요도와 긴급성에 따른 차등**
- 하드웨어적으로 인터럽트 라인마다 우선순위가 부여되어 있거나, 소프트웨어적으로 **인터럽트 마스크(Interrupt Mask)**를 설정하여 결정한다.
- 일반적으로 하드웨어 고장, 전원 이상과 같은 비마스크 인터럽트(NMI)가 가장 높으며, 그다음 타이머, 키보드/마우스, 네트워크 통신 순으로 우선순위가 정해진다. 낮은 우선순위의 인터럽트 처리 중에 높은 우선순위의 인터럽트가 오면 중첩 처리가 가능하다.

---

## 7. 시스템 콜은 어떻게 동작하는가? User Mode와 Kernel Mode 전환 과정은?

**사용자와 커널 사이의 인터페이스**
- 응용 프로그램이 커널의 권한이 필요한 서비스(파일 읽기, 메모리 할당 등)를 요청할 때 사용한다.
- **전환 과정:**
  1. 사용자 모드에서 실행 중인 앱이 시스템 콜을 호출한다.
  2. 소프트웨어 인터럽트(Trap)가 발생하며 CPU의 **Mode Bit**이 1(User)에서 0(Kernel)으로 바뀐다.
  3. 커널 영역의 시스템 콜 핸들러가 요청된 작업을 수행한다.
  4. 작업 완료 후 Mode Bit을 다시 1로 돌려주고 사용자 모드로 복귀하여 결과값을 전달한다.
