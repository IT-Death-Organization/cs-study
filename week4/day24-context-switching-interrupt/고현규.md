# 1. 컨텍스트 스위칭이란 무엇이며 왜 발생하는가?

> 컨텍스트 스위칭(Context Switching)이란 CPU가 현재 실행 중인 프로세스(또는 스레드)의 상태를 저장하고, 다음 실행할 프로세스의 상태를 읽어와 교체하는 작업을 말합니다.

우리가 사용하는 운영체제는 여러 프로그램을 동시에 실행하는 것처럼 보이게 하기 위해 CPU 자원을 아주 짧은 시간 단위로 쪼개서 각 프로세스에 할당합니다. 이때 작업의 문맥(Context)을 갈아끼우는 과정이 필요합니다.

### 발생원인

* 멀티태스킹 (Time Slice 만료)
    * 할당된 CPU 사용 시간(Time Quantum)을 모두 소모했을 때 발생합니다.
* I/O 인터럽트
    * 프로세스가 입출력 작업(파일 읽기, 네트워크 대기 등)을 요청하면, CPU는 기다리지 않고 다른 작업을 수행하기 위해 스위칭합니다.
* 시스템 콜/인터럽트 발생
    * 커널 모드에서 처리해야 하는 긴급한 작업이나 예외 상황이 발생했을 때 일어납니다.

<br>

# 2. 컨텍스트 스위칭 과정을 구체적으로 설명하면?

> 컨텍스트 스위칭은 운영체제의 커널(Scheduler)에 의해 제어되며, PCB(Process Control Block)라는 공간을 활용합니다.

### 구체적인 단계

1. 상태 보존
    * 현재 CPU 레지스터에 담긴 값, Program Counter(다음에 실행할 줄 위치) 등을 현재 프로세스의 PCB에 저장합니다.
2. 커널 모드 진입
    * 스케줄러가 다음 실행할 프로세스를 결정합니다.
3. 상태 복구
    * 새로 선택된 프로세스의 PCB에서 이전에 저장했던 실행 상태를 CPU 레지스터로 다시 불러옵니다.
4. 사용자 모드 복귀
    * CPU는 이제 새로운 프로세스의 마지막 실행 지점부터 다시 작업을 시작합니다.

<br>

참고로 스레드 간 스위칭 시에는 TCB(Thread Control Block)를 사용하며, 같은 프로세스 내의 스레드라면 공유 자원(힙, 데이터)을 그대로 두기 때문에 프로세스 스위칭보다 훨씬 가볍습니다.

<br>

# 3. 컨텍스트 스위칭의 오버헤드는 무엇이며 어떻게 줄일 수 있는가?

> 컨텍스트 스위칭은 실제 사용자의 계산 업무를 하는 시간이 아니라 시스템 유지를 위한 청소 시간 혹은 준비 시간입니다. 이 시간이 길어지면 시스템의 효율이 떨어지는데, 이를 오버헤드(Overhead)라고 합니다.

### 오버헤드의 원인

* CPU 사이클 소모
    * PCB를 저장하고 복원하는 연산 자체가 CPU를 점유합니다.
* 캐시 미스(Cache Miss)
    * 새로운 프로세스가 실행되면 기존 CPU 캐시에 쌓여있던 데이터들은 필요없어집니다. 메모리에서 다시 데이터를 가져와야 하므로 일시적으로 성능이 급락합니다.
* TLB 플러시
    * 가상 메모리 주소를 물리 주소로 변환하는 정보를 담은 TLB(Translation Lookaside Buffer)를 비워야 하는 비용이 큽니다.

### 해결 방법

* 멀티 스레드 활용
    * 프로세스보다 가벼운 스레드를 사용하여 캐시 정보를 최대한 재활용합니다.
* 적절한 타임 퀀텀 설정
    * 시간 단위가 너무 작으면 스위칭이 잦아져 오버헤드가 커지고, 너무 크면 응답성이 떨어집니다. 서비스 특성에 맞는 최적의 값을 찾아야 합니다.
* LWP(Light Weight Process) 사용
    * 커널 레벨 스레드와 사용자 레벨 스레드 사이의 매핑을 효율적으로 관리합니다.

<br>

# 4. 인터럽트는 왜 필요하며 어떻게 동작하는가?

> 인터럽트(Interrupt)는 CPU가 프로그램을 실행하고 있을 때, 외부/내부의 긴급한 이벤트가 발생했음을 알리고 이를 즉시 처리하도록 하는 메커니즘입니다.

### 필요성

만약 인터럽트가 없다면 CPU는 프린터가 인쇄를 끝냈는지, 마우스가 움직였는지(I/O 작업) 확인하기 위해 주기적으로 모든 장치를 체크해야 합니다. 인터럽트는 장치가 먼저 작업이 끝났다고 알려줌으로써 CPU가 다른 일에 집중할 수 있게 해줍니다.

### 동작 과정

1. 이벤트 발생
    * 하드웨어(키보드 등)나 소프트웨어(0으로 나누기 등)에서 신호를 보냅니다.

2. 중단 및 저장
    * CPU는 현재 수행 중인 명령을 마친 후, 현재 상태를 임시 저장합니다.

3. ISR 실행
    * 인터럽트 벡터 테이블을 참조하여 해당 이벤트에 맞는 처리 코드인 **ISR(Interrupt Service Routine)**을 실행합니다.

4. 복구
    * ISR이 끝나면 원래 하던 작업으로 돌아갑니다.

<br>

# 5. 인터럽트와 폴링의 차이는? 각각 언제 사용하는가?

| 비교 항목 | 인터럽트 (Interrupt) | 폴링 (Polling) |
| :--- | :--- | :--- |
| **방식** | 이벤트 발생 시 CPU에 신호를 보냄 | CPU가 주기적으로 장치 상태를 확인 |
| **주체** | 하드웨어 / 외부 장치 | CPU (소프트웨어) |
| **CPU 부하** | 이벤트가 없을 때 부하가 거의 없음 | 확인하는 작업 자체가 CPU 자원을 소모 |
| **반응 속도** | 즉각적임 | 주기(Interval)에 따라 지연이 생길 수 있음 |
---

### 언제 사용하는가?

* 인터럽트
    * 발생 시점을 예측할 수 없거나, 드물게 발생하는 경우(키보드 입력, 오류 발생 등)에 유리합니다.
* 폴링
    * 이벤트가 매우 빈번하거나, 인터럽트 처리 오버헤드가 폴링 확인 비용보다 클 정도로 아주 빠른 장치(고속 네트워크 카드 등)에서 간혹 사용됩니다.

<br>

# 6. 인터럽트 우선순위는 어떻게 결정되는가?

> 여러 인터럽트가 동시에 발생할 경우, 어떤 것을 먼저 처리할지 결정해야 합니다.

1. 하드웨어적 방식 (Daisy Chain)
    * 여러 장치를 직렬로 연결하여 CPU에 가까운 순서대로 높은 우선순위를 부여합니다.

2. 소프트웨어적 방식 (Polling)
    * CPU가 각 장치를 하나씩 체크하며 가장 급한 놈부터 찾아냅니다. (느림)

3. 중첩 인터럽트 (Masking)
    * 현재 중요한 ISR을 실행 중일 때, 그보다 낮은 순위의 인터럽트 신호는 무시(Mask)하고 더 높은 순위만 받아들이는 방식입니다.

<br>

보통 전원 공급 장치의 이상이나 하드웨어 오류 같은 Non-Maskable Interrupt(NMI)가 가장 높은 순위를 가집니다. 데이터 손실보다 시스템 파손 방지가 우선이기 때문입니다.

<br>

# 7. 시스템 콜은 어떻게 동작하는가? User Mode와 Kernel Mode 전환 과정은?

> 시스템 콜(System Call)은 사용자 프로그램이 운영체제의 커널 서비스(파일 접근, 메모리 할당 등)를 요청하기 위해 사용하는 인터페이스입니다.

### User Mode와 Kernel Mode 전환 과정은?

1. User Mode
    * 일반 애플리케이션은 제한된 자원에만 접근 가능합니다. (보안 목적)

2. Trap 발생
    * 프로그램이 파일 쓰기 같은 작업을 위해 시스템 콜을 호출하면, CPU에 소프트웨어 인터럽트(Trap)가 걸립니다.

3. Mode Bit 변경
    * CPU의 Mode Bit이 1(User)에서 0(Kernel)으로 바뀌며 커널 모드로 진입합니다.

4. 서비스 실행
    * 운영체제가 요청을 검증하고 대신 수행합니다.

5. 복귀
    * 작업이 끝나면 Mode Bit을 다시 1로 돌려주고 사용자 프로그램으로 제어권을 넘깁니다.
