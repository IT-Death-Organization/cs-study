## 1. 컨텍스트 스위칭이란 무엇이며 왜 발생하는가 ?

### 1-1. 컨텍스트 스위칭이란?

- 컨텍스트 스위칭 (문맥 교환)은 **CPU가 한 프로세스 (또는 스레드)에서 다른 프로세스로 전환**할 때, **현재 프로세스의 상태를 PCB(Process Control Block)에 저장**하고 새로운 프로세스의 상태를 로드하는 과정을 의미합니다.
- 멀티태스킹 환경에서 **여러 작업을 동시에** 하기 위해, 짧은 시간 동안 프로세스를 번갈아가며 실행하는 필수 메커니즘입니다.

### 1-2. 컨텍스트 스위칭의 주요 발생 원인

1. 멀티태스킹 : CPU 활용도를 높이고  여러 프로그램을 동시에 실행하는 것처럼 보이게 하기 위해
2. 인터럽트 발생 : 입출력(I/O) 요청, 하드웨어 인터럽트 등으로 인해 현재 작업보다 더 급한 처리가 필요할 때
3. CPU 시간 제한(Time Slice) 만료 : 시분할 시스템에서 프로세스가 할당된 CPI 시간을 모두 사용했을 때
4. 프로세스 스스로 대기 상태 진입 : I/O 작업, 파일 읽기 등을 기다려야 할 때

## 2. 컨텍스트 스위칭 과정을 구체적으로 설명하면 ?

1. **인터럽트 또는 시스템 콜 발생**: 스케줄러에 의해 CPU 할당 시간이 끝났음을 알리는 타이머 인터럽트가 발생하거나, 프로세스가 입출력을 요청하는 시스템 콜을 호출합니다.
2. **커널 모드 진입**: 프로세스가 실행되던 '유저 모드'에서 운영체제가 제어권을 갖는 '커널 모드'로 전환됩니다.
3. **현재 상태 저장 (Save)**: 현재 실행 중인 프로세스의 레지스터 값, 프로그램 카운터(PC), 스택 포인터 등을 해당 프로세스의 **PCB(Process Control Block)**에 저장합니다.
4. **다음 프로세스 선택 (Scheduling)**: 스케줄러가 다음에 실행할 프로세스를 준비 큐(Ready Queue)에서 선택합니다.
5. **새 상태 복구 (Restore)**: 선택된 프로세스의 PCB에서 이전에 저장했던 상태 정보를 CPU 레지스터에 다시 적재합니다.
6. **유저 모드 복귀**: CPU가 새로운 프로세스의 명령어를 실행할 수 있도록 다시 유저 모드로 전환하며 실행을 재개합니다.

## 3. 컨텍스트 스위칭의 오버헤드는 무엇이며 어떻게 줄일 수 있는가?

> **컨텍스트 스위칭 오버헤드**는 CPU가 프로세스를 전환하는 과정에서 발생하는 순수 관리용 시간 소비를 의미합니다. 즉, 실제 계산 업무를 처리하지 못하고 상태를 저장·복구하는 데 자원을 낭비하게 되는 현상입니다.


### 3-1. **주요 오버헤드 요인**

- **PCB 저장 및 복구**: 레지스터 세트, 프로그램 카운터 등의 정보를 메모리에 쓰고 읽는 비용입니다.
- **캐시 미스(Cache Miss)**: 새로운 프로세스가 실행되면 기존 CPU 캐시 데이터가 무효화되는데, 이를 채우기 위해 느린 메인 메모리에 다시 접근하면서 성능이 급격히 떨어집니다.
- **스케줄링 알고리즘 실행**: 다음 실행 대상을 고르는 로직 자체가 CPU 자원을 사용합니다.

### 3-2. 오버헤드를 줄이는 방법

1. **멀티 스레드 활용**: 프로세스보다 스레드 간 스위칭이 훨씬 가볍습니다. 스레드는 메모리 주소 공간을 공유하므로 **캐시 데이터를 유지**할 가능성이 높고 저장할 정보량도 적습니다.
2. **불필요한 컨텍스트 스위칭 최소화**: 인터럽트 빈도를 조절하거나, 동기화 도구(Lock) 사용 시 대기 시간을 최적화하여 전환 횟수를 줄입니다.
3. **스택리스 코루틴(Coroutines) 사용**: 운영체제가 아닌 애플리케이션 레벨에서 실행 흐름을 제어하여 커널 모드 진입 비용을 없앱니다. (예: Go의 Goroutine, Python의 Asyncio)
4. **하드웨어 지원**: CPU 가속 기능을 사용하여 레지스터 세트를 빠르게 교체하거나 관리합니다.

## 4. 인터럽트는 왜 필요하며 어떻게 동작하는가?

> **인터럽트(Interrupt)**는 CPU가 프로그램을 실행하고 있을 때, **외부나 내부에서 발생한 예외적인 사건을 즉각적으로 처리하기 위해 실행 중인 작업을 잠시 중단시키는 메커니즘**입니다.


### 4-1. 인터럽트가 필요한 이유

- **효율적인 자원 관리**: CPU가 입출력 장치의 작업 완료 여부를 계속 확인(Polling)하며 기다리는 낭비를 막고, 장치가 준비되었을 때만 응답하게 합니다.
- **실시간 대응**: 키보드 입력, 네트워크 데이터 수신, 하드웨어 오류 등 즉각적인 처리가 필요한 이벤트에 빠르게 반응할 수 있습니다.
- **멀티태스킹 구현**: 타이머 인터럽트를 통해 특정 시간이 지나면 강제로 제어권을 회수하여 여러 프로세스가 CPU를 나눠 쓰게 합니다.

### 4-2. 인터럽트 동작 과정

1. **인터럽트 요청**: 입출력 장치나 예외 상황 발생 시 CPU에 인터럽트 신호를 보냅니다.
2. **현재 상태 보존**: CPU는 현재 실행 중인 명령어(Instruction)를 마친 후, 중단된 위치와 레지스터 정보를 **PCB**나 **스택**에 임시 저장합니다.
3. **서비스 루틴 실행**: CPU는 **인터럽트 벡터**(종류별 처리 함수의 주소 테이블)를 참조하여 해당 인터럽트를 처리할 **ISR(Interrupt Service Routine)**로 점프해 코드를 실행합니다.
4. **복구 및 재개**: 처리가 끝나면 저장해두었던 이전 상태를 다시 불러와 원래 수행하던 작업으로 돌아갑니다.

## 5. 인터럽트와 폴링의 차이는? 각각 언제 사용하는가?

> **인터럽트는 장치가 CPU에게 신호를 보내는 방식이고, 폴링(Polling)은 CPU가 장치를 주기적으로 확인하는 방식으로,** 가장 큰 차이는 **"누가 주도권을 갖느냐"**에 있습니다.


| **구분** | **인터럽트 (Interrupt)** | **폴링 (Polling)** |
| --- | --- | --- |
| **주체** | **하드웨어/장치**가 CPU를 부름 | **CPU**가 장치 상태를 직접 확인 |
| **효율성** | 이벤트가 없을 때 CPU는 자기 일만 함 (높음) | 무의미한 확인 반복으로 CPU 리소스 낭비 (낮음) |
| **응답성** | 즉각적인 대응 가능 | 확인 주기(Polling Rate)에 따라 지연 발생 가능 |
| **구현** | 복잡함 (HW 지원 및 서비스 루틴 필요) | 단순함 (반복문으로 구현 가능) |

### **5-1. 인터럽트를 사용하는 경우 (대부분의 현대 시스템)**

- **발생 시점이 불분명할 때**: 키보드 입력, 마우스 클릭, 네트워크 패킷 수신 등 언제 일어날지 모르는 이벤트 처리에 최적입니다.
- **멀티태스킹 환경**: CPU가 입출력을 기다리지 않고 다른 프로세스를 실행해야 하는 일반적인 운영체제 환경에서 필수적입니다.

### **5-2. 폴링을 사용하는 경우 (특수한 상황)**

- **매우 빈번한 이벤트**: 데이터가 쉼 없이 쏟아져서 인터럽트를 매번 걸면 오히려 컨텍스트 스위칭 비용(오버헤드)이 더 클 때 사용합니다. (예: 초고속 네트워크 카드)
- **단순한 임베디드 시스템**: 운영체제가 없거나 하드웨어 구조가 아주 단순하여 인터럽트 컨트롤러를 달기 어려운 환경에서 씁니다.
- **짧은 대기 시간**: 장치의 작업 완료가 아주 순식간에 끝날 것이 확실할 때는 잠깐 루프를 돌며 기다리는 게 빠를 수 있습니다.

실제로 고성능 시스템에서는 평소엔 **인터럽트**로 처리하다가, 트래픽이 폭주하면 잠시 **폴링** 방식으로 전환하는 하이브리드 방식을 쓰기도 합니다.

## 6. 인터럽트 우선순위는 어떻게 결정되는가?

> 인터럽트 우선순위는 **시스템의 안정성과 긴급도**에 따라 결정되며, 크게 하드웨어와 소프트웨어 두 가지 방식으로 관리됩니다.


### **6-1. 우선순위 결정 원칙 (일반적 순서)**

보통 시스템에 치명적인 영향을 주거나 데이터 손실 가능성이 높은 순서대로 높은 순위를 가집니다.

1. **전원 이상 (Power Fail)**: 가장 높은 순위 (데이터 보존 및 종료 준비)
2. **기계 착오 (Machine Check)**: 하드웨어 오류 발생
3. **외부 신호 (External)**: 타이머 인터럽트 등 스케줄링 관련
4. **입출력 (I/O)**: 키보드, 네트워크, 디스크 작업 완료
5. **명령어 단계 (Program)**: 0으로 나누기, 잘못된 메모리 참조 (트랩)

### **6-2. 판별 및 제어 방법**

여러 인터럽트가 동시에 발생했을 때 어떤 것을 먼저 처리할지 결정하는 메커니즘입니다.

- **폴링(Polling) 방식 (소프트웨어)**: CPU가 각 장치를 하나씩 체크하며 인터럽트 원인을 찾습니다. 먼저 체크하는 장치가 자연스럽게 우선순위가 높지만, 속도가 느립니다.
- **데이지 체인(Daisy Chain) 방식 (하드웨어)**: 장치들을 직렬로 연결하여 CPU에 가까운 장치가 우선순위를 가집니다. 신호가 물리적 연결 순서대로 전달됩니다.
- **Vectored Interrupt (PIC 활용)**: **인터럽트 컨트롤러(PIC)**라는 전용 하드웨어가 각 장치의 우선순위를 판별하여 CPU에 알립니다. 현대 시스템에서 가장 많이 쓰이며 설정 변경이 유연합니다.

### **6-3. 인터럽트 네스팅 (Interrupt Nesting)**

현재 인터럽트를 처리 중일 때 **더 높은 순위**의 인터럽트가 들어오면, 현재 작업을 멈추고 새 인터럽트를 먼저 처리합니다. 반대로 낮은 순위의 인터럽트는 현재 작업이 끝날 때까지 대기(Masking)시킵니다.

## 7. 시스템 콜은 어떻게 동작하는가? User Mode와 Kernel Mode 전환 과정은?

> **시스템 콜(System Call)**은 유저 프로그램이 운영체제의 커널 서비스(파일 읽기, 메모리 할당, 네트워크 통신 등)를 요청하기 위한 유일한 통로입니다. 이 과정에서 보안과 시스템 보호를 위해 **유저 모드(User Mode)**에서 **커널 모드(Kernel Mode)**로의 전환이 일어납니다.
> 

### **7-1. 시스템 콜 동작 및 모드 전환 5단계**

1. **유저 프로그램의 요청**: 애플리케이션이 실행 중 라이브러리 함수(예: C의 `printf()`, `open()`)를 호출합니다. 이 함수 내부에서 실제 시스템 콜 번호를 레지스터에 저장하고 특정 명령어(예: x86의 `syscall` 또는 `int 0x80`)를 실행합니다.
2. **트랩(Trap) 발생**: 위 명령어가 실행되면 하드웨어적인 **소프트웨어 인터럽트(트랩)**가 발생합니다. 이때 CPU는 현재 실행 중인 유저 프로세스의 상태(PC, 레지스터 등)를 커널 스택에 저장합니다.
3. **모드 전환 (User → Kernel)**: CPU의 제어권이 운영체제로 넘어가며, CPU 내의 모드 비트(Mode Bit)가 유저 모드(1)에서 **커널 모드(0)**로 변경됩니다. 이제 모든 하드웨어 자원에 접근할 수 있는 권한을 얻습니다.
4. **서비스 루틴 실행**: 커널은 **시스템 콜 테이블**을 참조하여 요청된 번호에 맞는 커널 함수(예: `sys_write`)를 찾아 실행합니다.
5. **복귀 및 모드 전환 (Kernel → User)**: 작업이 완료되면 결과값을 반환하고, 저장했던 유저 프로세스의 상태를 복구합니다. 모드 비트를 다시 유저 모드(1)로 돌려놓으며 원래 프로그램의 다음 줄부터 실행을 재개합니다.

### **7-2. 모드를 나누는 이유?**

유저 프로그램이 직접 하드웨어나 메모리 전체에 접근하면 실수나 악의적인 공격으로 시스템 전체가 멈출 수 있기 때문입니다. 이를 **이중 모드(Dual-mode) 동작**이라 하며 하드웨어적으로 보호받습니다.
