## Q1. 컨텍스트 스위칭이란 무엇이며 왜 발생하는가?

[답변]
* 컨텍스트 스위칭
  - CPU가 현재 실행 중인 작업의 실행 상태 저장 -> 다른 작업의 실행 상태 복원
  - 실행 대상 바꾸는 과정
  - '컨텍스트': 프로세스나 스레드가 다시 이어서 실행되기 위해 필요한 CPU 상태 정보
    - 프로그램 카운터(PC), 레지스터 값, 스택 포인터(SP), 상태 레지스터 등

* 발생 이유
  - 실행 중인 프로세스의 타임 슬라이스 종료
  - 더 높은 우선순의 작업이 Ready
  - 현재 작업이 I/O 완료를 기다림 -> Block 상태 전환
  - 인터럽트 / 예외 발생
  - 시스템 콜 수행 -> 커널이 스케줄링 결정 내릴 때

---

## Q2. 컨텍스트 스위칭 과정을 구체적으로 설명하면?

[답변]
1. 현재 실행 중인 작업: 인터럽트 / 시스템 콜 / 예외 / 타이머 만료 발생
2. CPU: 현재 작업의 실행 잠시 중단 -> 커널 모드 진입
3. 운영체제: 현재 작업의 CPU 상태를 해당 작업의 PCB 또는 TCB에 저장
4. 저장되는 정보: 일반 레지스터, 프로그램 카운터, 스택 포인터, 플래그 레지스터, FPU/SIMD 레지스터 등
5. 현재 작업 상태: Running -> Ready / Blocked
6. 스케줄러: 다음 실행 작업 선택
7. 선택된 작업의 PCB/TCB: 이전에 저장된 CPU 상태 복원
8. 메모리 주소 공간이 다르면 -> 페이지 테이블 관련 정보 전환
9. CPU: 복원된 프로그램 카운터 위치부터 새 작업 실행

[보완]
* PCB(Process Control Block)
  - 프로세스의 상태, PID, 스케줄링 정보, 메모리 정보, 파일 정보, 레지스터 상태 등 저장하는 커널 자료구조
* TCP(Thread Control Block)
  - 스레드 단위 실행 상태를 저장하는 자료구조
 * 상태 전이
   - Running -> Ready: CPU 사용 시간 종료, 선점 발생
   - Running -> Blocked: I/O 대기, 이벤트 대기
   - Ready -> Running: 스케줄러가 CPU 할당
* 주소 공간 전환
  - 프로세스 간 전환에서는 가상 메모리 매핑도 바뀔 수 있어 TLB 무효화 등 추가 비용 발생 가능

## Q3. 컨텍스트 스위칭의 오버헤드는 무엇이며 어떻게 줄일 수 있는가?

[답변]
* 컨텍스트 스위칭 오버헤드: CPU가 실제 사용자 작업을 수행하지 않고, 작업 전환 자체에 시간을 쓰는 비용
* 대표적인 오버헤드
  - 레지스터 저장 및 복원 비용
  - 커널 모드 진입 및 복귀 비용
  - 스케줄러 실행 비용
  - 프로세스 간 전환 시 주소 공간 변경 비용
  - TLB 무효화 또는 재구성 비용
  - 캐시 적중률 저하로 인한 성능 하락

* 줄이는 방법
  - 불필요한 스레드/프로세스 수 줄임
  - 락 경합과 잦은 Block/Wakeup 줄임 -> 스케줄링 빈도 낮춤
  - 너무 짧은 타임 슬라이스 설정 피함
  - 가능한 경우 프로세스보다 스레드 활용
  - CPU affinity를 활용 -> 같은 코어에서 계쏙 실행되도록 유도
  - 이벤트 기반 비동기 I/O 사용 -> 대기 때문에 자주 전환되는 상황 줄임
  - 사용자 수준에서 busy waiting 대신 적절한 동기화 기법 사용

[보완]
* 스레드 전환보다 프로세스 전환이 더 비싼 이유
  - 메모리 맵이 달라지기 때문 -> CPU 캐시와 TLB에 있던 정보가 덜 유효해짐

* TLB 무효화
  - TLB에 저장된 주소 변환 정보 중 일부 또는 전부를 더 이상 유효하지 않다고 표시하고 제거하는 과정
  - 운영체제가 페이지 테이블을 바꿨는데 TLB에 예전 변환 정보가 남으면 CPU가 오래된 주소 변환 결과를 잘못 사용할 수 있음
  - TLB flush라고도 함

* CPU affinity
  - 특정 프로세스나 스레드가 가능한 한 특정 CPU 코어에서 계속 실행되도록 묶어두는 성향/설정
  - 운영체제 스케줄러는 보통 작업을 여러 코어에 분산시킴
  - 같은 코어의 캐시를 계속 활용할 가능성 커짐 -> 성능 좋아질 수 있음
  - 종류
    - Soft affinity: 가능하면 같은 코어에서 계속 실행하려는 정책. 필요하면 다른 코어로 옮길 수도 있음
    - Hard affinity: 아예 특정 코어에서만 실행되도록 강하게 제한

## Q4. 인터럽트는 왜 필요하며 어떻게 동작하는가?

[답변]
* 인터럽트
  - CPU가 현재 수행 중인 작업을 잠시 중단 -> 긴급하거나 중요한 사건 처리
  - 필요한 이유
    - CPU가 외부 장치의 상태를 계속 확인하지 않아도 됨 -> 비효율 줄임
    - I/O 완료, 타이머 만료, 하드웨어 오류 같은 비동기 사건에 빠르게 대응
    - 운영체제가 선점형 스케줄링을 구현

* 동작
  - 하드웨어 장치나 타이머가 인터럽트 신호 발생시킴
  - CPU
    - 현재 명령을 마무리할 수 있는 시점에서 인터럽트 인지
    - 현재 실행 상태 일부를 저장 -> 커널 모드로 전환
  - 인터럽트 벡터 테이블 참조 -> 해당 인터럽트 핸들러 주소로 이동
  - 운영체제 또는 핸들러가 인터럽트 원인 처리
  - 필요한 경우 추가 작업은 나중에 처리하도록 지연 -> 최소한의 핵심 작업만 우선 수행
  - 인터럽트 처리 후 저장한 문맥 복원 -> 원래 작업으로 돌아감

[보완]
* 인터럽트 종류
  - 하드웨어 인터럽트: 키보드 입력, 디스크 I/O 완료, 타이머 등 외부 장치에서 발생
  - 소프트웨어 인터럽트/트랩: 시스템 콜, 예외 처리 등 소프트웨어 실행 중 발생
* ISR
  - 인터럽트가 발생했을 때 실행되는 커널 코드

## Q5. 인터럽트와 폴링의 차이는? 각각 언제 사용하는가?

[답변]
* 인터럽트
  - 장치가 '이벤트가 발생했다'고 CPU에 알리는 방식
  - CPU: 평소 다른 일을 하다가 필요할 때만 처리
  - 이벤트 발생 시 반응성이 좋고 불필요한 확인 작업 감소
  - 사용 경우
    - 이벤트 발생 시점이 불규칙
    - CPU 효율이 중요
    - 사용자 입력, 네트워크 패킷 수신, 디스크 I/O 완료 처리

* 폴링
  - CPU가 주기적으로 장치 상태 직접 확인하는 방식
  - 이벤트가 없어도 계속 확인 -> CPU 자원을 더 사용
  - 구조가 단순하고 예측 가능한 제어 가능
  - 사용 경우
    - 매우 짧은 시간 안에 반복 확인해야 하는 경우
    - 구현 단순성이 중요
    - 인터럽트 처리 비용이 오히려 더 큰 경우
    - 임베디드 시스템: 특정 장치를 주기적으로 읽음

## Q6. 인터럽트 우선순위는 어떻게 결정되는가?

[답변]
* 인터럽트 우선순위
  - 여러 인터럽트가 동시에 또는 거의 동시에 발생했을 때 어떤 것을 먼저 처리할지 정하는 기준
  - 보통 하드웨어와 운영체제 정책에 의해 결정

* 결정 방식
  - 인터럽트 컨트롤러가 각 인터럽트 라인에 우선순위 부여
  - CPU는 더 높은 우선순위 인터럽트를 먼저 받아들임
  - 현재 처리 중인 인터럽트보다 높은 우선순위의 인터럽트 도착 -> 중첩 인터럽트 허용된 경우 선점 가능
  - 같은 우선순위 -> 고정 순서 또는 라운드 로빈 정책 사용
  - 반영 기준
    - 시간 민감성: 타이머, 실시간 장치
    - 데이터 손실 위험: 네트워크, 고속 입출력 장치
    - 시스템 안정성: 오류, 장애 관련 인터럽트
   
[보완]
* PIC/APIC
  - 인터럽트 우선순위와 분배를 담당하는 하드웨어 컨트롤러
  - 현대 시스템은 보통 APIC 계열 구조 사용
* 마스킹
  - 특정 우선순위 이하 인터럽트 잠시 막음
  - 중요한 처리 중: 낮은 우선순위 인터럽트 지연
* 중첩 인터럽트
  - 인터럽트 처리 도중 더 높은 우선순위 인터럽트가 끼어듦
  - 실시간성은 좋아지나 설계가 복잡

## Q7. 시스템 콜은 어떻게 동작하는가? User Mode와 Kernel Mode 전환 과정은?

[답변]
* 시스템 콜
  - 사용자 프로그램이 운영체제의 커널 기능을 안전하게 요청하는 방법
  - 사용자 프로그램: 직접 하드웨어 제어 X, 커널 자원 접근 X -> 파일 입출력, 프로셋스 생성, 메모리 할당, 네트워크 통신 등의 작업은 시스템 콜을 통해 요청

* 동작 과정
  - 사용자 프로그램ㅇ이 라이브러리 함수 호출
  - 해당 함수는 시스템 콜 번호와 인자를 정해진 레지스터나 스택에 배치
  - 이후 특수 명렬어 실행 -> 커널로 진입
  - CPU: user mode -> kernel mode
  - 커널: 시스템 콜 번호를 보고 해당 서비스 루틴 찾음
  - 전달받은 인자 검증 -> 필요한 작업 수행
  - 작업 결과를 반환값 형태로 준비
  - CPU: 다시 user mode로 복귀
  - 사용자 프로그램: 시스템 콜 결과 이어서 사용
