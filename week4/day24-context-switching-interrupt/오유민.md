### 📌 [Q1] 컨텍스트 스위칭이란 무엇이며 왜 발생하는가?

**컨텍스트 스위칭이란?**
- 운영체제에서 CPU가 인터럽드 등에 의해 한 프로세스(혹은 스레드)에서 다른 프로세스로 제어권을 넘겨줄 때, 현재 프로세스의 상태(Context)를 저장하고, 다음 프로세스를 수행하도록 교체하는 작업이다.
- Context(문맥)
  - 프로세스가 CPU를 사용하던 마지막 상태 정보를 의미하며, **프로세스 제어 블록(PCB)**에 저장된다.
  - 주요 구성 요소:
    - 프로그램 카운터(PC): 다음에 실행할 명령어의 주소
    - CPU 레지스터 값: 산술 연산 결과, 포인터 등
    - 프로세스 상태: 실행, 대기, 준비 등
    - 메모리 관리 정보: 페이지 테이블, 세그먼트 정보
    - 입출력 정보: 열린 파일, 입출력 버퍼 상태
- 컨텍스트 스위칭이 일어나는 동안 CPU는 아무 작업을 수행하지 못하므로 빈번한 스위칭은 성능 저하를 야기한다.

**발생 원인**
- 주요 발생 원인은 **CPU를 효율적으로 나누어 쓰기 위함**이다.
- **멀티 태스킹**: 실행 가능한 프로세스가 여러 개일때, 마치 동시에 실행되는 것처럼 보이게 하기 위해서 CPU 시간을 아주 짧게 쪼개어 번갈아 가며 할당
- **인터럽트** 발생: I/O 요청(디스크 읽기/쓰기, 네트워크 응답 등)이나 하드웨어 이벤트가 발생해 현재 작업보다 급한 처리가 필요할 때 발생
- **시스템 콜**: 프로세스가 OS의 자원을 사용하기 위해 사용자 모드 -> 커널 모드로 전환할 때 발생
- **우선순위 스케줄링**: 현재 실행 중인 프로세스보다 우선순위가 높은 프로세스가 나타났을 때(선점형 스케줄링) CPU를 넘겨주기 위해 발생

```
sequenceDiagram
    participant P1 as 프로세스 A
    participant OS as 운영체제 (Scheduler)
    participant P2 as 프로세스 B

    Note over P1: 프로세스 A 실행 중
    P1->>OS: 1. 인터럽트/시스템 콜 발생
    Note over OS: 2. 프로세스 A의 상태를 PCB A에 저장
    Note over OS: 3. 다음 수행할 프로세스 B 선택
    Note over OS: 4. PCB B에서 프로세스 B의 상태 복구
    OS->>P2: 5. 프로세스 B 실행 재개
    Note over P2: 프로세스 B 실행 중
```


### 📌 [Q2] 컨텍스트 스위칭 과정을 구체적으로 설명하면?

컨텍스트 스위칭은 **CPU 제어권이 사용자 프로세스에서 운영체제(커널)로 넘어갔다가 다시 다른 프로세스로 넘어가는 과정**이다. 커널 모드에서 **Dispatcher(디스패처)**에 의해 수행되며 단계는 다음과 같다.

1. **인터럽트 발생**: 인터럽트가 발생하면 CPU는 **현재 실행 중인 프로세스의 실행을 멈추고 커널 모드로 전환**한다. -> 프로세스 A가 실행 중인 상태에서 타이머 인터럽트(time slice 만료)/IO 요청/시스템 콜이 발생해 CPU 제어권이 OS의 커널로 넘어간다. 
2. **상태 저장**: 커널은 현재 프로세스 A의 상태(프로그램 카운터, 레지스터 값 등)를 프로세스 제어 블록(PCB)에 저장한다. (나중에 A가 다시 실행될 때 중단된 지점부터 다시 시작하기 위해 필수적)
3. **프로세스 상태 업데이트**: 프로세스 A의 상태를 Running에서 Ready 또는 Waiting으로 변경한다.
4. **프로세스 선택(스케쥴링)**: 스케줄러가 준비 큐(Ready Queue)에서 다음에 실행할 프로세스 B를 선택한다.
5. **상태 복구**: 선택된 프로세스 B의 상태를 B의 PCB에서 CPU 하드웨어 레지스터로 복구한다.
6. **실행 재개**: 프로세스 B의 상태를 running으로 변경하고, CPU는 복구된 프로그램 카운터(pc)가 가리키는 지점부터 프로세스 B를 실행한다.

### 📌 [Q3] 컨텍스트 스위칭의 오버헤드는 무엇이며 어떻게 줄일 수 있는가?

컨텍스트 스위칭 오버헤드란 시스템이 프로세스를 전환하는 과정에서 소모되는 시간과 메모리 등의 자원을 말한다. 이 시간동안에는 아무런 작업이 이루어지지 않으므로, 이를 줄이는 것이 성능 최적화의 핵심이다.

**오버헤드의 주요 원인**
- 교체 작업 자체의 비용: PCB 저장 및 복구, 스케줄링 알고리즘 실행 등에 소모되는 CPU 사이클
- 캐시 플러시(cache flush) 및 캐시 미스(cache miss): 가장 치명적인 비용, 새로운 프로세스가 실행되면 기존 CPU 캐시(L1, L2 등)에 저장된 데이터는 무의미해지며, 초기 실행 시 대량의 캐시 미스가 발생해 성능이 일시적으로 급락
- TLB (Translation Lookaside Buffer) 플러시: 프로세스마다 가상 메모리 주소 체계가 다르기 때문에 주소 변환 캐시인 TLB를 비워야 함 -> 메모리 접근 속도를 늦추는 원인

**오버헤드를 줄이는 방법**
- 프로세스 대신 스레드 사용하기
  - 스레느는 같은 프로세스 내의 메모리 공간(code, data, heap)을 공유하므로, 스레드 간 전환 시 주소 공간을 바꿀 필요가 없음
  - TLB를 플러시할 필요가 없고, 캐시 적중률이 비교적 유지되어 프로세스 스위칭보다 훨씬 가벼움
- 적절한 time slice 설정
  - 너무 짧은 경우: 응답성은 좋아지지만 컨텍스트 스위칭이 너무 빈번해 오버헤드 커짐
  - 너무 긴 경우: 오버헤드는 줄어들지만 멀티태스킹의 부드러움이 사라지고, 특정 작업이 CPU를 독점하게 됨
- CPU 친화도(Affinity) 활용
  - 프로세스가 실행되었던 CPU 코어에서 다시 실행되도록 스케줄링 (캐시 미스 방지)

### 📌 [Q4] 인터럽트는 왜 필요하며 어떻게 동작하는가?

**인터럽트**는 프로그램 상의 정상적인 실행 흐름을 멈추고, 즉각적인 처리가 필요한 외부 이벤트를 우선적으로 처리하기 위한 메커니즘으로, CPU의 높은 효율성을 유지하기 위해 꼭 필요하다.

**인터럽트가 필요한 이유**
- CPU 효율성 (비동기 처리): I/O처럼 CPU보다 훨씬 느린 장치의 결과를 기다리는 동안 CPU가 아무 일도 안 하거나(busy waiting) 계속 상태를 체크(polling)하는 낭비를 막아줌
- 즉각적인 반응성: 키보드 입력, 마우스 클릭, 네트워크 패킷 도착과 같은 실시간적인 외부 이벤트에 즉시 응답할 수 있게 함
- 시스템 보호 및 예외 처리: 프로그램 오류나 하드웨어 결함이 발생했을 때 시스템이 안전하게 멈추거나 복구할 기회를 제공
- 멀티태스킹 구현: 하드웨어 타이머가 주기적으로 인터럽트를 발생시켜 OS가 한 프로세스에서 다른 프로세스로 CPU를 강제로 넘겨줄 수 있게 함

**동작 과정** <br>
1. 인터럽트 요청 발생: 하드웨어 기기나 소프트웨어가 CPU에 신호를 보냄
2. 현재 작업 일시 중단: CPU는 현재 실행 중인 명령어를 마무리하고, 다음에 실행할 주소(pc)와 현재 상태를 별도의 스택이나 레지스터에 저장함
3. 인터럽트 벡터 테이블(IVT) 조회: 발생한 인터럽트 번호를 확인하고, 그 번호에 대응하는 처리 코드인 ISR(Interrupt Service Routine)의 주소를 메모리에서 찾음
4. ISR 실행: 해당 인터럽트를 처리하는 루틴(함수)을 실행함
5. 상태 복구 및 재개: ISR이 끝나면 저장해두었던 이전 프로세스의 상태를 복구하고, 중단된 지점부터 다시 실행

### 📌 [Q5] 인터럽트와 폴링의 차이는? 각각 언제 사용하는가?

**인터럽트**
- 장치가 CPU에게 신호를 보내 알림
- 주도권: 외부 장치
- CPU 효율 높음 (필요할 때만 작업 수행)
- 반응 속도 즉각적임
- 하드웨어 지원 및 구현이 복잡함

**폴링**
- CPU가 주기적으로 장치의 상태를 확인
- 주도권: CPU
- CPU 효율 낮음 (무의미한 체크로 CPU 자원 낭비)
- 주기에 따라 반응 속도 지연 발생 가능
- 구현이 매우 단순하고 직관적임

**인터럽트를 사용하는 경우(대부분의 현대적 시스템)**
- 발생 시점이 불분명할 때: 키보드 입력, 마우스 클릭, 네트워크 패킷 도착 등 언제 일어날지 모르는 이벤트 처리에 적합
- 멀티태스킹 환경: I/O 작업을 기다리는 동안 CPU가 다른 프로세스를 처리해야 하는 효율성이 중요한 시스템에서 사용

**폴링을 사용하는 경우**
- 이벤트가 매우 빈번할 때: 빈번한 인터럽트로 인한 컨텍스트 스위칭 비용 > 폴링으로 상태 체크하는 비용
- 하드웨어가 인터럽트를 지원하지 않을 때: 아주 간단한 임베디드 기기나 센서 제어 시
- 작업이 매우 짧고 예측 가능할 때


### 📌 [Q6] 인터럽트 우선순위는 어떻게 결정되는가?

일반적으로 데이터 손실 위험이 크거나 시스템상 치명적인 오류일수록 우선순위가 높음 (시스템 안정성(전원/에러) -> 효율성(타이머) -> 단순 입출력)
- 전원 공급 이상
- 하드웨어 오류
- 외부 신호: 타이머 인터럽트, I/O 장치 응답
- 입출력 콘솔: 키보드 마우스 등
- 소프트웨어 인터럽트

### 📌 [Q7] 시스템 콜은 어떻게 동작하는가? User Mode와 Kernel Mode 전환 과정은?

**시스템 콜이란?**
- User 프로그램이 OS Kernel의 서비스를 받기 위해 호출하는 인터페이스이며, 이 과정에서 안전을 위해 User Mode와 Kernel Mode 사이의 전환이 일어난다.

**User Mode**
- 사용자 어플리케이션이 실행되는 모드로, 하드웨어 직접 접근이나 특권 명령 실행이 제한됨 (mode bit = 1)

**Kernel Mode**
- OS 커널이 실행되는 모드로, 자원(메모리, 하드웨어)에 대한 모든 제어 권한을 가짐 (mode bit = 0)

**시스템 콜 동작 과정** <br>
: 사용자 프로그램이 파일을 읽거나 화면에 출력하는 등 자원 접근이 필요할 때 다음과 같은 단계로 시스템 콜 수행
- 시스템 콜 호출: 사용자 프로그램이 API를 통해 시스템 콜 함수를 호출
- 트랩(Trap) 발생: CPU에 소프트웨어 인터럽트인 **트랩**을 걸어, 실행 모드를 유저모드에서 커널 모드로 변환 (mode bit 1 -> 0)
- 커널 진입 및 핸들러 실행: CPU는 시스템 콜 번호를 확인하고, 시스템 콜 테이블을 참조해 해당 서비스 루틴(커널 함수)을 찾아 실행
- 서비스 수행: 커널이 요청받은 작업을 하드웨어 레벨에서 수행
- 결과 반환 및 복귀: 작업이 끝나면 실행 결과를 사용자 프로그램에 반환하고, 실행 모드를 다시 커널 -> 유저 모드로 전환
- 사용자 프로그램 재개: 시스템 콜 다음 줄부터 프로그램 실행을 이어나감

**유저 모드와 커널 모드를 나누는 이유**
- 보안 및 보호: 사용자 프로그램이 실수나 악의적인 의도로 직접 메모리를 조작하거나 다른 프로세스의 데이터를 삭제하는 것을 방지
- 추상화: 개발자가 하드웨어의 복잡한 동작 방식을 몰라도 OS가 제공하는 표준화된 인터페이스(시스템 콜)만으로 장치를 사용할 수 있게 함
