## 1. 힙이 완전 이진 트리여야 하는 이유

힙(Heap)은 **완전 이진 트리(Complete Binary Tree)** 구조를 유지한다.
완전 이진 트리란 **마지막 레벨을 제외한 모든 레벨이 꽉 차 있고, 마지막 레벨은 왼쪽부터 채워지는 트리**를 의미한다.

이 구조를 유지하는 이유는 다음과 같다.

### 1) 트리의 높이를 최소로 유지하기 위해

완전 이진 트리는 항상 **균형에 가까운 구조**를 유지한다.

트리 높이:

[
h = O(\log n)
]

따라서 힙에서 수행되는 연산:

-   삽입
-   삭제

모두 **트리 높이에 비례하여 O(log n)** 시간에 수행된다.

만약 완전 이진 트리가 아니라면 트리가 편향될 수 있고,
이 경우 높이가 (O(n))이 되어 힙의 성능이 크게 떨어진다.

---

### 2) 배열로 효율적으로 표현하기 위해

완전 이진 트리는 **노드 번호가 자연스럽게 정렬되는 구조**를 가진다.

예시

```
        10
      /    \
     20     30
    /  \   /
   40  50 60
```

배열 표현

```
index: 1  2  3  4  5  6
value:10 20 30 40 50 60
```

이때 부모와 자식 관계는 다음과 같이 계산된다.

```
left child  = 2i
right child = 2i + 1
parent      = i / 2
```

즉 **포인터 없이도 트리를 표현할 수 있다.**

---

## 2. 힙을 배열로 구현할 때의 장점

힙은 일반적으로 **배열(Array)** 로 구현된다.

### 1) 포인터가 필요 없다

트리를 노드 구조체로 구현하면

```
Node {
   value
   left
   right
}
```

처럼 포인터가 필요하다.

하지만 배열 기반 힙에서는 **인덱스 계산으로 부모/자식 접근**이 가능하다.

```
parent = i / 2
left = 2i
right = 2i + 1
```

따라서 **메모리 사용량이 줄어든다.**

---

### 2) Cache locality가 좋다

배열은 **연속된 메모리 공간**에 저장된다.

CPU cache 관점에서

-   배열 → 연속 접근
-   포인터 기반 트리 → 메모리 여기저기 접근

따라서 **배열 기반 힙이 실제 실행 성능이 더 좋다.**

---

### 3) 구현이 단순하다

삽입과 삭제 과정에서

-   부모 이동
-   자식 비교

모두 **인덱스 연산만으로 처리 가능하다.**

---

## 3. 힙의 삽입 과정 (Heapify-up)

힙에 새로운 원소를 삽입할 때는 **항상 마지막 위치**에 삽입한다.
그 후 **Heap property를 만족하도록 위로 이동한다.**

이를 **Heapify-up (sift-up, bubble-up)** 이라고 한다.

### 단계

1. 새로운 노드를 **배열의 마지막 위치**에 삽입
2. 부모 노드와 비교
3. 힙 조건을 만족하지 않으면 **swap**
4. 부모 위치로 이동하여 반복

---

### 예시 (Min Heap)

초기 힙

```
        5
      /   \
     10    20
    /  \
   30  40
```

배열

```
[5, 10, 20, 30, 40]
```

15 삽입

```
[5, 10, 20, 30, 40, 15]
```

트리

```
        5
      /   \
     10    20
    /  \   /
   30  40 15
```

15 < 20 이므로 swap

```
        5
      /   \
     10    15
    /  \   /
   30  40 20
```

Heap property 만족 → 종료

---

### 시간복잡도

최악의 경우 **루트까지 이동**

[
O(\log n)
]

---

## 4. 힙의 삭제(추출) 과정 (Heapify-down)

힙에서 삭제는 일반적으로 **루트 노드 삭제**를 의미한다.

Max Heap → 최대값
Min Heap → 최소값

삭제 과정에서는 **Heapify-down (sift-down)** 을 수행한다.

---

### 단계

1. 루트 노드 제거
2. **마지막 노드를 루트로 이동**
3. 자식 노드들과 비교
4. 힙 조건이 깨지면 **더 작은/큰 자식과 swap**
5. 아래로 내려가며 반복

---

### 예시 (Min Heap)

초기 힙

```
        5
      /   \
     10    15
    /  \   /
   30  40 20
```

루트 삭제

```
마지막 노드 20을 루트로 이동
```

```
        20
      /   \
     10    15
    /  \
   30  40
```

자식 중 더 작은 값과 교환

```
10 과 swap
```

```
        10
      /   \
     20    15
    /  \
   30  40
```

이제 heap property 만족 → 종료

---

### 시간복잡도

루트에서 리프까지 이동 가능

[
O(\log n)
]

---

## 5. 힙에서 특정 값을 탐색하는 시간복잡도

힙에서 **특정 값을 찾는 탐색은 O(n)** 이다.

이유는 힙이 **정렬된 구조가 아니기 때문**이다.

힙의 조건은 단 하나다.

Max Heap

```
parent ≥ child
```

Min Heap

```
parent ≤ child
```

즉 **부모와 자식 관계만 보장된다.**

예시

```
        10
      /    \
     20     30
    /  \   /  \
   40  50 60  70
```

여기서 **50을 찾으려면**
특정 방향으로만 탐색할 수 없다.

따라서 **모든 노드를 확인해야 한다.**

[
O(n)
]

---

## 6. 우선순위 큐를 배열이나 연결 리스트로 구현하면 생기는 문제

우선순위 큐(Priority Queue)는

**가장 높은 우선순위를 빠르게 꺼내는 자료구조**이다.

힙을 사용하는 이유는 **삽입과 삭제를 모두 효율적으로 하기 위해서**이다.

---

### 1) 배열 기반 구현

정렬되지 않은 배열

삽입

```
O(1)
```

삭제 (최댓값 찾기)

```
O(n)
```

매번 전체 탐색 필요

---

정렬된 배열

삽입

```
O(n)
```

위치를 찾고 shift 필요

삭제

```
O(1)
```

---

### 2) 연결 리스트 구현

정렬되지 않은 리스트

삽입

```
O(1)
```

삭제

```
O(n)
```

---

정렬된 리스트

삽입

```
O(n)
```

삭제

```
O(1)
```

---

### 3) 힙을 사용하는 경우

| 연산           | 시간복잡도 |
| -------------- | ---------- |
| 삽입           | O(log n)   |
| 삭제           | O(log n)   |
| 최대/최소 조회 | O(1)       |

즉 **삽입과 삭제가 균형 잡힌 성능**을 제공한다.

그래서 대부분의 언어의 **Priority Queue 구현은 힙 기반**이다.

예

-   Java `PriorityQueue`
-   C++ `priority_queue`
-   Python `heapq`
