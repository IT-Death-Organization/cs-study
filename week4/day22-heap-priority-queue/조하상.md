## 1. 힙이 완전 이진 트리여야 하는 이유는?

**배열 효율성과 높이의 균형**
- 힙은 최댓값이나 최솟값을 빠르게 찾기 위해 고안된 자료구조로, 항상 부모 노드가 자식 노드보다 크거나(Max Heap) 작아야(Min Heap) 한다.
- **완전 이진 트리** 형식을 유지함으로써 중간에 빈 공간 없이 노드를 채울 수 있다. 이는 곧 배열로 구현했을 때 메모리 낭비 없이 인덱스만으로 부모와 자식 관계를 즉시 계산할 수 있게 한다. 또한 트리의 높이를 항상 $O(\log n)$으로 유지하여 삽입과 삭제 연산의 성능을 보장한다.

---

## 2. 힙을 배열로 구현할 때의 장점은?

**메모리 절약과 연산의 간결함**
- **인덱스 접근:** 포인터(Point)를 사용하여 왼쪽, 오른쪽 자식을 가리킬 필요가 없다. 현재 인덱스가 $i$라면 왼쪽 자식은 $2i$, 오른쪽 자식은 $2i+1$, 부모는 $i/2$로 간단히 계산할 수 있다.
- **메모리 효율:** 포인터를 저장하기 위한 추가 공간이 필요 없으므로 물리적인 메모리 사용량이 줄어든다.
- **캐시 지역성:** 데이터가 메모리 상에 연속적으로 배치되므로 CPU 캐시 히트율이 높아져 실제 연산 속도가 빨라진다.



---

## 3. 힙의 삽입 과정을 Heapify-up 과정과 함께 설명하면?

**상향식 정렬**
1. 새로운 요소를 트리의 가장 마지막 노드(배열의 끝)에 추가한다.
2. 추가된 노드와 그 부모 노드의 값을 비교한다.
3. 힙의 속성(예: 부모가 자식보다 커야 함)을 위반한다면 두 노드의 위치를 바꾼다(**Heapify-up**).
4. 힙의 속성이 만족되거나 루트 노드에 도달할 때까지 2~3번 과정을 반복한다. 이 과정은 트리의 높이인 $O(\log n)$ 내에 완료된다.

---

## 4. 힙의 삭제(추출) 과정을 Heapify-down 과정과 함께 설명하면?

**하향식 정렬**
1. 루트 노드(최댓값 혹은 최솟값)를 제거한다.
2. 힙의 가장 마지막 노드를 루트 자리로 옮긴다.
3. 새로운 루트 노드와 그 자식 노드들을 비교한다.
4. 자식 노드 중 더 큰 값(Max Heap 기준)과 위치를 바꾼다(**Heapify-down**).
5. 자식 노드보다 자신이 더 크거나 리프 노드에 도달할 때까지 3~4번 과정을 반복한다. 삽입과 마찬가지로 $O(\log n)$이 소요된다.



---

## 5. 힙에서 특정 값을 탐색하는 시간복잡도는? 왜?

**$O(n)$의 시간 복잡도**
- 힙은 부모-자식 간의 크기 관계만 정의할 뿐, 이진 탐색 트리(BST)처럼 왼쪽 자식과 오른쪽 자식 간의 크기 순서가 정해져 있지 않다.
- 따라서 루트 노드를 제외한 특정 값을 찾으려면 전체 노드를 하나하나 확인해야 하므로 배열의 선형 탐색과 동일한 $O(n)$이 걸린다. 힙은 '탐색'보다는 '최우선 순위 데이터 추출'에 특화된 자료구조다.

---

## 6. 우선순위 큐를 배열이나 연결 리스트로 구현하면 어떤 문제가 있는가?

**연산 효율성의 저하**
- **배열(정렬된 상태):** 삽입 시 적절한 위치를 찾고 기존 데이터를 뒤로 밀어내야 하므로 $O(n)$이 걸린다.
- **연결 리스트(정렬된 상태):** 삽입 시 위치를 찾기 위해 처음부터 끝까지 탐색해야 하므로 $O(n)$이 걸린다.
- **결론:** 우선순위 큐를 힙으로 구현하면 삽입과 삭제 모두 $O(\log n)$으로 처리할 수 있어, 데이터의 양이 많아질수록 배열이나 리스트보다 압도적으로 성능이 유리하다.

| 구현 방식 | 삽입(Push) | 삭제(Pop/Peek) |
| :--- | :--- | :--- |
| 정렬된 배열 | $O(n)$ | $O(1)$ |
| 정렬된 연결 리스트 | $O(n)$ | $O(1)$ |
| **힙(Heap)** | **$O(\log n)$** | **$O(\log n)$** |
