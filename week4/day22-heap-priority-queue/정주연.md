## 🧠 학습 질문

## 힙이 완전 이진 트리여야 하는 이유는?

- **배열 매핑의 효율성**
    - 완전 이진 트리는 노드를 왼쪽부터 차례대로 채우는 구조이므로, 배열로 구현했을 때 인덱스 0번(또는 1번)부터 중간에 빈 간격 없이 꽉 채워 저장할 수 있음
- **트리 균형 유지**
    - 노드가 항상 왼쪽부터 채워지기 때문에 트리의 높이가 항상 logN을 유지함.
    - 이는 삽입과 삭제 시 수행되는 연산 횟수를 최악의 경우에도 O(log N)으로 보장해주는 핵심 근거가 됨

## 힙을 배열로 구현할 때의 장점은?

- **인덱스 산술 연산**
    - 포인터(Link)를 저장할 메모리 공간이 필요 없으며, 단순히 인덱스 계산만으로 부모와 자식 노드에 접근 가능함
        - 부모 인덱스: `i / 2`
        - 왼쪽 자식: `i * 2`
        - 오른쪽 자식: `i * 2 + 1`
- **메모리 절약 및 캐시 지역성**
    - 포인터 오버헤드가 없고 데이터가 메모리상에 연속적으로 배치됨
    - CPU 캐시 적중률(Cache Locality)이 높아져 실제 동작 속도가 빠름

## 힙의 삽입 과정을 Heapify-up 과정과 함께 설명하면?

- 단계
    - 새로운 요소를 트리의 가장 마지막 위치(배열의 맨 끝)에 추가함
    - 추가된 노드를 부모 노드와 비교함
    - 힙의 성질(최대 힙의 경우 자식이 부모보다 작아야 함)을 위반하면 부모와 위치를 바꿈(Swap)
    - 위반하지 않거나 루트 노드에 도달할 때까지 위 과정을 반복함
- **복잡도:** 트리의 높이에 비례하므로 O(log N)임

## 힙의 삭제(추출) 과정을 Heapify-down 과정과 함께 설명하면?

- **단계**
    1. 루트 노드(최댓값 또는 최솟값)를 제거하고 반환함
    2. 트리의 가장 마지막 노드를 루트 위치로 옮김
    3. 옮겨진 루트 노드를 자식 노드들과 비교함
    4. 자식들 중 우선순위가 더 높은(최대 힙에서는 더 큰 값) 노드와 위치를 바꿈
    5. 자식 노드가 더 작거나 단말 노드(Leaf)에 도달할 때까지 위 과정을 반복함
- **복잡도:** 역시 트리의 높이에 비례하므로 O(log N)임

## 힙에서 특정 값을 탐색하는 시간복잡도는? 왜?

- 힙은 이진 탐색 트리(BST)가 아님
- 부모가 자식보다 크다는 조건만 있을 뿐, 왼쪽 자식과 오른쪽 자식 사이의 크기 순서는 정해져 있지 않음(형제 간 순서 없음).
- 따라서 특정 값을 찾으려면 모든 노드를 다 뒤져야 하는 선형 탐색이 필요함

### 우선순위 큐를 배열이나 연결 리스트로 구현하면 어떤 문제가 있는가?

- **정렬되지 않은 배열/리스트**
    - 삽입은 O(1)로 빠르지만, 가장 높은 우선순위를 찾아서 삭제할 때 $O(N)$이 걸림
- **정렬된 배열/리스트**
    - 삭제는 $O(1)$로 빠르지만, 삽입할 때 적절한 위치를 찾아야 하고 뒤의 데이터를 밀어내야(Shifting) 하므로 $O(N)$이 걸림
- **힙의 우수성**
    - 힙은 삽입과 삭제 모두 O(log N)에 수행하므로, 데이터 양이 많아질수록 배열이나 리스트보다 훨씬 뛰어난 성능과 균형을 보여줌
