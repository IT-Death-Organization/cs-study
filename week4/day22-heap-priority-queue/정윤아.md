## Q1. 힙이 완전 이진 트리여야 하는 이유는?

[답변]
* 힙
  - 부모-자식 간의 대소 관계 유지 + 삽입/삭제 효율적 수행
  - 완전 이진 트리 유지
    - 트리의 높이가 항상 O(logn)으로 제한
      -> 삽입 시 위로 올리는 과정 / 삭제 시 아래로 내리는 과정 모두 빠르게 동작
    - 중간에 빈 자리 X -> 배열로 효율적으로 표현
    - 완전 이진 트리가 아니면 트리의 한 쪽으로 치우칠 수 있음 => 높이가 커져 삽입/삭제 성능이 나빠질 수 있음

[보완]
* 힙 조건
  - 최대 힙: 부모 노드 >= 자식 노드
  - 최소 힙: 부모 노드 <= 자식 노드
  - 정렬된 트리 X
  - 루트가 최댓값/최솟값이라는 점만 보장

---

## Q2. 힙을 배열로 구현할 때의 장점은?

[답변]
* 배열 구현
  - 부모/자식 위치를 인덱스 계산만으로 찾을 수 있음: 포인터 필요 X
  - 연결 구조를 위한 추가 메모리 필요 X => 공간 효율 좋음
  - 연속된 메모리 공간 사용 => 캐시 효율 좋음
  - 구현 비교적 단순, 삽입/삭제 시에도 노드 연결 바꾸는 게 아닌 원소 교환 중심으로 처리

---

## Q3. 힙의 삽입 과정을 Heapify-up 과정과 함께 설명하면?

[답변]
* 삽입 과정
  - 새 원소를 트리 마지막 위치에 추가 (배열 맨 뒤)
  - 부모와의 대소 관계가 힙 조건을 위반 -> 부모와 자리 바꿈 (Heapify-up)
  - 트리의 높이만큼 이동: O(logn)

---

## Q4. 힙의 삭제(추출) 과정을 Heapify-down 과정과 함께 설명하면?

[답변]
* 삭제 과정
  - 루트 노드 삭제
  - 마지막 원소를 루트로 이동
  - 루트에서 시작해 자식들과 비교
  - 힙 조건 위반 -> 더 적절한 자식과 자리 바꿈 (Heapify-down)
    - 최대 힙: 두 자식 중 더 큰 자식과 교환
    - 최소 힙: 두 자식 중 더 작은 자식과 교환
  - 트리의 높이만큼 이동: O(logn)

[보완]
* 삭제 과정 두 단계
  1. 구조 유지: 마지막 노드 루트로 올림
  2. 순서 복구: Heapify-down 수행

---

## Q5. 힙에서 특정 값을 탐색하는 시간복잡도는? 왜?

[답변]
* 탑색
  - 시간복잡도: O(n)
  - 전체 정렬 X
  - 원하는 값이 어느 위치에 있을지 바로 알 수 없음 => 최악의 경우: 모든 노드 확인
  - 최댓값/최솟값 빠르게 찾는 데 특화, 임의의 값 탐색엔 효율적이지 않음

---

## Q6. 우선순위 큐를 배열이나 연결 리스트로 구현하면 어떤 문제가 있는가?

[답변]
* 우선순위 큐
  - 원소를 우선순위와 함께 넣음 => 우선순위가 높은 원소가 먼저 나옴: 힙으로 구현
  - 일반 배열/연결 리스트로 구현: 삽입/삭제 중 하나가 비효율적
  - 정렬되지 않은 배열/연결 리스트로 구현
    - 삽입은 쉬우나, 우선순위가 가장 높은 원소를 찾으려면 전체 탐색 -> 삭제: O(n)
  - 정렬된 배열로 구현
    - 삭제(추출)는 빠르나, 삽입할 위치를 찾아 중간에 끼워 넣어야 함 -> 삽입: O(n)
  - 정렬된 연결 리스트로 구현
    - 삽입 위치 찾는 데 시간이 걸림 -> 삽입: O(n)
  - 힙으로 구현
    - 삽입, 삭제 모두 O(logn)
   
[보완]
* 우선순위 큐의 핵심 연산
  - 삽입
  - 최고 우선순위 원소 확인
  - 최고 우선순위 원소 제거
