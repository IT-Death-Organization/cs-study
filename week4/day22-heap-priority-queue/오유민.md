### 📌 [Q1] 힙이 완전 이진 트리여야 하는 이유는?

**힙이란?**
- 완전 이진 트리의 일종으로 우선순위 큐를 위하여 만들어진 자료구조이다.
- 여러 개의 값들 중에서 최댓값이나 최솟값을 빠르게 찾아내도록 만들어진 자료구조이다.
- 힙은 일종의 반정렬 상태(느슨한 정렬 상태) 를 유지한다. (큰 값이 상위 레벨, 작은 값이 하위 레벨에 있다는 정도만)
- 간단히 말하면 부모 노드의 키 값이 자식 노드의 키 값보다 항상 큰(작은) 이진 트리를 말한다.
- 힙 트리에서는 중복된 값을 허용한다. (이진 탐색 트리에서는 중복된 값을 허용하지 않는다.)
- 종류: 최대 힙(부모 노드의 키 값이 자식 노드의 키 값보다 크거나 같은 완전 이진 트리), 최소 힙(부모 노드의 키 값이 자식 노드의 키 값보다 작거나 같은 완전 이진 트리)

**완전 이진 트리란?**
- 트리의 모든 레벨이 완전히 채워져 있고, 마지막 레벨의 노드들이 왼쪽부터 채워져 있는 트리
- 트리의 높이가 낮게 유지되어 성능이 향상됨

**왜 완전 이진 트리여야 하는가?**
- 효율적인 배열 구현 -> 성능 향상
  - **완전 이진 트리는 별도의 포인터 없이 배열만으로 완벽하게 트리 표현 가능**
  - **인덱스 규칙** -> 1번 인덱스를 루트로 사용할 경우, 아래 연산만으로 부모/자식 노드 찾을 수 있음
    - 부모 인덱스: i / 2
    - 왼쪽 자식 인덱스: 2 * i
    - 오른쪽 자식 인덱스: 2 * i + 1
- 트리에 빈 공간이 없기 때문에 배열의 메모리 공간을 낭비 없이 사용 가능
- **트리 높이의 균형 & 시간복잡도 (O(log N))**
  - 힙의 주요 연산(삽입, 삭제)은 루트 노드부터 단말 노드까지의 경로를 따라 진행됨
  - **n개의 노드를 가진 완전 이진 트리의 높이는 항상 log N에 수렴함**
  - 만약 완전 이진 트리가 아니라 한쪽으로 치우친 트리라면, 최악의 경우 트리의 높이가 n이 되어 시간복잡도가 O(N)이 될 수 있음
  - **완전 이진 트리 구조는 모든 연산이 최악의 경우에도 O(log N)의 시간복잡도를 보장함**
- **구조 유지의 일관성**
  - 힙은 새로운 노드가 삽입될 때 항상 마지막 위치에 추가가 되고, 삭제될 때도 마지막 노드를 루트로 옮겨 재정렬함
  - **마지막 위치**라는 기준이 명확하려면, 트리가 왼쪽부터 차례대로 채워진 완전 이진 트리여야 함
  - 이 구조를 통해 데이터 삽입, 삭제 시 구조를 유지하는 알고리즘(Heapify-up, Heapify-down)을 효율적으로 구현할 수 있음


### 📌 [Q2] 힙을 배열로 구현할 때의 장점은?

- 메모리 공간의 효율성
  - 일반적인 트리는 각 노드마다 data, left_child, right_child, 때로는 parent까지도 가리키는 **포인터**를 저장해여 함
  - 배열로 구현할 경우 **별도의 포인터 저장 공간이 필요 없음**
  - 힙은 **완전 이진 트리이므로 배열의 중간에 빈 요소가 생기지 않고 0번부터 끝까지 꽉 채워짐**
- 캐시 지역성에 따른 성능 향상
  - 배열은 메모리 상 연속적인 공간에 배치됨 -> 연속된 메모리 접근은 CPU의 캐시 히트율을 높임
- 부모 - 자식 간 빠른 접근 가능 (산술 연산만으로)
  - 부모 인덱스: i / 2
  - 왼쪽 자식 인덱스: 2 * i
  - 오른쪽 자식 인덱스: 2 * i + 1
- **마지막 노드 찾기 쉬움**
  - 힙의 핵심 연산인 삽입, 삭제는 항상 트리의 마지막 노드를 대상으로 함
  - 일반적인 트리는 마지막 노드를 찾기 위해 루트부터 탐색 or 별도의 로직 필요
  - 배열: 그냥 배열의 끝이 곧 마지막 노드임 -> O(1)

### 📌 [Q3] 힙의 삽입 과정을 Heapify-up 과정과 함께 설명하면?

힙의 삽입 과정은 **일단 끝에 넣고, 제자리를 찾을 때까지 위로 올리기**이다. 이때 위로 끌어올리는 과정을 **Heapify-up**이라고 한다.

**예시: 최대 힙에 삽입**

**(1) 새로운 노드 삽입** <br>
완전 이진 트리 형태를 유지해야 하므로, 현재 트리의 가장 마지막 위치(배열의 끝)에 새로운 요소를 추가한다.

**(2) Heapify-up (제자리 찾기, 재구조화)** <br>
삽입된 노드가 부모 노드보다 클 경우 힙의 속성이 깨지게 된다. 이를 바로잡기 위해 다음 과정을 반복한다.
- 삽입된 노드와 그 부모 노드의 값을 비교
- 부모 노드보다 클 경우 자리를 바꿈
- 루트 노드에 도달하거나, 부모 노드보다 작아질 때까지 반복

**시간 복잡도: O(log N)**
- 최악의 경우: 트리의 마지막 위치에서 루트까지 올라감 = 트리의 높이
- 노드가 n개일 때 완전 이진 트리의 높이는 log N -> 삽입 연산은 **O(log N)**의 시간 복잡도를 가짐


### 📌 [Q4] 힙의 삭제(추출) 과정을 Heapify-down 과정과 함께 설명하면?

힙의 삭제 연산은 **루트 노드(최댓값 또는 최솟값)를 꺼내는 방식**으로 이루어진다.
**루트를 제거하고 마지막 노드를 올린 뒤, 제자리를 찾을 때까지 내리는 게 핵심**이며, 여기서 내리는 과정을 Heapify-down이라고 한다.

**예시: 최대 힙에서 삭제**

**(1) 루트 노드 추출 및 대체** <br>
- 최상단에 있는 루트 노드를 삭제하고 반환한다
- 트리의 가장 마지막 노드(배열의 끝 요소)를 루트로 옮겨 '완전 이진 트리'의 구조를 유지한다. 이때 힙의 성질(부모>자식)은 깨지게 된다.

**(2) Heapify-down** <br>
루트 자리에 새로 올라온 노드가 자식들보다 작을 수 있다. 이를 바로잡기 위해 아래로 내려보낸다.
- 현재 노드와 왼쪽/오른쪽 자식 노드 중 **더 큰 값**을 가진 자식을 찾는다.
- 자식의 값이 현재 노드보다 크다면 자리를 바꾼다.
- 자식 노드들보다 현재 노드가 크거나, 단말 노드에 도달할 때까지 과정을 반복하며 아래로 내려간다.

**시간 복잡도: O(log N)**
- 루트에서 단말 노드까지 = 최대 트리의 높이만큼 내려가야 하므로 O(log N)이 소요됨


### 📌 [Q5] 힙에서 특정 값을 탐색하는 시간복잡도는? 왜?

- 힙에서 특정 값을 탐색하는 시간 복잡도는 **O(N)**이며, 이는 힙이 반정렬 상태(느슨한 정렬 상태)를 유지하는 자료구조이기 때문이다.
- 형제 노드 간의 순서가 없음
  - 힙은 부모-자식 간의 크기 관계만 정의되어 있을 뿐, **왼쪽 자식과 오른쪽 자식 중 어느쪽이 더 큰지에 대한 규칙이 없다.** (따라서 값을 찾을 때 왼쪽/오른쪽 서브트리를 모두 뒤져봐야 한다)
- 특정 경로로 좁힐 수 없음 (No Search Key Logic)
  - 이진 탐색 트리(BST): 현재 노드보다 크면 오른쪽, 작으면 왼쪽으로 가야 한다는 규칙이 있어 한 번 비교할 때마다 탐색 범위가 1/2로 줄어들음 (O(log N))
  - 힙: 현재 부모 노드보다 작은 값을 찾는다면, 그 값이 왼쪽 자식 자리에 있을지 오른쪽 자식 아래에 있을지 판단할 근거가 없다. (최악의 경우 트리 전체를 모두 확인해야)
- 배열 선형 탐색과 동일
  - 힙을 배열로 구현했을 때, 데이터는 논리적으로는 트리 구조를 이루지만 물리적으로는 배열에 순차적으로 들어있다.
  - 특정 값을 찾으려면 단순히 배열의 첫 번째 요소부터 마지막 요소까지 하나씩 확인하는 선형 탐색을 수행해야 하므로 O(N)이 걸린다.

### 📌 [Q6] 우선순위 큐를 배열이나 연결 리스트로 구현하면 어떤 문제가 있는가?

우선순위 큐를 배열이나 연결 리스트로 구현하면 삽입 or 삭제 중 하나는 반드시 O(N)이 되어 성능 저하를 일으킨다. 우선순위 큐는 데이터의 추가와, 최우선 데이터의 인출이 빈번히 일어나기 때문에 삽입, 삭제 연산 모두 효율적인 힙(O(log N))이 가장 적합하다.

- 배열로 구현할 경우
  - 정리되지 않은 배열:
    - 삽입: 끝에 추가하면 되므로 O(1)
    - 삭제: 가장 우선순위가 높은 데이터를 찾기 위해 전체 스캔 -> O(N)
  - 정렬된 배열
    - 삽입: 새로운 데이터가 들어갈 자리를 찾고, 그 뒤의 데이터를 모두 뒤로 밀어야 하므로 O(N)
    - 삭제: 정렬되어 있으므로 맨 앞 or 맨 뒤에서 꺼냄 -> O(1)

- 연결 리스트로 구현할 경우
  - 정렬되지 않은 연결 리스트
    - 삽입: 맨 앞에 추가하면 되므로 O(1)
    - 삭제: 가장 우선순위가 높은 노드를 찾기 위해 전체 리스트를 순회 -> O(N)
  - 정렬된 연결 리스트
    - 삽입: 우선순위에 맞는 삽입 위치를 찾기 위해 루트부터 비교하며 순회 -> O(N)
    - 삭제: 첫 번째 노드를 꺼내면 되므로 O(1)

이렇듯 우선순위 큐를 배열이나 연결 리스트로 구현하면 반드시 O(N)의 시간 복잡도를 가지게 된다. 이는 데이터 개수가 늘어날 수록 기하급수적으로 성능이 안 좋아지게 되는 수치이다.

따라서 삽입, 삭제 모두에서 균형 잡힌 성능을 보장하는 **힙**이 **우선순위 큐 구현의 표준**이 된다.
