## 1. 힙이 완전 이진 트리여야 하는 이유는?
> - 힙은 **최댓값 또는 최솟값을 빠르게 찾아내기 위해 고안된 완전 이진 트리** 기반의 자료구조로, **우선순위 큐(Priority Queue)** 를 구현하기 위해 사용됩니다.
> - 완전 이진 트리는 트리의 **모든 레벨이 노드로 꽉 차 있고**, 마지막 레벨은 **왼쪽부터 차례대로 노드가 채워져 있는** 이진 트리입니다.

![title](https://user-images.githubusercontent.com/44635266/66712339-4d77ec80-edd6-11e9-8937-3de41f63759c.png)   

### 1-1. 배열을 이용한 효율적인 저장
힙은 연결리스트가 아닌 **배열**로 주로 구현되고, 완전 이진 트리는 노드가 **중간에 비는 곳 없이** 순차적으로 채워집니다.
- **별도의 포인터(참조값) 없이**, 현재 인덱스 i를 알면 수식만으로 부모와 자식의 위치를 즉시 알 수 있습니다. (부모 인덱스 = `i / 2`, 왼쪽 자식 인덱스 = `i * 2`, 오른쪽 자식 인덱스 = `(i * 2) + 1`)
-  배열의 0번 혹은 1번 인덱스부터 빈틈없이 데이터를 채울 수 있어 메모리 낭비가 전혀 없습니다.
### 1-2. 최악의 경우에도 성능 보장
힙의 주요 연산인 삽입과, 삭제는 노드를 루트에서 리프까지, 혹은 리프에서 루트까지 이동시키며 비교하는 과정을 거칩니다.
- 완전 이진 트리는 항상 꽉 채워진 형태를 유지하므로, n개의 노드를 가진 트리의 높이는 항상 **⌊log₂ n⌋** 이 됩니다.
- 트리가 한쪽으로 치우치는 **편향 트리** 가 될 가능성이 차단되므로, 삽입과 삭제 연산이 최악의 경우에도 항상 **O(log n)** 의 성능을 유지합니다. (만약 불완전한 트리였으면 탐색 깊이가 깊어져 성능이 **O(n)** 까지 떨어질 수 있음)
### 1-3. 목적에 충실한 구조
힙의 구조적 특징 덕분에 최상단 노드(Root)에는 항상 전체 데이터 중 최댓값 혹은 최솟값이 위치하게 됩니다.
- 배열의 첫번째 요소(heap[1])만 확인하면 되므로 최우선 순위 데이터를 O(1) 만에 찾아낼 수 있습니다.
> 힙은 **완전 이진 트리**라는 구조적 제약을 가짐으로써, **배열**이라는 단순하고 빠른 저장 수단을 사용할 수 있게 되고, 결과적으로 **성능의 일관성**(O(log n))을 확보하게 됩니다.

## 2. 힙을 배열로 구현할 때의 장점은?
> 힙은 구조적으로 **완전 이진 트리**의 형태를 띠고 있어, 배열로 구현했을 때 연결 리스트보다 훨씬 강력한 효율성을 발휘합니다.
### 2-1. 구현의 단순성과 삽입 연산의 용이성
- 힙의 삽입은 항상 트리의 가장 마지막 리프(Leaf) 노드에서 시작됩니다. 연결 리스트는 마지막 노드의 위치를 찾기 위해 루트부터 탐색해야 하거나 복잡한 포인터 관리가 필요하지만, 배열은 **마지막 인덱스(n)**에 데이터를 추가하면 됩니다.
- 부모와 자식 간의 연결을 위해 `left`, `right`, `parent` 와 같은 포인터를 저장할 메모리 공간을 아낄 수 있으며, 구현 코드가 훨씬 간결해집니다.
### 2-2. 수식을 통한 즉각적인 인덱스 접근
- 배열의 인덱싱 기능을 활용하면 별도의 탐색 없이 부모와 자식 노드의 위치를 수식만으로 즉시 계산할 수 있습니다.
1. 부모 노드p의 인덱스를 알 때, 왼쪽 자식은 2p, 오른쪽 자식은 2p+1로 바로 접근 가능합니다.
2. 반대로 자식 노드에서 부모 노드로 이동할 때도 i/2 연산 하나면 충분합니다.
### 2-3. 캐시 친화적인 특성과 성능 향상
- 배열은 메모리 상에 데이터가 **연속적으로 배치**됩니다. 이는 CPU가 데이터를 읽어올 때 주변 데이터까지 함께 캐시에 올리는 캐시 친화적인 특성을 제공해 실제 동작 속도를 높여줍니다.
- 이러한 하드웨어 효율성 덕분에 네트워크 라우팅이나 우선순위 큐처럼 응답 속도가 중요한 시스템 분야에서 실질적인 성능 향상을 도모할 수 있습니다.
### 2-4. 일관된 성능 유지(O(log n))
- 배열로 구현된 완전 이진 트리는 데이터가 빈틈없이 채워지므로 트리의 높이가 항상 최소한으로 유지됩니다.
- 데이터의 탐색(루트 접근), 삽입, 삭제 연산이 최악의 경우에도 O(log n)의 성능을 안정적으로 유지할 수 있도록 설계됩니다.


| 구분 | 배열 구현 | 연결 리스트 구현 |
| --- | --- | --- |
| 마지막 노드 찾기 | 인덱스(n)로 즉시 접근 | 루트부터 탐색하거나 별도 변수 관리 필요 |
| 메모리 효율 | 포인터 공간 불필요 (연속된 공간) | 노드당 2~3개의 포인터 공간 추가 필요 |
| 접근 속도 | 인덱스 계산을 통합 직접 접근 | 포인터를 따라가는 오버헤드 발생 |
| 캐시 효율 | 높음(데이터가 모여 있음) | 낮음(데이터가 메모리 곳곳에 분산됨) |

## 3. 힙의 삽입 과정을 Heapify-up 과정과 함께 설명하면?
1. **가장 마지막 위치에 삽입**: 트리의 가장 끝(배열의 마지막 인덱스)에 새로운 노드를 삽입하여 완전 이진 트리의 형태를 유지합니다.

2. **부모 노드와 비교**: 새로 삽입된 노드와 그 부모 노드의 값을 비교합니다.

3. **교환(Swap) 및 반복**: 삽입된 노드의 값이 부모보다 크다면 서로 위치를 바꿉니다.
- **종료 조건**: 부모 노드가 더 크거나, 삽입된 노드가 **루트(Root) 노드**에 도달하면 과정을 멈춥니다. 이 과정을**Heapify-up (또는 Up-heap, Percolate-up)** 이라고 합니다.

## 4. 힙의 삭제(추출) 과정을 Heapify-down 과정과 함께 설명하면?
1. **루트 노드 추출**: 최댓값(최대 힙) 또는 최솟값(최소 힙)이 있는 루트 노드를 반환하고 제거합니다.

2. **마지막 노드 이동**: 트리의 가장 마지막 노드(배열의 끝 요소)를 비어있는 루트 노드 자리로 옮깁니다. 이는 '완전 이진 트리'의 구조를 깨뜨리지 않기 위함입니다.

3. **자식 노드와 비교**: 새로운 루트 노드와 그 자식 노드들의 값을 비교합니다.

4. **교환(Swap) 및 반복 (Heapify-down)**:
- 최대 힙 기준: 두 자식 노드 중 **더 큰 값**을 가진 자식과 부모 노드를 교환합니다.
- 종료 조건: 부모가 자식보다 크거나, 리프(Leaf) 노드에 도달하여 더 이상 비교할 자식이 없으면 멈춥니다. 이 과정을 **Heapify-down** (또는 Down-heap, Percolate-down)이라고 합니다.

## 5. 힙에서 특정 값을 탐색하는 시간복잡도는? 왜?
> 힙에서 특정 값을 탐색하는 시간복잡도는 O(n) 이고, 그 이유는 힙은 값의 크기 순서만 보장하므로, 원하는 값이 어디에 있는지 알 수 없어 전 탐색이 필요하기 때문입니다.

### 5-1. 힙에서 특정 값을 탐색하는 시간복잡도가 O(n)인 이유
힙은 완전 이진 트리 형태로 구현되므로 부모-자식 관계만을 이용해 트리의 높이가 log n에 비례합니다. 하지만 특정 값이 힙의 어느 위치에 있는지 (루트가 아닌 중간 / 하위 노드) 예측할 수 없으므로, 루트부터 시작해 자식 노드로 내려가며 모든 원소를 확인해야 합니다. 따라서 최악의 경우(값이 없거나 가장 마지막에 있는 경우)에는 n번 비교가 필요해 O(n) 이 됩니다.

## 6. 우선순위 큐를 배열이나 연결 리스트로 구현하면 어떤 문제가 있는가?

- 우선순위 큐를 배열이나 연결 리스트로 구현하면 삽입·삭제 시 원소를 한 칸씩 이동하거나, 삽입 위치를 찾기 위해 **전체 원소를 순회**해야 하는 단점이 있어 최악의 경우 **O(n)** 의 시간복잡도가 발생합니다.
- 반면 힙 기반 구현은 삽입/삭제를 O(log n)로 줄여, 원소가 많거나 삽입/삭제가 빈번한 상황에서 효율적입니다.


| 구현 | 삽입 | 삭제 | 삽입 위치 탐 | 삭제 |
| --- | --- | --- | --- | --- |
| 배열 | O(n) | O(n) | O(n) |중간 삽입/삭제 시 원소 이동 필요 |
| 연결 리스트 | O(1)(참조만) | O(1) | O(n) | 삽입 위치 찾기 위해 순차 탐색 필요 |
- 배열: 삽입·삭제 과정에서 원소를 한 칸씩 밀거나 당겨야 하므로, 특히 중간 삽입/삭제가 빈번하면 성능 저하가 큽니다.
- 연결 리스트: 배열의 한계는 줄어들지만, 삽입 위치를 찾기 위해 첫 노드부터 끝까지 우선순위 비교를 진행해야 할 수 있어 탐색 비용이 O(n)이 될 수 있습니다.
