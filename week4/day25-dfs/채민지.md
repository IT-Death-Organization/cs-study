## 1. DFS는 어떻게 동작하며 왜 깊이를 우선 탐색하는가?

> **DFS(깊이 우선 탐색)**는 말 그대로 **갈 수 있는 경로가 있는 한 최대한 깊게 파고들며 탐색하는 방식**입니다.

![title](https://www.interviewbit.com/blog/wp-content/uploads/2021/12/DFS-Algorithm-1536x1190.png) 
### 1-1. DFS는 어떻게 동작하는가 ?

1. **시작 노드**를 방문하고 '방문 처리'를 합니다.
2. 현재 노드와 연결된 인접 노드 중 **방문하지 않은 노드**가 있다면 그쪽으로 이동합니다.
3. 더 이상 방문할 인접 노드가 없으면(막다른 길), 이전 단계로 **되돌아옵니다**
4. 모든 노드를 방문할 때까지 이 과정을 반복합니다.

⇒ 구현에는 주로 **재귀 함수**나 **스택(Stack)** 자료구조를 사용합니다.

### 1-2. 왜 깊이를 우선 탐색하는가 ?

- **모든 경우의 수 탐색:** 미로 찾기처럼 한 길을 끝까지 가보고 안 되면 돌아와서 다른 길을 찾는 방식에 최적화되어 있습니다.
- **그래프의 특징 파악:** 노드 간의 **연결 관계**나 사이클(회전)이 있는지 확인하는 데 유리합니다.
- **메모리 효율:** 경로상의 노드들만 기억하면 되기 때문에, 가로로 넓게 퍼지는 BFS(너비 우선 탐색)보다 특정 상황에서 메모리를 적게 쓸 수 있습니다.

⇒ 찾고자 하는 목표가 시작점 근처에 있다면 깊이 파고드는 특성상 시간이 오래 걸릴 수 있단 단점 존재

# 2. DFS를 스택으로 구현하는 이유는? 재귀로 구현하면 어떤 차이가 있는가?

> DFS를 스택(Stack)으로 구현하는 근본적인 이유는 **알고리즘의 동작 원리 자체가 '가장 나중에 발견한 길을 가장 먼저 탐색'하는 LIFO(Last-In-First-Out) 구조이기 때문**입니다.
- 재귀함수도 내부적으론 시스템의 콜 스택을 이용하므로 본질적으론 같은 논리를 따르나, 실제 구현상으론 다음과 같은 차이가 있습니다.

| **구분** | **재귀 (Recursion)** | **명시적 스택 (Explicit Stack)** |
| --- | --- | --- |
| **구현 난이도** | 코드가 간결하고 직관적임 | 코드가 다소 복잡할 수 있음 |
| **메모리 제한** | 시스템 **콜 스택 제한**에 걸려 `RecursionError` 발생 가능 | 힙(Heap) 메모리를 사용하므로 **상대적으로 여유로움** |
| **성능** | 함수 호출 오버헤드가 발생함 | 루프를 사용하므로 미세하게 빠를 수 있음 |
| **탐색 순서** | 인접 노드 방문 시 즉시 깊이 들어감 | 스택에 넣은 순서의 **역순**으로 방문함 |

## 3. 재귀 DFS의 시간복잡도와 공간복잡도는? (그래프 기준)

> 재귀 DFS의 시간 및 공간 복잡도는 그래프의 **노드(정점) 수 V와 간선수 E에 의해 결정됩니다.**

### 3-1. 시간 복잡도

- 시간복잡도 : O(V+E)
- 이유 : 모든 노드를 한 번씩 방문(V)하고, 각 노드에서 연결된 모든 간선을 한 번씩 확인(E)하기 때문입니다.
- 단, 인접 행렬로 구현할 경우 특정 노드에 연결된 노드를 찾기 위해 항상 모든 노드를 훑어야 하므로 O(V²)이 됩니다. (일반적인 인접 리스트 기준은 O(V+E))

### 3-2. 공간 복잡도

- 공간복잡도 : O(V)
- 이유
    - 어떤 노드를 방문했는지 기록하기 위해 노드 수만큼의 공간 O(V) 이 필요합니다.
    - 최악의 경우(예: 그래프가 한 줄로 길게 늘어선 편향 트리 형태일 때) 재귀 함수의 깊이가 노드 수만큼 깊어지므로 O(V) 의 스택 공간을 사용합니다.
    
    [](data:image/gif;base64,R0lGODlhAQABAIAAAP///wAAACH5BAEAAAAALAAAAAABAAEAAAICRAEAOw==)
    
    ⇒ 전체를 훑어야 하므로 시간은 **노드와 간선의 합**에 비례하고, 공간은 **노드의 개수**만큼 필요합니다.
    

## 4. DFS는 어떤 문제에 적합한가? 실제 활용 사례는?

- DFS는 **1. 끝까지 가봐야 알 수 있는 문제** 나 **2. 모든 조합을 확인해야 하는 문제**에 가장 적합합니다.

### 4-1. **DFS가 적합한 문제 유형**

- **경로의 존재 여부:** A 도시에서 B 도시까지 갈 수 있는 길이 하나라도 있는지 확인할 때 유리
- **사이클 검출:** 그래프 내에 순환 고리(Cycle)가 있는지 확인할 때 사용
- **백트래킹(Backtracking):** 체스의 퀸 배치(N-Queen)나 스도쿠처럼, 후보를 시도해보다가 아니면 되돌아와서 다른 길을 찾아야 하는 경우 필수적
- **단지 번호 붙이기:** 연결된 요소(Connected Components)를 찾아 그룹화할 때 효과적

### 4-2. 실제 활용 사례

- **미로 찾기:** 한 길을 끝까지 가보고 막히면 돌아오는 전형적인 DFS 방식
- **프로그래밍 언어 컴파일러:** 소스 코드의 문법 구조를 분석하거나, 변수 간의 의존성(어떤 함수가 어떤 함수를 부르는지)을 파악할 때 사용
- **퍼즐 게임 해결:** 루빅스 큐브나 퍼즐의 해답을 찾기 위해 가능한 모든 이동 조합을 탐색할 때 활용
- **네트워크 분석:** SNS에서 특정 사용자와 연결된 인맥 지도를 끝까지 타고 내려가며 분석할 때 사용

## 5. 백트래킹과 DFS의 관계는? 백트래킹은 어떻게 구현하는가?

### 5-1. 백트래킹과 DFS의 관계
![title](https://blog.kakaocdn.net/dn/DnCnI/btrqY9EvQTx/xZWxxAF4FDFZfjpVaS6aE1/img.png)   

- **DFS (전략):** 그래프나 트리의 모든 노드를 빠짐없이 방문하는 **탐색 알고리즘** 그 자체입니다.
- **백트래킹 (전술):** DFS로 탐색을 진행하되, 가고 있는 길이 정답이 될 가능성이 없으면 즉시 되돌아오는 **최적화 기법**입니다. 이를 **'가지치기(Pruning)'**라고 부릅니다.

즉, 모든 백트래킹은 DFS를 기반으로 하지만, 모든 DFS가 백트래킹인 것은 아닙니다. (일반 DFS는 모든 노드를 다 가보지만, 백트래킹은 가기 전에 '가도 될지' 먼저 따짐)

### 5-2. **백트래킹 구현 방법 (재귀 구조)**

백트래킹은 주로 **재귀 함수**로 구현하며, 아래와 같은 3단계 구조를 가집니다.

```python
def backtracking(node):
    # 1. 종료 조건 (정답을 찾았거나 바닥에 도달했을 때)
    if node == 목표:
        정답 처리 및 반환
        return

    for next_node in 후보들:
        # 2. 유망성 검사 (Promising: 이 길이 유효한가?)
        if 가볼 만한 길인가(next_node): 
            방문 표시 (선택)
            backtracking(next_node) # 3. 다음 단계 진행 (재귀)
            방문 표시 해제 (선택 취소 - 이 부분이 '백트래킹'의 핵심)
```

- 백트래킹 구현에서  가장 중요한 점은 재귀 호출이 끝난 뒤 **상태를 원상복구**하는 것입니다. ( ∵ 그래야 다음 후보를 탐색할 때 이전 선택의 영향을 받지 않고 깨끗한 상태에서 시작할 수 있음)

⇒ DFS는 끝까지 가보는 것이고, 백트래킹은 가다가 **이 길은 아니다** 싶으면 중간에 포기하고 돌아와서 시간을 아끼는 기술입니다.

## 6. DFS로 사이클을 탐지하는 방법은?

### **6-1. 무방향 그래프 (Undirected Graph)**

무방향 그래프에서는 단순히 "방문했던 노드인가?"만 확인하면 안 됩니다. 방금 직전에 온 노드(부모 노드)로 되돌아가는 것도 방문 재방문으로 처리될 수 있기 때문입니다.

- **방법:** DFS 탐색 시 현재 노드와 연결된 인접 노드를 확인합니다.
- **판단 기준:** 인접 노드가 이미 **방문(visited)** 상태인데, 그 노드가 현재 노드의 **부모(직전 노드)가 아니라면** 사이클이 존재하는 것으로 판단합니다.

### **6-2. 유방향 그래프 (Directed Graph)**

유방향 그래프는 방향성이 있기 때문에, 단순히 이전에 방문했다고 해서 사이클은 아닙니다. 현재 탐색 중인 **경로(Recursion Stack)** 상에 있는 노드를 다시 만나야만 사이클입니다.

- **방법:** 노드의 상태를 3가지(방문 전, 탐색 중, 탐색 완료)로 관리하거나 별도의 **재귀 스택(RecStack)** 배열을 사용합니다.
- **판단 기준:** 탐색 과정에서 아직 함수가 종료되지 않은 **'탐색 중'** 상태인 노드(역방향 간선, Back Edge)를 다시 방문하게 되면 사이클로 판단합니다.

| **그래프 종류** | **핵심 판단 조건** | **필요한 정보** |
| --- | --- | --- |
| **무방향** | 방문한 적이 있는데 부모 노드가 아님 | 방문 여부, 부모 노드 정보 |
| **유방향** | 현재 탐색 경로(스택)에 이미 존재하는 노드임 | 방문 여부, 재귀 스택 상태 |
