### 📌 [Q1] DFS는 어떻게 동작하며 왜 깊이를 우선 탐색하는가?

DFS는 그래프나 트리에서 **시작 노드에서 출발하여 최대한 깊이 들어갔다가, 더 이상 갈 곳이 없으면 되돌아와서 다른 경로를 탐색하는 알고리즘**이다.
DFS의 핵심은 **뒤 돌아보지 않고 끝까지 가보기**와 **막히면 직전 갈림길로 돌아오기(backtracking)**이다.

**동작 과정**<br>
- 시작 노드 방문: 탐색을 시작할 노드를 방문하고 '방문 처리'를 합니다.
- 인접 노드 확인: 현재 노드와 연결된 노드 중 방문하지 않은 노드가 있는지 확인합니다.
- 깊게 들어가기:
  - 방문하지 않은 인접 노드가 있다면, 그 노드로 이동하여 다시 1번 과정을 수행합니다. (재귀적 호출 또는 스택에 push)
- 되돌아오기 (Backtracking):
  - 더 이상 방문하지 않은 인접 노드가 없다면, 이전 노드로 돌아가서 다른 인접 노드를 찾습니다.
- 종료: 모든 노드를 방문할 때까지 반복합니다.

**왜 깊이를 우선 탐색하는가?**
- BFS보다 직관적이고 구현하기 쉬움
- 메모리 효율성 (공간 복잡도)
  - BFS(너비 우선)는 **현재 레벨의 모든 노드를 큐에 저장**해야 함, 만약 자식 노드가 많은 트리라면 메모리 사용량이 급격히 늘어남
  - DFS(깊이 우선)는 **현재 탐색 중인 경로의 노드들만 스택에 유지**하면 됨, 따라서 그래프의 폭에 비해 깊이가 깊지 않은 경우 **BFS보다 메모리 사용량이 적음**
- 다음과 같은 상황에서 유리함
  - 모든 경우의 수 탐색(backtracking)
  - 최단 경로가 아닌, **경로 자체가 중요한 경우** (ex. 경로의 가중치 합)

### 📌 [Q2] DFS를 스택으로 구현하는 이유는? 재귀로 구현하면 어떤 차이가 있는가?

DFS 구현 방법에는 **스택**을 사용하는 방법과 **재귀 호출**을 사용하는 방법 두 가지가 있다. 둘 다 LIFO 구조를 따르지만 차이가 있다.

**DFS를 스택으로 구현하는 이유**
- DFS의 핵심: 가장 **나중에** 발견한 갈림길로 되돌아 가는 것 -> **스택의 LIFO 구조**와 완벽하게 일치
- **역추적(backtracking)의 용이성**: 현재 노드에서 더 이상 갈 곳이 없을 때, **스택의 pop**을 통해 가장 최근에 머물렀던 부모 노드로 자연스럽게 돌아갈 수 있음
- **제한 없는 depth**: 시스템의 호출 스택(call stack) 크기 제한에 구애받지 않고, 메모리(heap)가 허용하는 한 아주 깊은 그래프도 탐색 가능

**재귀 DFS의 차이점**<br>
**[재귀]**
- **코드의 간결성**: 재귀 호출은 스택을 명시적으로 관리할 필요가 없어 코드가 훨씬 간결하고 직관적
- **메모리 위치**: **스택 영역(stack memory) 사용**
- **호출 스택 사용**: 재귀 호출 시에는 **함수 호출 스택(call stack)**에 함수 정보가 쌓임
  - 장점: 별도의 스택 자료구조를 만들 필요가 없음
  - 단점: 시스템의 호출 스택 크기 제한에 구애받음, 깊이가 깊어지면 **스택 오버플로우(Stack Overflow)** 발생 가능
- 탐색 순서: 인접 노드들을 순차적으로 방문 (1, 2, 3 순서)

**[스택]**
- **코드의 간결성**: 재귀보다 코드 복잡
- **메모리 위치**: **힙 영역(heap memory) 사용**
- 메모리(RAM)가 부족하지 않는 한 안전함
- 탐색 순서: 스택에 넣는 순서에 따라 반대로 방문할 수 있음 (3, 2, 1 순서)

**결정적인 차이: StackOverflow**
- 대부분의 프로그래밍 언어에서 시스템 호출 스택의 크기는 매우 작게(수 mb) 설정되어 있음
- 따라서 재귀 호출이 깊어지면 스택 오버플로우가 발생할 수 있음 -> 프로그램 강제 종료
- 반면 **스택을 사용한 DFS는 힙 메모리를 사용하므로 스택 오버플로우 걱정 없음** 
  - stack 메모리 영역(재귀): 보통 프로그램 실행 시 크기가 고정됨, 이 한도를 넘으면 바로 StackOverflow
  - heap 메모리 영역(스택): OS가 허용하는 한 자유롭게 늘어나는 동적인 영역, 컴퓨터에 꽂힌 전체 RAM 용량

**정리**
- 스택으로 구현하는 이유: DFS의 '되돌아오기' 로직이 스택의 LIFO 구조와 일치하기 때문
- 차이점: 재귀는 구현이 쉽지만 깊이 제한(StackOverflow)에 취약하고, 명시적 스택은 구현이 조금 더 복잡한 대신 대규모 데이터 처리에 안정적

### 📌 [Q3] 재귀 DFS의 시간복잡도와 공간복잡도는? (그래프 기준)

그래프 기준 재귀 DFS의 복잡도는 그래프를 **인접 리스트**로 표현했는지, **인접 행렬**로 표현했는지에 따라 달라진다.

V: 노드의 개수
E: 간선의 개수

시간 복잡도: DFS의 시간 복잡도는 **모든 노드를 한 번씩 방문하고, 각 노드에 연결된 모든 간선을 확인하는 데 걸리는 시간**이다.

**인접 리스트 사용 시 시간 복잡도: O(V + E)**
- 각 노드를 한 번씩 방문: O(V)
- 각 간선을 한 번씩 확인: O(E)
- == 전체 노드와 간선을 한 번씩 다 훑는 것이므로 O(V + E)
  - **V와 E 중 더 큰 것만 쓰면 안 되나? -> 안됨, 그 이유는**
    - 노드가 1,000개인데 간선이 0개일 수도 있고 (이 경우 $O(V)$)
    - 노드가 2개인데 간선이 1개일 수도 있습니다. (이 경우 $O(E)$) 그래프의 형태에 따라 두 값이 독립적일 수 있기 때문에, **"모든 노드를 방문하고($V$) 동시에 모든 간선을 검사한다($E$)"**는 의미에서 $O(V + E)$라고 표현하는 것이 가장 정확합니다.

**인접 행렬 사용 시 시간 복잡도: O(V^2)**
- 각 노드를 한 번씩 방문: O(V)
- 각 노드에 연결된 모든 간선을 확인: O(V)
- 노드 하나를 방문할 때마다, 다른 모든 노드와 연결되어 있는지 확인해야 함 -> V * V
- == 간선이 있든 없든 무조건 V번 확인해야 하므로 O(V^2)

공간 복잡도: DFS의 공간 복잡도는 **재귀 호출 시 스택에 쌓이는 노드의 개수**이다. 이때 재귀 DFS가 사용하는 메모리는 크게 두 가지이다.

**1. 호출 스택(Call Stack)**
- 재귀 호출 시 함수 정보가 쌓이는 공간
- 최악의 경우 그래프의 최대 깊이만큼 쌓일 수 있음 (한 번에 모든 노드가 스택에 쌓임, 최대 깊이는 노드의 개수와 비례) -> O(V)

**2. 방문 배열(Visited Array)**
- 방문한 노드를 기록하는 배열
- V개의 노드가 있으므로 V만큼의 공간 필요하므로 O(V)

따라서 재귀 DFS의 공간 복잡도는 O(V) + O(V) = O(V)이다.

**인접 리스트**
- 시간 복잡도: O(V + E)
- 공간 복잡도: O(V)

**인접 행렬**
- 시간 복잡도: O(V^2)
- 공간 복잡도: O(V)

### 📌 [Q4] DFS는 어떤 문제에 적합한가? 실제 활용 사례는?

DFS는 **최단 경로보다는 경로나 노드의 상태, 그래프의 구조 자체를 파악해야 하는 문제에** 적합하다.

**1. 모든 경우의 수 탐색 (Backtracking)**
- **백트래킹**: DFS를 기반으로 **해답이 될 수 없는 경로는 중간에 끊고 되돌아오는(Backtracking)** 알고리즘
- **활용**: 순열, 조합, 미로 찾기, N-Queen 문제 등

**2. 연결 요소(Connected Components) 찾기**
- **활용**: 그래프가 여러 개의 분리된 부분으로 이루어져 있을 때, 각 부분을 찾는 문제

**3. 위상 정렬(Topological Sorting)**
- **활용**: 작업의 선후 관계가 있는(작업 순서가 정해진) 그래프에서 작업 순서를 결정하는 문제

**4. 사이클 탐지(Cycle Detection)**
- **활용**: 그래프에 사이클(순환 경로)이 있는지 확인하는 문제

**실제 활용 사례**
- 파일 시스템 탐색: 컴퓨터의 폴더 구조를 끝까지 파고들며 특정 파일을 찾거나 용량을 계산할 때 사용합니다. (탐색기의 폴더 트리 구조)
- 미로 생성 및 해결: 복잡한 미로를 생성하거나, 미로에서 탈출 경로가 하나라도 있는지 확인할 때 쓰입니다.
- 의존성 관리: 소프트웨어 패키지 관리자(npm, pip 등)가 라이브러리 간의 의존성 관계를 파악하고 설치 순서를 정할 때 DFS 기반의 알고리즘을 사용합니다.
- 체크 알고리즘: 체스나 바둑 같은 인공지능이 "내가 여기 두면, 상대가 여기 두고, 그다음 내가..." 식으로 수십 수 앞을 깊게 내다볼 때(Decision Tree 탐색) 활용됩니다.

### 📌 [Q5] 백트래킹과 DFS의 관계는? 백트래킹은 어떻게 구현하는가?

**백트래킹과 DFS의 관계**
- 백트래킹은 DFS를 기반으로 한 알고리즘, DFS의 탐색 방식에 '유효성 검사'와 '중단' 기능을 더한 고도화 전략
- DFS는 **모든 경우의 수를 탐색**하는 알고리즘
- 백트래킹은 DFS를 기반으로 **해답이 될 수 없는 경로는 중간에 끊고 되돌아오는(Backtracking)** 알고리즘

**백트래킹 구현 방법**
- 주로 **재귀**를 통해 구현하며, Choose-Explore-Unchoose의 과정을 반복
  - 결정(Choose): 여러 후보 중 하나를 선택합니다.
  - 검사 및 재귀(Explore): 선택한 후보가 유효한지(Promising) 확인하고, 유효하다면 다음 단계로 깊게 들어갑니다.
  - 되돌리기(Unchoose/Backtrack): 정답을 못 찾았거나 다음 후보를 검사해야 할 때, 이전 단계로 돌아와서 방금 했던 선택을 취소합니다.

### 📌 [Q6] DFS로 사이클을 탐지하는 방법은?

DFS로 사이클을 탐지하는 방법은 **그래프가 방향성이 있는지, 없는지에 따라 탐지 로직이 약간 다르다.** 핵심은 "이미 방문한 노드를 다시 만났을 때, 그것이 단순한 이전 경로인지 or 순환 구조인지" 구분하는 것입니다.

**1. 무방향 그래프(Undirected Graph)**
- 이미 방문한 노드를 다시 만났을 때, **그 노드가 현재 노드의 부모 노드(바로 직전 노드)가 아니라면** 사이클로 간주
- 이유: 무방향 그래프에서는 A->B로 갔다가 B->A로 돌아오는 것이 당연하므로, 부모 노드를 제외한 이미 방문한 노드를 만나야만 진짜 사이클

**2. 방향 그래프(Directed Graph)**
- 이미 방문한 노드를 다시 만났을 때, **그 노드가 현재 탐색 경로(Recursion Stack)에 포함되어 있다면** 사이클로 간주
- 이유: 방향 그래프에서는 A->B->C로 갔다가 다시 A로 돌아오는 경우에만 사이클
- 이를 위해 **visited 배열 외에 recursionStack 배열(또는 Set)을 추가로 사용**
