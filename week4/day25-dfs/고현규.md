# 1. DFS는 어떻게 동작하며 왜 깊이를 우선 탐색하는가?

> DFS는 그래프나 트리 구조를 탐색하는 가장 기본적인 방법 중 하나로, 깊이를 우선 탐색합니다.

### DFS 동작 3단계

1. 방문
    * 현재 위차한 노드를 방문 처리하고 기록

2. 전진
    * 연결된 인접 노드 중 아직 가지 않은 곳이 있다면 그곳으로 "바로" 이동

3. 후퇴 (백트래킹)
    * 주변에 더 이상 갈 수 있는 곳 또는 안가본 곳이 없다면, 방금 왔던 길을 되돌아가서 다른 갈림길이 있는지 찾습니다.

<br>

### 탐색 순서

![DFS 트리 탐색 순서 이미지](https://kayuse88.github.io/assets/img/posts/dfs-bfs/Depth-first-tree.svg)

### 깊이를 우선하는 이유

모든 경우의 수를 다 따져야 할 때 깊이 우선 탐색이 필요합니다. 조합이나 순열처럼 이 길로 끝까지 갔을 때 뭐가 있는지 확인해야 하는 경우입니다.

모든 노드를 방문해야 할 때, 너비 우선 탐색(BFS)보다 메모리 사용량이 적을 수 있습니다.

<br>

# 2. DFS를 스택으로 구현하는 이유는? 재귀로 구현하면 어떤 차이가 있는가?

> DFS를 스택으로 구현하는 이유는 스택 오버플로우 방지를 위해서입니다.

재귀는 콜 스택(Call Stack)을 사용합니다. 그런데 이것은 크기가 제한적이기 때문에 깊이가 깊은 그래프를 재귀로 탐색하면 프로그램이 강제 종료될 수 있습니다.

반면 명시적 스택을 사용하면 힙(Heap)메모리를 사용하므로 훨씬 더 깊은 그래프를 탐색할 수 있습니다.

또한 재귀 호출은 컨텍스트 스위칭이나 매개변수 전달로 인한 오버헤드가 존재합니다.

<br>

# 3. 재귀 DFS의 시간복잡도와 공간복잡도는? (그래프 기준)

그래프의 노드의 갯수를 V, 간선의 갯수를 E라고 할 때 복잡도를 알아보겠습니다.

### 시간복잡도

DFS는 기본적으로 모든 노드를 한 번씩 방문하고, 모든 간선을 한 번씩 검사하는 것이 목표입니다.

이때 각 노드를 방문하는 부분을 보면 방문 여부(`visited`) 체크를 통해 딱 한번만 방문하게 됩니다. 따라서 $O(V)$입니다.

또한 각 노드에서 연결된 모든 간선을 확인하여 다음으로 이동해야 할 노드를 결정합니다. 인접 리스트로 구현했을 때 결과적으로 모든 간선을 확인하게 됩니다. 따라서 $O(E)$입니다.

전체 시간복잡도는 둘을 합친 $O(V+E)$입니다.

### 공간복잡도

DFS는 현재 탐색 중인 경로상의 노드들만 메모리에 유지합니다. 최악의 경우 그래프가 일직선 모양이어서 시작부터 끝까지 모든 정점이 재귀 호출 스택에 쌓이게 되므로 $O(V)$입니다.

<br>

# 4. DFS는 어떤 문제에 적합한가? 실제 활용 사례는?

> DFS는 그래프의 구조적 특징을 파악하거나 모든 경우의 수를 탐색해야 할 때 좋습니다.

### 경로 존재 여부 확인

지도(그래프)에서 A에서 B로 갈 수 있는가? Yes or No 문제를 풀 때 최적해를 찾을 필요 없이 가능 여부만 찾으면 되므로 DFS가 적절합니다.

### 사이클 탐지

그래프 내에 사이클(Cycle)이 있는지 확인할 때 DFS를 사용해서 이미 방문한 노드를 다시 방문한다면 사이클이 있다고 판단할 수 있습니다.

### 위상 정렬 (Topological Sorting)

어떤 순서가 정해져 있는 작업을 DFS로 탐색했다면 탐색을 마치고 나오는 그 순서를 기록하고 뒤집으면 올바른 순서가 됩니다.

예를 들면 학교 수강신청 선수과목 구조를 생각하면 이해가 쉽습니다. 어떤 과목을 시작으로 선수과목을 DFS로 탐색해 내려간다면 다시 나올 때 그 순서를 기록하고 뒤집어보면 저학년부터 시작해서 어떤걸 들어나가야 하는지 알 수 있습니다.

### 모든 조합/경우의 수 탐색 (Backtracking)

미로 찾기, 스도쿠 풀이, N-Queens 문제 등 모든 가능성을 탐색해야 할 때 DFS는 필수입니다.

# 5. 백트래킹과 DFS의 관계는? 백트래킹은 어떻게 구현하는가?

> DFS는 방식/도구이고, 백트래킹은 전략입니다.

쉽게 말하면, 답이 될 가능성이 없는 경로는 미리 포기하고 되돌아가는 설계 원리인 "백트래킹 전략"을

끝까지 가보고 아니면 돌아오는 특성을 가진 "DFS라는 도구"를 사용해서 구현/실현한다라고 생각하면 됩니다.

### 구현 방법 (Java)

```java
void backtracking(int level) {
    // 1. 종료 조건 (Base Case)
    if (level == 목표_깊이) {
        정답_처리();
        return;
    }

    for (int i = 0; i < 후보군_개수; i++) {
        // 2. 유망성 검사 (Pruning) - 이 길이 가망이 있는가?
        if (isPromising(i)) {
            
            // 3. 상태 변경 (방문 처리)
            visited[i] = true;
            
            // 4. 다음 단계로 전진 (재귀 호출)
            backtracking(level + 1);
            
            // 5. 상태 복구 (Backtrack)
            // 다시 돌아왔을 때는 방문 기록을 지워줘야 다른 경로에서 활용 가능함
            visited[i] = false;
        }
    }
}
```

# 6. DFS로 사이클을 탐지하는 방법은?

### 무방향 그래프라면

단순히 이미 방문한 노드를 또 방문하면이라는 조건은 조금 부족합니다. 이미 방문한 노드이면서 현재 노드의 바로 직전 노드(즉, 바로 위 부모)가 아니어야 합니다. T자형 구조를 탐색할 때 탐색이 어떻게 이루어지는지 생각해보면 쉽습니다.

### 유방향 그래프라면

노드를 단순히 방문 여부만 따지면 안되고 방문 안함, 현재 방문(탐색)중, 방문 완료 3가지 상태로 구분해야 합니다.

왜냐하면 그래프를 탐색 중 더 이상 길이 없는 끝을 만나고 되돌아가면서 반환된(방문 완료 처리된) 노드를 나중에 다른 경로로 다시 지나갈 수도 있기 때문입니다. 하지만 이 경우는 사이클이 아닙니다.

이것을 구현하기 위해서는 `visited[]` 배열 외에 `finished[]`라는 배열 혹은 반대로 `inStack[]` 배열을 추가로 관리해야 합니다.
