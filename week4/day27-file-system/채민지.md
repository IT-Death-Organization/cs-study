## 1. 파일 시스템의 역할은 무엇인가?
> 파일 시스템은 운영 체제(OS)가 저장 장치(하드디스크, SSD, USB 등)에 **데이터를 저장하고 관리하며, 필요할 때 쉽게 찾을 수 있도록 조직화하는 체계**입니다.

파일 시스템이 없다면 저장 장치는 단순한 **빈 저장 공간**일 뿐입니다. 운영체제가 데이터를 정확히 저장하고 다시 찾기 위해서는 **파일 시스템의 조직화된 구조**가 반드시 필요합니다.

| **기능** | **설명** |
| --- | --- |
| **구조적 저장** | 데이터를 파일 단위로 분할하고 디렉터리 구조로 정리 |
| **탐색 속도 향상** | 경로, 이름, 날짜 등의 정보를 활용해 빠른 탐색 가능 |
| **접근 제어** | 사용자/그룹 단위로 파일 읽기, 쓰기, 실행 권한 설정 |
| **오류 복구 지원** | 시스템 비정상 종료 시 복구 가능한 구조 제공 (예: 저널링) |
| **저장 공간 최적화** | 저장 공간을 효율적으로 할당해 낭비 최소화 |

## 2. 파일 할당 방법(연속, 연결, 인덱스)의 차이와 장단점은?
### 2-1. 연속 할당 (Contigious Allocation)
: 하나의 파일을 디스크 내 연속된 블록에 저장하는 방식입니다.
- 장점 : 블록 주소와 offset을 통해 빠르게 접근할 수 있습니다.
- 단점 : 내부/외부 단편화가 모두 발생하고, 파일 크기 측면에서 유연하지 않습니다.

### 2-2. 연결리스트 할당 (LinkedList Allocation)
: 파일을 구성하는 각 블록을 연결리스트로 구현하는 방식입니다. 디렉토리는 첫 번째와 마지막 블록에 대한 주소를 가집니다.
- 장점 : 외부 단편화를 해결했으며, 파일 크기 측면에서 매우 유연합니다.
- 단점 : 블록이 무작위로 할당되고, 개별적인 블록에 바로 접근하기 어렵고 느립니다.

## 2-3. 인덱스 할당 (Index Allocation)
: 파일에 저장된 블록들의 주소들을 인덱스 블록에 저장해둡니다. Index-node(i-node) 형태로 관리되며, 리눅스 / 유닉스에서 채택된 파일 저장 방식입니다.
- 장점 : 외부 단편화를 해결했으며, 직접 접근으로 속도가 빠릅니다.
- 단점 : 인덱스 블록을 모아두어 디스크가 낭비됩니다.

## 3. inode란 무엇이며 어떤 정보를 저장하는가?
> inode는 **파일 시스템 내에서 파일 또는 디렉토리에 대한 메타데이터를 저장**하는 자료구조입니다.

inode는 파일의 실제 데이터와 파일 이름을 제외한 거의 모든 상세 정보를 저장합니다.
### 3-1. inode에 저장되는 정보 (메타데이터)
- 파일 모드 및 종류 : 일반 파일, 디렉토리, 심볼릭 링크 등의 구분
- 파일 권한 : 소유자, 그룹 및 기타 사용자에 대한 읽기/쓰기/실행 권한
- 소유자 정보 : 파일 소유자의 사용자 ID(UID) 및 그룹 ID(GID)
- 파일 크기 : 바이트 단위의 파일 크기
- 타임스탬프 : 파일 생성 시간, 최근 접근 시간 (atime), 최근 수정 시간 (mtime) 등
- 링크 수 : 해당 inode를 가리키는 하드 링크의 개수
- 데이터 블록 포인터 : 파일의 실제 내용이 디스크의 어느 블록에 저장되어 있는지 알려주는 주소 정보
### 3-2. 주요 특징
- 파일 이름과의 분리 : 파일 이름은 디렉터리 파일 내에 '이름-inode 번호'쌍으로 존재하며, inode자체에는 저장되지 않습니다.
- 개수 제한 : 파일 시스템 생성 시 inode의 총 개수가 결정되므로, 디스크 용량이 남았더라도 inode를 모두 사용하면 더 이상 파일을 생성할 수 없습니다.
## 4. 디렉토리는 어떻게 구현되는가?
> - 디렉토리란 파일의 일종으로, 같은 집합에 속한 파일들의 이름, 속성, 크기 등과 같은 메타데이터를 저장하고 있습니다.
> - 파일 시스템에서 **'파일명'과 'inode 번호'를 매핑(Mapping)하여 관리하는 일종의 특수 파일**로, 사용자는 디렉토리를 파일을 담는 폴더로 인식하지만, 시스템 내부적으로는 파일들의 목록을 저장하고 있는 데이터 블록일 뿐입니다.

디렉토리는 파일을 체계적으로 정리하고 관리하는 상자로, 트리 구조로 여러 하위 폴더를 포함할 수 있습니다.
### 4-1. 내부 구현 방식
디렉토리 내에서 파일을 찾고 관리하기 위해 크게 두 가지 알고리즘이 사용됩니다.
### 선형 리스트 (Linear List)
- 구조 : 파일 이름과 해당 inode 포인터를 단순하게 나열한 리스트 형태입니다.
- 특징 : 구현이 매우 간단하지만, 파일을 찾을 때 처음부터 끝까지 검색해야 하므로 파일 수가 많아지면 속도가 느려집니다.
### 해시 테이블 (Hash Table)
- 구조 : 파일 이름을 해시 함수에 통과시켜 얻은 결과값으로 리스트 위치를 바로 찾아가는 방식입니다.
- 특징 : 검색 속도가 매우 빠르지만, 해시 충돌 (서로 다른 이름이 같은 해시값을 가짐) 처리 로직이 필요하며 관리가 더 복잡합니다.
### 4-2. 논리적 구조 (계층 구조)
대부분의 현대 운영 체제는 데이터를 효율적으로 조직하기 위해 트리 구조 (Tree Structure)를 사용합니다.
- 루트 디렉토리 (/) : 파일 시스템의 최상위 지점으로, 모든 경로의 시작점입니다.
- 하위 디렉토리 : 디렉토리 안에 또 다른 디렉토리를 포함함으로써 계층적인 분류가 가능합니다.
- 특별 항목 : 모든 디렉토리에는 자기 자신을 가리키는 `.` (현재 디렉토리)와 상위 디렉토리를 가리키는 `..` (부모 디렉토리) 항목이 자동으로 생성됩니다.
### 4-3. 디렉토리와 파일의 연결
디렉토리 파일의 데이터 블록 안에는 실제 파일 내용이 들어있지 않습니다. 대신 **[파일명 - inode 번호]** 쌍의 목록만 들어있으며, 시스템은 이 inode 번호를 통해 실제 데이터가 저장된 디스크 블록을 찾아갑니다. 이런 구조 덕분에 하나의 파일 (inode)이 여러 디렉토리에 다른 이름으로 존재할 수 있는 **하드 링크** 구현이 가능해집니다.

## 5. 디스크 스케줄링 알고리즘(FCFS, SSTF, SCAN, C-SCAN)은?
> 디스크 스케줄링 알고리즘은 디스크 헤드의 이동 경로에 따라 전체 처리 속도가 크게 달라지는 기법입니다. 각 알고리즘은 디스크 I/O의 효율성을 높이고 평균 응답 시간을 최소화하는 데 중점을 두고 있습니다.

### 5-1. FCFS(First-Come First-Served)
: 어느 스케줄링 알고리즘에서도 존재하는 가장 간단하고 가장 공평한 방법으로, 요청이 들어온 순서대로 처리하는 방식입니다.

```
200 cylinder dist: 0, 1, 2, ..., 199
Disk queue: 98, 183, 37, 122, 14, 124, 65, 67
현재 헤드가 가리키는 실린더(cylinder) 위치: 53
```

![title](https://user-images.githubusercontent.com/37871541/78577820-9c747b00-7869-11ea-9777-c784b57730e8.png)  
- 가로축은 0번부터 199번까지 실린더의 위치를 나타낸다. 여기서 파란색 선이 dist queue를 FCFS 방법으로 처리한 결과입니다.
- 위 그림의 결과를 본것처럼 큐에 들어온 순서가 큰값, 작은 값을 반복한다면 헤드가 움직이는 거리가 매우 커짐을 알 수 있습니다.

- 장점 : 구현이 쉽고 공평합니다.
- 단점 : 요청 위치가 멀리 떨어져 있으면 헤드가 앞뒤로 마구 움직여 효율이 매우 떨어집니다.
### 5-2. SSTF(Shortest-Seek-Time-First)
: 가장 짧은 탐색 시간을 먼저 선택하는 것입니다. 즉, 현재 헤드가 다음 요청을 처리하기 위해 움직여야 하는 거리가 가장 짧은 것을 선택하는 것입니다.
- 장점 : 탐색 시간을 크게 줄여 처리량이 높습니다.
- 단점 : 먼 곳의 요청은 계속 밀려나는 **기아 현상(Starvation)** 이 발생할 수 있습니다.
### 5-3. Scan
: 헤드가 디스크의 한쪽 끝에서 반대쪽 끝까지 이동하며 경로에 있는 모든 요청을 처리하고, 끝에 도달하면 방향을 바꿔 반대로 이동합니다.
- 장점 : 기아 현상이 없고, 헤드의 이동 경로가 예측 가능합니다.
- 단점 : 양 끝부분보다 가운데 위치한 요청이 더 자주 서비스받는 불평등이 생깁니다.
### 5-2. C-Scan (Circular SCAN)
: SCAN과 비슷하지만, **한쪽 방향으로만 요청을 처리**합니다. 끝에 도달하면 데이터를 처리하지 않고 시작 지점으로 급히 되돌아온 뒤 다시 한 방향으로 이동합니다.
![title](https://user-images.githubusercontent.com/37871541/78578737-e14ce180-786a-11ea-8b53-3e8438b21bf6.png)
- 장점 : 디스크의 모든 구역에 대해 SCAN보다 **균등한 대기 시간** 을 제공합니다.
- 단점 : 시작 지점으로 돌아가는 **공회전** 시간이 발생합니다.

| **알고리즘** | **특징** | **핵심 키워드** |
| --- | --- | --- |
| **FCFS** | 순차적 처리 | 공평성, 저효율 |
| **SSTF** | 최단 거리 우선 | 효율성, 기아 현상 위험 |
| **SCAN** | 양방향 이동 | 엘리베이터, 안정적 |
| **C-SCAN** | 한방향 이동 | 균등한 응답 시간 |

### 6. 저널링 파일 시스템은 왜 필요하며 어떻게 동작하는가?
> 저널링(Journaling) 파일 시스템은 **컴퓨터가 갑자기 꺼지거나 시스템 오류가 발생했을 때, 파일 시스템의 손상을 방지하고 빠르게 복구하기 위해 도입**되었습니다.
### 6-1.  왜 필요한가 ?
- **기존 복구 방식의 한계 극복** : 저널링이 없던 과거의 파일 시스템(ex. ext2)에서는 파일을 저장하는 도중 전원이 나가면, 재부팅 시 파일 시스템 전체 블록을 하나하나 스캔하여 오류를 찾고 복구하는 과정(`fcsk`)을 거쳐야 했습니다. 디스크 용량이 커질수록 이 스캔 과정은 수십 분에서 길게는 수 시간까지 소요됐습니다.
- **데이터 일관성 유지 (Data Consistency)** : 디스크에 새로운 파일을 쓰는 작업은 단일 단계로 끝나지 않습니다. '빈 공간 찾기 -> 아이노드(inode) 생성 -> 실제 데이터 쓰기 -> 메타데이터 업데이트' 등 여러 단계로 나뉘어 실행됩니다. 이 과정 중간에 시스템이 멈추면 데이터와 메타데이터의 정보가 불일치하여 파일 시스템 구조가 붕괴될 수 있는데, 저널링은 이러한 불일치를 방지합니다.
### 6-2. 어떻게 동작하는가 ?
저널링 파일 시스템은 디스크 내에 **저널(Journal)** 또는 **로그(Log)** 라는 별도의 전용 구역을 만들어둡니다. 핵심 동작 원리는 **"실제 저장소에 기록하기 전에, 내가 무엇을 변경할 것인지 미리 메모장(저널)에 적어둔다"** 는 것입니다. 이를 Write-Ahead Logging(WAL)이라고 합니다.
- 기본 동작 4단계
1. **저널 쓰기 :** 파일 변경 사항(메타데이터 및/또는 실제 데이터)을 디스크의 본래 위치에 바로 쓰지 않고, 먼저 저널 영역에 순차적으로 기록합니다.
2. **저널 커밋 :** 저널에 작업 내용이 온전히 다 적혔다는 것을 확인하는 '커밋 블록'을 기록합니다. 이 블록이 쓰여야 해당 작업이 유효한 것으로 인정됩니다.
3. **체크포인팅 :** 이제 저널에 기록된 내용을 바탕으로 디스크의 실제 위치(Main File System)에 데이터를 업데이트합니다.
4. **저널 비우기 :** 실제 파일 시스템에 안전하게 쓰기가 완료되면, 저널 영역에서 해당 기록을 지우고 새로운 작업을 기록할 수 있도록 공간을 확보합니다.
### 6-3. 장애 발생 시 복구 시나리오
저널링의 진가는 시스템이 다운되고 다시 부팅될 때 나타납니다. 운영체제는 부팅 시 전체 디스크를 검사하는 대신, **저널 영역만 확인**합니다.
- 저널을 쓰는 도중 (커밋 전) 다운된 경우: 저널 기록이 불완전하므로 해당 기록을 그냥 무시하고 폐기합니다. 실제 디스크 영역은 아예 건드리지 않았기 때문에 기존 데이터의 일관성이 완벽하게 유지됩니다.

- 저널 커밋 완료 후, 체크포인트(실제 디스크 기록) 도중 다운된 경우: 재부팅 시 저널을 확인해 보면 '완료된 커밋'이 남아 있습니다. 운영체제는 이 저널 내용을 읽어와서 실제 디스크에 쓰려다 만 작업을 **다시 실행**하여 업데이트를 마저 완료합니다.
> 이처럼 전체 디스크를 뒤질 필요 없이 저널에 적힌 **최근 작업 내역**만 확인하고 조치하면 되기 때문에, 시스템 복구가 단 몇 초 만에 완료됩니다.
