⸻

1️⃣ 파일 시스템의 역할

**파일 시스템(File System)**은
디스크 같은 보조기억장치에 데이터를 저장하고 관리하는 운영체제의 구성 요소이다.

즉, 디스크를 사용자가 파일 단위로 사용할 수 있게 추상화한다.

주요 역할

1️⃣ 파일 관리
	•	파일 생성(Create)
	•	파일 삭제(Delete)
	•	파일 읽기/쓰기(Read/Write)
	•	파일 이름 관리

2️⃣ 저장 공간 관리
	•	디스크 블록 할당
	•	빈 공간 관리

3️⃣ 메타데이터 관리
	•	파일 크기
	•	생성 시간
	•	접근 권한

4️⃣ 디렉토리 구조 제공
	•	파일을 계층적으로 관리

5️⃣ 보호 및 접근 제어
	•	사용자 권한 관리
	•	공유 및 잠금

✔ 정리
→ 파일 시스템 = 디스크 공간 + 파일 + 디렉토리 + 메타데이터 관리

⸻

2️⃣ 파일 할당 방법

파일은 디스크에 블록 단위로 저장된다.
이 블록을 어떻게 연결해서 저장할 것인가가 파일 할당 방식이다.

대표적인 3가지:

1️⃣ 연속 할당 (Contiguous Allocation)
2️⃣ 연결 할당 (Linked Allocation)
3️⃣ 인덱스 할당 (Indexed Allocation)

⸻

1️⃣ 연속 할당 (Contiguous Allocation)

파일을 연속된 디스크 블록에 저장

파일 A → [10][11][12][13][14]

장점

✔ 매우 빠른 접근
	•	순차 접근 빠름
	•	랜덤 접근도 빠름

✔ 구현 단순

단점

❌ 외부 단편화 발생

디스크가 이렇게 될 수 있음

[AAA][빈][BBBB][빈][CC]

큰 파일을 넣기 어려움

❌ 파일 크기 증가 어려움

⸻

2️⃣ 연결 할당 (Linked Allocation)

파일 블록을 포인터로 연결

[10] → [45] → [7] → [92]

각 블록에 다음 블록 주소 저장

장점

✔ 외부 단편화 없음

✔ 파일 크기 자유롭게 증가

단점

❌ 랜덤 접근 매우 느림

예

3번째 블록 접근

1 → 2 → 3

❌ 포인터 저장 공간 필요

❌ 포인터 손상 시 파일 손상

대표 구현
→ FAT(File Allocation Table)

⸻

3️⃣ 인덱스 할당 (Indexed Allocation)

인덱스 블록이 모든 블록 주소를 저장

Index block
↓
[10][45][7][92]

장점

✔ 랜덤 접근 가능

✔ 외부 단편화 없음

✔ 파일 확장 쉬움

단점

❌ 인덱스 블록 필요

❌ 작은 파일에서는 공간 낭비

대표 예

→ Unix inode

⸻

비교 정리

방식	장점	단점
연속	접근 속도 빠름	외부 단편화
연결	공간 효율 좋음	랜덤 접근 느림
인덱스	랜덤 접근 가능	인덱스 공간 필요


⸻

3️⃣ inode란 무엇인가

inode = index node

Unix/Linux 파일 시스템에서
파일의 메타데이터를 저장하는 구조체

✔ 파일 자체 데이터는 저장하지 않는다

파일 데이터는 디스크 블록에 있다.

inode는 그 위치 정보를 가진다.

⸻

inode에 저장되는 정보

대표적으로 다음이 들어간다.
	•	파일 타입 (file / directory)
	•	파일 크기
	•	소유자 UID
	•	그룹 GID
	•	접근 권한 (permission)
	•	생성/수정/접근 시간
	•	링크 수
	•	데이터 블록 주소

예

inode
 ├ file size
 ├ owner
 ├ permission
 ├ timestamps
 └ block pointers


⸻

파일 이름은 어디에 있을까?

inode에는 파일 이름이 없다

파일 이름은

➡ 디렉토리

에 저장된다.

directory
파일이름 → inode 번호


⸻

4️⃣ 디렉토리 구현

디렉토리는 실제로 특수한 파일

그 안에는

파일 이름 → inode 번호

가 저장된다.

예

Directory file

fileA → inode 31
fileB → inode 42
fileC → inode 11

파일 접근 과정

파일 열기

1. 디렉토리 검색
2. inode 번호 찾기
3. inode 읽기
4. 데이터 블록 접근


⸻

5️⃣ 디스크 스케줄링 알고리즘

디스크는 헤드 이동 비용이 매우 크다.

그래서 요청 순서를 최적화한다.

⸻

1️⃣ FCFS (First Come First Serve)

요청 순서대로 처리

요청
98 → 183 → 37 → 122

장점

✔ 공정

단점

❌ 헤드 이동 많음

⸻

2️⃣ SSTF (Shortest Seek Time First)

가장 가까운 요청 먼저 처리

예

현재 50

요청
10 40 60 90

→ 40 먼저

장점

✔ 평균 탐색 시간 감소

단점

❌ Starvation 가능

멀리 있는 요청이 계속 밀림

⸻

3️⃣ SCAN (엘리베이터 알고리즘)

헤드가 한 방향으로 이동하면서 처리

← ← ← ← ←
→ → → → →

엘리베이터처럼 움직임

특징

✔ 공정성 향상
✔ starvation 감소

⸻

4️⃣ C-SCAN (Circular SCAN)

한 방향만 서비스

→ → → → →
끝

처음으로 점프

→ → → →

특징

✔ 요청 대기 시간 균일

⸻

정리

알고리즘	특징
FCFS	단순, 이동 많음
SSTF	평균 탐색 시간 최소
SCAN	엘리베이터 방식
C-SCAN	한 방향 순환
