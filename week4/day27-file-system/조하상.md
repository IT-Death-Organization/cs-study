## 1. 파일 시스템의 역할은 무엇인가?

**데이터의 조직화와 추상화**
- 사용자가 보조 기억 장치(HDD, SSD 등)의 물리적인 구조를 몰라도 데이터를 쉽고 안전하게 저장하고 꺼낼 수 있도록 돕는 추상화 계층이다.
- 파일을 생성, 삭제, 수정하는 기능을 제공하며, 파일의 이름, 권한, 생성 날짜 등의 메타데이터를 관리한다. 또한 디스크의 빈 공간을 효율적으로 관리하여 저장 장치의 활용도를 높인다.

---

## 2. 파일 할당 방법(연속, 연결, 인덱스)의 차이와 장단점은?



| 방식 | 특징 | 장점 | 단점 |
| :--- | :--- | :--- | :--- |
| **연속 할당** | 파일을 디스크의 연속된 블록에 저장한다. | 순차 접근이 매우 빠르며 구현이 단순하다. | 외부 단편화가 발생하며, 파일 크기 확장이 어렵다. |
| **연결 할당** | 각 블록이 다음 블록의 포인터를 가진다. | 외부 단편화가 없으며 파일 확장이 용이하다. | 직접 접근(Random Access)이 불가능하고 포인터 유실 시 데이터 손실 위험이 크다. |
| **인덱스 할당** | 모든 포인터를 하나의 인덱스 블록에 모아 관리한다. | 직접 접근이 가능하며 외부 단편화 문제를 해결한다. | 작은 파일의 경우 인덱스 블록 자체가 메모리 낭비가 될 수 있다. |

## 3. inode란 무엇이며 어떤 정보를 저장하는가?

**파일의 메타데이터 저장소**
- 유닉스 계열 파일 시스템에서 각 파일을 나타내는 데이터 구조다. 파일의 이름과 실제 데이터를 제외한 파일에 대한 거의 모든 정보를 담고 있다.
- **저장 정보:** 파일 모드(권한), 소유자(UID/GID), 파일 크기, 생성/수정/접근 시간, 데이터 블록을 가리키는 포인터(Direct/Indirect 등)가 포함된다. 파일 이름은 인덱스 노드(inode)가 아닌 디렉토리 파일 내부에서 관리한다.



---

## 4. 디렉토리는 어떻게 구현되는가?

**파일 이름과 inode의 매핑**
- 운영체제 입장에서 디렉토리는 단순히 **"파일명과 그에 대응하는 inode 번호의 리스트"**를 담고 있는 특별한 종류의 파일이다.
- 가장 단순하게는 선형 리스트로 구현할 수 있으나, 탐색 성능을 높이기 위해 해시 테이블(Hash Table)을 사용하여 특정 파일 이름을 빠르게 조회하도록 구현하기도 한다. 사용자가 파일명을 호출하면 시스템은 디렉토리를 뒤져 해당 파일의 inode 번호를 찾아내고 실제 데이터에 접근한다.

## 5. 디스크 스케줄링 알고리즘(FCFS, SSTF, SCAN, C-SCAN)은?



- **FCFS (First-Come, First-Served):** 요청이 들어온 순서대로 처리한다. 공평하지만 디스크 헤드의 이동 거리가 길어져 비효율적일 수 있다.
- **SSTF (Shortest Seek Time First):** 현재 헤드 위치에서 가장 가까운 요청을 먼저 처리한다. 탐색 시간을 최소화하지만, 멀리 떨어진 요청은 계속 밀려나는 기아(Starvation) 현상이 발생할 수 있다.
- **SCAN:** 엘리베이터 알고리즘이라고도 하며, 헤드가 디스크의 한쪽 끝에서 반대쪽 끝까지 이동하며 경로상에 있는 모든 요청을 처리한다.
- **C-SCAN (Circular SCAN):** SCAN과 유사하지만, 반대쪽 끝에 도달하면 요청을 처리하지 않고 즉시 시작 지점으로 돌아와 다시 한 방향으로만 서비스를 제공한다. 대기 시간의 균등성을 보장한다.

## 6. 저널링 파일 시스템은 왜 필요하며 어떻게 동작하는가?

**데이터 무결성과 빠른 복구**
- 시스템이 갑자기 꺼졌을 때(Crash) 파일 시스템의 일관성이 깨지는 것을 방지하기 위해 사용한다.
- **동작 방식:** 실제 데이터를 쓰기 전에 변경 사항에 대한 로그(저널)를 먼저 기록한다. 만약 작업 도중 시스템이 멈추면, 재부팅 시 전체 디스크를 검사하는 대신 저널 로그만 확인하여 완료되지 않은 작업을 취소하거나(Rollback) 다시 실행(Redo)하여 빠르게 일관성을 복구한다.
