### 📌 [Q1] 파일 시스템의 역할은 무엇인가?

파일 시스템은 운영체제가 저장 매체(하드 디스크, SSD 등)에 데이터(파일)를 어떻게 저장, 접근, 관리할지를 정의하는 논리적인 구조와 규칙의 집합입니다.
- **주요 역할**: 
  - 파일의 저장 및 계층적 조직화(디렉토리)
  - 저장 공간 할당/회수 관리
  - 논리적 이름을 물리적 위치로 변환하는 주소 매핑
  - 접근 제어를 통한 파일 보호 및 보안
  - 시스템 오류 시의 데이터 무결성 유지

### 📌 [Q2] 파일 할당 방법(연속, 연결, 인덱스)의 차이와 장단점은?

- **연속 할당 (Contiguous Allocation)**: 파일을 디스크 상의 연속된 블록에 공간으로 통째로 저장합니다. 
  - *장점*: 순차 및 직접 접근 속도가 매우 빠릅니다 (디스크 헤드 탐색 시간 최소화). 
  - *단점*: 외부 단편화가 발생하며 파일 크기 확장이 어렵습니다.
- **연결 할당 (Linked Allocation)**: 파일을 빈 블록에 흩어 저장하고 다음 블록 주소로 연결합니다 (예: FAT). 
  - *장점*: 외부 단편화가 없고 파일 크기 확장이 자유롭습니다. 
  - *단점*: 직접 접근(Random Access)이 매우 느리며, 블록 하나가 손상되면 그 이후 데이터를 모두 잃을 수 있습니다.
- **인덱스 할당 (Indexed Allocation)**: 파일마다 모든 데이터 블록 주소를 모아둔 인덱스 블록을 하나씩 두는 방식입니다 (예: UNIX inode).
  - *장점*: 외부 단편화가 없으며, 직접 접근이 빠릅니다.
  - *단점*: 파일이 아주 작아도 인덱스 블록을 위한 추가 공간이 필요해 공간 낭비가 발생할 수 있습니다.

### 📌 [Q3] inode란 무엇이며 어떤 정보를 저장하는가?

- **inode (Index Node)**는 유닉스 계열 파일 시스템에서 파일 또는 디렉토리의 **메타데이터와 정보(포인터)**를 담고 있는 고정된 크기의 핵심 자료구조입니다.
- **저장하는 정보**: 
  - 파일 소유자/그룹
  - 파일 종류 및 접근 권한(rwx)
  - 파일 크기
  - 타임스탬프(수정/접근/생성 시간)
  - 하드 링크 수
  - **실제 데이터 블록을 가리키는 위치 포인터 배열** (단, 파일 이름은 inode가 아닌 디렉토리 영역에 저장됩니다)

### 📌 [Q4] 디렉토리는 어떻게 구현되는가?

- 디렉토리 또한 하나의 특별한 '파일'로 취급되며, 해당 파일 안에는 속한 **하위 파일/디렉토리들의 이름과 각 항목의 정보(inode 번호 등)를 매핑한 엔트리(Entry) 목록**으로 구성되어 있습니다.
- 정보 검색 효율화 및 관리를 위해 구현 방식에는 **선형 리스트(Linear List)**, **해시 테이블(Hash Table)**, 그리고 대규모 파일 검색에 가장 유리한 **트리 구조(주로 B-Tree 구조 등)** 방식이 사용됩니다.

### 📌 [Q5] 디스크 스케줄링 알고리즘(FCFS, SSTF, SCAN, C-SCAN)은?

디스크 헤드의 이동 경로를 최적화하여 탐색 시간을 줄이기 위해 디스크 I/O 요청 대기열을 처리하는 알고리즘들입니다.
- **FCFS**: 큐에 들어온 순서대로 처리. 디스크 이동 거리가 길어 비효율적일 수 있습니다.
- **SSTF**: 헤드의 현재 위치에서 가장 가까운 트랙의 요청부터 처리. 전체 탐색시간은 줄지만 먼 위치의 요청이 계속 밀리는 **기아 현상(Starvation)**이 발생할 수 있습니다.
- **SCAN**: 엘리베이터처럼 한쪽 방향 끝까지 이동하며 요청을 처리하고, 끝에 도달하면 꺾어서 반대 방향으로 처리하며 이동합니다.
- **C-SCAN**: 한쪽 방향으로만 요청을 처리. 끝 트랙까지 도달하면 방향을 바꿔 복귀하되, 되돌아올 때는 처리 없이 처음 위치로 헤드만 빠르게 점프하여 다시 단방향으로 순회합니다. 요청 처리가 보다 균등합니다.

### 📌 [Q6] 저널링 파일 시스템은 왜 필요하며 어떻게 동작하는가?

- 파일 시스템이 작동 중 강제 전원 종료나 시스템 크래시가 발생했을 때 **데이터 구조의 붕괴(일관성 문제)를 막고 복구 시간을 획기적으로 줄이기 위해** 사용됩니다.
- **동작 방식**: 디스크에 실제 데이터/구조를 반영하기 전, 어떤 작업을 할 것인지 **로그 영역(저널)에 먼저 기록**합니다. 기록(Commit)이 완료되면 원본 파일 영역을 실제로 갱신(Checkpoint)합니다. 사고가 발생하면 전체 디스크를 스캔하는 대신, 저널 기록만 짧게 확인해 트랜잭션 미완료 처리를 취소하거나 재수행(Redo)하여 빠르게 일관된 상태로 돌아갈 수 있습니다.
