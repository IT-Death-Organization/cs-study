운영체제(OS)에서 **파일 시스템(File System)** 은 저장 장치(HDD/SSD)의 데이터를 **파일과 디렉토리 구조로 관리하는 계층**입니다. 아래 질문들을 **운영체제 관점에서 구조적으로** 정리합니다.

# 1. 파일 시스템의 역할

파일 시스템은 **디스크에 저장된 데이터를 체계적으로 관리하는 시스템**입니다.

핵심 역할은 다음과 같습니다.

### 1️⃣ 파일 관리 (File abstraction)

사용자는 디스크의 물리적 블록을 직접 다루지 않습니다.

대신 다음과 같은 **추상화**를 사용합니다.

```
/home/user/report.txt
```

파일 시스템은 이 경로를 **실제 디스크 블록 위치로 매핑**합니다.

즉

```
파일 이름 → inode → 데이터 블록
```

---

### 2️⃣ 디스크 공간 관리

디스크는 **block 단위**로 관리됩니다.

예

```
1 block = 4KB
```

파일 시스템은

-   어떤 block이 사용 중인지
-   어떤 block이 비어있는지

관리합니다.

대표 방법

-   bitmap
-   free list

---

### 3️⃣ 파일 접근 제공

파일 시스템은 다음 API를 제공합니다.

대표 system call

```
open()
read()
write()
close()
lseek()
```

---

### 4️⃣ 메타데이터 관리

파일 자체 데이터 외에도 다음 정보를 관리합니다.

예

-   파일 크기
-   생성 시간
-   권한
-   소유자

이 정보는 보통 **inode**에 저장됩니다.

---

# 2. 파일 할당 방법

파일이 디스크에 저장되는 방법입니다.

대표적으로 3가지가 있습니다.

```
1. Contiguous Allocation
2. Linked Allocation
3. Indexed Allocation
```

---

# 2-1 Contiguous Allocation (연속 할당)

파일을 **연속된 블록에 저장**합니다.

예

```
파일 A
block 10
block 11
block 12
block 13
```

### 장점

1️⃣ 구현이 매우 단순
2️⃣ 순차 접근 성능이 매우 좋음
3️⃣ random access도 빠름

왜냐하면

```
block = start + offset
```

으로 계산 가능하기 때문입니다.

---

### 단점

**외부 단편화 (External Fragmentation)**

예

```
[파일A][빈공간][파일B][빈공간][파일C]
```

큰 파일을 저장할 **연속 공간이 없을 수 있음**

또한

파일 크기가 증가하면 문제가 발생합니다.

---

# 2-2 Linked Allocation (연결 할당)

파일을 **연결 리스트 형태**로 저장합니다.

```
block 5 → block 20 → block 3 → block 90
```

각 block이 다음 block을 가리킵니다.

---

### 장점

1️⃣ 외부 단편화 없음
2️⃣ 파일 크기 확장 쉬움

---

### 단점

1️⃣ random access 불가능

예

```
100번째 block
```

찾으려면

```
처음부터 100번 따라가야함
```

즉

```
O(n)
```

---

2️⃣ 포인터 공간 낭비

각 block에 **next pointer** 필요

---

# 2-3 Indexed Allocation (인덱스 할당)

파일마다 **index block**을 하나 둡니다.

```
index block

[5][20][3][90]
```

각 항목이 데이터 block을 가리킵니다.

---

### 장점

1️⃣ random access 가능

```
index[10]
```

바로 접근 가능

---

2️⃣ 외부 단편화 없음

---

### 단점

1️⃣ index block 추가 공간 필요
2️⃣ 매우 큰 파일이면 index block이 부족

그래서 **multi-level index** 사용합니다.

예

```
Unix inode
```

---

# 3. inode란 무엇인가

**inode (index node)** 는 파일의 **메타데이터를 저장하는 구조체**입니다.

Linux/Unix 파일 시스템의 핵심입니다.

---

### inode가 저장하는 정보

대표적으로

```
파일 타입
파일 크기
파일 권한
소유자
생성 시간
수정 시간
데이터 블록 포인터
link count
```

---

중요한 점

**파일 이름은 inode에 저장되지 않습니다**

구조는 다음과 같습니다.

```
directory entry
   ↓
filename → inode number
              ↓
           inode
              ↓
       data block
```

즉

```
파일 이름은 directory가 관리
```

---

# 4. 디렉토리는 어떻게 구현되는가

디렉토리는 사실 **특수한 파일**입니다.

내용은 다음과 같습니다.

```
filename → inode number
```

예

```
home directory

report.txt → inode 1001
photo.jpg → inode 2003
notes.md → inode 1040
```

---

따라서 파일 열 때 과정은

```
/home/user/report.txt
```

OS 동작

```
1. root directory 찾기
2. home directory inode
3. user directory inode
4. report.txt inode
5. data block
```

---

디렉토리 구현 방식

대표적으로

### 1️⃣ Linear list

```
filename → inode
```

단순한 리스트

단점

```
검색 O(n)
```

---

### 2️⃣ Hash table

```
hash(filename)
```

빠른 lookup

---

### 3️⃣ B-tree

대규모 파일 시스템에서 사용

예

-   NTFS
-   ext4

---

# 5. 디스크 스케줄링 알고리즘

디스크는 **seek time**이 매우 중요합니다.

헤드 이동 시간이 성능을 좌우합니다.

그래서 요청을 **효율적으로 처리하는 알고리즘**이 필요합니다.

---

# 5-1 FCFS (First Come First Serve)

요청 순서대로 처리합니다.

예

```
queue

98 → 183 → 37 → 122
```

---

### 장점

공정함

---

### 단점

헤드 이동이 매우 많음

예

```
98 → 183 → 37
```

왔다갔다

---

# 5-2 SSTF (Shortest Seek Time First)

현재 헤드에서 **가장 가까운 요청** 선택

예

```
현재 50

요청
10 55 70 90

→ 55 선택
```

---

### 장점

평균 seek time 감소

---

### 단점

**starvation 가능**

멀리 있는 요청이 계속 밀릴 수 있음

---

# 5-3 SCAN (엘리베이터 알고리즘)

헤드가 **한 방향으로 이동하면서 처리**

```
→ → → → → →
```

끝까지 가면 방향 변경

---

예

```
0 ----- 199
     ↑
```

오른쪽으로 가며 처리

---

### 장점

공정성 증가

---

# 5-4 C-SCAN (Circular SCAN)

SCAN의 변형입니다.

헤드 이동 방식

```
→ → → → → →
끝
↓
처음으로 점프
```

다시 오른쪽으로 이동

---

### 장점

요청 대기 시간 균등

---

# 6. 저널링 파일 시스템

대표 예

-   ext3
-   ext4
-   NTFS

---

## 왜 필요한가

문제

```
write 중 전원 꺼짐
```

예

```
inode 수정 완료
data block 미완료
```

파일 시스템이 **깨질 수 있음**

이를 **inconsistency**라고 합니다.

---

# 저널링 동작 과정

파일 시스템 변경 전

**journal (log)** 에 기록합니다.

순서

```
1. 변경 내용을 journal에 기록
2. journal commit
3. 실제 파일 시스템 변경
4. journal clear
```

---

### 크래시 발생 시

OS 재부팅 후

journal 확인

```
commit 되었으면 → 재적용
commit 안됐으면 → rollback
```

그래서 **빠르게 복구 가능**
