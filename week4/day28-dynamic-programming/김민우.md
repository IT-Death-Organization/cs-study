# 1. 동적 프로그래밍(Dynamic Programming, DP)이란?

동적 프로그래밍은 **복잡한 문제를 작은 부분 문제(subproblem)로 나눈 뒤**,
그 **부분 문제의 답을 저장해 재사용**하면서 전체 문제를 해결하는 기법이다.

핵심은:

* 큰 문제를 작은 문제로 나눌 수 있고
* 작은 문제들이 **반복해서 등장**하며
* 작은 문제의 답을 이용해 큰 문제의 답을 만들 수 있을 때

중복 계산을 줄여서 효율적으로 푸는 방식이다.

즉, DP는 한마디로 말하면:

> **“같은 계산을 여러 번 하지 않기 위해, 한 번 구한 부분 문제의 답을 저장해 두고 재사용하는 방법”**

이다.

---

## 왜 필요한가?

어떤 문제는 재귀적으로 풀 수는 있는데,
그대로 풀면 같은 부분 문제를 계속 반복해서 계산해서 너무 느려진다.

대표 예: 피보나치 수열

[
F(n) = F(n-1) + F(n-2)
]

재귀로 그냥 풀면:

* `F(5)`를 구할 때 `F(3)`이 여러 번 계산되고
* `F(4)` 안에서도 `F(3)`, `F(2)`가 또 반복 계산됨

그래서 시간복잡도가 지수적으로 커진다.

하지만 `F(3)`을 한 번만 계산해서 저장해두면,
이후에는 꺼내서 쓰기만 하면 되므로 훨씬 빨라진다.

---

# 2. 어떤 문제에 적용 가능한가?

DP는 아무 문제에나 쓰는 게 아니라, 보통 다음 같은 유형에 적용된다.

## 대표 적용 분야

### 1) 경우의 수 세기

* 계단 오르기
* 격자 경로 수
* 동전으로 금액 만들기 경우의 수

### 2) 최적화 문제

* 최소 비용
* 최대 이익
* 최장 증가 부분 수열(LIS)
* 배낭 문제(Knapsack)
* 행렬 곱셈 순서 최적화

### 3) 문자열 문제

* 최장 공통 부분 수열(LCS)
* 편집 거리(Edit Distance)
* 팰린드롬 분할

### 4) 그래프/경로 문제

* 최단 경로의 일부 유형
* DAG 위의 DP
* 트리 DP

### 5) 게임/상태 전이 문제

* 특정 상태에서 이길 수 있는지
* 비트마스크 DP
* 상태 압축 DP

# 3. DP의 두 가지 조건

DP가 성립하려면 보통 다음 두 조건이 중요하다.

---

## 1) 최적 부분 구조(Optimal Substructure)

큰 문제의 최적해가 **작은 부분 문제의 최적해**로부터 만들어질 수 있는 성질이다.

즉,

> 전체 문제의 가장 좋은 답이, 부분 문제들의 가장 좋은 답을 조합해서 구성될 수 있어야 한다.

### 예시: 계단 오르기

한 번에 1칸 또는 2칸 올라갈 수 있다고 하자.

`dp[n] = n번째 칸까지 가는 방법 수`

그러면 마지막 이동만 생각하면:

* `n-1`에서 1칸 올라왔거나
* `n-2`에서 2칸 올라왔음

따라서

[
dp[n] = dp[n-1] + dp[n-2]
]

즉, `n`까지 가는 최적/정답은 더 작은 문제들의 답으로 구성된다.

---

### 예시: 최소 비용 경로

어떤 칸 `(i,j)`까지 가는 최소 비용은 보통

* 위에서 오는 최소 비용
* 왼쪽에서 오는 최소 비용

중 작은 값에 현재 칸 비용을 더한 것

즉,

[
dp[i][j] = \min(dp[i-1][j], dp[i][j-1]) + cost[i][j]
]

여기서도 큰 문제의 최소 비용이 작은 문제의 최소 비용을 기반으로 결정된다.

---

## 2) 중복 부분 문제(Overlapping Subproblems)

큰 문제를 풀다 보면 **같은 부분 문제가 여러 번 반복해서 등장하는 성질**이다.

### 예시: 피보나치

`F(5)`를 재귀로 계산하면

* `F(4)`와 `F(3)`을 구하고
* `F(4)` 안에서 또 `F(3)`, `F(2)`를 구하고
* `F(3)`이 여러 번 반복된다

즉, 같은 부분 문제를 매번 다시 푸는 낭비가 생긴다.

DP는 이걸 저장해서 해결한다.

---

## 두 조건의 의미를 같이 보면

* **최적 부분 구조**: 작은 답으로 큰 답을 만들 수 있어야 함
* **중복 부분 문제**: 같은 작은 문제를 여러 번 계산하게 됨

둘 다 있어야 DP가 특히 강력하다.

---

# 4. Memoization(Top-down)과 Tabulation(Bottom-up)의 차이

DP 구현 방식은 크게 두 가지다.

---

## 1) Memoization = Top-down

재귀적으로 문제를 내려가며 풀되,
한 번 계산한 결과를 **메모(배열, map)** 해두고 다시 쓰는 방식이다.

### 구조

* 큰 문제부터 시작
* 필요한 작은 문제만 재귀적으로 호출
* 이미 계산한 값이면 바로 반환

### 피보나치 예시

```python
memo = [-1] * (n + 1)

def fib(n):
    if n <= 1:
        return n
    if memo[n] != -1:
        return memo[n]
    memo[n] = fib(n-1) + fib(n-2)
    return memo[n]
```

---

## 2) Tabulation = Bottom-up

가장 작은 문제부터 차례대로 테이블을 채우며
큰 문제의 답까지 올라가는 방식이다.

### 구조

* 기저 상태부터 시작
* 점화식 순서대로 반복문으로 계산
* 최종적으로 원하는 답에 도달

### 피보나치 예시

```python
dp = [0] * (n + 1)
dp[1] = 1

for i in range(2, n + 1):
    dp[i] = dp[i-1] + dp[i-2]
```

---

# 5. 각 방식의 장단점

---

## Memoization(Top-down)의 장점

### 1) 점화식을 그대로 코드로 옮기기 쉽다

재귀식과 거의 같은 형태라서 사고 흐름이 자연스럽다.

### 2) 필요한 상태만 계산한다

전체 상태를 다 채우지 않고, 실제로 필요한 것만 계산할 수 있다.

### 3) 문제 구조를 이해하기 쉽다

특히 처음 점화식을 세울 때 더 직관적이다.

---

## Memoization(Top-down)의 단점

### 1) 재귀 호출 오버헤드가 있다

함수 호출이 반복되므로 반복문보다 느릴 수 있다.

### 2) 스택 오버플로우 위험

입력이 크면 재귀 깊이가 너무 깊어질 수 있다.

### 3) 구현 실수 가능성

방문 여부, 초기값, 캐시 처리 등을 조심해야 한다.

---

## Tabulation(Bottom-up)의 장점

### 1) 반복문 기반이라 보통 더 안정적이다

재귀 스택 문제 없음.

### 2) 성능이 좋은 경우가 많다

호출 오버헤드가 적다.

### 3) 공간 최적화가 쉬운 경우가 많다

예를 들어 피보나치는 이전 두 값만 필요하므로 O(1) 공간으로 줄일 수 있다.

---

## Tabulation(Bottom-up)의 단점

### 1) 계산 순서를 직접 설계해야 한다

어떤 순서로 테이블을 채워야 하는지 잘못 잡으면 틀린다.

### 2) 필요 없는 상태까지 다 채울 수 있다

전체 테이블을 다 채우는 과정이 낭비일 수 있다.

### 3) 점화식은 쉬워도 구현이 직관적이지 않을 수 있다

특히 상태가 복잡하면 순서 설계가 어렵다.

---

# 6. 어떤 경우에 어떤 방식을 선택하는가?

## Top-down이 좋은 경우

* 점화식을 세우는 게 먼저 중요할 때
* 상태 전이가 복잡할 때
* 실제로 일부 상태만 필요한 문제일 때
* 처음 아이디어 검증할 때

## Bottom-up이 좋은 경우

* 입력 크기가 커서 재귀가 위험할 때
* 시간/공간 최적화가 중요할 때
* 상태 순서가 명확할 때
* 실전 코딩테스트에서 안정적으로 구현하고 싶을 때

---

## 실전 감각으로 말하면

* **처음 사고할 때는 Top-down이 편한 경우가 많고**
* **최종 제출 코드는 Bottom-up이 더 안정적인 경우가 많다**

하지만 문제마다 다르다.
트리 DP, DFS 기반 DP는 Top-down 느낌이 자연스러운 경우도 많다.

---

# 7. 분할 정복과 DP의 차이

둘 다 큰 문제를 작은 문제로 나눈다는 점은 비슷하다.
하지만 핵심 차이는 **부분 문제가 서로 겹치느냐**다.

---

## 분할 정복(Divide and Conquer)

큰 문제를 독립적인 작은 문제로 나눈 뒤,
각각 풀어서 합친다.

### 예

* 병합 정렬
* 퀵 정렬
* 이진 탐색

여기서는 보통 부분 문제가 **서로 겹치지 않는다**.

예를 들어 병합 정렬은:

* 왼쪽 절반 정렬
* 오른쪽 절반 정렬

이 둘은 서로 다른 부분 배열이라 중복 계산이 거의 없다.

---

## DP

작은 문제로 나누는데,
그 작은 문제들이 **서로 중복된다**.

예를 들어 피보나치는 `fib(3)` 같은 문제가 여러 번 등장한다.
그래서 저장해두는 것이 중요하다.

---

## 차이 요약

### 분할 정복

* 부분 문제가 독립적
* 중복이 적음
* 저장 없이도 효율적일 수 있음

### DP

* 부분 문제가 중복됨
* 저장이 핵심
* 최적 부분 구조 필요

---

## 언제 DP를 써야 하는가?

재귀적 분해는 되는데,
**같은 부분 문제를 반복 계산하고 있다면** DP를 고려해야 한다.

즉,

> “재귀는 되는데 너무 느리다”
> → 같은 부분 문제가 반복되는지 확인
> → 반복되면 DP 후보

---

# 8. DP 문제를 풀 때 점화식을 세우는 과정

이게 제일 중요하다.
많은 학생이 DP를 외워서 풀려고 하는데, 사실 핵심은 **상태 정의**와 **점화식 설계**다.

DP는 보통 다음 순서로 푼다.

---

## 1단계: 상태(state)를 정의한다

가장 먼저 해야 할 일은

> `dp[x]`가 무엇을 의미하는지

를 정확히 정하는 것이다.

예:

* `dp[i]` = i번째까지 고려했을 때의 최댓값
* `dp[i]` = i에 도달하는 방법 수
* `dp[i][j]` = i번째 물건까지 고려하고, 무게 한도가 j일 때 최대 가치
* `dp[i][j]` = 문자열 A의 앞 i개, B의 앞 j개를 고려한 최적값

상태 정의가 애매하면 점화식도 흔들린다.

---

## 2단계: 마지막 선택 또는 현재 상태가 어디서 왔는지 본다

많은 DP는 **“마지막에 무슨 일이 일어났는가?”**를 보면 점화식이 나온다.

예: 계단 오르기
`n`번째 칸에 도착하려면:

* `n-1`에서 1칸 왔거나
* `n-2`에서 2칸 왔음

그래서
[
dp[n] = dp[n-1] + dp[n-2]
]

예: 배낭 문제
i번째 물건을 넣는지/안 넣는지로 나뉜다.

* 안 넣으면 `dp[i-1][w]`
* 넣으면 `dp[i-1][w-weight[i]] + value[i]`

그래서
[
dp[i][w] = \max(dp[i-1][w],\ dp[i-1][w-w_i] + v_i)
]

---

## 3단계: 점화식을 만든다

상태 전이를 식으로 적는다.

이때 보통:

* 경우를 나누거나
* 마지막 행동을 기준으로 나누거나
* 이전 상태에서 현재 상태로 어떻게 오는지 쓴다

---

## 4단계: 기저 조건(base case)을 정한다

재귀든 반복문이든 시작점이 필요하다.

예:

* `dp[0] = 0`
* `dp[1] = 1`
* 빈 문자열끼리의 편집 거리는 0
* 물건을 하나도 안 고르면 최대 가치는 0

기저 조건이 잘못되면 전체가 틀린다.

---

## 5단계: 계산 순서를 정한다

Bottom-up이면 특히 중요하다.

점화식에서 `dp[i]`가 `dp[i-1]`, `dp[i-2]`에 의존한다면
작은 인덱스부터 채워야 한다.

2차원 DP면:

* 행 우선
* 열 우선
* 대각선 순서

등 문제에 맞는 순서를 잡아야 한다.

---

## 6단계: 중복 계산/공간 최적화 여부를 본다

예를 들어 현재 상태가 이전 한두 줄만 필요하면 전체 테이블을 둘 필요가 없다.

---

# 9. 점화식을 세우는 사고법

DP 점화식이 막힐 때는 아래처럼 생각하면 된다.

---

## 방법 1: “dp[i]를 무엇으로 정의할까?”

가장 먼저 뜻을 분명히 한다.

예:

* `dp[i]` = i번째까지의 최적값
* `dp[i]` = i를 만드는 방법 수

---

## 방법 2: “마지막 선택은 무엇인가?”

많은 문제는 마지막 행동 기준으로 분기된다.

예:

* 마지막 계단을 1칸 올라왔나 2칸 올라왔나?
* 마지막 문자를 선택했나 안 했나?
* 현재 물건을 넣었나 안 넣었나?

---

## 방법 3: “현재 상태는 어떤 이전 상태들에서 올 수 있는가?”

그래프처럼 생각하면 쉽다.

예:

* `(i,j)`는 `(i-1,j)`와 `(i,j-1)`에서 온다
* `i`는 `i-1`, `i-2`에서 온다

---

## 방법 4: “최대/최소/합/가능 여부 중 무엇을 구하는가?”

연산 형태가 달라진다.

* 경우의 수 → 보통 `+`
* 최댓값 → `max`
* 최솟값 → `min`
* 가능 여부 → `or`, `bool`

---

# 10. 예시로 보는 DP 설계

---

## 예제 1: 피보나치

### 상태

`dp[i] = i번째 피보나치 수`

### 점화식

[
dp[i] = dp[i-1] + dp[i-2]
]

### 기저 조건

[
dp[0]=0,\ dp[1]=1
]

---

## 예제 2: 1, 2칸 계단 오르기

### 상태

`dp[i] = i번째 칸까지 가는 방법 수`

### 점화식

[
dp[i] = dp[i-1] + dp[i-2]
]

### 이유

마지막 이동이 1칸 또는 2칸뿐이기 때문

---

## 예제 3: 동전으로 금액 만들기(최소 개수)

### 상태

`dp[x] = 금액 x를 만드는 데 필요한 최소 동전 개수`

### 점화식

각 동전 `c`에 대해
[
dp[x] = \min(dp[x], dp[x-c] + 1)
]

### 기저 조건

[
dp[0]=0
]

---

## 예제 4: 배낭 문제

### 상태

`dp[i][w] = i번째 물건까지 고려했을 때, 가방 용량 w에서의 최대 가치`

### 점화식

* 물건 i를 안 넣는 경우
* 물건 i를 넣는 경우

[
dp[i][w] = \max(dp[i-1][w],\ dp[i-1][w-w_i]+v_i)
]

---

# 11. DP를 사용할 때 흔한 실수

## 1) 상태 정의가 애매함

`dp[i]`가 정확히 뭔지 불분명하면 점화식이 꼬인다.

## 2) 기저 조건 누락

특히 `dp[0]`, 시작 위치, 빈 문자열 처리 등

## 3) 계산 순서 오류

Bottom-up에서 아직 안 구한 값을 참조하면 틀린다.

## 4) 중복 문제 없는 데 억지로 DP 사용

그냥 그리디나 분할정복이 더 맞는 경우도 많다.

## 5) 상태 수가 너무 커서 메모리 초과

필요하면 상태 압축, 공간 최적화 필요

---

# 12. 한 번에 정리

## 동적 프로그래밍이란?

큰 문제를 작은 문제로 나누고,
작은 문제의 답을 저장해서 재사용하며 푸는 방법

## 적용 가능한 문제

* 경우의 수
* 최적화
* 문자열
* 경로
* 상태 전이 문제

## 필수 조건

* 최적 부분 구조
* 중복 부분 문제

## 구현 방식

* Memoization(Top-down): 재귀 + 저장
* Tabulation(Bottom-up): 반복문 + 테이블 채우기

## 분할 정복과 차이

* 분할 정복: 부분 문제가 독립적
* DP: 부분 문제가 중복됨

## 점화식 세우는 방법

1. 상태 정의
2. 마지막 선택/이전 상태 분석
3. 점화식 작성
4. 기저 조건 설정
5. 계산 순서 결정
