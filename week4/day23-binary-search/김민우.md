# 1. 이진 탐색의 전제 조건은 무엇이며 왜 필요한가?

## 전제 조건

이진 탐색(Binary Search)의 핵심 전제는 **탐색 대상이 정렬되어 있어야 한다**는 점이야.

보통 오름차순 정렬 기준으로 설명하면,

* `arr[mid] < target` 이면
  → `mid` 왼쪽은 전부 target보다 작다고 확신할 수 있어야 함
* `arr[mid] > target` 이면
  → `mid` 오른쪽은 전부 target보다 크다고 확신할 수 있어야 함

이 확신이 가능해야 절반을 버릴 수 있다.

## 왜 정렬이 필요할까?

이진 탐색은 매 단계마다 배열의 절반을 날리는 방식이야.
그런데 절반을 버리려면, 현재 중간값과 target을 비교했을 때 **어느 쪽에 target이 있을지 논리적으로 결정**할 수 있어야 해.

예를 들어:

```text
[1, 3, 5, 7, 9]
target = 7
mid = 5
```

`5 < 7` 이니까 왼쪽 절반은 볼 필요가 없고 오른쪽만 보면 된다.

하지만 정렬이 안 되어 있으면:

```text
[8, 1, 7, 3, 5]
mid = 7
```

`7`과 target을 비교해도 왼쪽/오른쪽 중 어디를 버려야 할지 알 수 없다.
즉, **절반 제거의 근거가 사라진다.**

## 본질

이진 탐색은 단순히 “가운데를 보는 알고리즘”이 아니라,

**비교 결과로 탐색 구간을 절반씩 줄일 수 있는 단조성(monotonicity)** 이 있어야 동작하는 알고리즘이야.

그래서 배열 자체가 정렬되어 있거나,
혹은 “어떤 조건을 만족하는 구간이 한쪽으로 몰려 있는 구조”가 있어야 한다.

예:

* 정렬 배열에서 값 찾기
* `x 이하 개수`
* 특정 조건을 만족하는 최소값 찾기
* Parametric Search

# 2. 이진 탐색이 O(log n)인 이유를 수학적으로 설명하면?

## 직관

매 단계마다 탐색 범위를 절반으로 줄인다.

처음 크기: `n`
1번 후: `n/2`
2번 후: `n/4`
3번 후: `n/8`

k번 후에는:

```text
n / 2^k
```

탐색 범위가 1이 될 때 종료되므로,

```text
n / 2^k = 1
```

양변에 `2^k`를 곱하면:

```text
n = 2^k
```

로그를 취하면:

```text
k = log2(n)
```

따라서 시간 복잡도는:

```text
O(log n)
```

## 조금 더 엄밀하게

정확히는 탐색 구간 길이가 1 이하가 될 때까지 반복하므로,

```text
n / 2^k <= 1
```

즉,

```text
2^k >= n
```

따라서,

```text
k >= log2(n)
```

결국 필요한 반복 횟수는 대략 `⌈log2 n⌉` 수준이 된다.

## 예시

* 원소 8개 → 최대 3번
* 원소 16개 → 최대 4번
* 원소 1,024개 → 최대 10번
* 원소 1,000,000개 → 약 20번

즉, 데이터가 엄청 커져도 반복 횟수는 조금만 증가한다.

## 재귀식으로 봐도 됨

이진 탐색의 시간 복잡도는 다음처럼 쓸 수 있어:

```text
T(n) = T(n/2) + O(1)
```

한 번 비교하고 절반만 남기니까.

이를 계속 풀면:

```text
T(n) = T(n/4) + O(1) + O(1)
     = T(n/8) + O(1) + O(1) + O(1)
     ...
```

총 `log n`번 반복되므로:

```text
T(n) = O(log n)
```

# 3. 이진 탐색 구현 시 mid 계산을 `(low + high) / 2` 대신 `low + (high - low) / 2`로 하는 이유는?

## 이유: 오버플로우 방지

겉보기엔 둘이 같아 보여.

```java
mid = (low + high) / 2;
mid = low + (high - low) / 2;
```

수학적으로는 동일한 값이야.

하지만 컴퓨터에서는 `low + high`가 **int 범위를 초과할 수 있음**.

예를 들어 Java `int` 최대값은 약 21억(`2,147,483,647`)인데,

```java
low = 2_000_000_000
high = 2_100_000_000
```

이면

```java
low + high
```

는 int 범위를 넘어서 음수가 되거나 이상한 값이 된다.

그러면 `mid`가 잘못 계산되고,

* 인덱스 에러
* 무한 루프
* 잘못된 결과

가 발생할 수 있다.

## 왜 `low + (high - low) / 2`는 안전한가?

`high - low`는 현재 탐색 구간 길이이므로 일반적으로 범위를 넘기지 않는다.

그리고

```java
low + (high - low) / 2
```

는 “low에서 high까지의 중간만큼 이동”하는 구조라서,
직접 큰 수 두 개를 더하지 않아도 된다.

## 예시

```java
int low = 2_000_000_000;
int high = 2_100_000_000;

// 위험
int mid1 = (low + high) / 2;

// 안전
int mid2 = low + (high - low) / 2;
```

## 정리

면접에서 말할 포인트는 이거야:

> `(low + high) / 2`는 두 값의 합이 정수 범위를 넘을 수 있어서 오버플로우 위험이 있다.
> `low + (high - low) / 2`는 같은 중간값을 계산하면서도 합 연산으로 인한 overflow를 피할 수 있다.

# 4. Lower Bound와 Upper Bound의 차이는? 각각 언제 사용하는가?

이 부분은 정말 중요하다.
많이 헷갈리는 이유는 “찾는 값”이 아니라 **조건을 만족하는 첫 위치**를 찾는 개념으로 이해해야 하기 때문이야.

---

## Lower Bound

### 정의

**target 이상(`>= target`)이 처음 나타나는 위치**

즉,

```text
처음으로 arr[i] >= target 을 만족하는 i
```

### 예시

```text
arr = [1, 2, 2, 2, 4, 5]
target = 2
```

Lower Bound는 인덱스 `1`

왜냐하면 `arr[1] = 2`가 처음으로 `2 이상`이 되는 위치이기 때문.

---

## Upper Bound

### 정의

**target 초과(`> target`)가 처음 나타나는 위치**

즉,

```text
처음으로 arr[i] > target 을 만족하는 i
```

### 예시

```text
arr = [1, 2, 2, 2, 4, 5]
target = 2
```

Upper Bound는 인덱스 `4`

왜냐하면 `arr[4] = 4`가 처음으로 `2 초과`가 되는 위치이기 때문.

---

## 차이 한 줄 요약

* Lower Bound: `>= target`의 첫 위치
* Upper Bound: `> target`의 첫 위치

---

## 왜 필요한가?

### 1) 특정 값의 존재 여부 확인

Lower Bound 위치를 구해서 그 값이 진짜 target인지 보면 된다.

```java
int idx = lowerBound(arr, target);
if (idx < arr.length && arr[idx] == target) {
    // 존재함
}
```

---

### 2) 중복 원소 개수 구하기

```text
개수 = upperBound(target) - lowerBound(target)
```

예:

```text
arr = [1, 2, 2, 2, 4, 5]
target = 2

lowerBound = 1
upperBound = 4
개수 = 4 - 1 = 3
```

---

### 3) 삽입 위치 찾기

정렬을 유지하며 삽입하려면 보통 lower bound 위치에 넣으면 된다.

---

### 4) 범위 질의

예를 들어 `[L, R]` 구간 값의 개수는

```text
upperBound(R) - lowerBound(L)
```

처럼 구할 수 있다.

---

## 구현

### Lower Bound

```java
static int lowerBound(int[] arr, int target) {
    int low = 0;
    int high = arr.length; // high는 닫힌 구간이 아니라 열린 끝

    while (low < high) {
        int mid = low + (high - low) / 2;

        if (arr[mid] >= target) {
            high = mid;
        } else {
            low = mid + 1;
        }
    }
    return low;
}
```

### Upper Bound

```java
static int upperBound(int[] arr, int target) {
    int low = 0;
    int high = arr.length;

    while (low < high) {
        int mid = low + (high - low) / 2;

        if (arr[mid] > target) {
            high = mid;
        } else {
            low = mid + 1;
        }
    }
    return low;
}
```

---

## 왜 `high = mid`, `low = mid + 1`인가?

이건 bound 문제는 “정답 후보를 버리면 안 되기 때문”이야.

예를 들어 lower bound에서 `arr[mid] >= target`이면
mid 자체가 정답일 수 있다.
그래서 `high = mid - 1`로 버리면 안 되고, `high = mid`로 남겨야 한다.

이게 일반적인 “값 찾기” 이진탐색과 bound 탐색의 가장 큰 차이점이다.

# 5. 이진 탐색을 응용할 수 있는 문제 유형은? (Parametric Search)

## 핵심 아이디어

Parametric Search는
“정답 그 자체를 바로 찾는 게 아니라, **어떤 값이 가능한지 불가능한지 판정하는 함수**를 만들고 그 함수에 대해 이진 탐색하는 방식”이야.

즉,

* 어떤 값 `x`가 조건을 만족하는가?
* `x`가 가능하면 그보다 작은/큰 값도 가능한가?

이렇게 **가능/불가능이 한쪽으로 단조롭게 나뉘면** 이진 탐색 가능.

---

## 단조성(monotonicity)

예를 들어 어떤 문제에서:

* `x = 10` 가능
* `x = 9` 가능
* `x = 8` 가능
* `x = 7` 불가능
* `x = 6` 불가능

이렇게 가능/불가능이 경계 하나를 기준으로 나뉘면 된다.

```text
불가능 불가능 불가능 가능 가능 가능
```

또는 반대:

```text
가능 가능 가능 불가능 불가능 불가능
```

이런 구조면 경계값을 이진 탐색으로 찾을 수 있다.

---

## 대표 문제 유형

### 1) 최소값을 찾는 문제

“조건을 만족하는 최소의 x”

예:

* K개의 공유기를 설치할 때 가능한 최소 거리?
* 작업 시간을 제한할 때 필요한 최소 속도?
* 특정 예산 내에서 가능한 최소 비용?

### 2) 최대값을 찾는 문제

“조건을 만족하는 최대의 x”

예:

* 랜선 자르기에서 만들 수 있는 최대 길이
* 나무 자르기에서 가져갈 수 있는 최대 절단 높이
* 제한 시간 안에 가능한 최대 점수/용량

---

## 대표 예시 1: 랜선 자르기

랜선 길이들이 있고, N개 이상 만들 수 있는 최대 길이를 구하라.

길이 `L`로 잘랐을 때 만들 수 있는 개수:

```java
count += cable / L;
```

* `L`이 작을수록 많이 만들 수 있음
* `L`이 클수록 적게 만들 수 있음

즉,

* 어떤 길이까지는 가능
* 그보다 크면 불가능

단조성이 성립하므로 이진 탐색 가능.

---

## 대표 예시 2: 나무 자르기

절단기 높이 `H`를 정했을 때 얻는 나무 길이 합이 `M` 이상이 되도록 하는 최대 `H`.

* `H`가 낮을수록 많이 가져감
* `H`가 높을수록 적게 가져감

역시 단조적이다.

---

## Parametric Search 템플릿

### 조건을 만족하는 최대값 찾기

```java
long low = 1;
long high = maxValue;
long answer = 0;

while (low <= high) {
    long mid = low + (high - low) / 2;

    if (can(mid)) {
        answer = mid;
        low = mid + 1;   // 더 큰 값도 가능한지 확인
    } else {
        high = mid - 1;
    }
}
```

### 조건을 만족하는 최소값 찾기

```java
long low = 1;
long high = maxValue;
long answer = high;

while (low <= high) {
    long mid = low + (high - low) / 2;

    if (can(mid)) {
        answer = mid;
        high = mid - 1;  // 더 작은 값도 가능한지 확인
    } else {
        low = mid + 1;
    }
}
```

---

## Parametric Search를 알아보는 신호

문제에서 이런 표현이 보이면 의심해봐:

* 최소/최대 가능한 값
* 적어도/최소한/이상/이하
* 조건을 만족하는 가장 큰 값
* 조건을 만족하는 가장 작은 값
* 시간, 거리, 용량, 길이, 속도, 예산 같은 “수치 정답”

이런 문제는 대개

1. 정답 후보 범위를 정하고
2. `can(mid)`를 만들어
3. 이진 탐색한다.

# 6. 회전된 정렬 배열에서 이진 탐색을 어떻게 적용하는가?

이건 정렬 배열이 한 번 꺾여 있는 경우야.

예:

```text
[4, 5, 6, 7, 0, 1, 2]
```

원래는 오름차순인데, 어느 지점을 기준으로 회전됨.

---

## 핵심 아이디어

배열 전체는 완전히 정렬되어 있지 않지만,
**mid를 기준으로 보면 적어도 한쪽 구간은 항상 정렬되어 있다.**

예를 들어:

```text
[4, 5, 6, 7, 0, 1, 2]
         mid=7
```

왼쪽 `[4,5,6,7]`은 정렬되어 있음.

또 어떤 경우엔 오른쪽이 정렬되어 있다.

즉, 매번

1. 왼쪽이 정렬인지 확인
2. 아니면 오른쪽이 정렬인지 확인
3. target이 그 정렬된 구간 안에 있는지 보고
4. 절반을 버린다

---

## 판별 기준

현재 구간이 `[low, high]`이고 `mid`가 있을 때

### 왼쪽이 정렬된 경우

```java
if (arr[low] <= arr[mid])
```

그러면 왼쪽 `[low ~ mid]`는 정렬 상태.

이때 target이 왼쪽 구간 안에 있으면:

```java
if (arr[low] <= target && target < arr[mid])
    high = mid - 1;
else
    low = mid + 1;
```

---

### 오른쪽이 정렬된 경우

왼쪽이 정렬이 아니면 오른쪽이 정렬이야.

```java
else
```

이때 오른쪽 `[mid ~ high]`가 정렬 상태.

target이 오른쪽 구간 안에 있으면:

```java
if (arr[mid] < target && target <= arr[high])
    low = mid + 1;
else
    high = mid - 1;
```

---

## 코드

```java
static int searchRotated(int[] arr, int target) {
    int low = 0;
    int high = arr.length - 1;

    while (low <= high) {
        int mid = low + (high - low) / 2;

        if (arr[mid] == target) return mid;

        // 왼쪽 구간이 정렬된 경우
        if (arr[low] <= arr[mid]) {
            if (arr[low] <= target && target < arr[mid]) {
                high = mid - 1;
            } else {
                low = mid + 1;
            }
        }
        // 오른쪽 구간이 정렬된 경우
        else {
            if (arr[mid] < target && target <= arr[high]) {
                low = mid + 1;
            } else {
                high = mid - 1;
            }
        }
    }

    return -1;
}
```

---

## 예시로 이해

```text
arr = [4, 5, 6, 7, 0, 1, 2]
target = 0
```

### 1단계

* low = 0, high = 6
* mid = 3, arr[mid] = 7

왼쪽 `[4,5,6,7]` 정렬됨
target=0은 이 구간에 없음
→ 오른쪽으로 이동

### 2단계

* low = 4, high = 6
* mid = 5, arr[mid] = 1

왼쪽 `[0,1]` 정렬됨
target=0은 이 구간에 있음
→ 왼쪽으로 이동

### 3단계

* low = 4, high = 4
* mid = 4, arr[mid] = 0
  → 찾음

---

## 중복이 있으면?

예를 들어

```text
[2, 2, 2, 3, 4, 2]
```

이런 경우에는 `arr[low] <= arr[mid]`만으로 어느 쪽이 정렬인지 애매할 수 있다.

예:

```text
arr[low] == arr[mid] == arr[high]
```

그러면 한쪽 정렬 여부를 명확히 판단할 수 없어져서,
보통 `low++`, `high--` 같은 처리를 추가해야 한다.

이 경우 최악 시간복잡도는 `O(log n)`이 아니라 `O(n)`까지 떨어질 수 있다.
