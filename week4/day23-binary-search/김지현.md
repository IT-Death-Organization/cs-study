## 1. 이진 탐색(Binary Search)의 전제 조건은 무엇이며 왜 필요한가?

**전제 조건:**
데이터가 **정렬(sorted)** 되어 있어야 한다.

**이유**

이진 탐색은 **탐색 범위를 절반씩 줄여가는 알고리즘**이다.
이를 위해서는 **중간 값(mid)과 목표 값(target)의 크기 비교가 의미가 있어야 한다.**

예를 들어 배열이 오름차순 정렬되어 있다고 가정하자.

```
[1, 3, 5, 7, 9, 11, 13]
```

target = 9일 때

1. mid = 7
2. 9 > 7 → **오른쪽 절반만 탐색**

정렬이 되어 있기 때문에
**왼쪽에 있는 값들은 전부 7보다 작다는 것이 보장된다.**

즉

```
왼쪽 영역 전체 제거
```

이것이 가능하다.

하지만 정렬이 되어 있지 않다면

```
[7, 1, 13, 5, 9, 3]
```

mid = 13이라도
target이 어느 쪽에 있는지 **판단할 수 없다.**

따라서

**Binary Search의 핵심 전제 = monotonic order (정렬된 상태)**

---

## 2. 이진 탐색이 O(log n)인 이유 (수학적 설명)

이진 탐색은 매 단계마다 **탐색 범위를 절반으로 줄인다.**

초기 데이터 개수

```
n
```

1단계 후

```
n / 2
```

2단계 후

```
n / 4
```

k단계 후

```
n / 2^k
```

탐색이 끝나는 시점은

```
n / 2^k = 1
```

양변에 log₂를 취하면

```
n = 2^k
```

따라서

```
k = log₂(n)
```

즉 탐색 횟수는

```
O(log n)
```

예시

| n             | 최대 탐색 횟수 |
| ------------- | -------------- |
| 1,000         | 약 10          |
| 1,000,000     | 약 20          |
| 1,000,000,000 | 약 30          |

데이터가 **10억 개여도 30번 정도 비교**면 끝난다.

## 3. mid 계산을 `low + (high - low) / 2`로 하는 이유

보통 mid를 이렇게 계산한다.

```
mid = (low + high) / 2
```

하지만 이 방식은 **정수 오버플로우 위험**이 있다.

예시 (32bit int)

```
low = 2,000,000,000
high = 2,100,000,000
```

```
low + high = 4,100,000,000
```

int 범위를 초과한다.

따라서 안전한 방식은

```
mid = low + (high - low) / 2
```

이 계산은 다음과 같다.

```
mid = low + (high - low)/2
    = (low + high) / 2
```

수학적으로 동일하지만

```
(high - low)
```

는 항상 **범위 안에 있는 값**이기 때문에
오버플로우가 발생하지 않는다.

그래서 실제 라이브러리와 실무 코드에서도 이 방식이 사용된다.

---

## 4. Lower Bound와 Upper Bound

### Lower Bound

**target 이상이 처음 나타나는 위치**

```
arr = [1,2,2,2,3,5]
target = 2
```

```
lower_bound = index 1
```

조건

```
arr[i] >= target
```

---

### Upper Bound

**target보다 큰 값이 처음 나타나는 위치**

```
arr = [1,2,2,2,3,5]
target = 2
```

```
upper_bound = index 4
```

조건

```
arr[i] > target
```

---

### 차이

| 함수        | 조건     |
| ----------- | -------- |
| lower_bound | ≥ target |
| upper_bound | > target |

---

### 활용

**1️⃣ 특정 값 개수 구하기**

```
count = upper_bound - lower_bound
```

예

```
[1,2,2,2,3,5]
2의 개수

= 4 - 1
= 3
```

---

**2️⃣ 삽입 위치 찾기**

정렬 유지하면서 삽입할 때

```
insert position = lower_bound
```

## 5. 이진 탐색 응용 — Parametric Search

이진 탐색은 **값을 찾는 알고리즘이 아니라**

**조건을 만족하는 최소/최대 값을 찾는 알고리즘으로 확장할 수 있다.**

이를

```
Parametric Search
```

라고 한다.

핵심 아이디어

```
조건이 monotonic 해야 한다.
```

즉

```
false false false true true true
```

처럼 한 번 바뀌면 다시 돌아가지 않는 형태.

---

### 예시 1 — 랜선 자르기

문제

```
랜선을 잘라서 N개 만들 때
가능한 최대 길이
```

길이 L을 가정하고

```
L로 잘라서 N개 이상 만들 수 있는가?
```

가능하면

```
더 길게 시도
```

불가능하면

```
더 짧게
```

---

### 예시 2 — 최소 시간 찾기

```
심사대 n개
m명 처리
최소 시간
```

시간 T를 가정하고

```
T 시간 동안 처리 가능한 인원 >= m ?
```

가능하면

```
더 작은 시간 탐색
```

---

즉 Binary Search 대상이

```
데이터
```

가 아니라

```
답의 범위
```

가 된다.

## 6. 회전된 정렬 배열에서 이진 탐색

문제 예시

```
[4,5,6,7,0,1,2]
```

원래 배열

```
[0,1,2,4,5,6,7]
```

가 **회전(rotation)** 된 상태.

이 경우 일반 이진 탐색이 깨진다.

하지만 중요한 성질이 있다.

```
항상 한쪽 구간은 정렬되어 있다.
```

---

### 알고리즘

1️⃣ mid 계산

```
mid = (low + high) / 2
```

2️⃣ 어느 쪽이 정렬되어 있는지 확인

```
if arr[low] <= arr[mid]
```

왼쪽 구간 정렬

```
[low ... mid]
```

---

3️⃣ target이 왼쪽에 있는지 확인

```
arr[low] <= target < arr[mid]
```

있다면

```
high = mid - 1
```

없다면

```
low = mid + 1
```

---

### 오른쪽 정렬 케이스

```
else
```

오른쪽 정렬

```
[mid ... high]
```

target이 여기에 있으면

```
low = mid + 1
```

없으면

```
high = mid - 1
```

---

### 시간복잡도

```
O(log n)
```
