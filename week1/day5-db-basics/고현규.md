# 1. 데이터베이스가 파일 시스템보다 나은 이유는?

과거에는 데이터를 일반 텍스트 파일이나 엑셀 같은 파일 시스템에 저장했습니다. 하지만 데이터 양이 방대해지면서 데이터베이스(DBMS)가 등장하게 되었습니다.

### 데이터 중복 및 불일치 해결
파일 시스템은 같은 데이터가 여러 파일에 중복 저장되기 쉽고, 한 곳을 수정했을 때 다른 곳이 갱신되지 않는 데이터 불일치(Inconsistency) 문제가 발생합니다. DBMS는 이를 통합 관리하여 최소화합니다.

### 데이터 무결성 유지
데이터베이스는 제약 조건(Constraints)을 통해 데이터가 일정한 규칙을 지키도록 강제합니다. 예를 들어, 나이 필드에 문자가 들어오는 것을 원천 차단합니다.

### 보안 및 동시 제어
파일 시스템은 여러 사람이 동시에 수정할 때 데이터가 덮어씌워질 위험이 크지만, DBMS는 트랜잭션과 잠금(Locking) 메커니즘을 통해 여러 사용자의 동시 접근을 안전하게 관리합니다.

<br>

# 2. RDBMS와 NoSQL의 차이는? 각각의 장단점은?



| 특징 | RDBMS (관계형) | NoSQL (비관계형) |
| :--- | :--- | :--- |
| **데이터 구조** | 엄격한 스키마, 2차원 테이블 형태 | 유연한 스키마, 다양한 저장 방식 |
| **관계 및 조인** | JOIN을 통한 데이터 관계 표현 능숙 | JOIN이 없거나 매우 제한적 |
| **확장성** | 수직적 확장(Scale-up) 위주 | 수평적 확장(Scale-out) 용이 |
| **장점** | 데이터 무결성 보장, SQL 표준 지원 | 빠른 쓰기 속도, 대량 데이터 처리 최적화 |
| **단점** | 스키마 변경이 어렵고 확장 비용이 큼 | 데이터 중복 발생 가능, 일관성 저하 가능성 |

<br>

# 3. RDBMS는 언제 사용하고, NoSQL은 언제 사용하는가?

### RDBMS를 선택해야 할 때
1. 데이터의 구조가 명확하고 변경될 가능성이 적을 때
2. 금융 서비스처럼 데이터의 정확성과 일관성이 절대적으로 중요할 때
3. 복잡한 쿼리와 JOIN 연산이 많이 필요할 때

### NoSQL을 선택해야 할 때
1. 데이터의 구조가 불명확하거나 자주 변경될 때
2. 읽기/쓰기 작업이 매우 빈번하여 고성능 처리가 필요할 때
3. 대규모의 데이터를 여러 서버에 분산하여 저장해야 할 때

<br>

# 4. CAP 이론이란 무엇인가? (일관성, 가용성, 분할 내성)

분산 데이터베이스 시스템에서 다음 세 가지 조건을 모두 만족하는 것은 불가능하며, 대개 두 가지만 선택 가능하다는 이론입니다.



* Consistency (일관성): 모든 노드가 같은 시간에 같은 데이터를 보여줘야 함.
* Availability (가용성): 일부 노드에 장애가 생겨도 요청에 응답할 수 있어야 함.
* Partition Tolerance (분할 내성): 노드 간 통신 단절이 발생해도 시스템이 유지되어야 함.

현대적인 분산 시스템(NoSQL 등)은 대개 네트워크 장애를 대비해야 하므로 P(분할 내성)를 기본으로 두고, C(일관성)를 택할지 A(가용성)를 택할지를 결정합니다.

<br>

# 5. NoSQL의 종류와 차이는?

| 종류 | 특징 | 대표 DB |
| :--- | :--- | :--- |
| **Key-Value** | 가장 단순한 형태, 키를 통한 빠른 조회 | Redis, Memcached |
| **Document** | JSON/XML 형태로 데이터 저장, 자유로운 구조 | MongoDB, CouchDB |
| **Column-Family** | 행이 아닌 열(Column) 단위로 저장, 대용량 통계 적합 | Cassandra, HBase |
| **Graph** | 데이터 간의 관계를 노드와 간선으로 연결 | Neo4j |

<br>

# 6. 스키마가 있는 것과 없는 것의 장단점은?

### 스키마가 있는 경우 (Structured)
* 장점: 데이터가 규격화되어 있어 에러를 사전에 방지할 수 있고, 데이터 품질이 보장됩니다.
* 단점: 초기 설계가 어렵고, 서비스 성장 중 데이터 구조를 바꾸려면 시스템을 멈추거나 복잡한 마이그레이션이 필요합니다.

### 스키마가 없는 경우 (Schemaless)
* 장점: 어떤 형태의 데이터든 즉시 저장할 수 있어 개발 속도가 매우 빠릅니다. 데이터 구조 변경이 자유롭습니다.
* 단점: 데이터가 제각각이라 애플리케이션 레벨에서 데이터를 읽을 때 검증 로직이 복잡해질 수 있으며, 데이터 중복 관리가 어렵습니다.
