### 데이터베이스가 파일 시스템보다 나은 이유

파일 시스템의 한계를 극복하기 위해 등장한 것이 DBMS임 (*DBMS: Database Management System)

- **데이터 중복성 및 불일치 해결:** 파일 시스템은 동일한 데이터를 여러 파일에 중복 저장하기 쉬워 데이터 일관성이 깨질 위험이 크지만, DB는 통합 관리를 통해 중복을 최소화함
- **데이터 무결성 유지:** 제약 조건(Constraints)을 통해 데이터의 정확성을 보장함
- **동시 접근 제어:** 여러 사용자가 동시에 접근할 때 데이터가 꼬이지 않도록 트랜잭션과 잠금(Lock) 메커니즘을 제공함
- **장애 복구:** 시스템 장애 시 트랜잭션 로그를 통해 이전의 안정적인 상태로 복구(Recovery)할 수 있음
- **보안성:** 사용자별로 접근 권한을 세밀하게 제어할 수 있음

### RDBMS와 NoSQL의 차이점 & 장단점


| **구분** | **RDBMS (Relational)** | **NoSQL (Non-Relational)** |
| --- | --- | --- |
| **데이터 구조** | 정해진 스키마, 테이블 형태 | 유연한 구조 (JSON, Key-Value 등) |
| **확장 방식** | 수직적 확장 (Scale-up) | 수평적 확장 (Scale-out) |
| **관계** | 외래 키(FK)를 통한 조인(Join) 가능 | 일반적으로 조인이 없거나 매우 제한적 |
| **일관성** | ACID 원칙 준수 (강한 일관성) | CAP 이론에 따른 선택적 일관성 (BASE) |
- RDBMS
    - 장점
        - 데이터 구조가 명확하고 정합성이 높으며, SQL이라는 표준화된 언어로 복잡한 쿼리 수행이 가능
    - 단점
        - 스키마 변경이 어렵고, 대용량 데이터 처리 시 확장 비용이 많이 듦
- NoSQL
    - 장점
        - 스키마가 자유로워 데이터 구조 변경이 잦은 환경에 유리하며, 저렴한 서버 여러 대로 데이터를 분산 저장(Sharding)하기 쉬움
    - 단점
        - 데이터 중복이 발생할 수 있고, 복잡한 데이터 관계를 표현하기 어려움

### RDBMS vs NoSQL  언제 사용

- **RDBMS:**
    - 데이터 구조가 명확하고 거의 변하지 않을 때
    - 금융 시스템처럼 데이터의 정확성과 일관성(ACID)이 절대적으로 중요할 때
    - 여러 테이블 간의 복잡한 조인이 자주 일어날 때
- **NoSQL:**
    - 데이터 구조가 불분명하거나 자주 바뀌는 경우
    - 읽기/쓰기 성능이 매우 중요하고 초당 트래픽이 엄청날 때 (예: 소셜 미디어 피드)
    - 데이터 양이 기하급수적으로 늘어나 수평적 확장이 필수적일 때

### CAP 이론 (Consistency, Availability, Partition Tolerance)

분산 데이터베이스 시스템에서 다음 세 가지 속성을 모두 만족하는 것은 불가능하며, **최대 두 가지만 선택 가능**하다는 이론

- **일관성(Consistency):** 모든 노드가 동시에 같은 데이터를 보여주어야 함
- **가용성(Availability):** 특정 노드에 장애가 생겨도 시스템은 항상 응답해야 함
- **분할 내성(Partition Tolerance):** 노드 간 통신 단절이 발생해도 시스템이 계속 동작해야 함 (현대 분산 시스템에서는 필수 요소임)

### NoSQL의 종류와 차이점

- **Key-Value:** 가장 단순한 형태. 속도가 매우 빠르며 캐싱이나 세션 관리에 쓰임 (예: Redis, Riak)
- **Document:** 데이터를 JSON/BSON 형태로 저장. 객체 지향 프로그래밍과 구조가 비슷해 개발 효율이 좋음 (예: MongoDB, CouchDB)
- **Column-Family:** 대량의 데이터에 대해 특정 열을 기준으로 집계나 분석을 할 때 유리함 (예: Cassandra, HBase)
- **Graph:** 노드와 간선으로 관계를 표현. 소셜 네트워크나 추천 엔진처럼 복잡한 관계 추적에 최적화됨 (예: Neo4j)

### 스키마가 있는 것 vs 없는 것 (Schema vs Schemaless)

- **Schema**
    - 장점
        - 데이터 품질 유지, 저장 공간 효율성, 필드 누락 방지
    - 단점
        - 유연성 부족, 스키마 변경 시 서비스 중단(Migration) 위험
- **Schemaless**
    - 장점
        - 비정형 데이터 저장 가능, 빠른 프로토타이핑 및 배포 가능
    - 단점
        - 애플리케이션 레벨에서 데이터 정합성을 검증해야 하므로 관리 로직이 복잡해질 수 있음
