1. 정규화가 필요한 이유

정규화의 목적은 다음 세 가지입니다.
	1.	데이터 중복 최소화
	2.	무결성 유지
	3.	이상 현상(Anomaly) 방지

중복 데이터가 많으면 수정·삭제·삽입 시 일관성이 깨질 수 있습니다.

⸻

이상 현상(Anomaly)

중복으로 인해 발생하는 비정상적 데이터 변경 문제
	1.	삽입 이상
불필요한 정보가 없으면 삽입 불가
	2.	삭제 이상
하나의 정보 삭제 시 원치 않는 정보까지 함께 삭제
	3.	갱신 이상
같은 데이터가 여러 행에 존재 → 일부만 수정되면 불일치 발생

⸻

2. 함수적 종속성 (Functional Dependency)

정의:

X → Y
X 값이 같으면 Y 값은 반드시 하나로 결정됨

예:
학번 → 이름
주문번호 → 고객ID

종속성 종류:
	1.	완전 함수 종속
기본키 전체에 종속
	2.	부분 함수 종속
기본키 일부에만 종속
	3.	이행 종속
A → B, B → C 이면 A → C

⸻

3. 정규형 정리

1NF (제1정규형)

조건:
	•	모든 속성은 원자값
	•	반복 그룹 제거

목표:
컬럼에 배열, 리스트 저장 금지

⸻

2NF (제2정규형)

조건:
	•	1NF 만족
	•	부분 함수 종속 제거

즉, 복합키의 일부에만 종속되는 속성 제거

⸻

3NF (제3정규형)

조건:
	•	2NF 만족
	•	이행 종속 제거

A → B → C 구조 제거

⸻

BCNF (Boyce-Codd Normal Form)

조건:
	•	모든 함수적 종속성 X → Y에서
	•	X는 반드시 후보키

3NF보다 더 엄격함.

⸻

4. 3NF와 BCNF 차이

3NF는 다음을 허용합니다.

X → Y에서
	•	X가 후보키이거나
	•	Y가 기본 속성(Prime Attribute)

이 예외 때문에 3NF인데도 이상 현상이 남을 수 있습니다.

BCNF는 예외를 허용하지 않습니다.

모든 결정자는 반드시 후보키여야 합니다.

⸻

5. BCNF가 필요한 경우

주로 다중 후보키 구조에서 발생

예시:

수강(학생, 과목, 교수)

조건:
	•	한 과목은 한 교수만 담당
	•	한 교수는 한 과목만 담당

함수적 종속성:
교수 → 과목
과목 → 교수

후보키:
(학생, 과목)
(학생, 교수)

이 경우 3NF는 만족하지만
교수 → 과목에서 교수가 후보키가 아니므로
BCNF는 위반

따라서 분해 필요

⸻

6. 정규화의 장단점

장점:
	•	데이터 중복 감소
	•	무결성 보장
	•	유지보수 용이
	•	저장 공간 절약

단점:
	•	JOIN 증가
	•	조회 성능 저하 가능
	•	설계 복잡도 증가

⸻

7. 왜 항상 정규화를 하지 않는가

읽기 성능이 더 중요할 수 있기 때문

특히:
	•	대규모 트래픽 서비스
	•	조회 중심 시스템
	•	리포팅 시스템

JOIN 비용이 커질 수 있음

⸻

8. 역정규화 (Denormalization)

정의:
의도적으로 중복을 허용하여 조회 성능 개선

사용하는 경우:
	1.	JOIN 비용이 클 때
	2.	읽기 비중이 매우 클 때
	3.	집계 데이터를 자주 조회할 때

성능 향상 이유:
	•	JOIN 제거
	•	쿼리 단순화
	•	랜덤 I/O 감소

⸻

9. 실무에서는 어디까지 정규화하는가

일반적으로:

3NF까지가 표준

BCNF는 특수한 다중 후보키 구조에서 사용

실무 전략:
	1.	먼저 3NF까지 설계
	2.	병목 쿼리 분석
	3.	필요한 부분만 역정규화

⸻
