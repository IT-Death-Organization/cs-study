## Q1. 정규화가 필요한 이유는? 이상 현상(Anomaly)이란?

[답변]
* 정규화가 필요한 이유
  - 데이터 중복 줄임 => 갱신 이상(Anomaly) 방지
  - 데이터 일관성/무결성 높임
  - 적절한 정규화를 통해 한 곳에서만 데이터 관리 가능
  - "함수적 종속성"에 맞게 속성 배치 => '한 사실'은 '한 곳'에만 저장
* 이상 현상(Anomaly)
  - 중복된 데이터 구조로 인해, 데이터 조작 과정에서 발생하는 논리적 문제
  - 대부분 한 테이블이 여러 주제(엔티티)의 정보를 동시에 담는 경우 발생
    1. 삽입 이상(Insert Anomaly)
       : 특정 정보를 넣기 위해 불필요한 다른 정보까지 함께 삽입 or 필수값이 없어 삽입 불가
    3. 갱신 이상
       : 같은 값이 여러 곳에 중복 => 일부만 수정하면 데이터 불일치 발생
    5. 삭제 이상
       : 어떤 데이터를 삭제 => 의도치 않게 다른 정보까지 삭제

[보완]
* 함수적 종속성
  - 관계형 데이터에서 속성들 사이의 결정 관계
  - X -> Y
    - 같은 테이블에서 X 값이 같으면 Y 값은 반드시 같다 (X가 Y를 결정)
    - ex. 학번 -> 학생이름, 주민번호 -> 생년월일

## Q2. 제1정규형, 제2정규형, 제3정규형, BCNF의 조건과 차이는?

[답변]
* 제1정규형(1NF)
  - 조건
    - 모든 속성 값이 원자값(atomic)
    - 반복 그룹(배열/리스트 형태 컬럼)이 없어야 함
  - '한 칸'에 '값 하나'
* 제2정규형(2NF)
  - 조건
    - 1NF 만족
    - 기본키가 복합키인 경우 -> 부분 함수 종속(Partial Dependency)이 없음
  - 복합키 일부에만 매달리 컬럼 제거
* 제3정규형(3NF)
  - 조건
    - 2NF 만족
    - 이행 함수 종속이 없음: 비주요 속성이 다른 비주요 속성에 의존 X
  - 비주요 속성 간 종속 제거
* BCNF
  - 조건
    - 모든 비자명 함수적 종속 X->Y에서 X가 항상 슈퍼키
    - 3NF보다 강한 형태: 3NF에서 남을 수 있는 일부 이상을 추가로 제거
  - 결정자(determinant)는 반드시 키

[보완]
* 용어
  - 결정자: 함수적 종속 X->Y에서 X
  - 슈퍼키: 튜플을 유일하게 식별 가능한 속성 집합
  - 후보키: 최소 슈퍼키
  - 비주요 속성: 어떤 후보키에도 포함되지 않는 속성
* 부분 함수 종속
  - 복합키를 사용하는 테이블에서, 어떤 속성이 키 전체가 아니라 키의 일부에만 종속되는 경우
  - (A, B)가 키인데, A만으로 C 결정
* 이행 함수 종속
  - X->Y, Y->Z가 성립 => X->Z가 되는 종속
  - 비주요 속성이 다른 비주요 속성을 결정할 수도 있음
* 비자명 함수적 종속
  - 함수적 종속 X->Y에서 Y가 X의 부분집합이 아닌 경우
  - ex. 자명: (학번, 과목번호) -> 학번
        비자명: 학번 -> 이름

## Q3. 함수적 종속성(Functional Dependency)이란 무엇인가?

[답변]
* 함수적 종속성(FD)
  - 관계형 데이터베이스에서 속성 간의 결정 관계
  - X->Y라고 할 때, X값이 같으면 Y값은 반드시 같다: X가 Y를 결정
  - 정규화는 결국 FD를 분석해, 중복과 이상을 줄이도록 테이블을 분해하는 과정

---

## Q4. 정규화의 장단점은? 왜 항상 정규화를 하지 않는가?

[답변]
* 장점
  - 데이터 중복 감소 -> 저장공간 절감 및 관리 단순화
  - 삽입/갱신/삭제 이상 방지 -> 데이터 일관성과 무결성 향상
  - 변경에 강함(유지보수 용이)
* 단점
  - 테이블이 분해되며 JOIN 증가 -> 조회 성능 저하 가능
  - 쿼리 복잡도 증가(개발 난이도 및 운영 복잡도 상승)
  - 특정 조회 패턴에서는 '한 번에' 가져올 수 있던 데이터가 여러 테이블에 흩어짐
* 항상 정규화를 하지 않는 이유
  - 데이터 설계 목표가 정합성만이 아닌, 정합성과 성능/단순성의 균형이기 때문
  - 조회 비중이 큰 시스템의 경우 정규화로 인한 JOIN 비용 체감 가능

## Q5. 역정규화는 언제 하는가? 어떤 경우에 성능 향상이 되는가?

[답변]
* 역정규화
  - 정규화로 인해 증가한 JOIN, 집계 비용 등을 줄이기 위해 의도적으로 중복 허용 -> 구조 단순화
  - 고려되는 상황
    - 조회가 압도적으로 많음 -> JOIN 병목
    - 자주 사용되는 집계/파생 값을 매번 계산하기 부담스러운 경우
    - 화면/리프토에서 항상 함꼐 조회되는 컬럼 묶음이 존재하는 경우
    - 분산 환경/대용량 환경 -> 조인 비용이 특히 큰 경우
* 성능 향상 케이스
  - JOIN 제거/축소: 조회 시 필요한 테이블 수 줄임 -> I/O와 CPU 절감
  - 집계/계산 결과 미리 저장: ex. 주문 합계, 누적 포인트, 좋아요 수
  - 캐시 성격의 컬럼 추가: 자주 조회되는 '표시용 값'

[보완]
* 역정규화는 성능을 얻는 대신 갱신 시 정합성 유지 비용(트랜잭션, 트리거, 애플리케이션 로직) 증가
* '어디서 값을 갱신', '불일치 발생 시 복구 전략' 함께 설계

## Q6. 실무에서는 보통 몇 정규형까지 정규화하는가?

[답변]
* 실무에서는 보통 제3정규형까지 정규화하는 경우가 많음
* 시스템 특성에 따라 BCNF까지 적용 - 항상 필수는 아니며 설계 복잡도와 성능 요구사항을 함께 고려
* 이후 조회 성능이 중요한 구간 -> 부분 역정규화

[보완]
* OLTP(Online Transaction Processing)
  - 사용자 요청에 따라 짧고 빈번한 트랜잭션(삽입/수정/삭제)을 처리하는 운영 시스템
  - 3NF 중심으로 정규화하여 정합성 강화 -> 필요 시 선택적 역정규화
* OLAP(Online Analytical Processing)
  - 데이터를 모아 조회/집계/분석(리포트, 대시보드, 통계)하는 분석 시스템
  - 조회/집계 효율을 위해 스타 스키마(사실/차원) 등 비정규화 구조 적극 활용
