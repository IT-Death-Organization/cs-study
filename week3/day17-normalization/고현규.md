# 1. 정규화가 필요한 이유는? 이상 현상(Anomaly)이란?

> 정규화는 관계형 데이터베이스 설계에서 데이터의 중복을 최소화하고, 데이터를 구조적으로 조직화하여 이상 현상(Anomaly)을 방지하기 위한 일련의 과정입니다.

### 1) 왜 정규화가 필요한가?
잘못 설계된 테이블에 데이터를 마구 밀어넣다 보면, 같은 정보가 여러 곳에 중복되어 저장됩니다. 이는 단순히 저장 공간을 낭비하는 것을 넘어, 데이터의 일관성(Consistency)과 무결성(Integrity)을 파괴하는 원인이 됩니다. 정규화는 이를 위해 테이블을 무손실 분해하여 "한 곳에는 하나의 사실만(One fact in one place)" 있도록 만듭니다.

### 2) 이상 현상(Anomaly)의 종류
정규화를 거치지 않은 테이블에서 발생하는 치명적인 세 가지 문제입니다.

* **삽입 이상 (Insertion Anomaly):** 데이터를 저장할 때 원하지 않는 정보까지 강제로 입력해야 하거나, 특정 정보가 없으면 데이터 자체를 입력하지 못하는 현상입니다.
    * 예: 학생이 수강 신청을 하지 않으면 학생 정보(학번, 이름)조차 등록할 수 없는 경우
* **갱신 이상 (Update Anomaly):** 중복된 데이터 중 일부만 수정되어 데이터 간의 불일치가 발생하는 현상입니다.
    * 예: 한 학생의 전화번호가 바뀌었는데, 여러 개의 수강 기록 중 일부만 수정되어 어떤 번호가 진짜인지 알 수 없게 된 경우
* **삭제 이상 (Deletion Anomaly):** 특정 정보를 삭제할 때, 그와 연관된 유지해야 할 다른 정보까지 함께 지워지는 현상입니다.
    * 예: 특정 과목의 수강 기록을 지웠는데, 그 과목을 듣던 유일한 학생의 개인정보까지 통째로 날아가는 경우

### 관심사의 분리 (Separation of Concerns)
정규화는 결국 관심사의 분리를 DB 수준에서 실현하는 것입니다. 서로 다른 도메인의 데이터가 한 바구니에 담겨 있으면 수정이나 삭제 시 의도치 않은 사이드 이펙트가 발생합니다. 정규화는 데이터를 각각의 독립된 논리적 단위로 쪼개어, 하나를 고칠 때 다른 정보가 오염되지 않도록 보호하는 방어막 역할을 합니다.

# 2. 제1정규형, 제2정규형, 제3정규형, BCNF의 조건과 차이는?

> 정규화는 단계를 거치며 점진적으로 데이터 구조를 개선해 나가는 과정입니다. 각 단계는 이전 단계의 조건을 모두 만족해야 합니다.

| 정규형 | 조건 (Key point) | 해결하려는 문제 |
| :--- | :--- | :--- |
| **제1정규형 (1NF)** | 모든 도메인(컬럼)이 원자 값(Atomic Value, 하나의 값)이어야 함. | 한 칸에 여러 값이 들어가는 경우 해결 |
| **제2정규형 (2NF)** | 부분 함수적 종속성을 제거해야 함. (완전 함수적 종속 만족) | 기본키의 일부에만 종속된 속성 분리 |
| **제3정규형 (3NF)** | 이행적 함수적 종속성을 제거해야 함. | 기본키가 아닌 속성 간의 종속 관계 분리 |
| **BCNF** | 모든 결정자가 후보키여야 함. | 후보키가 여러 개일 때 발생하는 종속성 해결 |

### 1) 제1정규형 (1NF)
테이블의 컬럼이 원자 값만을 가져야 합니다. 예를 들어, 한 학생의 '취미' 컬럼에 '축구, 농구'처럼 여러 값이 들어있으면 안 됩니다.

### 2) 제2정규형 (2NF)
기본키가 복합키(둘 이상의 컬럼 조합)일 때, 기본키의 일부분에만 종속되는 속성이 있으면 안 됩니다. 모든 속성은 기본키 전체에 의존해야 합니다.

### 3) 제3정규형 (3NF)
기본키를 거쳐서 다른 속성에 종속되는 관계를 끊어야 합니다. 즉, $A \rightarrow B$이고 $B \rightarrow C$일 때 $A \rightarrow C$가 성립하는 관계(이행적 종속)를 분리합니다.

### 4) BCNF (Boyce-Codd Normal Form)
제3정규형을 조금 더 강화한 것으로, 모든 결정자가 후보키인 상태입니다. 주로 복합키가 존재하고 서로 중첩될 때 발생하며, "모든 결정자는 키여야 한다"는 원칙을 지키는 단계입니다.

# 3. 함수적 종속성(Functional Dependency)이란 무엇인가?

> 함수적 종속성은 어떤 속성 $X$의 값을 알면 다른 속성 $Y$의 값을 유일하게 결정할 수 있는 관계를 말합니다. ($X \rightarrow Y$)

### 1) 결정자와 종속자
* **결정자 (Determinant):** $X \rightarrow Y$ 관계에서 $X$. 값을 결정하는 주체. (예: 학번)
* **종속자 (Dependent):** $X \rightarrow Y$ 관계에서 $Y$. $X$에 의해 값이 결정되는 대상. (예: 이름)

### 2) 종속성의 종류
* **완전 함수적 종속**
    * 속성 $Y$가 기본키 $X$ 전체에 대해서만 종속되는 경우.
* **부분 함수적 종속**
    * 복합키 $X$ 중에서 일부분에만 속성 $Y$가 종속되는 경우.
* **이행적 함수적 종속**
    * $A \rightarrow B$이고 $B \rightarrow C$여서 $A \rightarrow C$가 성립하는 경우.

# 4. 정규화의 장단점은? 왜 항상 정규화를 하지 않는가?

> 정규화는 무결성을 지키는 강력한 도구이지만, 성능(Performance) 측면에서는 명확한 트레이드오프가 존재합니다.

### 1) 정규화의 장점
* **데이터 무결성 유지**
    * 중복이 제거되므로 데이터 불일치(이상 현상)를 원천 차단합니다.
* **유연한 구조**
    * 테이블이 논리적으로 쪼개져 있어 새로운 요구사항이 생겨도 구조 변경이 용이합니다.
* **저장 공간 절약**
    * 동일한 데이터가 여러 번 저장되지 않아 저장 효율이 높습니다.

### 2) 정규화의 단점 (왜 항상 하지 않는가?)
* **조인(Join) 비용 발생**
    * 데이터를 확인하기 위해 여러 테이블을 합쳐야 하므로 응답 시간이 길어집니다.
* **쿼리의 복잡성**
    * 정규화가 깊어질수록 데이터를 가져오기 위한 SQL문이 복잡해지고 유지보수가 어려워질 수 있습니다.
* **CPU/메모리 사용량 증가**
    * 수많은 조인 연산은 데이터베이스 엔진에 큰 부하를 줍니다.

<br>

이론적으로는 높은 단계의 정규형이 완벽해 보이지만, 실제 서비스에서는 사용자에게 0.1초라도 빨리 결과를 보여주는 것이 더 중요할 때가 많습니다. 따라서 읽기 작업이 압도적으로 많은 조회 시스템에서는 의도적으로 정규화를 덜 하거나, 다음 5번에서 소개할 역정규화를 선택하기도 합니다.

<br>

# 5. 역정규화는 언제 하는가? 어떤 경우에 성능 향상이 되는가?

> 역정규화(Denormalization)는 시스템의 성능 향상 및 개발 편의성을 위해 정규화된 데이터 모델을 의도적으로 중복, 통합, 분리하는 과정입니다.

### 1) 역정규화를 하는 시점
* **조인이 너무 잦을 때** 
    * 특정 데이터를 조회할 때마다 5개 이상의 테이블을 조인해야 한다면 성능 저하가 심각해집니다.
* **대량의 범위를 자주 조회할 때** 
    * 통계 쿼리나 집계 쿼리처럼 넓은 범위의 데이터를 계산해야 할 때 미리 계산된 값을 저장해두는 방식입니다.

### 2) 성능 향상의 사례
* **중복 컬럼 추가** 
    * 조인을 피하기 위해 자주 사용하는 컬럼을 다른 테이블에도 중복해서 기록합니다.
* **통계 테이블 생성** 
    * 매번 `SUM`이나 `AVG`를 계산하는 대신, 일별/월별 합계 데이터를 미리 저장해둔 테이블을 조회합니다.
* **테이블 분할 (Sharding/Partitioning)**
    * 너무 거대해진 테이블을 가로(Row) 또는 세로(Column)로 쪼개어 검색 속도를 높입니다.

### 최후의 수단
역정규화는 무결성을 희생하고 성능을 얻는 행위입니다. 중복 데이터를 허용하기 때문에 데이터 수정 시 모든 곳을 업데이트해야 하는 "갱신 이상"의 위험을 수동으로 관리해야 하죠. 따라서 인덱스 최적화, 쿼리 튜닝, 캐싱(Redis 등)을 먼저 시도해보고, 그래도 답이 없을 때 신중하게 도입해야 하는 고급 기술입니다.

# 6. 실무에서는 보통 몇 정규형까지 정규화하는가?

> 실무 데이터베이스 설계의 표준은 **제3정규형(3NF) 또는 BCNF**까지입니다.

### 1) 왜 3NF/BCNF인가?
* 대부분의 이상 현상은 3NF까지만 진행해도 해결됩니다. 
* BCNF 이상의 단계(4NF, 5NF 등)는 현실 세계의 데이터 관계에서는 거의 발생하지 않는 특수한 경우이거나, 조인 비용이 정규화로 얻는 이득보다 훨씬 커지기 때문에 실무에서는 거의 고려하지 않습니다.

### 2) 실무에서의 설계 흐름
1. 우선 BCNF까지 철저하게 정규화를 진행하여 논리적으로 완벽한 설계를 만듭니다.
2. 시스템 구축 후 성능 테스트를 진행합니다.
3. 성능 병목이 발생하는 특정 지점에 대해서만 부분적으로 역정규화를 적용합니다.
