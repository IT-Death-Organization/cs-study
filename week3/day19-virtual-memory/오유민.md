### 📌 [Q1] 가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?

가상 메모리(Virtual Memory)는 **실제 컴퓨터의 물리 메모리(RAM)보다 큰 메모리 공간을 가상으로 제공**하는 기술이다.
가상 메모리가 필요한 이유는 다음과 같다.

- 물리 메모리의 한계 극복
  - 프로그램 크기가 RAM보다 커도 실행 가능

- 프로세스 간 메모리 격리(보호)
  - 각 프로세스가 독립적인 주소 공간을 가지게 해서, 다른 프로세스의 메모리에 접근할 수 없도록 함

- 메모리 사용 효율 증가
  - 실제로 필요한 부분만 메모리에 올려 사용, 나머지는 디스크에 보관

**물리 메모리보다 큰 프로그램을 실행할 수 있는 원리**
- **프로그램 전체를 메모리에 한 번에 올리지 않기 때문에** 가능
- 원리: <br>
프로그램을 page 단위로 나눔 (보통 4KB) -> 실행할 때 필요한 페이지만 RAM에 올림 / 나머지는 디스크(스왑 영역)에 둠 -> 실행 중 필요한 페이지가 RAM에 없으면 **Page Fault** 발생 -> OS가 디스크에서 해당 페이지를 RAM으로 로드 
  - **페이징(Paging)**: 프로그램을 페이지(Page)라는 고정 크기의 블록으로 분할
  - **페이지 테이블(Page Table)**: 각 페이지가 물리 메모리의 어느 위치에 있는지 매핑하는 테이블
  - **요구 페이징(Demand Paging)**: 실제로 필요한 페이지만 메모리에 로드

ex)
RAM: 8GB, 실행 프로그램: 20GB
```
프로그램 전체 20GB
   ↓
페이지 단위로 분할
   ↓
지금 필요한 2GB만 RAM에 로드
나머지 18GB는 디스크에 저장
```
필요할 때마다 페이지를 RAM <-> 디스크 간 교체하면서 실행 (Demand Paging)


### 📌 [Q2] 페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?

페이지 폴트란 **프로세스가 접근하려는 페이지가 현재 물리 메모리(RAM)에 없을 때 발생하는 인터럽트**이다.
페이지 폴트 자체는 오류가 아니라 정상적인 가상 메모리 동작 과정이다.

```
CPU: 이 메모리 주소 사용해야지
↓
MMU 확인
↓
RAM에 해당 페이지 없음
↓
Page Fault 발생
```

**발생 시 처리 과정**
```
(1) CPU가 가상 주소 접근 시도
- 프로세스가 어떤 데이터를 읽거나 쓰려고 함

(2) Page Table 확인 / Page Fault 발생
- mmu가 페이지 테이블을 확인
- **해당 페이지의 Present bit가 0이면 Page Fault 발생** (CPU가 OS에게 인터럽트 전달)

(3) OS가 디스크에서 Page 찾음
- 해당 페이지가 저장된 디스크(스왑 영역) 위치 확인

(4) RAM에 빈 Frame 확보
- 빈 프레임이 있으면 바로 로드
- 없으면 **페이지 교체 알고리즘**으로 내보낼 페이지 선택 (ex. LRU, FIFO 등)

(5) Page를 디스크에서 RAM으로 로드 & 실행 재개
- 디스크에서 해당 페이지를 RAM으로 로드
- 해당 페이지의 Present bit를 1로 변경 (Page Table 업데이트)
- Frame 번호 기록
- CPU가 다시 접근 시도 -> 이번엔 Page Fault 없이 성공

```

```
프로그램 실행
      ↓
CPU 메모리 접근
      ↓
페이지 테이블 확인
      ↓
RAM에 없음
      ↓
Page Fault 발생
      ↓
OS가 디스크에서 페이지 가져옴
      ↓
RAM에 적재
      ↓
프로그램 실행 재개
```

### 📌 [Q3] 요구 페이징(Demand Paging)은 어떻게 동작하는가?

요구 페이징이란 **프로그램을 실행할 때 모든 페이지를 메모리에 올리지 않고, 실제로 필요할 때만 메모리에 로드하는 방식**이다.

**프로그램은 보통 전체 코드와 데이터를 항상 사용하지는 않는다.** (ex. 프로그램 크기: 1GB, 실제 사용량: 100MB)
전체를 메모리에 올리면 낭비가 심함 -> **필요한 페이지만 로드**한다.

**동작 과정**
```
(1) 프로그램 시작
- 처음엔 첫 페이지만 메모리에 로드

(2) 실행 중 페이지 접근
- 필요한 페이지가 RAM에 있으면 바로 사용
- 없으면 Page Fault 발생

(3) Page Fault 처리
- OS가 디스크에서 해당 페이지를 RAM으로 로드
- Page Table 업데이트
- 프로그램 재개
```

**장점**
- 메모리 사용 효율 증가
- 프로그램 크기 제한 완화
- 다중 프로그래밍 성능 향상

### 📌 [Q4] 페이지 교체 알고리즘(FIFO, LRU, LFU, Optimal)의 차이는?

**페이지 교체 알고리즘**이란 **페이지 폴트가 발생했을 때, 어떤 페이지를 내보낼지 결정하는 알고리즘**이다.

**FIFO (First-In, First-Out)**
- 가장 먼저 들어온 페이지(메모리에 가장 오래 있었던) 제거
- 구현이 간단 & 큐로 관리 가능
- 단, 사용 중인 페이지도 제거될 수 있음
- Belady's Anomaly 발생 가능 (프레임 수가 늘어나도 오히려 페이지 폴트가 늘어나는 현상)
```
A → B → C → D
메모리 가득
새 페이지 E 필요

→ A 제거
```

**LRU (Least Recently Used)**
- 가장 오랫동안 사용되지 않은 페이지를 내보냄
- Locality 반영, 실제 프로그램 패턴과 잘 맞음
- 성능이 좋지만 구현이 복잡함
```
최근 사용
C → A → B

새 페이지 필요
→ 가장 오래 안쓴 C 제거
```

**LFU (Least Frequently Used)**
- 가장 적게 사용된 페이지를 내보냄
- 자주 사용하는 페이지 유지할 수 있음
- 카운트 관리 필요, 초기 사용 횟수 적은 페이지가 불리


**Optimal (OPT, 최적)**
- 앞으로 가장 오랫동안 사용되지 않을 페이지를 내보냄 (미래 접근을 미리 알고 있다고 가정)
- 페이지 폴트 최소화, 이론상 최적의 성능
- 단, 미래를 알 수 없어서 실제로 구현 불가능
```
앞으로 사용 순서
A B C A B D

→ 가장 늦게 사용될 페이지 제거
```

### 📌 [Q5] LRU를 실제로 구현하는 방법은? (연결 리스트 + 해시맵)

**LRU(Least Recently Used) 구현 방법**
- LRU는 **가장 오랫동안 사용되지 않은 페이지를 내보내는 알고리즘**이다.
- **이중 연결 리스트(Doubly Linked List)**와 **해시맵(HashMap)**을 함께 사용한다.
- 이 방식의 목표는 모든 연산을 O(1) 시간 복잡도로 처리하는 것이다.

**왜 해시맵 + 연결 리스트를 사용할까?**
- LRU는 **데이터 조회(get), 데이터 삽입(put)** 연산이 핵심이다.
- LRU는 아래 작업을 항상 해야 한다.
  - 최근 사용한 데이터는 앞으로 이동
  - 가장 오랫동안 사용되지 않은 데이터는 뒤로 이동(제거)
  - 이때 **키(key)**로 빠르게 찾아야 하므로 **해시맵O(1)**을 사용하고, **연결 리스트**로 사용 순서를 관리해야 한다.

```
HashMap
key → Node

Node
key
value
prev
next
```
```
Head (Most Recently Used)
 ↓
[A] ↔ [B] ↔ [C]
                 ↑
           Tail (Least Recently Used)
```

**구현 원리**
1. 해시맵(HashMap)
  - key: 페이지 번호
  - value: 연결 리스트의 node
  - **key -> node 매핑**
  - O(1) 시간 복잡도로 페이지를 빠르게 찾을 수 있음

2. 연결 리스트(Doubly Linked List)
  - 가장 최근에 사용된 페이지를 맨 앞으로
  - 가장 오랫동안 사용되지 않은 페이지를 맨 뒤로
  - **head -> tail 순서로 사용 순서 관리**
  - O(1) 시간 복잡도로 페이지를 추가, 삭제, 이동할 수 있음

**동작 과정** <br>

```
캐시 크기(Capacity)가 3이라고 가정하고, 데이터 A -> B -> C -> B -> D 순서로 입력될 때의 과정입니다.

[1단계: 빈 캐시에 데이터 추가]
입력: A
상태: [A]
설명: 캐시가 비어 있으므로 A를 맨 앞에 추가합니다.

[2단계: 데이터 추가 (계속)]
입력: B
상태: [B, A]
설명: 새로운 데이터 B를 맨 앞에 추가합니다. (A는 뒤로 밀려남)

[3단계: 캐시가 가득 참]
입력: C
상태: [C, B, A]
설명: C를 맨 앞에 추가합니다. 이제 캐시 용량 3이 모두 찼습니다.

[4단계: 기존 데이터 재참조 (Cache Hit)]
입력: B
상태: [B, C, A]
설명: 캐시에 이미 B가 있습니다. B를 가장 최근에 사용된 것으로 간주하여 맨 앞으로 이동시킵니다.

[5단계: 새로운 데이터 추가 및 교체 (Cache Miss & Eviction)]
입력: D
상태: [D, B, C]
설명: 캐시가 가득 찼는데 새로운 데이터 D가 들어옵니다. 이때 가장 오랫동안 사용되지 않은(맨 뒤에 있는) A를 제거하고 D를 맨 앞에 추가합니다.
```

**정리** <br>
LRU는 보통 HashMap과 Doubly Linked List를 함께 사용해 구현합니다. HashMap을 통해 O(1)로 데이터를 찾고, 연결 리스트를 통해 최근 사용 순서를 관리합니다. 데이터에 접근하면 해당 노드를 리스트의 head로 이동시키고, 캐시가 가득 찬 경우 tail에 있는 가장 오래 사용되지 않은 노드를 제거합니다.


### 📌 [Q6] 스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?

스레싱은 **하드디스크의 페이지 교체 횟수가 너무 많아져서, CPU가 실제 프로세스를 실행하는 시간보다 페이지를 교체하는 데 더 많은 시간을 소비하는 현상**이다.

**발생 원인**
- 주로 다중 프로그래밍 환경에서 발생
- **페이지 폴트가 너무 자주 발생**
- **메모리가 부족해서 페이지를 계속 교체해야 함**
- **CPU가 실제 작업을 하지 못하고 페이지 교체만 반복** (페이지를 하드디스크에서 가져오는 동안 CPU는 계속 대기하게 되고, 이로 인해 CPU 이용률이 급감한다. OS는 이를 보고 CPU가 놀고 있다고 착각해 새로운 프로세스를 메모리에 올리려고 한다. 그러면 메모리가 더 부족해지고, 페이지 폴트가 더 자주 발생한다. = 악순환)

**해결 방안** <br>
-> 프로세스가 필요한 만큼의 프레임을 충분히 할당해주면 된다.
- **Working Set Model(워킹 셋 모델)**
  - 프로세스가 일정 시간동안 자주 참조하는 페이지들의 집합을 '워킹 셋'으로 정의한다.
  - 이 워킹 셋이 한꺼번에 메모리에 올라갈 수 있을 때만 프로세스를 실행한다.
  - 만약 전체 워킹 셋을 수용할 메모리가 부족하면, 아예 어떤 프로세스를 통째로 swap-out 시켜서 다른 프로세스들에게 충분한 공간을 몰아준다.
- **Page Fault Frequency(페이지 폴트 빈도)**
  - 페이지 폴트 발생 빈도를 모니터링하여 프레임 할당향을 직접 조절하는 방식
  - 페이지 폴트가 너무 잦으면 프레임을 더 할당하고(해당 프로세스에 프레임이 너무 적다는 뜻), 너무 적으면 프레임을 줄인다.(프레임이 낭비되고 있는 상황)
- **구조적 해결**
  - RAM 용량 증설

### 📌 [Q7] Working Set Model이란 무엇인가?

**Working Set Model(워킹 셋 모델)** <br>
  - 프로세스가 일정 시간동안 자주 참조하는 페이지들의 집합을 '워킹 셋'으로 정의한다.
  - 이 워킹 셋이 한꺼번에 메모리에 올라갈 수 있을 때만 프로세스를 실행한다.
  - 만약 전체 워킹 셋을 수용할 메모리가 부족하면, 아예 어떤 프로세스를 통째로 swap-out 시켜서 다른 프로세스들에게 충분한 공간을 몰아준다.
