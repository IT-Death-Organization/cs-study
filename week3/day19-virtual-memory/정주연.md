## 🧠 학습 질문

## 가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?

- 이유
    - RAM은 한정돼 있는데, 프로그램은 그보다 더 큰 주소 공간을 쓰려고 함.
    - 여러 프로세스를 동시에 돌리려면 각자 **독립적인 주소 공간**이 필요함.
- 원리
    - CPU는 “가상 주소”만 보고 실행함.
    - 하드웨어(MMU)가 가상 주소를 “페이지 단위”로 실제 물리 메모리 또는 디스크(스왑 공간)에 매핑함.
    - 실제로는 **프로그램 전체를 RAM에 올리지 않고, 당장 필요한 페이지만 RAM에 올리고 나머지는 디스크에 두는 방식**이라, 물리 메모리보다 큰 프로그램도 부분부분 나눠서 돌릴 수 있음.

## 페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?

- 정의
    - CPU가 어떤 가상 주소(페이지)에 접근했는데, 그 페이지가 **현재 RAM에 없는 경우** 발생하는 예외임.
- 처리 과정 (간단 흐름)
    1. CPU가 메모리 접근 → MMU가 페이지 테이블을 봄.
    2. 해당 페이지가 “메모리에 없음(not present)”이면 페이지 폴트 예외 발생, 커널로 트랩.
    3. OS가 디스크(스왑/파일)에서 그 페이지를 읽어서 RAM의 빈 프레임에 적재.
    4. 페이지 테이블 갱신 (해당 가상 페이지 → 새 물리 프레임).
    5. 예외를 일으킨 명령을 다시 실행 → 이번엔 정상적으로 메모리 접근 성공.

## 요구 페이징(Demand Paging)은 어떻게 동작하는가?

- 개념
    - “필요한 페이지만 그때그때 메모리에 올리는 방식”임.
- 동작
    - 프로세스 전체를 한 번에 올리지 않고, 처음에는 거의 안 올린 상태로 시작.
    - 코드 실행 중 어떤 페이지가 처음 필요해질 때, 그 순간 페이지 폴트가 발생하고, 그때 디스크에서 가져와 RAM에 올림.
    - 자주 쓰는 페이지만 RAM에 남고, 덜 쓰는 페이지는 나중에 교체되어 나감.
- 장점: 메모리를 덜 쓰고, 더 많은 프로세스를 동시에 돌릴 수 있음.

## 페이지 교체 알고리즘(FIFO, LRU, LFU, Optimal)의 차이는?

RAM이 꽉 찼는데 새 페이지를 가져와야 할 때, “어떤 페이지를 내보낼지”를 정하는 알고리즘임.

- FIFO (First-In First-Out)
    - 가장 먼저 들어온 페이지를 가장 먼저 내보냄.
    - 구현 쉬움, 하지만 “오래 있었다고 해서 덜 사용된다는 보장은 없음”(Belady anomaly 등).
- LRU (Least Recently Used)
    - “가장 오래 동안 사용되지 않은” 페이지를 내보냄.
    - 최근에 사용된 페이지는 앞으로도 쓸 가능성이 높다는 가정.
    - 실제 접근 시간 패턴과 꽤 잘 맞아서 성능 좋음, 하지만 구현이 약간 비쌈.
- LFU (Least Frequently Used)
    - 사용 횟수가 가장 적은 페이지를 교체.
    - “자주 쓰인 페이지는 앞으로도 쓸 것”이라는 관점.
    - 오래전에 한 번 몰아서 쓰이고, 최근엔 안 쓰이는 페이지도 남아 있을 수 있어 한계가 있음.
- Optimal (OPT, Belady’s optimal)
    - 앞으로 “가장 오랫동안 사용되지 않을” 페이지를 내보냄.
    - 미래를 알아야 하므로 현실에서는 구현 불가, 이론적인 **최적 기준**으로만 사용(다른 알고리즘 성능 비교용).

## LRU를 실제로 구현하는 방법은? (연결 리스트 + 해시맵)

캐시/LRU 구현에서 자주 나오는 패턴임.

- 요구사항
    - “어떤 key의 페이지에 O(1)로 접근”하고,
    - “가장 오래 안 쓰인 페이지를 O(1)로 제거”해야 함.
- 구조
    1. **이중 연결 리스트(doubly linked list)**
        - 리스트의 앞쪽: 최근에 사용된 페이지.
        - 리스트의 뒤쪽: 가장 오래 사용되지 않은 페이지.
    2. **해시맵(HashMap: key → 해당 노드 포인터)**
        - key(페이지 번호 등)를 바로 리스트 노드에 매핑해서 찾기 O(1).
        - 동작
            - 어떤 페이지를 접근하면
                - 해시맵으로 노드를 찾고, 해당 노드를 리스트 맨 앞으로 이동.
            - 새 페이지를 넣어야 하고 캐시가 꽉 찼으면
                - 리스트 맨 뒤 노드를 제거(가장 오래 안 쓰인 것), 해시맵에서도 삭제.
        - 이렇게 하면 get/put 모두 평균 O(1)에 LRU를 구현할 수 있음.

## 스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?

- 정의
    - CPU가 실제 일은 거의 못 하고, **페이지 폴트 처리 + 디스크 스왑에 대부분의 시간을 쓰는 상태**임.
- 왜 발생?
    - 메모리에 올라온 페이지들이 너무 적절하지 않아서, 프로그램이 필요로 하는 working set을 충분히 담지 못할 때.
    - 다수의 프로세스를 과도하게 동시에 돌려서, 각 프로세스에 할당된 프레임 수가 너무 적을 때.
- 결과: 페이지 폴트 빈도가 폭증, 디스크 I/O 폭주, CPU 사용률 떨어짐.
- 해결 방법
    - 동시에 돌리는 프로세스 수를 줄여서 각 프로세스당 프레임을 더 많이 줌.
    - 물리 메모리를 늘리거나, 메모리 사용량이 큰 작업을 조절.
    - Working set 기반 할당, 페이지 교체 정책 튜닝 등으로 “자주 쓰는 페이지”를 충분히 메모리에 유지.

## Working Set Model이란 무엇인가?

- 개념
    - 어떤 시점 t에서 한 프로세스가 일정 시간 구간 Δ 동안 **실제로 참조한 페이지들의 집합**을 그 프로세스의 working set이라고 부름.
    - 이 집합이 “지금 이 프로세스가 원활히 실행되기 위해 메모리에 있어야 할 페이지들”이라고 보는 모델임.
- 목적
    - 각 프로세스의 working set 크기만큼 프레임을 보장해 주면 페이지 폴트가 줄어들고, 스래싱을 피할 수 있음.
    - OS가 “working set 추정 + 그에 따른 프레임 할당 조절”을 통해, 메모리 압박이 심하면 일부 프로세스를 잠시 swap out하는 등의 정책을 세울 수 있음.
