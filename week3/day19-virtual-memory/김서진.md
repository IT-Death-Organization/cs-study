## 1.  가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?
필요성: 물리 메모리(RAM)의 크기는 물리적 한계가 명확하다. 가상 메모리는 프로세스 전체가 메모리에 올라오지 않아도 실행 가능하게 함으로써 메모리 관리 효율성을 극대화하고, 시스템이 동시에 실행할 수 있는 프로세스의 수(Degree of Multiprogramming)를 늘려준다. 또한, 메모리 보호 및 공유가 용이해진다.

원리: **가상 주소 공간(Virtual Address Space)**을 사용하여 논리 주소와 물리 주소를 분리한다. 프로그램은 연속된 거대한 메모리 공간을 사용하는 것처럼 인식하지만, 실제로는 당장 필요한 부분(Page)만 물리 메모리에 적재하는 부분 로딩(Partial Loading) 방식을 취한다. 나머지 데이터는 디스크의 **스왑 영역(Swap Space)**에 저장해 두었다가 필요 시 교체하며 실행한다.

---

## 2.  페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?
페이지 폴트란 CPU가 참조하려는 페이지가 현재 물리 메모리에 적재되어 있지 않아, 페이지 테이블의 Valid-Invalid Bit가 'Invalid'인 상태를 의미한다.

처리 과정:

Trap 발생: CPU가 주소 변환을 시도하다가 Invalid bit를 확인하면 운영체제에 트랩(소프트웨어 인터럽트)을 건다.

커널 모드 전환: 운영체제는 해당 참조가 유효한지(잘못된 접근인지 아니면 단순히 메모리에 없는 것인지) 확인한다.

데이터 탐색: 디스크(스왑 영역)에서 해당 페이지를 찾는다.

프레임 확보: 물리 메모리 내 빈 프레임을 찾는다. 빈 프레임이 없다면 페이지 교체 알고리즘을 통해 대상(Victim Page)을 선정하여 디스크로 쫓아낸다.

I/O 수행: 디스크에서 메모리로 페이지를 읽어온다. 이 동안 해당 프로세스는 대기 상태(Wait)가 된다.

테이블 갱신: 로딩이 끝나면 페이지 테이블을 업데이트하고 Valid bit로 변경한다.

명령어 재시작: 중단되었던 명령어를 처음부터 다시 실행한다.

## 3.  요구 페이징(Demand Paging)은 어떻게 동작하는가?
3. 요구 페이징(Demand Paging)은 어떻게 동작하는가?
프로그램 실행 시 프로세스 전체를 메모리에 올리지 않고, **실제로 참조되는 페이지가 요청될 때(On-demand)**만 메모리에 적재하는 기법이다.

Lazy Swapper: 페이지가 필요하지 않으면 절대로 메모리에 올리지 않는 전략을 사용한다.

Pure Demand Paging: 극단적으로는 단 한 페이지도 메모리에 올리지 않고 시작할 수 있다. 이 경우 실행 직후 연속적인 페이지 폴트가 발생하며 필요한 데이터를 점진적으로 채워 나간다.

장점: 입출력 양 감소, 메모리 사용량 절약, 응답 시간 단축, 시스템 처리량 증가를 기대할 수 있다.

---

## 4.  페이지 교체 알고리즘(FIFO, LRU, LFU, Optimal)의 차이는?
알고리즘,결정 기준,특징 및 한계
FIFO: 먼저 들어온 순서,
"구현이 매우 간단하지만, Belady's Anomaly(프레임이 늘어나도 폴트가 증가하는 현상)가 발생할 수 있다."

Optimal: 미래에 가장 늦게 쓰일 것
"이론상 최저의 페이지 폴트율을 보장하지만, 미래의 참조 순서를 알 수 없으므로 실제 구현은 불가능하다."

LRU: 가장 오랫동안 안 쓰인 것
**시간 지역성(Temporal Locality)**에 기반한다. 성능이 매우 우수하여 실무에서 가장 선호되는 방식이다.

LFU: 참조 횟수가 가장 적은 것
페이지의 인기도를 측정한다. 초기에만 반짝 많이 쓰이고 이후 안 쓰이는 페이지가 메모리에 계속 남는 단점이 있다.

---

## 5.  LRU를 실제로 구현하는 방법은? (연결 리스트 + 해시맵)
단순한 리스트로 구현 시 탐색에 O(n)이 소요되어 오버헤드가 크다. 이를 해결하기 위해 두 자료구조를 조합한다.이중 연결 리스트(Doubly Linked List): 페이지의 순서를 유지한다. 가장 최근에 사용된 페이지는 Head에, 가장 오래된 페이지는 Tail에 위치시킨다. 노드 삽입/삭제가 O(1)이다.해시맵(Hash Map): 특정 페이지 번호를 Key로, 해당 노드의 주소를 Value로 저장한다. 이를 통해 특정 페이지가 리스트 내 어디에 있는지 $O(1)$로 즉시 찾아낸다.동작: 페이지 참조 시 해시맵에서 위치를 찾고(O(1)), 해당 노드를 리스트의 Head로 옮긴다(O(1)). 메모리가 가득 차면 Tail의 노드를 삭제한다.

## 6.  스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?
스래싱은 프로세스가 실제 실행 시간보다 페이지 교체에 따른 I/O 작업에 더 많은 시간을 소비하여 시스템의 전체 처리량이 급격히 저하되는 상태를 말한다.

발생 원인: 너무 많은 프로세스를 동시에 실행(MPD 과도화)하여, 개별 프로세스에 할당된 프레임 수가 부족해질 때 발생한다. 페이지 폴트가 빈번해지면 CPU 이용률이 낮아지고, OS는 이를 보고 더 많은 프로세스를 투입하는 악순환에 빠진다.

해결 방법:

Degree of Multiprogramming 조절: 동시 실행 프로세스 수를 줄인다.

Working Set Model: 프로세스가 일정 시간 동안 필요로 하는 페이지 집합을 한꺼번에 할당한다.

PFF(Page Fault Frequency): 페이지 폴트 빈도의 상한선과 하한선을 두어, 프레임 수를 동적으로 조절한다.

---
## 7.  Working Set Model이란 무엇인가?
프로세스가 일정 시간 동안 특정 주소 영역을 집중적으로 참조하는 지역성(Locality) 원리를 이용한 모델이다.정의: 시각 $t$에서 최근 $\Delta$(윈도우 크기) 시간 동안 참조된 페이지들의 집합을 Working Set이라 정의한다.운영 방식: 운영체제는 각 프로세스의 Working Set 크기만큼 프레임이 확보될 때만 해당 프로세스에 메모리를 할당한다. 만약 남는 프레임이 Working Set 총합보다 작다면, 일부 프로세스를 Swap-out 시켜 스래싱을 방지한다.효과: 다중 프로그래밍의 정도를 동적으로 제어하며 시스템 전체의 안정성을 유지한다.
