# 1. 가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?

> 가상 메모리는 프로세스 전체가 메모리에 올라오지 않아도 실행이 가능하도록 하는 메모리 관리 기법입니다. 물리 메모리(RAM)를 논리적으로 추상화를 하는데, 쉽게 말해 사용자에게 큰 메모리가 있는것처럼 속이는 것입니다.

### 가상 메모리가 필요한 이유
한정된 물리 메모리를 가지고 그보다 더 큰 메모리를 요구하는 프로그램을 돌릴 수 있게 해줍니다.

각 프로세스에게 독립된 가상 주소 공간을 줘서 다른 프로세스의 메모리 영역을 침범하지 못하게 막아서 보호와 격리를 실현시킵니다.

프로그램의 일부만 메모리에 올리기 때문에 남는 공간을 활용해서 더 많은 프로세스를 동시에 띄웁니다.

### 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리
부분 적재(Partial Loading)와 지역성(Locality) 덕분입니다.

프로그램이 실행돼도 모든 코드가 다 쓰이진 않습니다. 당장 사용중인 코드만 메모리에 올리고, 사용하지 않는 코드는 보조 기억 장치(Disk/SSD)의 스왑 영역에 둡니다.

CPU는 가상 주소(Logical Address)를 바라보고 명령을 내립니다. 그리고 하드웨어(MMU)가 그때그때 이 가상주소를 실제 RAM의 주소로 매핑해줍니다. 만약 요청한 주소의 데이터가 RAM에 없으면 그때 보조 기억 장치에서 읽어오는 방법을 사용합니다.

<br>

# 2. 페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?

> 위에서 말한 "만약 요청한 주소의 데이터가 RAM에 없으면" 이라는 상황이 바로 페이지 폴트(Page Fault)입니다.

정확히 말하면, 페이지 폴트란 프로세스가 접근하려고 하는 페이지가 현재 물리 메모리(RAM)에 올라와 있지 않아, 하드웨어(MMU)가 OS에 알리는 인터럽트입니다.

### 페이지 폴트의 처리 과정 6단계
1) Trap 발생
    * CPU가 페이지 테이블을 확인했는데 '유효 비트(Valid bit)'가 0(Invalid)이라면, MMU는 운영체제에 트랩(Trap)을 걸어 제어권을 넘깁니다.

2) 상태 보존
    * OS는 현재 실행 중인 프로세스의 상태(레지스터, PC 등)를 잠시 보관합니다.

3) 디스크 탐색
    * OS는 내부적인 페이지 테이블(Backing Store 위치 정보 등)을 확인하여 해당 페이지가 하드디스크의 어느 위치에 있는지 찾아냅니다.

4) 페이지 적재 (Page-in)
    * 비어있는 물리 메모리 공간(프레임)을 찾아 디스크에서 해당 페이지를 읽어옵니다. 그런데 약 빈 프레임이 없다면, 페이지 교체 알고리즘을 통해 누군가를 쫒아냅니다.

5) 테이블 업데이트
    * 페이지가 성공적으로 RAM에 올라오면, 페이지 테이블의 해당 엔트리를 업데이트하고 유효 비트를 1(Valid)로 바꿉니다.

6) 명령어 재실행
    * 아까 중단되었던 명령어를 다시 실행하면, 이번에는 페이지 폴트 없이 데이터에 접근할 수 있습니다.

# 3. 요구 페이징(Demand Paging)은 어떻게 동작하는가?

> 요구 페이징은 말 그대로 요구가 있을 때만 페이지를 메모리에 적재하는 기법입니다. 프로그램 실행 시 모든 데이터를 RAM에 올리지 않고, 실제로 참조되는 부분만 가져오는 방식입니다.

### 동작 원리
요구 페이징은 게으른 스와퍼(Lazy Swapper)를 사용합니다.
* 프로세스 실행 시, 당장 필요한 페이지만 메모리에 올림
* 나머지 페이지들은 디스크(Backing Store)에 그대로 둠
* 실행 중 아직 메모리에 없는 페이지를 참조하려 할 때, 페이지 폴트를 발생 -> 해당 페이지를 메모리에 적재

극단적으로 첫 번째 명령어를 실행할 때조차 아무런 페이지를 올리지 않고 시작할 수도 있습니다. 시작부터 페이지 폴트가 발생해서 초기엔 좀 느리지만, 점차 안정화됩니다.

### 장점
사용하지 않는 코드는 메모리에 두지 않기 때문에 메모리를 절약할 수 있습니다.

전체 프로그램을 올릴 때까지 기다릴 필요가 없어서 응답 속도가 빨라집니다.

메모리 절약으로 인해 더 많은 프로세스를 동시에 실행할 수 있습니다.

이러한 모든 점은 프로그램이 가지는 "지역성(Locality)" 덕분입니다. 만약 프로그램이 지역성을 가지지 않는다면 역효과가 날 것입니다.

<br>

# 4. 페이지 교체 알고리즘(FIFO, LRU, LFU, Optimal)의 차이는?

> 메모리가 꽉 찼을 때 어떤 페이지를 디스크로 내보낼지에 대한 결정은 가상 메모리 성능의 핵심입니다. 페이지 폴트를 줄이기 위한 고민이 필요합니다.

### 1) FIFO (First-in, First-out)

* **원리**
    * 메모리에 가장 먼저 들어온 페이지를 가장 먼저 내보냅니다.

* **특징**
    * 구현이 간단하고 오버헤드가 적습니다.

* **문제점**
    * Belady의 역설(Belady's Anomaly) 현상이 발생할 수 있습니다. 즉, 프레임(메모리 공간)을 늘려줬는데 오히려 페이지 폴트가 더 많이 발생하는 현상입니다.

### 2) Optimal (최적 교체)

* **원리**
    * 앞으로 가장 오랫동안 사용되지 않을 페이지를 찾아 교체합니다.

* **특징**
    * 모든 알고리즘 중 페이지 폴트가 가장 적은 이론적 최적해입니다.

* **문제점**
    * 미래에 어떤 페이지가 쓰일지 미리 알아야 하므로 구현이 불가능합니다. 주로 다른 알고리즘의 성능을 비교하는 벤치마크용(Upper bound)으로 쓰입니다.

### 3) LRU (Least Recently Used)

* **원리**
    * 가장 오랫동안 사용되지 않은 페이지를 교체합니다.

* **특징**
    * 최근에 사용한 데이터가 또 곧바로 사용될 가능성이 높다는 시간 지역성(Temporal Locality)에 기반합니다.
    * 실무에서 가장 많이 쓰이며, Optimal에 근접한 좋은 성능을 보여줍니다.

### 4) LFU (Least Frequently Used)

* **원리**
    * 참조 횟수가 가장 적은 페이지를 교체합니다.

* **특징**
    * 활발하게 사용되는 페이지는 남겨두겠다는 전략입니다.

* **문제점**
    * 초반에만 반짝 많이 쓰이고 나중엔 안 쓰이는 데이터가 메모리에 계속 남는 '과거의 영광' 문제가 발생할 수 있으며, 참조 횟수를 기록하는 오버헤드가 큽니다.

<br>

# 5. LRU를 실제로 구현하는 방법은? (연결 리스트 + 해시맵)

> LRU는 최근에 사용한 데이터들을 남겨놔야 하기 때문에, 데이터 접근 시 순서를 맨 앞으로 옮기는 작업과 가장 오래된 데이터를 찾고 지우는 작업을 빠르게 수행해야 합니다. 이 작업들을 수행하기 위해 이중 연결 리스트(Doubly Linked List)와 해시맵(Hashmap)을 결합해 사용합니다.

### 해시맵의 역할
특정 데이터가 어디에 있는지 바로 찾기($O(1)$) 위함입니다.

Key는 데이터의 식별자, Value는 연결 리스트의 노드 주소를 가리킵니다.

### 이중 연결 리스트의 역할
데이터의 순서를 관리합니다. 가장 최근에 사용한 데이터는 머리쪽에, 가장 오래된 데이터는 꼬리쪽에 배치합니다.

특정 노드를 삭제하거나 위치를 옮길 때 앞뒤 링크만 바꿔주면 되므로 $O(1)$ 시간에 수행할 수 있습니다.

### 동작
1. 데이터 조회 (GET)
    * 해시맵에서 해당 데이터를 찾습니다.
    * 데이터가 있다면, 연결 리스트에서 해당 노드를 맨 앞(머리)쪽으로 이동시킵니다.

2. 데이터 삽입 (PUT)
    * 이미 있는 경우: 값을 업데이트하고 맨 앞으로 옮깁니다.
    * 없어서 새로 추가하는 경우:
        * 만약 메모리가 꽉 차있다면 연결 리스트의 맨 뒤(꼬리)에 있는 노드를 삭제하고 해시맵에서도 지웁니다.
        * 새 노드를 만들어 맨 앞(머리)에 추가하고 해시맵에 등록합니다.

<br>

이것이 Java의 LinkedHashMap이 내부적으로 동작하는 원리이기도 합니다.

# 6. 스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?

> 스래싱이란 프로세스가 실제 명령을 실행하는 시간보다 페이지를 교체(Swap-in/out)하는데 더 많은 시간을 소비햐여, 시스템 전체 성능(CPU 이용률)이 급격하게 떨어지는 현상을 말합니다.

### 1) 발생 과정

1. 다중 프로그래밍 정도(Degree of Multiprogramming)의 과도한 증가
    * 너무 많은 프로세스를 동시에 실행시킵니다.

2. 프레임 부족
    * 개별 프로세스가 원활하게 돌아가기 위해 필요한 최소한의 메모리 조각(프레임)을 할당받지 못하게 됩니다.

3. 페이지 폴트 폭발
    * 각 프로세스가 실행될 때마다 필요한 페이지가 없어 빈번하게 페이지 폴트가 발생합니다.

4. CPU 대기
    * CPU는 데이터를 읽어오는 디스크 I/O 작업이 끝날 때까지 기다립니다.

5. OS의 오해
    * OS는 CPU가 노는 것을 보고 할 일이 없는 줄 알고 프로세스를 더 띄웁니다.

6. 시스템 마비
    * 메모리 부족은 심화되고, CPU는 하루 종일 페이지 교체만 하다가 시스템이 사실상 멈추게 됩니다.

### 2) 해결 방법

* 다중 프로그래밍 정도 조절
    * 실행 프로세스 수를 줄여 메모리를 여유롭게 합니다.

* 워킹 셋(Working Set) 모델
    * 프로세스가 특정 시간 동안 자주 참조하는 페이지 묶음을 미리 파악하여, 그 묶음만큼의 공간이 확보될 때만 프로세스를 실행시킵니다.

* Page Fault Frequency 조절
    * 페이지 폴트 발생 빈도를 모니터링 후, 폴트가 너무 많다면 프레임을 더 주고, 너무 적으면 프레임을 줄여서 균형을 맞춥니다.

<br>

# 7. Working Set Model이란 무엇인가?

> 워킹 셋 모델은 프로세스가 일정 시간 동안 자주 참조하는 페이지들의 집합(Working Set)을 정의하고, 이 집합이 메모리에 한꺼번에 올라갈 수 있을 때만 프로세스를 실행시키는 모델입니다.

### 지역성에 기반한 설계
프로세스는 실행되는 동안 메모리의 특정 영역을 집중적으로 참조하는 경향이 있습니다.

* 워킹 셋 윈도우($\Delta$)
    * 초근 얼마만큼의 시간 동안 참조된 페이지들을 워킹 셋으로 묶을 지 결정하는 고정된 시간 범위

* Working Set(WS)
    * 최근 $\Delta$ 시간 동안 참조된 서로 다른 페이지들의 집합

### 스래싱 방지 매커니즘
운영체제는 모든 활성 프로세스의 워킹 셋 크기 합($\sum WSS_i$)을 모니터링합니다.

만약 이 합이 전체 RAM의 크기보다 크다면, OS는 프로세스 하나를 통째로 중단 시키고 해당 메모리를 다른 프로세스들에게 나눠줍니다.

반대라면 여유가 있다고 판단하여 새로운 프로세스를 추가로 실행시킵니다.

### 장단점
* 장점: 스래싱을 효과적으로 방지하며 CPU 이용률을 최적으로 유지합니다.
* 단점: 워킹 셋을 지속적으로 모니터링하고 업데이트하는 과정 자체가 시스템에 오버헤드를 줄 수 있습니다.

<br>

어설프게 많은 프로세스를 한꺼번에 실행시키기보다 감당 가능한 몇개를 확실히 끝내겠다는 철학이 담겨있는 느낌입니다.
