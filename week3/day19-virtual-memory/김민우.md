## 1. 가상 메모리가 필요한 이유

가상 메모리는 **프로그램이 실제 RAM보다 큰 메모리를 사용하는 것처럼 보이게 하는 메커니즘**이다.

### 필요 이유

**1) 프로그램 크기가 RAM보다 클 수 있음**

- RAM = 8GB, 프로그램 = 20GB인 경우 전부 RAM에 올릴 수 없다.
- 가상 메모리는 **디스크를 RAM의 확장처럼 사용**한다.

**2) 메모리 보호 (Memory Protection)**

- 각 프로세스는 **독립적인 가상 주소 공간**을 가진다.
- 예: Process A와 B 모두 `0x0000 ~ 0xFFFF`를 사용하지만, 실제로는 **다른 물리 메모리**에 매핑된다.
- 다른 프로세스의 메모리 침범을 방지한다.

**3) 메모리 효율 향상**

- 프로그램 전체를 RAM에 올릴 필요 없이, 실제로 사용하는 부분만 로드한다.
- -> **Demand Paging**

### 핵심 아이디어

```
Virtual Address -> Page Table -> Physical Address
```

- CPU는 **가상 주소**를 사용한다.
- MMU가 **물리 주소로 변환**한다.

---

## 2. 물리 메모리보다 큰 프로그램 실행 원리

핵심: **Paging + Disk**

프로그램을 **Page 단위**로 나누고, RAM에는 일부만 올린다.

```
프로그램: [Page0][Page1][Page2][Page3][Page4]
RAM:      [Page0][Page3]
Disk:     [Page1][Page2][Page4]
```

필요한 페이지만 RAM으로 가져온다 -> **Demand Paging**

---

## 3. Page Fault

### 정의

프로세스가 **RAM에 없는 페이지에 접근할 때 발생하는 인터럽트**

### 발생 과정

1. CPU가 가상 주소 접근
2. MMU가 Page Table 확인 -> `Present bit = 0` (RAM에 없음)
3. **Page Fault** 발생, OS 개입

### Page Fault 처리 과정

1. Page Fault trap 발생
2. OS가 디스크에서 페이지 찾음
3. RAM의 빈 프레임 확인
4. 없으면 **Page Replacement** 수행
5. 디스크 -> RAM 로드
6. Page Table 갱신
7. 명령어 재실행

---

## 4. Demand Paging

### 정의

페이지를 **필요할 때만 메모리에 로드하는 방식**

### 동작

```
초기 상태: RAM 비어있음 / Disk에 프로그램 전체 존재
CPU -> Page0 접근 -> Page Fault -> Disk에서 RAM으로 Page0 로드
```

### 특징

| 구분 | 내용 |
|------|------|
| 장점 | 메모리 절약, 큰 프로그램 실행 가능 |
| 단점 | Page Fault 발생 시 디스크 I/O로 인해 매우 느림 |

## 5. 페이지 교체 알고리즘

RAM이 꽉 찼을 때 **어떤 페이지를 제거할지 결정**하는 알고리즘

### (1) FIFO (First In First Out)

- 가장 먼저 들어온 페이지를 제거한다.
- 단점: **Belady's anomaly** - 페이지 프레임 수를 늘려도 Page Fault가 증가하는 현상이 발생할 수 있다.

### (2) LRU (Least Recently Used)

- **가장 오래 사용되지 않은 페이지**를 제거한다.
- 실제 locality를 반영하여 좋은 성능을 보인다.
- 단점: 구현 비용이 크다.

### (3) LFU (Least Frequently Used)

- **사용 횟수가 가장 적은 페이지**를 제거한다.
- 단점: 과거 사용 기록에 의존하여 최근성을 반영하지 못한다.

### (4) Optimal

- **앞으로 가장 늦게 사용될 페이지**를 제거한다 (미래를 안다고 가정).
- 이론적 최적이지만 실제 구현은 불가능하다.
- 다른 알고리즘 성능 비교의 기준으로 사용된다.

### 알고리즘 비교

| 알고리즘 | 기준 | 특징 |
|----------|------|------|
| FIFO | 적재 순서 | 구현 간단, Belady's anomaly |
| LRU | 최근 사용 시점 | 성능 우수, 구현 비용 높음 |
| LFU | 사용 빈도 | 과거 기록 의존 |
| Optimal | 미래 사용 시점 | 이론적 최적, 구현 불가 |

## 6. LRU 구현 방법

### HashMap + Doubly Linked List

- `get`: O(1), `put`: O(1)

### 구조

```
HashMap: key -> node (O(1) lookup)
LinkedList: 최근 사용 순서 유지 (Head = most recent, Tail = least recent)
```

### 동작 예시

```
현재 상태: Head - [4] <-> [2] <-> [1] - Tail

get(1) 호출

결과: Head - [1] <-> [4] <-> [2] - Tail
```

캐시가 꽉 찬 경우 -> Tail 노드 제거 (eviction)

### HashMap + LinkedList를 쓰는 이유

- HashMap: O(1) 노드 조회
- LinkedList: O(1) 노드 삭제 및 이동

## 7. Thrashing

### 정의

페이지 교체가 너무 많이 발생해 **CPU보다 디스크 작업이 많아지는 현상**

### 상황

```
RAM 부족 -> Page Fault -> Disk I/O -> Page Fault -> Disk I/O -> ...
CPU 사용률 급감
```

### 원인

- 실행 중인 프로세스 수가 너무 많음
- Working Set 크기 합계 > RAM 용량

### 해결 방법

1. **프로세스 수 감소** - 멀티프로그래밍 정도를 줄인다.
2. **Working Set 기반 메모리 할당** - 각 프로세스의 Working Set에 맞게 메모리를 할당한다.
3. **Local Replacement** - 프로세스별로 페이지 교체를 독립적으로 수행한다.

---

## 8. Working Set Model

### 정의

프로세스가 **최근 일정 시간 동안 사용한 페이지 집합**

### 표기

```
W(t, Delta)

t 시점에서 최근 Delta 시간 동안 사용한 페이지 집합
```

### 예시

```
최근 접근 순서: 1 2 3 4 2 3
Working Set: {1, 2, 3, 4}
```

### 의미

- Working Set에 속한 페이지들이 RAM에 있어야 Page Fault가 적게 발생한다.

### Thrashing 방지

```
sum(Working Set 크기) > RAM 용량
```

위 조건이면 OS가 일부 프로세스를 suspend한다.
