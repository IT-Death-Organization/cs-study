## 1. 가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?

**메모리 관리의 추상화와 효율성**
- 가상 메모리는 프로세스 전체가 메모리에 올라오지 않아도 실행이 가능하도록 하여 물리 메모리 크기의 한계를 극복하기 위해 필요하다.
- **실행 원리:** 각 프로세스에게는 독립적이고 연속적인 가상 주소 공간을 제공하고, 실제 실행에 필요한 부분만 물리 메모리에 올린다. 나머지는 디스크의 스왑 영역(Swap Area)에 보관하며, CPU가 가상 주소를 참조할 때마다 MMU가 이를 물리 주소로 변환하여 접근한다. 이를 통해 물리 메모리보다 용량이 큰 프로그램도 마치 전체가 메모리에 있는 것처럼 속여서 실행할 수 있다.

---

## 2. 페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?

**정의 및 처리 메커니즘**
- CPU가 참조하려는 페이지가 현재 물리 메모리에 없어 페이지 테이블의 유효 비트가 'Invalid'인 상태를 말한다.
- **처리 과정:**
  1. CPU가 페이지를 참조할 때 페이지 테이블에서 해당 페이지가 메모리에 없음을 확인하고 트랩(Trap)을 발생시킨다.
  2. 운영체제가 제어권을 넘겨받아 디스크의 스왑 영역에서 해당 페이지를 찾는다.
  3. 물리 메모리 내의 빈 프레임(Free Frame)을 찾는다. (빈 프레임이 없으면 교체 알고리즘 실행)
  4. 디스크에서 찾은 페이지를 메모리의 빈 프레임에 적재한다.
  5. 페이지 테이블의 해당 항목을 물리 주소로 업데이트하고 유효 비트를 'Valid'로 설정한다.
  6. 중단되었던 프로세스의 명령어를 재개한다.



---

## 3. 요구 페이징(Demand Paging)은 어떻게 동작하는가?

**지연 로딩(Lazy Loading) 방식**
- 프로세스 실행 시 모든 페이지를 한꺼번에 메모리에 올리지 않고, **실제로 해당 페이지가 필요할 때(요청될 때)**만 메모리에 적재하는 방식이다.
- 이를 통해 메모리 낭비를 줄이고 초기 응답 속도를 향상시키며, 더 많은 프로세스를 메모리에 동시에 수용할 수 있다. 페이지 폴트가 발생할 때만 디스크에서 데이터를 읽어오므로 전체적인 시스템 자원 효율성을 높인다.

---

## 4. 페이지 교체 알고리즘(FIFO, LRU, LFU, Optimal)의 차이는?

- **FIFO (First-In-First-Out):** 메모리에 가장 먼저 들어온 페이지를 우선적으로 교체한다. 구현이 매우 간단하나, 활발히 사용 중인 페이지를 내보낼 수 있으며 Belady's Anomaly 현상이 발생할 수 있다.
- **LRU (Least Recently Used):** 가장 오랫동안 사용되지 않은 페이지를 교체한다. 과거의 패턴이 미래를 결정한다는 전제로 설계되었으며 성능이 뛰어나 실무에서 가장 널리 쓰인다.
- **LFU (Least Frequently Used):** 참조 횟수가 가장 적은 페이지를 교체한다. 최근에 들어왔으나 아직 참조 횟수가 적은 페이지가 희생될 수 있다는 단점이 있다.
- **Optimal (최적 교체):** 앞으로 가장 오랫동안 사용되지 않을 페이지를 교체한다. 이론적으로 최소의 페이지 폴트율을 보장하지만, 미래를 예측할 수 없으므로 실제 시스템에서는 구현이 불가능하다.

---

## 5. LRU를 실제로 구현하는 방법은? (연결 리스트 + 해시맵)

**효율적인 $O(1)$ 연산 조합**
- **이중 연결 리스트(Doubly Linked List):** 페이지 노드들을 사용 순서대로 연결한다. 참조된 노드는 리스트의 맨 앞(Head)으로 옮기고, 교체 시에는 맨 뒤(Tail)에 있는 노드를 제거한다.
- **해시맵(Hash Map):** 페이지 번호를 키로, 리스트 상의 노드 주소를 값으로 저장한다. 이를 통해 연결 리스트를 일일이 탐색하지 않고도 특정 페이지 노드에 $O(1)$의 시간 복잡도로 즉시 접근할 수 있다.
- 두 자료구조를 조합함으로써 페이지 참조와 교체 작업을 모두 $O(1)$ 내에 처리할 수 있게 한다.

## 6. 스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?

**과도한 페이지 교체와 성능 급락**
- **발생 원인:** 다중 프로그래밍의 정도가 너무 높아져서 각 프로세스가 할당받는 메모리 프레임 수가 부족할 때 발생한다. 프로세스들이 페이지를 찾지 못해 계속 페이지 폴트가 일어나고, CPU는 실제 연산보다 페이지 교체(디스크 I/O)에 대부분의 시간을 쏟게 되어 전체 시스템 성능이 급격히 저하된다.
- **해결 방법:** 동시에 실행되는 프로세스 수(다중 프로그래밍의 정도)를 조절하여 각 프로세스가 필요한 최소한의 프레임을 확보하게 한다. 또는 Working Set Model을 도입하여 현재 활발하게 사용되는 페이지 집합을 메모리에 보장해준다.



---

## 7. Working Set Model이란?

**지역성(Locality)을 활용한 자원 관리**
- 프로세스가 일정 시간 동안 집중적으로 참조하는 페이지들의 집합을 'Working Set'이라고 정의한다.
- 운영체제는 시스템 전체의 가용 프레임이 각 프로세스의 Working Set 크기 합보다 클 때만 프로세스를 실행한다. 만약 메모리가 부족해 특정 프로세스의 Working Set을 다 채워주지 못할 상황이라면, 해당 프로세스의 메모리를 전부 스왑 아웃시켜서 스래싱을 방지하고 다른 프로세스들의 실행 성능을 보장한다.
