## Q1. 가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?

[답변]
* 가상 메모리가 필요한 이유
  - 실제 메모리보다 큰 '연속적'인 메모리 공간이 있는 것처럼 보이게 함
  - 물리 메모리의 실제 크기에 직접 제약 받지 않고 실행
* 물리 메모리보다 큰 프로그램 실행 원리
  - 운영체제: 프로그램 전체를 한 번에 RAM에 올리지 않고, 현재 필요한 일부만 물리 메모리에 적재
    -> 나머지는 디스크의 스왑 영역 또는 실행 파일 영역에 둠
  - CPU는 가상 주소 사용: MMU가 페이지 테이블 참조 -> 물리 주소로 변환
  - 실제 필요한 페이지만 물리 메모리에 적재

[보완]
* 스왑 영역: RAM에서 밀려난 페이지 임시 저장소
* 실행 파일 영역: 프로그램 원본 파일에서 다시 읽어올 수 있는 부분
* MMU: Memory Management Unit
  - 가상 주소 -> 물리 주소 변환 하드웨어
  - MMU가 페이지 테이블 등의 정보를 바탕으로 실제 RAM의 어느 위치인지 찾아줌
* TLB: Translation Lookaside Buffer
  - 주소 변환 결과 캐싱 장치

## Q2. 페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?

[답변]
* 페이지 폴트(Page Fault)
  - 프로세스가 어떤 가상 페이지에 접근 -> 해당 페이지가 물리 메모리(RAM)에 존재하지 않아 발생하는 예외

* 발생 시 처리 과정
- CPU의 페이지 접근 -> MMU가 페이지 테이블 확인 -> 메모리에 없음 확인 -> 페이지 폴트 트랩 발생
  -> 운영체제 개입: 필요한 페이지를 디스크에서 찾아 물리 메모리로 적재
    - 빈 프레임 O: 적재
    - 빈 프레임 X: 페이지 교체 알고리즘 => 기존 페이지 하나 내보냄
  -> 페이지 적재 끝나면 페이지 테이블 갱신 -> 중단됐던 명령어 재실행

[보완]
* 페이지 폴트가 항상 비정상 상황은 아님
* 요구 페이징 환경
  - 정상적으로 자주 발생
* 접근한 주소 자체가 잘못된 경우 단순 페이지 폴트 처리 X -> 세그멘테이션 폴트 / 접근 위반 예외
* 처리 순서
  - CPU 주소 접근
  - 페이지 테이블 확인
  - 유효하지 않거나 메모리에 없음 표시 확인
  - 커널로 트랩 발생
  - 해당 페이지의 적법성 검사
  - 디스크에서 페이지 적재
  - 페이지 테이블/TLB 갱신
  - 명령 재시작

## Q3. 요구 페이징(Demand Paging)은 어떻게 동작하는가?

[답변]
* 요구 페이징
  - 프로세스 실행 시 모든 페이지를 미리 메모리에 올리지 않고
  - 실제 참조되는 시점에 해당 페이지 적재하는 방식

* 동작
  - 프로세스 시작: 극히 일부 페이지만 메모리에 올라감
    - 나머지는 디스크에 남아있음
  - 프로그램이 아직 메모리에 없는 페이지 참조 -> 페이지 폴트
  - 운영체제: 해당 페이지를 보조기억장치에서 읽어옴 -> 물리 메모리에 적재
  - 이후 다시 같은 페이지에 접근 -> 이미 메모리에 적재

 [보완]
* 장점
  - 초기 적재 시간 감소
  - 메모리 사용량 감소
  - 더 많은 프로세스 실행 가능
* 단점
  - 페이지 폴트 발생 -> 디스크 I/O 필요 -> 성능 저하
  - 요구 페이징이 효과적이려면 프로그램의 지역성 특성이 잘 나타나야 함
    => 프로그램은 특정 시간 동안 일부 페이지를 집중적으로 참조하는 경향

## Q4. 페이지 교체 알고리즘(FIFO, LRU, LFU, Optimal)의 차이는?

[답변]
* FIFO: First-In First-Out
  - 가장 먼저 들어온 페이지 가장 먼저 교체
  - 구현 단순 but 오래 있었다고 해서 앞으로 안 쓰인다는 보장 X
  - 성능 비효율적일 수 있음: Belady의 anomaly 발생 가능
* LRU: Least Recently Used
  - 가장 오랫동안 사용되지 않은 페이지 교체
  - 최근에 사용된 페이지는 곧 사용될 가능성이 높다는 가정에 기반
  - 실제 접근 패턴을 비교적 잘 반영 -> 성능이 좋지만 구현 비용이 큼
* LFU: Least Frequently Used
  - 참조 횟수가 가장 적은 페이지 교체
  - 자주 사용된 페이지 유지
  - 과거에 많이 쓰였지만 앞으로는 안 쓰일 페이지가 계속 남는 문제
* Optimal
  - 앞으로 가장 오랫동안 사용되지 않을 페이지 교체
  - 이론적으로 가장 좋은 성능
  - 미래의 참조를 알아야 함 -> 실제 구현 불가능, 비교 기준용으로 사용

[보완]
* Belady의 anomaly: 프레임 수를 늘렸는데도 오히려 페이지 폴트가 증가하는 현상 -> 주로 FIFO에서 나타날 수 있음

## Q5. LRU를 실제로 구현하는 방법은? (연결 리스트 + 해시맵)

[답변]
* LRU는 일반적으로 해시맵 + 이중 연결 리스트 조합으로 구현
  - 해시맵: 특정 페이지가 현재 캐시 또는 메모리에 있는지 O(1)에 찾기 위해 사용
  - 이중 연결 리스트: 페이지의 최근 사용 순서 관리
  - 가장 최근에 사용한 페이지: 리스트의 head로 이동
  - 가장 오래 사용하지 않은 페이지: 리스트의 tail로 이동
  - 페이지 참조 동작
    - 페이지가 이미 존재: 해당 노드를 리스트에서 제거 후 head로 이동
    - 페이지가 없음: 새 노드를 head에 삽입
    - 용량 초과: tail의 노드 제거 -> 해시맵에서 삭제

[보완]
* 이중 연결 리스트 사용 이유
  - 특정 노드를 O(1)에 제거하기 위해
  - 안 그러면 head에서부터 타고 tail까지 가야 함

## Q6. 스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?

[답변]
* 스래싱
  - 프로세스가 실제 작업보다 페이지 교체에 대부분의 시간을 소비하는 현상
  - 페이지 폴트가 매우 자주 발생 -> CPU는 계산보다 디스크 I/O를 기다리는 시간이 많아짐

* 주된 원인
  - 프로세스에 할당된 프레임 수가 너무 적음 -> 현재 작업에 필요한 페이지 집합을 메모리에 유지하지 못함
  - 특히 다중 프로그래밍 수준이 너무 높아 많이 프로세스가 동시에 메모리를 차지할 때 발생하기 쉬움
 
* 발생 과정
  - 프로세스 수 증가
  - 각 프로세스에 할당되는 프레임 수 감소
  - 페이지 폴트 증가
  - CPU 이용률 감소
  - 운영체제가 CPU 이용률을 높이기 위해 더 많은 프로세스를 추가
  - 메모리 압박 심화
  - 스래싱 악화

* 해결 방법
  - 각 프로세스에 충분한 프레임 할당
  - 다중 프로그래밍 수준 낮춤
  - Working Set 기반으로 필요한 페이지 수 추정 및 보장
  - 페이지 폴트 빈도(PFF)를 기준으로 프레임 수를 동적으로 조절
  - 물리 메모리 증설
  => 현재 실행 중인 프로세스들의 지역성을 수용할 수 있을 만큼 메모리 확보

## Q7. Working Set Model이란 무엇인가?

[답변]
* Working Set Model
  - 어떤 프로세스가 최근 일정 시간 동안 실제로 사용한 페이지들의 집합을 그 프로세스의 작업 집합으로 보는 모델
    - Working Set은 시간에 따라 바뀜
  - 이 작업 집합은 -> 프로세스가 현재 원활히 실행되기 위해 메모리에 유지되어야 하는 페이지들의 집합으로 간주
  - 운영체제는 각 프로세스의 Working Set 크기를 기준으로 필요한 프레임 수를 추정
  - 전체 프로세스들의 Working Set 합 > 물리 메모리 크기
    => 스래싱 발생 가능성 증가
  - 스래싱을 방지하고 메모리 할당을 조절하는 기준으로 사용
    - 어떤 프로세스를 계속 실행시킬지
    - 얼마나 많은 프레임을 줄지
    - 일부 프로세스를 일시 중단할지
    - 등 판단

[보완]
* 보통 W(t, Δ) 형태로 표현
  - t: 현재 시점
  - Δ: 최근 참조 구간(시간 또는 참조 횟수 범위)
    - 너무 작으면 실제 필요한 페이지 충분히 반영 X
    - 너무 크면 불필요한 페이지까지 포함
  - W(t, Δ): 시점 t에서 최근 Δ 동안 참조된 페이지들의 집합
