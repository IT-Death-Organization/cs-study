## 가상 메모리 (Virtual Memory)

### 가상 메모리가 필요한 이유는? 물리 메모리보다 큰 프로그램을 실행할 수 있는 원리는?

가상 메모리는 **프로그램이 실제 물리 메모리(RAM)보다 큰 주소 공간을 사용할 수 있도록 하는 메모리 관리 기법**이다.

핵심 아이디어는 **프로그램이 사용하는 주소(virtual address)**와 **실제 RAM의 주소(physical address)**를 분리하는 것이다.

운영체제는 **페이지 테이블(Page Table)**을 이용해 다음과 같이 주소를 변환한다.

```
Virtual Address
      ↓
Page Table
      ↓
Physical Address (RAM)
```

프로그램은 **전체 메모리가 존재하는 것처럼 보이지만**, 실제로는 일부 페이지만 RAM에 올라가 있다.
나머지 데이터는 **디스크의 swap 영역**에 저장된다.

즉,

1. 프로그램은 **가상 주소 공간 전체를 사용**
2. 실제로 필요한 페이지만 **RAM에 로드**
3. 나머지는 **디스크에 저장**

이 구조 덕분에 **물리 메모리보다 큰 프로그램 실행이 가능**하다.

## Page Fault

### 페이지 폴트(Page Fault)란 무엇이며 발생 시 처리 과정은?

페이지 폴트는 **프로세스가 접근한 페이지가 현재 RAM에 존재하지 않을 때 발생하는 인터럽트**이다.

예:

```
CPU → 메모리 접근
      ↓
페이지 테이블 확인
      ↓
RAM에 없음
      ↓
Page Fault 발생
```

처리 과정:

1. CPU가 가상 주소 접근
2. 페이지 테이블 확인
3. 해당 페이지가 **RAM에 없음 (valid bit = 0)**
4. **Page Fault trap 발생 → 커널 모드 전환**
5. OS가 디스크에서 해당 페이지 읽기
6. RAM에 빈 프레임이 없으면 **페이지 교체 알고리즘 실행**
7. 페이지를 RAM에 로드
8. 페이지 테이블 업데이트
9. 중단된 명령 재실행

요약

```
Page Fault
→ 디스크에서 페이지 읽기
→ RAM에 적재
→ 페이지 테이블 수정
→ 명령 재실행
```

## Demand Paging

### 요구 페이징(Demand Paging)은 어떻게 동작하는가?

Demand Paging은 **페이지가 실제로 필요할 때만 메모리에 로드하는 방식**이다.

프로그램 시작 시

```
프로세스 시작
↓
RAM에는 거의 아무 페이지도 없음
↓
접근할 때마다 Page Fault 발생
↓
필요한 페이지만 로드
```

특징

-   초기 로딩 빠름
-   메모리 사용 효율적
-   디스크 I/O 증가 가능

이 방식은 **Locality of Reference**에 기반한다.

대부분 프로그램은

-   **Temporal locality** (최근 사용한 데이터 재사용)
-   **Spatial locality** (근처 주소 접근)

를 가지기 때문에 일부 페이지만 있어도 실행 가능하다.

## 페이지 교체 알고리즘

RAM이 가득 차면 어떤 페이지를 제거할지 결정해야 한다.

### FIFO (First In First Out)

가장 먼저 들어온 페이지를 제거한다.

```
Queue 구조
[1][2][3][4]

새 페이지 필요
→ 1 제거
```

장점

-   구현 간단

단점

-   **Belady's Anomaly 발생 가능**

---

### LRU (Least Recently Used)

**가장 오래 사용되지 않은 페이지 제거**

```
최근 사용 기록 기준

최근 사용 → 유지
오래 미사용 → 제거
```

장점

-   locality 특성 반영
-   성능 좋음

단점

-   구현 비용 높음

---

### LFU (Least Frequently Used)

**사용 횟수가 가장 적은 페이지 제거**

```
사용 횟수 기준

page A : 10
page B : 2
page C : 1

→ C 제거
```

단점

-   오래된 사용 기록이 계속 남음
-   locality 반영이 부족

---

### Optimal (OPT)

**앞으로 가장 늦게 사용될 페이지 제거**

```
future access 기준
```

예

```
future access
A → 2
B → 10
C → 50

→ C 제거
```

특징

-   **이론적으로 최적**
-   실제 구현 불가능 (미래 정보 필요)
-   다른 알고리즘 성능 비교 기준

---

## LRU 실제 구현

LRU는 일반적으로 **HashMap + Doubly Linked List**로 구현한다.

구조

```
HashMap : key → node
Linked List : 사용 순서 관리
```

동작

### 접근

```
key 조회
↓
HashMap O(1)
↓
Linked List에서 해당 노드를 head로 이동
```

### 삽입

```
새 노드 생성
↓
head에 삽입
↓
HashMap 저장
```

### 제거

```
capacity 초과
↓
tail 노드 제거 (가장 오래된 페이지)
↓
HashMap 삭제
```

시간 복잡도

```
get : O(1)
put : O(1)
```

## Thrashing

### 스래싱(Thrashing)은 왜 발생하며 어떻게 해결하는가?

Thrashing은 **페이지 교체가 너무 자주 발생하여 CPU보다 디스크 I/O에 대부분 시간을 쓰는 상태**이다.

현상

```
page fault 폭증
↓
디스크 I/O 증가
↓
CPU idle 증가
↓
성능 급격히 하락
```

주요 원인

1. **프로세스 수 과다**
2. **각 프로세스에 할당된 프레임 부족**
3. **Working set을 유지할 메모리 부족**

---

### 해결 방법

1️⃣ **프로세스 수 감소**

multiprogramming level 감소

2️⃣ **Working Set 유지**

각 프로세스가 필요한 최소 페이지 확보

3️⃣ **Page Fault Frequency 제어**

page fault rate 기준으로 프레임 조정

---

## Working Set Model

Working Set은 **프로세스가 일정 시간 동안 실제로 사용하는 페이지 집합**이다.

정의

```
W(t, Δ)
```

-   t : 현재 시간
-   Δ : 관찰 시간 윈도우

예

```
최근 Δ 시간 동안 사용된 페이지

{2, 5, 7, 10}
```

이 페이지들이 **현재 실행에 필요한 최소 페이지 집합**이다.

운영체제는

```
각 프로세스 working set 크기 합
≤
전체 물리 메모리
```

를 유지하려고 한다.

만약 초과하면

```
Thrashing 발생
```

따라서 OS는

-   일부 프로세스를 **suspend**
-   multiprogramming level 감소

하여 해결한다.
