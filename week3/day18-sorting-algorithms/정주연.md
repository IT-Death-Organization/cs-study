## 🧠 학습 질문

## 병합 정렬의 분할 정복 원리를 설명하면?

- 병합 정렬은 분할 → 정복(정렬) → 병합의 전형적인 분할 정복 알고리즘
- 작은 배열을 정렬하는 건 쉬움 → 그걸 반복해서 전체를 정렬
- 분할(Divide)
    - 배열을 정확히 반씩 나눠서 두 개의 하위 배열로 쪼갬
- 정복(Conquer)
    - 하위 배열을 재귀적으로 병합 정렬
    - 길이가 1 이하가 되면 이미 정렬된 것으로 보고 종료
- 병합(Combine)
    - 정렬된 두 하위 배열을 두 포인터를 이용해 하나의 정렬된 배열로 합침

## 병합 정렬이 항상 O(n log n)인 이유는?

- 크기가 n인 배열에 대해
    - 한 단계에서 반으로 나눈 두 하위 배열을 정렬한 뒤 병합
    - 병합(merge)에 드는 비용은 항상 원소 n개를 한 번씩 보면서 합치므로 O(n)
- 이때 높이는
    - 매 단계마다 배열 크기가 반으로 줄어듦  (n→ n/2 → n/4 → … 1)
    - 이 단계의 수: logn (밑이 2) 정도임
- 전체 시간: 각 레벨마다 O(n) * 레벨 개수 O(log n) → O(n log n)
- 입력이 이미 정렬되어 있든, 역정렬이든, 랜덤이든 반씩 나누고, n만큼 병합하는 구조가 동일함 → 최선/평균/최악이 모두 O(n log n)으로 일정

## 병합 정렬의 공간복잡도가 O(n)인 이유는?

- 두 정렬된 배열을 하나로 합칠 때 임시 공간이 필요함
- ex)
    - 길이 n 배열을 정렬할 때, 보통 길이 n짜리 보조 배열을 하나 생성
    - 왼쪽·오른쪽 배열에서 작은 값부터 보조 배열에 채우고
    - 다 끝나면 그 보조 배열 내용을 원래 배열로 복사
- 입력 배열 외에 원소 수만큼의 추가 공간이 필요 → 추가 공간 O(n)

## 퀵 정렬이 평균적으로 병합 정렬보다 빠른 이유는?

- 입력 배열 외에 원소 수만큼의 추가 공간이 필요 → 추가 공간 O(n)
- 병합 정렬
    - 매 단계 병합할 때 큰 보조 배열을 써야 해서 메모리 쓰기가 많이 발생
- 퀵 정렬
    - in-place 정렬이 가능함(추가 배열 없이, 배열 안에서만 swap) → 메모리 접근 패턴이 간단하고 캐시 친화적
    - 평균적으로 비교·이동 횟수, 메모리 사용량 측면에서 병합 정렬보다 상수가 작아서 실제 실행 시간이 더 빠른 경우가 많음

## 퀵 정렬의 최악의 경우는 언제 발생하며 시간복잡도는?

- 퀵 정렬 흐름
    - 피벗 고름 → 피벗 기준 작/큰  나눔 → 두 구간에서 재귀적으로 퀵정렬
- 최악
    - 분할이 극단적으로 한쪽으로 치우침
    - ex) 항상 첫 번째 원소를 피벗으로 고르는데, 입력이 이미 정렬된 배열/역정렬된 배열인 경우 → 왼쪽에 (n-1)개, 오른쪽에 0개 (또는 반대) 이런 식 → 한 칸 줄어드는 선형 깊이의 재귀가 됨
    - 재귀식: T(n) = T(n-1) + O(n) → O(n^2)
    - 최악 시간복잡도는 O(n^2)이며, 치우친 분할이 계속 반복될 때 발생

## 피벗 선택 방법(첫 번째, 마지막, 중간, 랜덤)에 따른 성능 차이는?

- 첫 번쨰
    - 구현이 가장 간단함
    - 하지만 입력이 이미 정렬되었거나, 거의 정렬된 경우 최악 O(n²) 상황이 자주 발생
    - 실전에서는 이 방식만 쓰면 위험함
- 마지막
    - 첫 번째와 같은 문제.
    - 특정 패턴의 입력(오름차순, 내림차순 등)에 매우 약함
    - 단순 예제 코드를 위해 자주 쓰이지만, 실제 라이브러리 수준에서는 잘 안 씀
- 중간
    - “위치상 가운데”를 고르는 것이라, 입력 패턴과 조금 덜 상관있을 수 있음
    - 그래도 값의 분포를 보장하진 못함 (값이 한쪽에 몰린 경우 등)
- 랜덤
    - 임의 인덱스를 피벗으로 고름
    - 입력이 어떤 패턴이든 균형 잡힌 분할이 나올 확률이 높아짐 → 평균적으로 퀵 정렬의 O(n log n)을 안정적으로 기대할 수 있음
    - 최악 O(n²)는 여전히 존재하지만, “그 최악 패턴이 우연히 계속 걸릴 확률”을 매우 낮추는 효과가 있음

## 힙 정렬의 시간복잡도와 공간복잡도는? 왜 O(n log n)인가?

- 시간복잡도
    - 배열로 최대 힙(max-heap)을 구성 (build heap)
        - bottom-up 방식으로 만들면 O(n) 에 힙 구성 가능
    - 힙의 루트(가장 큰 값)를 배열의 끝과 swap → 힙 크기 줄이기 → 다시 heapify << 이 과정 n번 반복
        - 한 번 root를 꺼낼 때마다 heapify에 O(log n)  → n * O(log n) = O(n log n)
    - 시간복잡도:  O(n log n)
- 공간복잡도
    - 힙 정렬은 배열을 그 자체를 힙으로 보고 in-place로 정렬
    - 추가로 필요한 건 몇 개의 변수(인덱스, tmp 변수) 정도 → O(1) 추가 공간
    - 즉, 공간복잡도:  O(1)

## 실무에서 어떤 정렬 알고리즘을 사용하는가? (Tim Sort, Intro Sort)

- Timsort
    - Python의 `list.sort()`, `sorted()`, Java의 `Arrays.sort()` (Object 버전) 등이 사용하는 알고리즘
    - **Merge Sort + Insertion Sort** 기반의 하이브리드, 거기에 “이미 어느 정도 정렬되어 있는 구간(run)”을 감지해서 최적화
    - 실무 데이터는 완전 랜덤보다 “부분적으로 이미 정렬된” 경우가 많음 → 이런 상황에서 매우 빠름
    - 안정 정렬(stable sort)임 → 같은 값의 상대 순서가 유지
- IntroSort
    - **Quick Sort + Heap Sort + Insertion Sort** 섞은 알고리즘
    - 처음에는 퀵 정렬로 시작
    - 재귀 깊이가 너무 깊어져서 최악 패턴이 의심되면 Heap Sort로 갈아타서 O(n log n)을 보장
    - 작은 구간에 대해서는 Insertion Sort로 마무리해서 상수 시간 최적화
    - 평균적으로 매우 빠르고, 최악의 경우에도 O(n log n) 보장 → 라이브러리 구현에 적합
