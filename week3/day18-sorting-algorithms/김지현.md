## 1. 병합 정렬(Merge Sort)의 **분할 정복(Divide and Conquer)** 원리

병합 정렬은 **Divide → Conquer → Combine** 구조로 동작한다.

### 1️⃣ Divide (분할)

배열을 **절반씩 계속 분할**한다.

```
[8, 3, 5, 4, 7, 6, 1, 2]

→ [8,3,5,4] | [7,6,1,2]
→ [8,3] [5,4] | [7,6] [1,2]
→ [8] [3] [5] [4] | [7] [6] [1] [2]
```

배열 크기가 **1이 될 때까지 재귀적으로 분할**한다.

---

### 2️⃣ Conquer (정복)

각 부분 배열은 이미 **정렬된 상태**라고 가정한다.
(원소가 1개이기 때문)

---

### 3️⃣ Combine (병합)

두 개의 정렬된 배열을 **정렬 상태를 유지하며 병합**한다.

```
[8] [3] → [3,8]
[5] [4] → [4,5]

→ [3,8] [4,5]
→ [3,4,5,8]
```

병합 과정에서 **두 배열의 포인터를 비교하면서 작은 값부터 넣는다.**

---

## 2. 병합 정렬이 항상 **O(n log n)** 인 이유

병합 정렬의 시간복잡도는 다음 두 요소로 결정된다.

### 1️⃣ 분할 깊이

배열을 계속 **2로 나누기** 때문에

```
n → n/2 → n/4 → n/8 ...
```

트리 높이는

```
log₂ n
```

---

### 2️⃣ 각 레벨에서의 병합 비용

각 레벨에서 모든 원소를 **한 번씩 비교하며 병합**한다.

따라서

```
각 레벨 비용 = O(n)
```

---

### 3️⃣ 전체 시간복잡도

```
전체 시간 = 레벨 수 × 레벨 비용

= log n × n
= O(n log n)
```

그리고 병합 정렬은 **입력 데이터와 상관없이 항상 동일한 구조**로 실행되므로

```
best = average = worst = O(n log n)
```

---

## 3. 병합 정렬의 **공간복잡도 O(n)** 이유

병합 정렬은 **병합 과정에서 새로운 배열이 필요하다.**

예시

```
왼쪽 배열: [3,8]
오른쪽 배열: [4,5]

→ 결과 배열: [3,4,5,8]
```

이때 **결과를 저장할 임시 배열(temp array)** 이 필요하다.

```
temp[n]
```

따라서

```
공간복잡도 = O(n)
```

또한 재귀 호출 스택이 존재한다.

```
recursion stack = O(log n)
```

하지만

```
O(n) + O(log n) ≈ O(n)
```

---

## 4. 퀵 정렬이 평균적으로 병합 정렬보다 빠른 이유

이론적으로 둘 다

```
O(n log n)
```

이지만 실제 성능은 퀵 정렬이 더 빠른 경우가 많다.

### 이유 1: **메모리 locality (cache 친화성)**

퀵 정렬은 **in-place 정렬**이다.

```
배열 내부에서 swap
```

반면 병합 정렬은

```
추가 배열 생성
메모리 복사 발생
```

→ **CPU cache miss 증가**

---

### 이유 2: **추가 메모리 사용 없음**

```
Merge Sort → O(n) extra memory
Quick Sort → O(log n) stack only
```

---

### 이유 3: **데이터 이동 비용**

병합 정렬

```
n log n 번 데이터 복사
```

퀵 정렬

```
swap 중심
```

→ 실제 CPU 비용이 더 낮다.

---

## 5. 퀵 정렬의 **최악의 경우**

퀵 정렬은 **피벗이 항상 최솟값 또는 최댓값일 때** 발생한다.

예시

```
이미 정렬된 배열

[1,2,3,4,5]
```

피벗 = 첫 번째

```
1 | [2,3,4,5]
```

다음

```
2 | [3,4,5]
```

계속 반복

---

### 분할 구조

```
n → n-1 → n-2 → n-3
```

즉

```
T(n) = T(n-1) + O(n)
```

결과

```
O(n²)
```

---

## 6. 피벗 선택 방법에 따른 성능 차이

### 1️⃣ 첫 번째 원소

```
pivot = arr[0]
```

문제

```
이미 정렬된 배열 → 최악 O(n²)
```

---

### 2️⃣ 마지막 원소

```
pivot = arr[n-1]
```

첫 번째와 동일한 문제 발생 가능.

---

### 3️⃣ 중간 원소

```
pivot = arr[n/2]
```

일반적으로 **균형 분할 확률 증가**

---

### 4️⃣ 랜덤 피벗

```
pivot = random index
```

확률적으로

```
평균 O(n log n)
```

보장.

---

### 5️⃣ Median-of-three (실무에서 많이 사용)

세 값 중 중앙값 사용

```
arr[first]
arr[middle]
arr[last]
```

예

```
[1, 100, 50] → pivot = 50
```

→ **편향 분할 방지**

---

## 7. 힙 정렬(Heap Sort)

### 시간복잡도

```
O(n log n)
```

### 공간복잡도

```
O(1)
```

(in-place)

---

### 왜 O(n log n)인가

힙 정렬 과정

### 1️⃣ Heap Build

배열을 **Max Heap으로 변환**

```
O(n)
```

---

### 2️⃣ 최대값 반복 추출

과정

```
1. root와 마지막 swap
2. heapify
```

heapify 비용

```
O(log n)
```

이를 **n번 반복**

```
n × log n
```

---

### 전체 시간

```
build heap = O(n)
sort phase = O(n log n)

→ O(n log n)
```

---

## 8. 실무에서 사용하는 정렬 알고리즘

실무에서는 **순수 Quick Sort / Merge Sort를 그대로 쓰지 않는다.**

대부분 **하이브리드 알고리즘**이다.

---

### 1️⃣ TimSort (Python, Java)

Python

```
list.sort()
sorted()
```

Java

```
Arrays.sort(Object[])
```

구조

```
Merge Sort + Insertion Sort
```

특징

-   이미 정렬된 데이터 감지
-   작은 배열 → insertion sort
-   큰 배열 → merge

시간복잡도

```
best: O(n)
average: O(n log n)
worst: O(n log n)
```

---

### 2️⃣ IntroSort (C++)

C++

```
std::sort()
```

구조

```
Quick Sort + Heap Sort + Insertion Sort
```

동작

```
기본: Quick Sort
재귀 깊이 너무 깊어짐 → Heap Sort
작은 배열 → Insertion Sort
```

그래서

```
worst-case O(n log n) 보장
```

## 정렬 알고리즘 비교 요약

| 알고리즘   | 평균       | 최악       | 공간     |
| ---------- | ---------- | ---------- | -------- |
| Merge Sort | O(n log n) | O(n log n) | O(n)     |
| Quick Sort | O(n log n) | O(n²)      | O(log n) |
| Heap Sort  | O(n log n) | O(n log n) | O(1)     |
| TimSort    | O(n log n) | O(n log n) | O(n)     |
| IntroSort  | O(n log n) | O(n log n) | O(log n) |
