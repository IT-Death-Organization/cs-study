### 📌 [Q1] 병합 정렬의 분할 정복 원리를 설명하면?

**병합 정렬이란?** <br>
하나의 리스트를 n 개의 균등한 크기로 분할해 분할된 부분 리스트를 각각 정렬한 다음, 다시 합쳐서 전체를 정렬하는 방식이다. 이 과정에서 분할 정복(Divide and Conquer) 전략을 사용한다.

**병합 정렬의 특징**
- 항상 O(n log n)의 시간복잡도를 가진다.
- 안정 정렬(Stable Sort)이다.
- 추가 메모리 공간 O(n)이 필요하다.
- 대용량 데이터, 외부 정렬에 적합하다.

**분할 정복(Divide and Conquer)의 3단계**
1. **분할(Divide)**: 배열을 더 이상 나눌 수 없을 때(원소가 1개일 때)까지 절반씩 계속 나눈다. (원소가 1개인 배열은 이미 정렬된 상태라고 본다.)

```
[5, 3, 8, 4]
→ [5, 3]   [8, 4]
→ [5] [3]  [8] [4]
```

2. **정복(Conquer)**: 부분 배열을 정렬한다. 배열의 크기가 충분히 작지 않으면 (크기가 2 이상이면) **재귀 호출**을 통해 다시 분할 단계를 거친다.

```
[5] [3] → [3, 5]
[8] [4] → [4, 8]
```

3. **결합(Combine)**: 정렬된 부분 배열들을 하나의 정렬된 배열로 합친다.

```
[3, 5]  [4, 8]
→ [3, 4, 5, 8]
```

**분할 정복의 과정** <br>
큰 문제
 → 작은 문제로 분할
   → 더 작은 문제로 분할
     → (1개 단위 도달)
 → 정렬하면서 병합
 → 다시 병합
 → 최종 정렬 완성

 ### 📌 [Q2] 병합 정렬이 항상 O(n log n)인 이유는?

- 배열을 log n번 나눈다 (트리 높이, 배열의 크기가 1이 될 때까지 나눔)
- 각 단계에서 n개의 원소를 정렬한다. (n번의 비교/연산)
- 전체 시간 복잡도 = (트리의 높이) x (각 단계의 연산 횟수) = log n x n = O(n log n)

 ### 📌 [Q3] 병합 정렬의 공간복잡도가 O(n)인 이유는?

정렬 과정에서 분할된 배열을 다시 합칠 때 **원래 배열의 크기만큼의 새로운 임시 메모리 공간이 필요하기 때문**이다.

병합 정렬은 데이터를 쪼개는 '분할' 단계에서는 추가 공간을 거의 쓰지 않지만, 다시 합치는 '병합' 단계에서 메모리를 사용하게 된다.
  - 두 개의 정렬된 부분 배열을 하나의 완성된 배열로 만들 때, 결과를 담아둘 **별도의 임시 배열**이 반드시 필요하다.
  - 예를 들어, 8개의 원소를 정렬한다면 병합 과정에서 최대 8개의 데이터를 담을 수 있는 추가 공간이 있어야 정렬된 값을 안전하게 채워 넣을 수 있다.

**그렇다면 왜 O(n)인가?**
- **동시성 관점**에서 보면, 병합은 한 번에 모든 층에서 동시에 일어나지 않는다. 특정 시점에 수행되는 병합 작업 중 가장 큰 규모는 결국 **전체 데이터 개수인 n을 넘지 않는다.**
- 재귀 호출로 인해 스택 프레임이 쌓이면서 발생하는 공간은 **O(log n)**으로, 전체 O(n) 공간 복잡도에 비해 미미하다.
- 따라서 병합 정렬의 공간 복잡도는 임시 배열의 크기인 **O(n)**이다.


 ### 📌 [Q4] 퀵 정렬이 평균적으로 병합 정렬보다 빠른 이유는?

 **퀵 정렬이란?** <br>
 분할 정복 기반 정렬 알고리즘으로, 하나의 기준값(pivot)을 정해 기준값보다 작은 값은 왼쪽으로, 큰 값은 오른쪽으로 이동시켜 정렬하는 방식이다.

 **퀵 정렬의 특징**
 - 평균적으로 O(n log n)의 시간복잡도를 가진다.
 - 최악의 경우 O(n^2)의 시간복잡도를 가진다.
 - 공간 복잡도는 O(log n)이다.

 **퀵 정렬과 병합 정렬의 시간 복잡도**
 둘의 평균 시간 복잡도는 O(n log n)으로 동일하지만, 퀵 정렬이 평균적으로 더 빠르다. 왜냐하면 **하드웨어 효율성**과 **상수 시간** 때문이다.

 **(1) 지역성(locality)와 캐시 적중률(Cache Hit)**
 둘의 가장 큰 차이는 **참조 지역성**에서 발생한다.
 - 퀵 정렬
  - 인접한 데이터끼리 비교 & 자리 바꿈 -> CPU 캐시 효율, Cache Hit 좋음 (데이터 연속적으로 활용할 확률 높음)
  - 캐시 적중률 높음 -> 실제 성능 빠름
 - 병합 정렬
  - 데이터 쪼개고 합치는 과정에서 계속 새로운 배열을 생성 및 참조
  - 2개의 배열을 합치는 과정에서 메모리 여기저기 흩어진 데이터 접근
  - 캐시 미스 잦음 -> 실제 성능 느림

 **(2) 상수 시간(Constant Factor)**
- 시간 복잡도는 데이터가 무한히 커질 때의 경향성일 뿐, **실제 실행 시간은 앞에 붙는 상수(k)에 의해 결정된다.
- 퀵 정렬은 병합 정렬보다 내부 루프가 훨씬 단순해 이 **상수값(k)이 병합 정렬보다 작음**

**(3) 추가 메모리 할당 및 복사 비용**
- 병합 정렬은 정렬 과정에서 O(n)의 추가 메모리가 반드시 필요하지만, 퀵 정렬은 별도의 추가 배열 없이 원본 배열 내에서 요소의 위치만 바꾸는 **제자리 정렬** 방식이라 실행 시간이 더 단축된다.

 ### 📌 [Q5] 퀵 정렬의 최악의 경우는 언제 발생하며 시간복잡도는?

 퀵 정렬의 성능은 배열을 얼마나 균등하게 나누느냐에 달려 있다.

 **최악의 경우**
 - 배열이 항상 한쪽으로 치우치게 분할될 때 (배열 내의 최솟값 or 최댓값)
- **이미 정렬된 배열**
  - 맨 앞 or 맨 뒤 원소를 피벗으로 잡는 경우, 한 쪽은 원소가 0개, 한 쪽은 n-1개로 나뉘는 불균형한 분할 반복됨
- **역순으로 정렬된 배열**
  - 이미 정렬된 배열과 마찬가지로 분할이 한쪽으로 치우침
- **모든 원소가 같은 배열**

**퀵 정렬 최악의 경우 시간 복잡도: $O(n^2)$**
- 분할이 불균형하게 일어날 경우, **퀵 정렬의 구조가 이진 트리 형태가 아닌 편향된 트리(skewed tree) 형태가 된다.
- 층의 개수: 배열이 n, n-1, n-2, ..., 1로 줄어들기 때문에 트리의 높이는 n이 된다.
- 각 층의 연산 횟수: n (각 층에서 피벗과 나머지 원소를 비교)
- 전체 시간 복잡도: n x n = $O(n^2)$


 ### 📌 [Q6] 피벗 선택 방법(첫 번째, 마지막, 중간, 랜덤)에 따른 성능 차이는?

 **(1) 첫 번째 or 마지막 원소 선택 (Fixed Pivot)**
 - 구현이 가장 간단
 - 이미 정렬된 배열, 역순 배열, 중복된 원소가 많은 배열에서 최악의 경우 발생
 - 실무에서 거의 사용하지 않음
 - 시간 복잡도: 평균 O(n log n), 최악 O(n^2)

 **(2) 중간값 선택 (Middle Pivot)**
 - 고정 피벗보다 성능이 좋음
 - 1과 같은 최악의 경우를 피할 확률은 높아지지만, 특정 케이스에서는 여전히 O(n^2)이 될 수도 있음
 - 시간 복잡도: 평균 O(n log n), 최악 O(n^2)

 **(3) 랜덤 선택 (Random Pivot)**
 - 배열에서 랜덤으로 원소를 피벗으로 선택
 - 어떤 형태의 입력값이 들어와도 기댓값 O(n log n)을 보장
 - 최악의 경우가 발생할 확률이 매우 낮음
 - 시간 복잡도: 평균 O(n log n), 최악 O(n^2)

 ### 📌 [Q7] 힙 정렬의 시간복잡도와 공간복잡도는? 왜 O(n log n)인가?

 **힙 정렬이란?** <br>
 힙(Heap) 자료구조를 이용한 정렬 알고리즘으로, 최댓값(또는 최솟값)을 빠르게 꺼낼 수 있는 힙의 성질을 이용해 정렬한다.

 **힙 이란?** <br>
 힙은 **완전 이진 트리** 형태의 자료구조로, 부모 노드의 값이 자식 노드의 값보다 크거나 같은 **최대 힙(Max Heap)** 또는 부모 노드의 값이 자식 노드의 값보다 작거나 같은 **최소 힙(Min Heap)**을 말한다. 보통은 최대 힙을 사용해 오름차순 정렬을 한다.

 **힙 정렬의 과정** <br>

 ex) [4, 10, 3, 5, 1]

 1. 최대 힙 만들기 (Heapify) -> 배열을 최대 힙 구조로 바꾼다.
 ```
         10
       /  \
      5    3
     / \
    4   1
```
[10, 5, 3, 4, 1]

2. 루트(최댓값) 제거 -> 루트를 맨 뒤 값과 교환, 힙 크기 줄이기, 다시 Heapify
```
[1, 5, 3, 4 | 10]
→ heapify
→ [5, 4, 3, 1 | 10]
```

3. 반복
```
[1, 3, 4, 5, 10]
```

**힙 정렬의 시간 복잡도: $O(n log n)$**
- <1> 힙 만들기: $O(n)$
  - 각 노드가 힙의 속성 만족하도록 재배치함
- <2> 힙에서 원소 제거: $n$번 반복, 각 원소 제거 시 $O(log n)$ 소요 -> $O(n log n)$
  - 루트 노드 추출: 총 n개의 원소에 대해 반복 -> n
  - 힙 재구조화: 루트가 빠진 자리에 마지막 노드 올리고, 다시 아래로 내려가면서 힙 속성 만족하도록 재배치 -> 트리 높이가 lon n이므로 log n
  - 따라서 n번 반복 * log n = n log n
- 전체 시간 복잡도: $O(n log n)$

**힙 정렬의 공간 복잡도: $O(1)$**
- 제자리 정렬(in-place sorting)로 추가 메모리 불필요
  - 배열 자체를 이진 트리로 해석해서 처리함
  - 요소들의 위치를 서로 바꾸는(swap) 연산만 사용하므로 추가 메모리 불필요

**힙 정렬의 장점**
- 어떤 데이터가 들어와도 항상 O(n log n)의 시간 복잡도를 보장
- 메모리 사용량 적음

**힙 정렬의 단점**
- 참조 지역성이 좋지 않아 퀵 정렬보다는 실제 실행 속도 느림
- 동일한 값들의 상대적 순서가 바뀔 수 있는 불안정 정렬임

 ### 📌 [Q8] 실무에서 어떤 정렬 알고리즘을 사용하는가? (Tim Sort, Intro Sort)

실무에서는 퀵 정렬이나 병합 정렬을 그대로 사용하기보다, 각 알고리즘의 장점은 취하고 단점은 보완한 **하이브리드 정렬 알고리즘**을 사용한다.

대표적인 예시가 **팀 정렬(Tim Sort)**과 **인트로 정렬(Intro Sort)**이다.

**팀 정렬(Tim Sort)**
- 병합 정렬 + 삽입 정렬
- 실무 데이터가 이미 어느정도 정렬되어 있는 경우가 많다는 점을 활용
- 데이터를 작은 덩어리로 나누고, 각 덩어리를 삽입 정렬로 정렬한 뒤 병합
- 최선의 경우 O(n)
- 안정적
- 파이썬(sort(), sorted()), 자바(Arrays.sort()) 등에서 사용

**인트로 정렬(Intro Sort)**
- 퀵 정렬 + 힙 정렬 + 삽입 정렬
- 퀵 정렬의 빠른 속도 + 힙 정렬의 안정성
- 원리
  - 1. 퀵 정렬로 시작
  - 2. 재귀 깊이가 일정 수준 넘어가면 힙 정렬(O(n log n) 보장)로 전환
  - 3. 배열 크기가 작아지면 삽입 정렬로 전환
- 퀵 정렬의 빠른 속도 + 최악의 경우 리스크 방지
- C++ STL 등에서 사용
