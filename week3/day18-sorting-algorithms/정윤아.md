## Q1. 병합 정렬의 분할 정복 원리를 설명하면?

[답변]
* 분할(Divide): 배열을 원소 1개가 될 때까지 반으로 분할
* 정복(Conquer): 원소 1개짜리 배열은 이미 정렬된 상태로 간주
* 결합(Combine): 두 개의 정렬된 부분 배열 -> 앞에서부터 비교하면 더 작은 값을 결과 배열에 담음 (병합)

---

## Q2. 병합 정렬이 항상 O(n log n)인 이유는?

[답변]
* 분할 깊이(레벨 수): 매번 반으로 나눔 -> log2n 수준으로 고정
* 각 레벨에서 수행되는 병합 작업 총량은 n (전체 원소를 한 번씩 결과에 배치)
* 전체 시간: 레벨 수(logn) x 레벨당 작업량(n) = O(nlogn)
* 입력 데이터 상태와 상관 없이 병합 과정은 동일한 비교/복사 수행 -> 평균/최악이 거의 동일

[보완]
* 점화식: T(n) = 2T(n/2) + O(n) -> T(n) = O(nlogn)

---

## Q3. 병합 정렬의 공간복잡도가 O(n)인 이유는?

[답변]
* 병합 과정에서 두 정렬된 배열을 합치기 위해 추가 배열(보조 버퍼)을 사용 -> 전체 원소 수만큼 임시 저장 공간 필요
  => 추가 메모리: O(n)

---

## Q4. 퀵 정렬이 평균적으로 병합 정렬보다 빠른 이유는?

[답변]
* 퀵 정렬
  - 평균적으로 in-place(제자리) 분할 가능 -> 추가 메모리 접근이 적음 + 캐시 친화적
  - 평균적으로 비교 횟수/상수 계수가 작음
  - 분할 후 각 부분을 정렬하는 과정이 실제 하드웨어에서 효율적으로 동작하는 경우가 많음
  - 병합 정렬: 병합 시 보조 배열에 삽입, 복사 -> 메모리 대역폭 비용이 상대적으로 큼

[보완]
* 빅오가 아닌, 상수 계쑤와 메모리 접근 패턴의 차이에서 주로 발생
* 데이터 특성에 따라 병합 정렬이나 TimSort가 유리할 수 있음

---

## Q5. 퀵 정렬의 최악의 경우는 언제 발생하며 시간복잡도는?

[답변]
* 최악의 경우: 분할이 매번 극단적으로 불균형 (ex. 한쪽이 n-1, 다른 쪽이 0에 가깝게 분할)
  - ex. 이미 정렬된 배열에 대해 피벗을 첫 원소/마지막 원소로 고정 선택
    => 점화식: T(n) = T(n-1) + O(n), T(n) = O(n^2)

[보완]
* 중복 원소가 매우 많을 떄도 분할이 나빠질 수 있음(2-way partition일 때)
  => 3-way partition 등 적용

---

## Q6. 피벗 선택 방법(첫 번째, 마지막, 중간, 랜덤)에 따른 성능 차이는?

[답변]
* 첫 번째/마지막 원소 피벗
  - 구현 단순
  - 입력이 이미 정렬/역정렬이면 분할이 한쪽으로 쏠림 -> 최악 O(n^2) 발생 가능
* 중간 원소 피벗
  - 정렬/역정렬 같은 패턴 입력에서 첫/마지막보다 나음
  - 값의 분포에 따라 여전히 불균형 가능성 ('인덱스 중앙'이나, '값의 중앙'은 아닐 수 있음)
* 랜덤 피벗
  - 특정 입력 패턴에 의해 망가지는 상황 줄임 -> 기대 시간복잡도 O(nlogn)이 되는데 유리
* median-of-three(3개 중 중앙값) 등 휴리스틱
  - 첫/중간/마지막 등의 후보에서 중앙값 선택 -> 극단 분할 줄임, 평균 성능 개선

[보완]
* '좋은 피벗': 양쪽 크기를 비슷하게 만듦
* 랜덤 피벗: 항상 좋은 분할을 보장하지 않음. 최악 패턴 입력에 고정적으로 당하는 위험 확률적으로 완화

---

## Q7. 힙 정렬의 시간복잡도와 공간복잡도는? 왜 O(n log n)인가?

[답변]
* 시간복잡도: O(nlogn)
  - 힙 구성(build heap): O(n)
  - n번의 삭제(최댓값/최솟값 추출) 과정: 각 추출마다 heapify가 O(logn) -> 총 O(nlogn)
* 공간복잡도: 일반적인 배열 기반 구현에서 O(1) 추가 공간(제자리 정렬)

[보완]
* heapify: 힙화. 힙 성질을 깨뜨린 원소를 다시 힙 성질이 성립하도록 고치는 과정

---

## Q8. 실무에서 어떤 정렬 알고리즘을 사용하는가? (Tim Sort, Intro Sort)

[답변]
* 순서 정렬보단 하이브리드 정렬 사용
* Tim Sort
  - 병합 정렬 기반 + 삽입 정렬
  - '부분적으로 정렬된' 데이터에서 매우 강함
  - 안정 정렬(stable)이 필요할 때 선호
  - 대표적으로 Python의 sorted()/list.sort() 등에 사용
* Intro Sort
  - 기본은 퀵 정럴 -> 재귀 깊이가 커져 최악이 의심되면 힙 정렬로 전환 => 최악 O(nlogn) 보장
  - 평균 성능은 퀵 정렬 수준, 최악 방어는 힙 정렬 -> C++의 std::sort 등
