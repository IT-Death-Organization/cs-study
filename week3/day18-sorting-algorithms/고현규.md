# 1. 병합 정렬의 분할 정복 원리를 설명하면?

> 분할 정복(Divide and Conquer)이란 하나의 큰 문제를 더이상 나눌 수 없을 때까지 나눈 후, 각각을 해결하고 다시 합쳐서 정답을 찾아가는 알고리즘입니다. 병합 정렬(Merge Sort)은 분할 정복 알고리즘의 대표적인 예시입니다.

### 1) 분할
한번에 정렬하기 어려운 큰 배열을 반으로 자릅니다. 그리고 이 과정을 재귀적으로 반복해서 배열 조각의 크기가 1이 될 때까지 반복합니다. 크기가 1이 된 배열은 그 자체로 정렬이 되었다고 볼 수 있습니다.

### 2) 정복
작게 쪼개진 배열들을 정렬합니다.

### 3) 결합
나누었던 두 배열을 하나의 정렬된 배열로 합쳐나갑니다. 이때, 두 배열의 첫 번째 원소부터 하나씩 비교하고 더 작은 값을 새로운 임시 배열에 추가합니다.

<br>

# 2. 병합 정렬이 항상 O(n log n)인 이유는?

> 병합 정렬은 배열을 절반으로 나누는 단계($\log n$)와 각 단계에서 데이터를 병합하는 단계($n$)가 곱해져 항상 일정한 성능을 보장하는 알고리즘입니다.

데이터 개수가 $n$개라면 절반씩 나누는 층수는 $\log_2 n$입니다.

병합 단계에서 일어나는 비교 및 이동 연산의 합은 항상 $n$입니다. 왜냐하면, 전체 데이터 $n$개를 한 번씩 훑으면서 새 배열에 옮기기 때문입니다.

따라서 병합 정렬의 시간복잡도는 $O(n \log n)$입니다.

어떠한 데이터가 주어져도 항상 일정한 성능을 보장하는 병합 정렬은 가장 안정적인 알고리즘입니다. 운에 따라 결과가 다른 퀵 정렬과는 다르죠.

<br>

# 3. 병합 정렬의 공간복잡도가 O(n)인 이유는?

> 병합 정렬에서 정렬된 두 조각을 하나로 합치는 과정에서 데이터를 잠시 보관할 임시 배열이 반드시 필요하기 때문입니다.

원본 배열과 똑같은 크기의 임시 배열을 만들어두고 병합 시 왼쪽 배열과 오른쪽 배열의 원소를 비교해서 더 작은 숫자를 임시 배열에 복사해두고, 병합이 끝나면 다시 원본 배열로 데이터를 복사하는 과정을 거칩니다.

또한 재귀 호출 스택을 위한 공간($O(\log n)$)도 필요합니다. 하지만 큰 항은 O(n)이므로, 전체 공간복잡도는 $O(n)$ 으로 봅니다.

제자리 정렬(In-place Sort)이 가능하여 공간 복잡도가 $O(1)$ 또는 $O(\log n)$인 퀵 정렬이나 힙 정렬과는 다릅니다.

메모리 용량이 수 MB 단위였던 과거에는 $O(n)$이라는 공간 복잡도는 큰 부담이었습니다. 하지만 현대에는 얘기가 다르죠. 오히려 데이터의 순서가 바뀌지 않는 안정성이 더 중요해지면서, 병합 정렬은 현대의 정렬 알고리즘의 뼈대가 되었습니다.

<br>

# 4. 퀵 정렬이 평균적으로 병합 정렬보다 빠른 이유는?

> 시간 복잡도는 둘 다 $O(n \log n)$이지만, 퀵 정렬은 평균적으로 더 빠릅니다.

퀵 정렬은 추가 메모리 할당이 필요 없는 제자리 정렬(In-place Sort)이며, 데이터가 메모리상에서 인접하게 배치되어 캐시 적중률(Cache Locality)이 매우 높기 때문입니다. CPU는 한 번 읽은 메모리 근처의 데이터를 캐시에 미리 올려두는 방식으로 속도를 높이는데, 퀵 정렬은 배열 안에서 인접한 요소들끼리 비교하여 자리를 바꾸기 때문에 캐시 적중률이 매우 높아집니다. 하지만 병합 정렬은 데이터를 나누고 합치는 과정에서 참조하는 메모리 주소가 흩어져있게 되기 때문에 캐시 효율이 상대적으로 떨어집니다.

두 정렬의 시간 복잡도가 같은 이유는 시간 복잡도를 나타낼 때 쓰는 빅오 표기법은 증가 추세만 보여줄 뿐, 실제 실행 시 곱해지는 상수는 무시하기 때문입니다.

실제 알고리즘의 수행을 보면 병합 정렬에선 데이터를 임시 배열로 복사하고, 다시 원본으로 옮기는 과정(n만큼의 이동)이 매 단계마다 필요합니다. 반면, 퀵 정렬은 추가 복사가 없고 내부에서 데이터의 위치만 바꾸는 단순 연산이 위주라서 실제 연산 횟수는 훨씬 적습니다.

또한 추가적인 공간 할당도 오버헤드가 되어 성능에 영향을 미칠 수 있습니다.

이러한 이유들로 인해 퀵 정렬이 "퀵"이라 불리게 되었습니다.

<br>

# 5. 퀵 정렬의 최악의 경우는 언제 발생하며 시간복잡도는?

> 퀵 정렬은 특정 상황에선 $O(n^2)$의 시간 복잡도를 가질 수 있습니다. 피벗(Pivot)이 매번 배열의 최솟값이나 최댓값으로 선택되어, 파티션이 한쪽으로 극단적으로 치우치는 경우 발생합니다.

가장 대표적인 최악의 상황은 이미 정렬된 배열(Sorted Array) 혹은 역순으로 정렬된 배열에 대해 첫 번째나 마지막 원소를 피벗으로 선택할 때입니다. 퀵 정렬은 피벗을 기준으로 작은 값은 왼쪽, 큰 값은 오른쪽으로 분리하므로, 이미 정렬된 배열의 경우 피벗은 항상 최소값이나 최댓값이 되어 파티션이 한쪽으로 치우치게 됩니다.

또한 구조적으로 배열이 0과 $n-1$개로 쪼개지면서 트리 높이가 $n$이 되어 트리 구조가 아니라 연결 리스트처럼 길어집니다.

시간복잡도의 경우 매 단계마다 $n$, $n-1$, $n-2$, ... , $1$번의 비교 연산을 수행해야 하므로 $O(n^2)$의 시간 복잡도를 가집니다.

<br>

# 6. 피벗 선택 방법(첫 번째, 마지막, 중간, 랜덤)에 따른 성능 차이는?

> 피벗(Pivot)을 어떻게 선택하느냐는 퀵 정렬의 성능이 $O(n \log n)$이 될지 $O(n^2)$이 될지를 결정짓는 가장 중요한 요소입니다.

### 1) 첫 번째 또는 마지막 원소 (First/Last)
가장 구현하기 쉬운 방법입니다. 하지만 배열이 이미 정렬되어 있거나 역순인 경우, 피벗이 항상 최솟값이나 최댓값이 되어 최악의 성능($O(n^2)$)을 보여줍니다. 실무에서는 거의 사용하지 않습니다.

### 2) 중간 원소 (Middle)
배열의 정가운데 원소를 피벗으로 잡습니다. 이미 정렬된 배열에서도 파티션을 어느 정도 균등하게 나눌 수 있어 첫 번째/마지막 선택 방식보다 훨씬 안정적입니다.

### 3) 랜덤 선택 (Random)
배열 중 무작위로 하나를 골라 피벗으로 사용합니다. 확률적으로 최악의 상황을 피할 수 있게 해주지만, 난수 생성 자체에 오버헤드가 발생할 수 있다는 단점이 있습니다.

### 4) Median-of-Three (세 값의 중앙값)
실무에서 가장 많이 선호되는 방식입니다. 배열의 **첫 번째, 중간, 마지막** 세 원소 중 중앙값을 피벗으로 선택합니다.
* **장점:** 정렬된 데이터에서도 균형 잡힌 분할을 보장할 확률이 매우 높으며, 구현이 크게 복잡하지 않으면서도 최악의 경우를 효과적으로 방지합니다.

# 7. 힙 정렬의 시간복잡도와 공간복잡도는? 왜 O(n log n)인가?

> 힙 정렬(Heap Sort)은 완전 이진 트리의 일종인 힙(Heap) 자료구조를 이용하여 정렬하는 방식입니다.

### 1) 시간 복잡도: 항상 $O(n \log n)$
힙 정렬은 데이터의 상태와 상관없이 항상 일정한 성능을 보장합니다.
* **Heapify 과정**
    * 일반 배열을 힙 구조로 만드는 데 $O(n)$이 소요됩니다.
* **삭제 및 재정렬**
    * 루트 노드(최댓값)를 꺼내고 다시 힙 구조를 유지(Heapify)하는 과정이 $\log n$만큼 걸립니다. 이를 모든 요소($n$개)에 대해 수행하므로 $n \times \log n$이 됩니다.
* **결론**
    * $O(n) + O(n \log n) \approx O(n \log n)$

### 2) 공간 복잡도: $O(1)$
병합 정렬과 달리 추가적인 임시 배열이 필요하지 않습니다. 주어진 배열 내부에서 요소들의 위치만 바꾸며 정렬을 수행하는 제자리 정렬(In-place Sort)이기 때문에 공간 복잡도는 $O(1)$입니다.

---
### 퀵/병합 정렬과의 비교
* **vs 병합 정렬**
    * 시간 복잡도는 같고 공간은 힙 정렬이 덜 쓰지만, 병합 정렬은 안정 정렬(Stable)인 반면 힙 정렬은 불안정 정렬입니다.
* **vs 퀵 정렬**
    * 힙 정렬은 최악의 경우에도 $O(n \log n)$을 보장하지만, 실제 평균 속도는 퀵 정렬이 더 빠릅니다. 힙 정렬은 인덱스가 사방으로 튀기 때문에 캐시 적중률(Cache Locality)이 낮기 때문입니다.

<br>

이러한 힙 정렬은 전체를 정렬하기보다 가장 큰/작은 값 몇 개만 필요할 때 혹은 메모리 사용량이 극도로 제한된 환경에서 안정적인 $O(n \log n)$이 필요할 때 최고의 효율을 보여줍니다.

# 8. 실무에서 어떤 정렬 알고리즘을 사용하는가? (Tim Sort, Intro Sort)

> 현대 프로그래밍 언어의 표준 라이브러리들은 하나의 알고리즘만 쓰지 않고, 상황에 맞춰 여러 알고리즘을 섞어 쓰는 하이브리드 정렬(Hybrid Sort)을 사용합니다.

### 1) Tim Sort (팀 정렬)
* **기반** 
    * 병합 정렬 + 삽입 정렬
* **사용 언어** 
    * Java (Collections.sort), Python, Swift 등
* **특징**
    * 데이터가 이미 어느 정도 정렬되어 있는 실무 데이터의 특성을 이용합니다. 작은 덩어리는 삽입 정렬로 해결하고, 큰 덩어리는 병합 정렬로 합칩니다. 안정 정렬(Stable)이며 최선의 경우 $O(n)$까지 속도가 나옵니다.

### 2) Intro Sort (인트로 정렬)
* **기반** 
    * 퀵 정렬 + 힙 정렬 + 삽입 정렬
* **사용 언어** 
    * C++ (std::sort)
* **특징**
    * 기본적으로는 가장 빠른 퀵 정렬을 사용하다가, 재귀 깊이가 너무 깊어져서 최악의 경우($O(n^2)$)가 우려되면 힙 정렬로 전환하여 성능을 방어합니다. 데이터 개수가 아주 적어지면 삽입 정렬로 마무리합니다.
