## 1. BST에서 탐색이 평균 O(log n)인 이유는?
>연결 리스트는 최악의 경우 모든 노드를 다 훑어야 하므로 **O(n)** 이 걸리는 반면, BST의 경우 트리가 균형을 이루고 있다면, 한 번의 비교를 거칠 때마다 탐색해야 할 노드의 수가 약 절반씩 줄어듭니다. 이는 정렬된 배열에서의 **이진 탐색** 원리와 동일하며, 이 과정이 반복될 때의 시간복잡도가 **O(logn)** 이 됩니다.

## 2. BST의 삭제에서 자식이 2개인 노드를 삭제할 때 처리 방법은?
> 삭제할 노드의 값을 후속자 값으로 초기화하고, 후속자 노드를 삭제하는 방식으로 처리합니다. 후속자는 오른쪽 서브트리의 최소값을 가지므로, 왼쪽 서브트리의 노노드를 삭제해야 합니다.
### 2-1. 대체 노드 선택
삭제할 노드를 대신해 그 자리에 들어올 수 있는 후보는
1. **In-order Successor (중위 후속자)** : 삭제할 노드보다 **큰 값들 중 가장 작은 값** (오른쪽 서브트리의 가장 왼쪽 노드)
2. **In-order Predecessor (중위 선행자)** : 삭제할 노드보다 **작은 값들 중 가장 큰 값**

보통은 **In-order Successor**를 사용하는 것이 일반적입니다.

### 2-2. 삭제 프로세스 (Successor 기준)
 자식이 2개인 노드 V를 삭제하는 단계는 다음과 같습니다.
1. **후속자 찾기** : 노드 V의 오른쪽 서브 트리로 이동한 후, 거기서 계속 왼쪽 자식을 따라 내려가 더 이상 왼쪽 자식이 없는 노드(S)를 찾습니다. ▷ 이 노드가 후속자
2. **값 복사** : 후속자(S)가 가진 값을 삭제할 노드(V)의 위치에 덮어씌웁니다. 이 때 V의 원래 값은 사라지지만, 트리의 BST 조건은 유지됩니다.
3. **후속자 노드 삭제** : 원래 후속자가 있던 노드(S)를 트리에서 삭제합니다.
- 후속자 노드는 **가장 왼쪽** 노드이므로, **왼쪽 자식은 절대 가질 수 없습니다.**(∵ 있었다면 더 내려갔을 것이기 때문)
- ∴ S는 자식이 없거나(리프), 오른쪽 자식만 1개 있는 상태가 됩니다.(이는 자식이 0~1개인 노드 삭제 케이스로 처리 가능)

##3. BST가 편향 트리가 되면 시간복잡도가 O(n)이 되는 이유는?
> 트리의 모양이 배열과 같아지기 때문에 h=n이 되어 **O(n)** 의 시간복잡도를 갖게 됩니다.

![title](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Fblog.kakaocdn.net%2Fdna%2FtVBlB%2FbtsGdtaFzbF%2FAAAAAAAAAAAAAAAAAAAAABkJpPQq69OKRiESEPkwRkPaFeRzOh479khDklg1mcJz%2Fimg.png%3Fcredential%3DyqXZFxpELC7KVnFOS48ylbz2pIh7yKj8%26expires%3D1774969199%26allow_ip%3D%26allow_referer%3D%26signature%3DyE6UqnMde4An2Uh%252FvPUb3vLLuLw%253D)   
이진탐색 트리는 h, 즉 트리가 얼마나 낮은 높이를 갖게 되는지에 따라 성능이 좌우됩니다. (검색, 삽입과 같은 연산이 트리의 높이 h만큼 진행됨)

BST의 탐색 원래는 "현재 노드보다 작으면 왼쪽, 크면 오른쪽"으로 가며 후보군을 절반씩 쳐내는 것인데, 편향 트리에서는 이 '절반'의 이점이 사라집니다. 편향 트리는 한쪽으로만 늘어 연결리스트처럼 보이며, 높이가 n-1까지 커질 수 있습니다. 이 때 연산이 높이만큼 진행되므로 **O(n)** 이 됩니다.

| 구분 | 균형 BST (Best/Average) | 편향 BST (Worst) |
| --- | --- | --- |
| 구조 | 좌우 자식이 균형 있게 배치됨 | 3 |
| 높이(h) | log₂ n | n |
| 탐색 시간 | O(log n) | O(n) |


## 4. AVL 트리는 어떻게 균형을 유지하는가? 회전의 종류는?
AVL 트리는 **Balance Factor(BF)** 라는 개념을 도입해 엄격하게 균형을 관리합니다.

**BF = (왼쪽 서브트리의 높이) - (오른쪽 서브트리의 높이)**
- 모든 노드에서 BF의 절댓값이 **1 이하(-1, 0, 1)** 여야 하며, 이를 벗어나면 **회전(Rotation)** 을 통해 균형을 맞춥니다.

### 4-1. 회전의 종류
1. **LL (Left-Left)**: 왼쪽 자식의 왼쪽 서브트리에 노드가 추가된 경우 → **우회전(Right Rotation)** 1회
1. **RR(Right-Right)**: 오른쪽 자식의 오른쪽 서브트리에 노드가 추가된 경우 → **좌회전(Left Rotation)** 1회
1. **LR (Left-Right)**: 왼쪽 자식의 오른쪽 서브트리에 노드가 추가된 경우 → **좌회전 후 우회전**
1. **RL (Right-Left)**: 오른쪽 자식의 왼쪽 서브트리에 노드가 추가된 경우 → **우회전 후 좌회전**

## 5. Red-Black 트리의 특성은 무엇인가?
> Red-Black 트리는 AVL보다 균형을 조금 덜 엄격하게 잡지만 삽입/삭제 시 오버헤드가 적어 실무에서 가장 많이 쓰이는 균형 이진 탐색 트리입니다.

![title](https://img1.daumcdn.net/thumb/R800x0/?scode=mtistory2&fname=https:%2F%2Fblog.kakaocdn.net%2Fdn%2FQBRWd%2FbtsKhIBAoCc%2FCrqHbLvGvmgKvPmqokpk01%2Fimg.png)   

Red-Black 트리의 경우 다음 5가지 속성을 반드시 만족해야 합니다.
1. 모든 노드는 **Red** 혹은 **Black** 이다.
2. **Root** 노드는 항상 **Black** 이다.
3. 모든 **Leaf(NIL)** 노드는 **Black** 이다.
4. **Red** 노드의 자식은 항상 **Black** 이다. (∵ Red 가 연속해서 나올 수 없음)
5. 임의의 노드에서 리프 노드에 이르는 모든 경로에는 **동일한 개수의 Black 노드**가 있다. (Black-Height)
## 6. AVL 트리와 Red-Black 트리 중 어떤 것을 선택해야 하는가?


| 비교 항목 | AVL 트리 | Red-Black 트리 |
| --- | --- | --- |
| 균형 수준 | **엄격함** (BF 1 이하) | **느슨함**(높이 차이 최대 2배) |
| 탐색 성능 | **매우 빠름** (높이가 더 낮음) | 빠름 |
| 삽입/삭제 | 느림 (회전이 잦음) | **빠름** (회전이 적음) |
| 사용 사례 | 변동이 적고 **조회가 빈번**할 때 | **데이터 삽입/삭제가 잦을 때** |
>- Dictionary나 정적 인덱스처럼 한 번 구축해놓고 검색만 주구장창 한다면 AVL 트리가 유리합니다.
>- Java의 TreeMap, HashMap(Treeify), C++의 STL(map/set), Linux 스케줄러처럼 데이터가 계속 변하는 범용적인 상황에서는 Red-Black 트리가 적합합니다.
