# 1. BST에서 탐색이 평균 O(log n)인 이유는?

> 기본적으로 이진 탐색 트리(BST)는 모든 노드가 왼쪽 자식, 부모, 오른쪽 자식이라는 약속된 형태가 있기 때문에 탐색 시 매 단계마다 탐색 범위를 절반으로 줄여나갈 수 있습니다.

데이터의 갯수가 $n$개일 때, 트리가 균형있게 잘 생성되었다면 트리의 높이는 대략 $log_2 n$이 됩니다.

따라서 탐색 시 매 단계마다 탐색 범위를 절반으로 줄여나갈 수 있기 때문에 탐색 시간복잡도는 평균적으로 $O(\log n)$이 됩니다.

<br>

# 2. BST의 삭제에서 자식이 2개인 노드를 삭제할 때 처리 방법은?

> BST의 형태(왼쪽 자식, 부모, 오른쪽 자식)를 유지하면서 노드를 삭제해야 합니다.

### 1) 후계자 찾기 (In-order Successor / Predecessor)
두 가지 후보 중 하나를 선택합니다. 보통 오른쪽 자식 중 가장 작은 값을 선호합니다.

* **In-order Successor**: 삭제할 노드의 오른쪽 자식 중 가장 작은 값
* **In-order Predecessor**: 삭제할 노드의 왼쪽 자식 중 가장 큰 값

### 2) 삭제 및 교체
1. 삭제하려는 노드 `X`를 찾습니다.
2. `X`의 후계자 `S`를 찾습니다.
3. `X`의 값을 `S`의 값으로 복사합니다.
4. `S`를 삭제합니다.

후계자는 왼쪽 서브트리에서 최댓값이거나 오른쪽 서브트리에서 최솟값이기 때문에 왼쪽 후계자는 오른쪽면의 자식이 절대 없고, 오른쪽 후계자는 왼쪽면의 자식이 절대 없습니다. 따라서 쉽게 교체 가능합니다.

<br>

# 3. BST가 편향 트리가 되면 시간복잡도가 O(n)이 되는 이유는?

> 편향 트리란 노드들이 한쪽 방향으로만 치우쳐서 자라난 형태를 말합니다. 더이상 트리라기보단, 연결 리스트에 가깝습니다.

BST의 장점(높이가 $\log_2 n$)이 사라지므로 탐색, 삽입, 삭제의 시간 복잡도는 최악의 경우 $O(n)$이 됩니다.

보통 이미 정렬된 데이터가 들어올 때 이런 일이 발생합니다.

* 예: [1, 2, 3, 4, 5] 순으로 삽입이 이루어진다면 매번 현재 루트보다 큰 값이 들어오므로 계속 오른쪽으로만 자라날 것입니다.

<br>

# 4. AVL 트리는 어떻게 균형을 유지하는가? 회전의 종류는?

> AVL 트리란 발명자의 이름인 Adelson-Velsky and Landis에서 유래한 이름으로, 데이터가 들어올 때마다 스스로 높이를 계산해 균형을 맞춥니다.

AVL 트리는 모든 노드에서 좌우의 서브트리의 높이 차이를 1 이하로 유지하며, 균형이 깨질 때 회전을 통해 스스로 구조를 재조정합니다.

### 1) 균형 인수 (Balance Factor, BF)

AVL 트리의 핵심 지표입니다.

$$BF = height(Left\ Subtree) - height(Right\ Subtree)$$

모든 노드의 BF가 -1, 0, 1중 하나여야 합니다.

이 규칙이 깨지는 순간 회전을 수행합니다.

### 2) 회전 (Rotation)
4가지의 회전이 있습니다.

1. LL 회전
    * 왼쪽 자식의 왼쪽에 노드가 추가된 경우
        * 우회전(Right Rotation) 1회로 해결
2. RR 회전
    * 오른쪽 자식의 오른쪽에 노드가 추가된 경우
        * 좌회전(Left Rotation) 1회로 해결
3. LR 회전
    * 왼쪽 자식의 오른쪽에 노드가 추가된 경우
        * 좌회전 후 우회전 (2회)
4. RL 회전
    * 오른쪽 자식의 왼쪽에 노드가 추가된 경우
        * 우회전 후 좌회전 (2회)

# 5. Red-Black 트리의 특성은 무엇인가?

> Red-Black 트리는 노드에 색상(Red 또는 Black)이라는 속성을 부여하고, 5가지 규칙을 통해 트리의 균형을 유지합니다.

### 5가지 필수 규칙
1. **노드 색상:** 모든 노드는 빨간색 아니면 검은색입니다.
2. **루트 노드:** 루트 노드는 무조건 검은색입니다.
3. **리프 노드(NIL):** 모든 리프 노드(데이터가 없는 말단 노드)는 검은색입니다.
4. **No Double Red:** 빨간색 노드의 자식은 반드시 검은색이어야 합니다.
5. **Black Height:** 임의의 노드에서 그 자손인 리프 노드에 이르는 모든 경로에는 동일한 개수의 검은색 노드가 있어야 합니다.

<br>

이 5가지 규칙을 유지한 채로 새로운 노드를 삽입할 때(보통 Red로 삽입) 위 규칙이 깨지면 주변 노드의 색을 바꾸거나(Recoloring) 트리를 비트는(Rotation) 작업을 수행하여 규칙을 다시 복구합니다.

Red-Black 트리는 AVL 트리처럼 좌우 높이차를 1 이하로 칼같이 맞추지 않습니다. 대신 검은색 노드의 갯수를 맞추고 약간의 높이 차이는 허용합니다.

이 덕분에 트리 재정렬의 횟수가 AVL 트리보다 적습니다. 데이터의 추가와 삭제가 빈번한 실무환경에서 뛰어난 성능을 보여주기 때문에 Java의 `TreeMap이나` `HashMap`의 내부 구조, 그리고 리눅스 커널 등 실무에서 가장 널리 쓰이는 트리입니다.
 
<br>

# 6. AVL 트리와 Red-Black 트리 중 어떤 것을 선택해야 하는가?

> 조회가 압도적으로 많다면 AVL 트리, 삽입/삭제가 빈번하다면 Red-Black 트리를 선택합니다.

| 구분 | AVL 트리 | Red-Black 트리 |
| :--- | :--- | :--- |
| **균형 강도** | 매우 엄격 (높이 차이 $\le 1$) | 비교적 느슨 (검은색 노드 기준) |
| **탐색 속도** | 빠름 (트리 높이가 더 낮음) | 보통 (AVL보다 미세하게 느림) |
| **삽입/삭제 속도** | 느림 (회전이 많음) | 빠름 (회전 횟수가 적음) |
| **구현 난이도** | 보통 | 매우 복잡함 |
| **실무 사용** | 특정 조회 최적화 시스템 | 대부분의 표준 라이브러리 (STL, Java 등) |
---
<br>

### AVL 트리를 선택하는 경우
- 데이터가 한 번 구축되면 거의 변하지 않는 경우 (Read-Only, Read-Intensive)
- 검색 엔진의 사전(Dictionary)이나 고정된 설정값 조회 시스템처럼 탐색 속도가 절대적으로 중요한 경우

### Red-Black 트리를 선택하는 경우
- 데이터가 빈번하게 추가/삭제가 이루어지는 일반적인 상황
