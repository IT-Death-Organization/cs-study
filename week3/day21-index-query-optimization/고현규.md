# 1. 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가?

만약 인덱스라는게 없다면 1000만개의 데이터가 있을 때 사용자는 원하는 데이터를 찾기 위해 1000만개의 데이터를 모두 확인하는 대참사가 일어납니다.

이 사태를 막기 위해 우리는 인덱스를 사용합니다.

### 조회 성능 향상
조회 성능을 향상시키기 위해 정렬된 구조와 포인터를 사용합니다.

인덱스는 특정 컬럼의 값과 그 값이 위치한 주소(RawID)를 따로 뽑아 정렬된 상태로 저장합니다.

데이터가 정렬되어있기 때문에 $O(\log n)$의 속도로 데이터를 찾을 수 있습니다.

또한 인덱스를 이용해 필요한 데이터만 가져오기 때문에 비싼 작업인 I/O를 엄청 줄일 수 있습니다.

이런 장점(조회 속도, 특히 `WHERE`, `JOIN`, `ORDER BY`)을 얻기 위해 쓰기 성능을 희생합니다.

<br>

# 2. 인덱스는 어떻게 동작하는가? B-Tree 구조를 설명하면?

> B-Tree는 균형 트리(Balanced Tree)의 한 종류로, 자식 노드가 2개 이상일 수 있다는 다원 탐색 트리이고, 데이터가 삽입/삭제될 때 스스로 균형을 유지하여 어떤 데이터를 찾더라도 일정한 탐색 시간($O(\log n)$)을 보장합니다. 대부분의 데이터베이스(MySQL, Oracle, PostgreSQL 등)에서 기본으로 사용하는 가장 표준적인 인덱스 구조입니다.

### B-Tree 구조

- 루트 노드(Root Node)
    - 트리의 가장 상단에 위치한 노드
    - 하위 브렌치로 가기 위한 가이드 역할을 함
- 브랜치 노드(Branch Node)
    - 루트 노드와 리프 노드 사이에 위치한 노드
    - 데이터의 범위를 나누어 탐색 경로를 좁혀주는 징검다리
- 리프 노드(Leaf Node)
    - 트리의 가장 하단에 위치한 노드
    - 실제 데이터의 주소값을 가지고있음
    - 인덱스 컬럼의 값들이 정렬된 상태로 저장되어있음

<br>

일반적인 이진 트리와 달리 한 노드에 많은 키 값을 담을 수 있습니다. DB는 데이터를 페이지 단위로 읽어오는데, B-Tree는 한 노드를 페이지 크기에 맞춰 설계하여 단 한 번의 디스크 읽기로 많은 경로 정보를 얻습니다. 덕분에 트리의 높이를 매우 낮게 유지(보통 3~4단계)할 수 있어 수억 건의 데이터도 쉽게 찾아냅니다.

# 3. Hash Index와 B-Tree Index의 차이는? 각각 언제 사용하는가?

> Hash Index는 해시 함수를 이용해 키 값을 주소로 변환하는 방식이고, B-Tree Index는 값을 정렬된 트리 구조로 관리하는 방식입니다.

Hash Index는 정렬되어있지 않다는 점에서 B-Tree Index와 차이가 있습니다.

### Hash Index

Hash Index는 키 값을 해시 함수에 넣어 주소를 바로 찾습니다.

따라서 $O(1)$의 매우 빠른 조회 속도를 가집니다.

하지만 데이터들이 정렬되어있지 않기 때문에 범위 검색(`>`, `<`)이나 부분 일치 검색(`LIKE`)이 불가능합니다.

따라서 오직 동등 비교(`=`)만 필요한 상황에서 사용합니다.

### B-Tree Index

B-Tree Index는 값을 정렬된 트리 구조로 관리하는 방식입니다.

조회가 $O(\log n)$의 속도로 준수하고, 부등호(`>`, `<`) 를 사용한 범위 검색과 정렬(`ORDER BY`) 결과를 그대로 활용할 수 있습니다.

대부분의 일반적인 상황(검색 요청 처리)에서 최적이며, 관계형 데이터베이스의 기본 인덱스 방식입니다.

---
<br>

보통 쿼리에서 `id=123`처럼 딱 하나만 찾는 경우보다 `price > 10000` 이나 `name LIKE '김` 처럼 범위 조회, 정렬하는 경우가 훨씬 많기 때문에 B-Tree Index가 기본으로 사용됩니다.

# 4. 클러스터드 인덱스와 논클러스터드 인덱스의 차이는?

> 클러스터드 인덱스는 데이터 자체가 인덱스 순서로 저장되는 방식이고, 논클러스터드 인덱스는 데이터는 그대로 두고 주소값만 별도로 관리하는 방식입니다.

### 클러스터드 인덱스 (Clustered Index)
- **정의** 
    * 테이블당 단 하나만 존재할 수 있으며, 보통 기본키(Primary Key)가 이 역할을 합니다.
- **특징** 
    * 인덱스의 리프 노드에 실제 행 데이터가 들어있습니다. 즉, 인덱스 순서가 곧 물리적인 저장 순서입니다.
    * 삽입 시 데이터 재정렬 비용이 큽니다.
- **장점** 
    * 인덱스만 찾으면 바로 데이터가 거기 있으므로 조회 속도가 매우 빠릅니다.

단어 자체가 가나다 순으로 배치된 국어사전과 같습니다.

### 논클러스터드 인덱스 (Non-Clustered Index)
- **정의** 
    * 테이블당 여러 개를 만들 수 있는 일반적인 인덱스(Secondary Index)입니다.
- **특징** 
    * 리프 노드에 데이터 자체가 아닌, 데이터가 위치한 주소(클러스터드 인덱스 키)가 들어있습니다.
- **조회 과정** 
    * 인덱스 페이지에서 주소를 찾은 뒤, 다시 실제 데이터 페이지로 찾아가는 '랜덤 I/O'가 추가로 발생합니다.

본문은 따로 있고 쪽수만 적힌 일반 책의 색인과 같습니다.
<br>

# 5. 인덱스의 단점은 무엇인가? (저장 공간, CUD 성능 저하)

> 인덱스는 빠른 조회를 얻는 대신, 저장 공간과 쓰기 성능을 지불하는 트레이드오프입니다. 따라서 정말 필요한 컬럼에 전략적으로 사용해야합니다.

### 쓰기(INSERT, UPDATE, DELETE) 성능 저하

새로운 데이터를 넣을 때마다 인덱스 페이지도 매번 정렬하고 수정해야 합니다.

특히 인덱스가 많을수록 하나의 데이터를 넣을 때 고쳐야 할 인덱스 파일이 늘어나 전체적인 서비스 성능이 급격히 떨어집니다.

### 추가 저장 공간 필요

인덱스는 별도의 파일로 관리됩니다. 보통 테이블 크기의 약 10~20% 정도의 추가 공간을 차지하며, 인덱스가 많아지면 저장공간이 부족해질 수 있습니다.

### 인덱스 단편화

데이터의 삭제와 수정이 빈번하면 인덱스 내부 공간에 구멍이 생겨 성능이 저하됩니다. 이 경우 인덱스를 재구성해줘야 하는 유지보수 비용이 발생합니다.

# 6. 어떤 컬럼에 인덱스를 설정해야 하는가? (카디널리티, 선택도)

> 인덱스의 효율은 데이터가 얼마나 잘 걸러지는가에 달려있습니다.

### 카디널리티 (Cardinality)

전체 데이터 중 중복도가 낮은 정도를 말합니다.

주민번호, ID, 이메일과 같은 컬럼은 데이터가 다 제각각이기 때문에 카디널리티가 높습니다. Good

성별, 결제여부와 같은 컬럼은 데이터가 거의 겹치기 때문에 카디널리티가 낮습니다. Bad

카디널리티가 높은 컬럼에 인덱스를 걸어야 탐색 범위를 확 줄일 수 있습니다.

### 선택도 (Selectivity)

특정 조건으로 조회했을 때 얼마나 적은 양의 데이터가 선택되는지를 나타냅니다. 

보통 선택도가 5~10% 이내인 컬럼이 인덱스로 사용하기 적합합니다. 너무 많은 데이터가 선택되면 인덱스를 쓰는 것보다 그냥 다 읽는게 더 빠를 수도 있습니다.

# 7. 복합 인덱스(Composite Index)는 언제 사용하며 컬럼 순서는 왜 중요한가?

> 두 개 이상의 컬럼을 합쳐서 만드는 인덱스를 복합 인덱스라고 하며, 정렬 순서에 따른 '좌측 접두사(Leftmost Prefix)' 규칙이 핵심입니다.

### 컬럼 순서의 중요성

복합 인덱스는 첫 번째 컬럼으로 정렬한 뒤, 그 안에서 두 번째 컬럼으로 다시 정렬합니다. 
- 예: `(성별, 학번)` 인덱스인 경우
    - '남성' 데이터를 먼저 모으고, 그 안에서 '학번'순으로 정렬됩니다.
- 만약 `WHERE 학번 = 202111242`이라는 쿼리를 날리면, 이 인덱스는 사용될 수 없습니다. 첫 단추인 성별이 없으면 탐색 자체가 불가능하기 때문입니다.

### 설계 원칙

- `WHERE` 절에서 자주 사용되는 컬럼을 우선순위로 둡니다.
- 카디널리티가 높은 컬럼(잘 안겹치는)을 앞쪽에 배치하는 것이 일반적으로 탐색 범위를 빠르게 줄여 성능에 유리합니다.

<br>

전화번호부에서 김철수를 찾을 때 당연히 '김'씨쪽을 먼저 간후 철수를 찾지, 철수를 먼저 찾진 않는 것과 같습니다.

# 8. 실행 계획(Execution Plan)은 어떻게 보고 분석하는가?

> 실행 계획은 데이터베이스가 쿼리를 처리하기 위해 어떤 경로를 선택했는지 보여주는 '내비게이션 지도'입니다.

### 확인 방법
SQL 문 앞에 `EXPLAIN` 키워드를 붙여서 실행합니다.

```sql
EXPLAIN SELECT * FROM users WHERE id = 1;
```

### 분석
- **type (접근 방식):** 성능을 결정짓는 가장 중요한 지표입니다.
    - `const/eq_ref`: 최고 (PK나 유니크 키로 단 한 건 조회)
    - `ref`: 좋음 (인덱스로 여러 건 조회)
    - `range`: 보통 (인덱스로 범위 조회)
    - `ALL`: 최악 (인덱스 안 타고 전체 테이블 스캔)
- **key:** 실제로 어떤 인덱스를 사용했는지 보여줍니다.
- **rows:** 쿼리 처리를 위해 읽어야 할 예상 행의 수입니다. 이 수치가 낮을수록 효율적인 쿼리입니다.
