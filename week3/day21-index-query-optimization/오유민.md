### 📌 [Q1] 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가?

**Why** <br>
인덱스는 특히 대용량 데이터베이스의 조회 성능을 향상시키는 중요한 기술이다.
만약 인덱스가 없다면, 데이터베이스는 우리가 찾는 정보가 어디 있는지 모르기 때문에 **처음부터 끝까지 모든 데이터를 탐색해야 한다.**

하지만 인덱스가 있다면, 데이터베이스는 **인덱스를 통해 찾고자 하는 데이터가 어디 있는지**를 빠르게 찾을 수 있다.

**How** <br>
인덱스는 데이터를 특정한 기준에 따라 **정렬된** 별도의 목록으로 저장한다.
- **이진 탐색과 트리 구조**
  - 인덱스는 주로 **B-Tree** 구조로 저장된다.
  - B-Tree는 **이진 탐색 트리**의 일종으로, 특정 값이 어디 있는지 빠르게 찾을 수 있도록 정렬된 구조를 유지한다. (가나다/ABC순처럼)
- **탐색 범위의 축소**
  - 데이터를 미리 정렬해 둔 목록(인덱스)을 먼저 보고 데이터가 위치한 주소값으로 바로 점프하기 때문에 성능이 매우 향상된다.

### 📌 [Q2] 인덱스는 어떻게 동작하는가? B-Tree 구조를 설명하면?

**B-Tree(Balanced Tree)란?** <br>
: B-Tree는 자식 노드가 2개 이상인 **다진 탐색 트리**이다.
- **항상 균형을 유지(Balanced)**
  - 어떤 값을 찾더라도 root에서 leaf 노드까지 가는 거리가 항상 동일 -> 성능이 향상 일정하게 유지됨
  - 노드 구성
    - root node: 최상위 노드
    - branch node(internal node): 자식 노드가 있는 중간 노드, 하위 노드로 가기 위한 key 값과 포인터를 가지고 있음
    - leaf node: 자식 노드가 없는 최하위 노드, 주소값(RID, Row ID)을 저장하는 노드

**B-Tree 인덱스 동작 과정**
```
ex) 1~100이 있는 트리에서 35 찾기
(1) 루트 노드 방문
: 루트 노드에 [20, 50]이라는 기준값이 있다면, 35는 20보단 크고 50보다 작으므로 그 사이의 포인터를 따라 중간 노드로 내려감

(2) 브랜치 노드 방문
: 1에서 내려온 노드에 [30, 40]이 있다면, 35는 30보다 크고 40보다 작으므로 다시 그 사이의 포인터를 따라 리프 노드로 내려감

(3) 리프 노드 도달
: 리프 노드에 [35]이 있다면, 35는 바로 해당 노드의 주소값(RID)을 반환

(4) 데이터 접근
: 찾은 위치의 주소값(RID)을 사용하여 데이터에 바로 접근
```
```
[ Root Node ]
      |
    [ 20 | 50 ]
   /     |      \
[ 1~19 ] [21~49] [51~100]  <-- Branch Nodes
             |
          [ 35 | Pointer to Data ] <-- Leaf Nodes
```

**왜 이진 트리가 아니라 B-Tree인가?**
- **디스크 I/O 최적화**
  - 일반적인 이진 트리는 자식이 딱 2개 뿐이라 트리가 매우 깊어짐
  - **B-Tree는 하나의 노드에 수십~수백 개의 키를 담을 수 있어 트리의 높이가 매우 낮게 유지됨**
- **성능 유리**
  - 데이터베이스는 하드디스크나 SSD에서 데이터를 읽어오는 비용이 가장 비싼데, **트리의 높이가 낮을수록 디스크를 읽는 횟수가 줄어들어** 성능이 압도적으로 향상됨

### 📌 [Q3] Hash Index와 B-Tree Index의 차이는? 각각 언제 사용하는가?

**Hash Index** <br>
: 해시 함수를 이용해 키 값을 특정 숫자로 변환하고, 이를 인덱스로 사용하는 방식 (메모리 상의 HashTable처럼)
- 조회하려는 값에 해시 함수를 적용해 결과(버킷 주소)가 나오면 그 주소로 바로 점프함
- **장점**
  - 정확히 일치하는 값을 찾을 때 속도 매우 빠름 (평균 시간 복잡도: O(1))
- **단점**
  - **범위 검색 불가**: 해시 값은 원래 값의 크기와 상관없이 생성되므로, age > 20과 같은 범위 검색이 불가능
  - **정렬 불가**: 데이터가 정렬되어 있지 않아 ORDER BY를 사용할 수 없음
  - **부분 일치 검색 불가**: 문자열의 뒷부분만 찾는 등의 검색 불가능
- 메모리 DB, 단일 값 조회 시 사용

**B-Tree Index** <br>
: B-Tree 구조를 사용하여 정렬된 트리 구조로 인덱스를 저장하는 방식
- 루트 노드부터 리프 노드까지 비교하며 내려감 (O(log n))
- **장점**
  - **범위 검색 가능**: age > 20과 같은 범위 검색이 가능
  - **정렬 가능**: 데이터가 정렬되어 있어 ORDER BY를 사용할 수 있음
  - **부분 일치 검색 가능**: 문자열의 뒷부분만 찾는 등의 검색 가능
- **단점**
  - 해시 인덱스보다 미세하게 느림
- 일반적인 모든 RDBMS에서 사용

대부분의 실무 상황(정렬, 범위 검색, 조인 등)에서는 B-Tree Index가 훨씬 유리하기 때문에 거의 모든 RDBMS는 B-Tree Index를 기본으로 사용한다.
반면, 오직 정확히 일치하는 단일값 조회만 빈번하고, 범위 검색이나 정렬이 전혀 필요 없는 (ex. Redis 같은 in-memory 저장소나 특수 목적용 테이블)에서는 Hash Index를 사용한다.

### 📌 [Q4] 클러스터드 인덱스와 논클러스터드 인덱스의 차이는?

**클러스터드 인덱스**
- 테이블 자체가 인덱스인 구조로, 테이블의 데이터를 인덱스 키 값에 따라 물리적으로 다시 정렬
- 리프 노드에 실제 데이터 페이지가 있음 (인덱스를 찾으면 바로 데이터가 그 자리에 있는 셈)
- 테이블당 단 하나만 생성할 수 있음 (물리적 정렬은 한 가지 기준으로만 가능하기 때문)
- 보통 Primary Key를 설정하면 자동으로 클러스터드 인덱스가 됨
- 조회 속도 매우 빠름
- ex) 전화번호부, 사전

**논클러스터드 인덱스**
- 데이터 파일은 건드리지 않고, 별도의 인덱스 페이지를 생성
- 리프 노드에 실제 데이터가 아닌, 데이터가 위치한 주소값(RID 또는 클러스터드 키)이 저장되어 있음
  - 인덱스 페이지에서 주소 찾기 -> 이 주소를 가지고 실제 데이터 페이지로 이동(Look up)해야 함
- 테이블당 여러 개 생성 가능
- 조회 속도는 클러스터드 인덱스보다 느리지만 빠름 (주소로 한 번 더 가야 함)
- ex) 전공 서적 맨 뒤의 찾아보기 (색인은 키워드별로 정렬되어 있고 해당 페이지 번호가 적혀있는 것처럼)

### 📌 [Q5] 인덱스의 단점은 무엇인가? (저장 공간, CUD 성능 저하)

**추가적인 저장 공간 필요**
- 인덱스는 데이터베이스 안에 별도로 생성되는 객체로, 별도의 저장 공간이 필요하다.
- 원본 테이블의 데이터를 정렬한 목록을 따로 저장해야 하므로, 보통 테이블 크기의 10~15%정도 추가 공간이 필요하다.

**CUD(CREATE, UPDATE, DELETE) 성능 저하**
- 인덱스는 조회(SELECT) 성능은 비약적으로 높여주지만, CUD(CREATE, UPDATE, DELETE) 연산 시, 인덱스도 함께 업데이트 해야 하므로, 추가적인 시간이 소요된다.
- 따라서 인덱스는 읽기 전용 기술이며, 쓰기 작업이 빈번한 시스템에서는 인덱스를 최소화해야 한다.

### 📌 [Q6] 어떤 컬럼에 인덱스를 설정해야 하는가? (카디널리티, 선택도)

- **카디널리티(Cardinality)**가 높을수록(중복도가 낮을수록) 인덱스 효율이 좋다. (ex. 성별보다는 이메일, 아이디)
- **선택도(Selectivity)**가 낮은 컬럼(약 5~10% 이내)을 선택하여 인덱스를 통해 걸러지는 데이터 양을 최소화해야 한다.
- 수정이 빈번한 컬럼보다는 조회가 잦은 WHERE, JOIN, ORDER BY 절에 자주 사용되는 컬럼이 우선순위가 높다.
- 데이터 양이 너무 적은 테이블은 인덱스보다 전체 스캔이 빠를 수 있으므로 규모를 고려해야 한다.

### 📌 [Q7] 복합 인덱스(Composite Index)는 언제 사용하며 컬럼 순서는 왜 중요한가?

- 여러 개의 조건이 동시에 결합되어 조회되는 패턴이 많을 때, 단일 인덱스 여러 개보다 복합 인덱스 하나가 더 효율적이다.
- 컬럼 순서는 탐색 범위를 결정하는 핵심으로, 카디널리티가 높은(중복이 적은) 컬럼을 앞쪽에 배치하는 것이 유리하다.
- 인덱스의 첫 번째 컬럼이 조건절(WHERE)에 포함되지 않으면 인덱스를 제대로 타지 못할 수 있으므로 쿼리 패턴을 분석해 순서를 정해야 한다.

### 📌 [Q8] 실행 계획(Execution Plan)은 어떻게 보고 분석하는가?

- 쿼리 앞에 EXPLAIN 키워드를 붙여 데이터베이스가 어떤 방식으로 데이터를 찾을 것인지 미리 확인할 수 있다.
- type 항목을 통해 ALL(전체 스캔)인지 range, ref, const(인덱스 사용)인지 확인하는 것이 가장 중요하다.
- key를 통해 의도한 인덱스가 실제로 쓰였는지, rows를 통해 쿼리 처리 과정에서 얼마나 많은 행을 검사하는지 체크하여 최적화한다.
- Extra 항목에 Using filesort나 Using temporary가 보인다면 성능 저하 요인이므로 인덱스 수정을 고려해야 한다.
