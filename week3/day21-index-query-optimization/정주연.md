## 🧠 학습 질문

## 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가?

- DB에서 테이블의 검색 성능 높여줌
    - e.g. RDB에서는 B+Tree구조로 된 index를 사용하여 검색 속도를 향상
- 인덱스가 없다면
    - 특정 데이터를 찾기 위해 테이블의 모든 레코드를 처음부터 끝까지 읽어야 하는 **Full Table Scan**이 발생
    - 데이터가 많아질수록 응답 속도가 기하급수적으로 느려지는 원인이 됨
- 성능 향상 원리
    - 컬럼의 값과 해당 레코드가 저장된 물리적 주소를 쌍으로 저장
    - 데이터를 찾을 때 테이블 전체를 뒤지는 대신 정렬된 인덱스로 접근 → **이진 탐색**과 유사한 방식으로 빠르게 위치를 찾아냄

## 인덱스는 어떻게 동작하는가? B-Tree 구조를 설명하면?

- Root Node, Branch Node, Leaf Node로 구성됨
- 각 노드는 정렬된 상태를 유지함
- 노드 내의 키 값들을 비교하며 자식 노드로 내려감
- Leaf Node에 도달하면 실제 데이터 레코드의 위치(RID/Primary Key)를 찾아 데이터에 접근
- 장점
    - 데이터가 추가되거나 삭제되어도 트리의 균형을 유지 → 어떤 데이터를 찾더라도 O(log N)의 일정한 시간 복잡도를 보장

## Hash Index와 B-Tree Index의 차이는? 각각 언제 사용하는가?

| **구분** | **Hash Index** | **B-Tree Index** |
| --- | --- | --- |
| **원리** | 해시 함수를 통해 값을 고정된 길이의 주소로 변환 | 트리 구조를 따라 순차적으로 탐색 |
| **조회 성능** | $O(1)$ (매우 빠름) | $O(\log N)$ |
| **범위 검색** | **불가능** (순서가 없음) | **가능** (정렬되어 있음) |
| **사용처** | 정확히 일치하는 값(`=`)을 찾을 때 | 부등호(`<`, `>`)나 정렬이 필요한 경우 |

## 클러스터드 인덱스와 논클러스터드 인덱스의 차이는?

- **클러스터드 인덱스(Clustered Index)**
    - **물리적 정렬**
        
        ⇒ 실제 데이터 레코드 자체가 인덱스 순서에 따라 물리적으로 정렬되어 저장됨
        
    - 테이블당 단 **1개**만 존재함 (보통 Primary Key가 자동으로 지정됨)
    - Leaf Node가 곧 실제 데이터 페이지임
- **논클러스터드 인덱스(Non-Clustered Index):**
    - **논리적 정렬
    ⇒** 별도의 공간에 인덱스 페이지를 만듦. 실제 데이터는 정렬되지 않은 채 남아있음
    - 테이블당 **여러 개** 생성 가능함. Leaf Node에는 실제 데이터가 아닌, 데이터를 찾기 위한 '주소값'이나 'Primary Key'가 들어있음

## 인덱스의 단점은 무엇인가? (저장 공간, CUD 성능 저하)

- **저장 공간 소모**
    - 인덱스라는 별도의 구조를 유지해야 하므로 추가적인 디스크 공간(보통 테이블 크기의 10~20%)이 필요함
- **CUD(쓰기) 성능 저하**
    - 데이터가 추가(Insert), 수정(Update), 삭제(Delete)될 때마다 인덱스도 함께 업데이트하고 트리 균형을 맞춰야 하므로 오버헤드가 발생함

## 어떤 컬럼에 인덱스를 설정해야 하는가? (카디널리티, 선택도)

- **카디널리티(Cardinality)가 높은 컬럼**
    - 중복된 값이 적고 유니크한 값이 많을수록 좋음 (예: 주민번호 > 이름 > 성별)
- **선택도가 낮은 컬럼**
    - 전체 레코드 중 특정 조건으로 걸러지는 비율이 낮을수록(보통 5~10% 이내) 인덱스 효율이 높음
- `WHERE`, `JOIN`, `ORDER BY` 절에서 자주 사용되는 컬럼을 우선적으로 고려함

## 복합 인덱스(Composite Index)는 언제 사용하며 컬럼 순서는 왜 중요한가?

- **개념**
    - 두 개 이상의 컬럼을 합쳐서 하나의 인덱스로 만드는 것임
- **순서의 중요성**
    - 복합 인덱스는 **왼쪽 컬럼부터 순서대로** 정렬됨.
    - 예를 들어 `(이름, 나이)` 순서로 인덱스를 만들면 '이름'으로 먼저 정렬된 후, 이름이 같은 경우에만 '나이'로 정렬됨
- **주의사항:** 첫 번째 컬럼을 조건절에서 누락하면 인덱스를 제대로 타지 못함 (Leftmost Prefix 법칙)

## 실행 계획(Execution Plan)은 어떻게 보고 분석하는가?

SQL 문 앞에 `EXPLAIN` 키워드를 붙여서 확인하며, DB 옵티마이저가 어떻게 쿼리를 실행할지 미리 보는 것임

- **핵심 체크 포인트**
    - `const`, `eq_ref`, `ref` 등은 인덱스를 잘 타는 좋은 신호임. `ALL`이 뜨면 Full Table Scan이므로 인덱스 점검이 필요함
    - **key:** 실제로 어떤 인덱스를 사용하는지 확인 가능함
    - **rows:** 쿼리 처리를 위해 조사해야 하는 예상 행 수임. 적을수록 효율적임
    - **Extra:** `Using filesort`나 `Using temporary`가 뜨면 정렬이나 임시 테이블 작업 때문에 성능이 떨어질 수 있으니 쿼리 튜닝이 필요함
