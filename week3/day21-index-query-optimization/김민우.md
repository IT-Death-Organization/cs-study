# 1. 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가

## 문제

DB는 기본적으로 **테이블을 Heap 구조(순차 저장)** 로 저장합니다.

예를 들어

```
SELECT * FROM user WHERE id = 100;
```

인덱스가 없다면 DB는

```
1행
2행
3행
...
N행
```

모든 행을 검사 (Full Table Scan) 해야 합니다.

시간복잡도

```
O(n)
```

---

## 인덱스가 있으면

인덱스는 **정렬된 자료구조**입니다.

예

```
id index

10
25
40
100
180
```

DB는 이 구조에서 바로 위치를 찾을 수 있습니다.

시간복잡도

```
O(log n)
```

따라서

```
Full Scan   → O(n)
Index Scan  → O(log n)
```

대용량 데이터에서 성능 차이가 크게 발생합니다.

---

# 2. 인덱스는 어떻게 동작하는가 (B-Tree 구조)

대부분 DB 인덱스는 **B-Tree** 입니다.

## 구조

```
        [50]
       /    \
   [20]     [80]
  /   \     /   \
10   30   60   90
```

특징

* 정렬된 트리
* 균형 트리 (높이가 항상 일정)
* 한 노드에 여러 key 저장

---

## 검색 과정

예

```
WHERE id = 60
```

검색

```
루트 50
→ 60 > 50
→ 오른쪽 이동

노드 80
→ 60 < 80
→ 왼쪽 이동

60 발견
```

시간복잡도

```
O(log n)
```

---

## DB에서 실제 구조

```
Root
 ├─ Internal Node
 │   ├─ Internal Node
 │   │   ├─ Leaf Node
 │   │   └─ Leaf Node
 │   └─ Internal Node
 └─ Internal Node
```

Leaf Node에는

```
index key + row pointer
```

또는

```
index key + 실제 row
```

가 저장됩니다.

---

# 3. Hash Index vs B-Tree Index

| 구분    | B-Tree   | Hash    |
| ----- | -------- | ------- |
| 검색 방식 | Tree 탐색  | Hash 계산 |
| 시간복잡도 | O(log n) | O(1)    |
| 범위 검색 | 가능       | 불가능     |
| 정렬    | 가능       | 불가능     |

---

## Hash Index

동작

```
hash(key) → bucket
```

예

```
hash(100) → bucket 3
```

장점

```
= 검색 매우 빠름
```

단점

```
범위 검색 불가능
```

예

```
WHERE id > 100
```

불가능합니다.

---

## B-Tree

다음 연산이 가능합니다.

```
=
>
<
BETWEEN
ORDER BY
```

그래서 대부분의 DB는 **B-Tree 인덱스**를 사용합니다.

---

# 4. 클러스터드 인덱스 vs 논클러스터드 인덱스

## Clustered Index

데이터 자체가 정렬됩니다.

```
[ID 순으로 저장된 테이블]

10
20
30
40
50
```

인덱스

```
ID → 실제 데이터
```

특징

```
테이블 자체가 인덱스 구조
```

그래서

```
클러스터드 인덱스는 1개만 존재
```

대표 예

* MySQL InnoDB Primary Key

---

## Non-Clustered Index

데이터는 따로 있고

```
index → pointer
```

구조

```
index

10 → row address
20 → row address
30 → row address
```

특징

```
여러 개 생성 가능
```

---

# 5. 인덱스의 단점

## 1. 저장 공간 증가

```
table data
+ index data
```

추가 디스크 공간이 필요합니다.

---

## 2. 쓰기 성능 저하

INSERT

```
1. 테이블 insert
2. 인덱스 insert
3. B-tree 재정렬
```

UPDATE

```
index 수정
```

DELETE

```
index 삭제
```

따라서

```
CUD 성능 감소
```

---

## 3. 관리 비용

인덱스가 많으면

```
optimizer 비용 증가
```

---

# 6. 어떤 컬럼에 인덱스를 설정해야 하는가

핵심 기준

## 1. 카디널리티 (Cardinality)

값의 종류 개수

예

```
user_id   → 매우 높음
gender    → 2개
```

좋은 인덱스

```
카디널리티가 높은 컬럼
```

---

## 2. 선택도 (Selectivity)

```
조건으로 선택되는 데이터 비율
```

예

```
WHERE gender='M'
```

전체의 약 50%

→ 인덱스 효과 낮음

좋은 예

```
WHERE user_id = 1234
```

→ 하나의 row만 조회

---

## 3. 자주 조회되는 컬럼

다음 구문에 자주 사용되는 컬럼

```
WHERE
JOIN
ORDER BY
GROUP BY
```

---

# 7. 복합 인덱스 (Composite Index)

예

```
INDEX (age, city)
```

정렬 구조

```
(age, city)

(20, Seoul)
(20, Tokyo)
(25, Seoul)
(30, Busan)
```

---

## 사용 가능한 쿼리

가능

```
WHERE age = 20
WHERE age = 20 AND city = 'Seoul'
```

불가능

```
WHERE city = 'Seoul'
```

이것을

```
Leftmost Prefix Rule
```

이라고 합니다.

---

## 컬럼 순서가 중요한 이유

예

```
INDEX (age, city)
```

쿼리

```
WHERE city='Seoul'
```

→ 인덱스 사용 불가

하지만

```
INDEX (city, age)
```

이면 사용 가능합니다.

---

# 8. 실행 계획 (Execution Plan) 분석

DB가 **어떤 방식으로 쿼리를 실행하는지 보여주는 것**입니다.

예 (MySQL)

```
EXPLAIN SELECT * FROM user WHERE id=10;
```

---

## 주요 컬럼

### type

접근 방식

좋은 순서

```
system
const
eq_ref
ref
range
index
ALL
```

최악

```
ALL
```

Full Table Scan을 의미합니다.

---

### key

사용된 인덱스

```
key: idx_user_id
```

---

### rows

예상 스캔 row 수

```
rows: 100000
```

값이 많을수록 성능 문제가 될 수 있습니다.

---

### extra

추가 정보

예

```
Using index
Using where
Using temporary
Using filesort
```

특히

```
Using temporary
Using filesort
```

는 성능 문제가 발생할 가능성이 있습니다.

---

# 면접에서 좋은 답변 구조

보통 다음 흐름으로 설명하면 좋습니다.

1. 인덱스 목적

```
Full Scan → Index Scan
O(n) → O(log n)
```

2. 내부 구조

```
대부분 B-Tree
균형 트리
```

3. 인덱스 종류

```
Hash vs B-tree
Clustered vs Non-clustered
```

4. 설계 기준

```
Cardinality
Selectivity
복합 인덱스
```

5. 단점

```
저장공간 증가
쓰기 성능 저하
```
