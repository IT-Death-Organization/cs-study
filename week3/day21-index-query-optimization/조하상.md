## 1. 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가?

**전체 스캔 방지와 탐색 효율성**
- 인덱스는 데이터베이스 테이블의 검색 속도를 높이기 위해 사용하는 별도의 자료구조다. 책의 맨 뒤에 있는 '찾아보기(색인)'와 같은 역할을 한다.
- 인덱스가 없다면 특정 데이터를 찾기 위해 테이블의 처음부터 끝까지 모든 레코드를 읽어야 하는 **Full Table Scan**이 발생한다. 하지만 인덱스를 활용하면 정렬된 정보를 바탕으로 이진 탐색과 유사한 방식으로 데이터를 찾아가기 때문에 조회 성능이 비약적으로 향상된다.

---

## 2. 인덱스는 어떻게 동작하는가? B-Tree 구조를 설명하면?

**균형 잡힌 다진 트리 구조**
- 대부분의 RDBMS는 인덱스를 구현할 때 **B-Tree(Balanced Tree)** 구조를 사용한다. 이는 모든 리프 노드(Leaf Node)가 동일한 깊이를 가지도록 설계된 트리다.
- **Root Node(최상위)** → **Branch Node(중간)** → **Leaf Node(최하위)** 순으로 구성되며, 각 노드 내의 데이터는 항상 정렬된 상태를 유지한다.
- 리프 노드에는 실제 데이터 레코드의 주소(또는 클러스터드 키 값)가 들어있어, 루트부터 키 값을 비교하며 내려가면 단 몇 번의 이동만으로 원하는 데이터에 도달할 수 있다.



---

## 3. Hash Index와 B-Tree Index의 차이는? 각각 언제 사용하는가?

- **Hash Index:** 키 값을 해시 함수를 통해 고유한 값으로 변환하여 매핑한다. $O(1)$의 매우 빠른 속도를 자랑하지만, **동등 비교(=)** 연산에서만 사용 가능하다. 범위 검색이나 정렬이 불가능하므로 주로 메모리 기반 DB나 임시 테이블에서 빠른 단건 조회가 필요할 때 사용한다.
- **B-Tree Index:** 데이터를 정렬된 상태로 유지하므로 동등 비교뿐만 아니라 **범위 검색(Between, <, >)**, 부동 일치 검색(Like 'abc%'), 정렬 처리에 모두 유효하다. 범용성이 높아 대부분의 실무 인덱스로 사용한다.

---

## 4. 클러스터드 인덱스와 논클러스터드 인덱스의 차이는?

- **클러스터드 인덱스 (Clustered Index):** 테이블당 단 하나만 존재할 수 있으며, 물리적으로 데이터가 인덱스 순서에 따라 정렬되어 저장된다. 책의 페이지 번호와 같으며, 주로 기본키(PK)가 이 역할을 수행한다. 리프 노드 자체가 실제 데이터 페이지다.
- **논클러스터드 인덱스 (Non-Clustered Index):** 물리적인 데이터 정렬은 그대로 두고 별도의 인덱스 페이지를 구성한다. 책 뒤의 찾아보기와 같으며, 리프 노드에는 데이터가 아닌 '데이터가 있는 위치 정보(포인터)'를 담고 있다. 한 테이블에 여러 개 생성이 가능하다.



---

## 5. 인덱스의 단점은 무엇인가? (저장 공간, CUD 성능 저하)

- **저장 공간 차지:** 인덱스는 별도의 자료구조이므로 테이블 크기의 약 10% 내외에 해당하는 추가 저장 공간이 필요하다.
- **CUD 성능 저하:** 데이터의 삽입(Insert), 수정(Update), 삭제(Delete)가 발생할 때마다 인덱스 구조도 실시간으로 정렬하고 갱신해야 한다. 특히 인덱스가 많을수록 쓰기 작업 시 발생하는 오버헤드가 커져 전체적인 성능이 떨어질 수 있다.

---

## 6. 어떤 컬럼에 인덱스를 설정해야 하는가? (카디널리티, 선택도)

좋은 인덱스 효율을 위해 다음 기준을 고려한다.
- **카디널리티(Cardinality):** 중복된 값이 적고 고유한 값이 많을수록(높을수록) 인덱스 효율이 좋다. (예: 주민번호 > 성별)
- **선택도(Selectivity):** 한 인덱스를 통해 전체 데이터 중 5~10% 이내를 걸러낼 수 있을 때 효과적이다.
- **활용도:** `WHERE` 절, `JOIN` 조건, `ORDER BY` 절에서 자주 사용되는 컬럼을 선택한다.

---

## 7. 복합 인덱스(Composite Index)는 언제 사용하며 컬럼 순서는 왜 중요한가?

**가장 왼쪽 접두사 원칙 (Leftmost Prefix Rule)**
- 두 개 이상의 컬럼을 합쳐서 하나의 인덱스를 만드는 방식이다. 여러 조건을 동시에 조회할 때 단일 인덱스 여러 개보다 효율적이다.
- **순서의 중요성:** 인덱스는 정의된 컬럼 순서대로 정렬된다. 첫 번째 컬럼이 조건에 없으면 두 번째 컬럼부터는 인덱스를 타지 못할 확률이 높다. 따라서 **카디널리티가 높은 컬럼을 앞쪽에 배치**하는 것이 검색 범위를 좁히는 데 유리하다.



---

## 8. 실행 계획(Execution Plan)은 어떻게 보고 분석하는가?

**EXPLAIN 명령어를 통한 분석**
- 쿼리 앞에 `EXPLAIN`을 붙여 데이터베이스가 어떻게 데이터를 찾을지 미리 확인한다.
- **주요 확인 항목:**
  - **type:** `const`, `eq_ref`, `ref`는 좋지만, `ALL`(Full Scan)이나 `index`는 성능 최적화가 필요하다는 신호다.
  - **key:** 실제로 인덱스를 탔는지, 어떤 인덱스를 사용했는지 확인한다.
  - **rows:** 쿼리 처리를 위해 읽어야 할 예상 레코드 수다. 이 숫자가 너무 크다면 인덱스 재설계나 쿼리 튜닝이 필요하다.
