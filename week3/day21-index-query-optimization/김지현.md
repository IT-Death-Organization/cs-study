## 1. 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가?

데이터베이스에서 인덱스(Index)는 **테이블의 데이터를 빠르게 찾기 위한 자료구조**이다.
기본적으로 테이블을 조회할 때 인덱스가 없으면 **Full Table Scan**이 발생한다.

### Full Table Scan

```
SELECT * FROM users WHERE id = 100;
```

테이블 구조

| id  | name |
| --- | ---- |
| 1   | A    |
| 2   | B    |
| 3   | C    |
| ... | ...  |
| 100 | Z    |

인덱스가 없다면 DB는 **1번부터 100번까지 모든 row를 순차적으로 확인**한다.

시간복잡도 관점

```
O(n)
```

### Index Scan

인덱스가 있으면 **별도의 정렬된 자료구조**에서 위치를 찾는다.

```
users_id_index
1 → row1
2 → row2
3 → row3
...
100 → row100
```

이 경우 탐색 복잡도

```
O(log n)
```

즉, **전체 데이터를 탐색하지 않고 필요한 위치로 바로 이동**한다.

정리

| 방식            | 동작                    | 시간복잡도 |
| --------------- | ----------------------- | ---------- |
| Full Table Scan | 전체 행 탐색            | O(n)       |
| Index Scan      | 인덱스 탐색 후 row 접근 | O(log n)   |

# 2. 인덱스는 어떻게 동작하는가? (B-Tree 구조)

대부분의 RDBMS(MySQL, PostgreSQL, Oracle)는 **B-Tree 인덱스**를 사용한다.

### B-Tree 특징

1. **정렬된 상태 유지**
2. **균형 트리**
3. 모든 leaf depth 동일

예시

```
                [30]
           /           \
       [10,20]        [40,50]
      /   |   \       /   |   \
    ... ... ...     ... ... ...
```

검색 과정

```
SELECT * FROM users WHERE id = 42
```

1. Root 노드 확인
2. 범위 비교
3. 해당 child 이동
4. Leaf node에서 실제 row pointer 확인

시간복잡도

```
O(log n)
```

### Leaf Node

Leaf node에는 보통

```
key + row pointer
```

예시

```
[10 → row1]
[20 → row2]
[30 → row3]
```

즉

```
인덱스 → 실제 데이터 위치
```

---

# 3. Hash Index vs B-Tree Index

| 구분      | B-Tree       | Hash        |
| --------- | ------------ | ----------- |
| 구조      | 트리         | 해시 테이블 |
| 검색      | O(log n)     | O(1)        |
| 범위 검색 | 가능         | 불가능      |
| 정렬      | 가능         | 불가능      |
| LIKE 검색 | 가능(prefix) | 불가능      |

### Hash Index

```
hash(id) → bucket
```

예시

```
hash(10) → bucket1
hash(20) → bucket3
hash(30) → bucket2
```

### 사용 상황

#### B-Tree

```
WHERE id = 10
WHERE id > 10
WHERE id BETWEEN
WHERE name LIKE 'abc%'
ORDER BY
```

#### Hash

```
WHERE id = 10
```

즉

```
Hash → 정확한 값 검색
B-tree → 대부분 상황
```

그래서 실제로는 **B-Tree가 거의 기본 인덱스**이다.

# 4. 클러스터드 인덱스 vs 논클러스터드 인덱스

## Clustered Index

테이블 데이터 자체가 **인덱스 순서로 저장**

```
[1,A]
[2,B]
[3,C]
[4,D]
```

특징

-   데이터 자체가 정렬
-   테이블당 1개
-   Primary Key가 보통 사용됨

MySQL InnoDB

```
PK = Clustered Index
```

---

## Non-Clustered Index

데이터와 **별도의 인덱스 구조**

```
index
name → row pointer

A → row1
B → row2
C → row3
```

실제 조회

```
Index → row pointer → 데이터
```

즉

```
1. 인덱스 탐색
2. 테이블 접근
```

이를

```
bookmark lookup
```

이라고 한다.

---

# 5. 인덱스의 단점

### 1. 저장 공간 증가

인덱스는 **별도의 자료구조**

```
table
+ index
+ index
+ index
```

큰 테이블에서 공간이 크게 증가.

---

### 2. CUD 성능 저하

INSERT / UPDATE / DELETE 시

```
table 변경
+
index 업데이트
```

예시

```
INSERT user
→ index에도 삽입
```

특히 B-tree는

```
node split
rebalance
```

발생 가능.

---

### 3. 너무 많은 인덱스 문제

```
index 10개
```

INSERT 시

```
index 10개 업데이트
```

→ write 성능 감소

# 6. 어떤 컬럼에 인덱스를 설정해야 하는가?

핵심 기준

```
카디널리티 (Cardinality)
선택도 (Selectivity)
```

---

## Cardinality

컬럼 값의 **유니크 정도**

예시

| 컬럼    | 카디널리티 |
| ------- | ---------- |
| user_id | 높음       |
| gender  | 낮음       |

좋은 인덱스

```
user_id
email
order_id
```

나쁜 인덱스

```
gender
status
boolean
```

---

## Selectivity

조건이 얼마나 **데이터를 줄이는가**

예시

```
WHERE age = 25
```

전체 1M rows 중

```
age 25 → 10 rows
```

→ 좋은 인덱스

하지만

```
status = ACTIVE
```

전체 1M 중

```
900k
```

→ 인덱스 효과 없음

---

# 7. 복합 인덱스 (Composite Index)

여러 컬럼을 묶은 인덱스

```
INDEX (A, B, C)
```

예시

```
WHERE A=1 AND B=2
```

### Leftmost Prefix Rule

복합 인덱스는 **왼쪽부터 사용 가능**

```
INDEX (A,B,C)
```

가능

```
A
A,B
A,B,C
```

불가능

```
B
B,C
C
```

---

### 컬럼 순서 중요한 이유

예시

```
INDEX (country, city)
```

쿼리

```
WHERE country='KR' AND city='Seoul'
```

효율적.

하지만

```
WHERE city='Seoul'
```

→ 인덱스 사용 불가.

---

### 순서 전략

보통

```
1. 카디널리티 높은 컬럼
2. WHERE 자주 사용
3. JOIN 조건
```

순으로 배치.

---

# 8. 실행 계획 (Execution Plan) 분석

DB는 쿼리를 실행하기 전에 **실행 계획을 생성**한다.

MySQL

```
EXPLAIN SELECT * FROM users WHERE id=10;
```

예시 결과

| id  | type | key          | rows |
| --- | ---- | ------------ | ---- |
| 1   | ref  | idx_users_id | 1    |

---

### 중요한 컬럼

#### type

접근 방식

| type  | 설명            |
| ----- | --------------- |
| ALL   | Full Table Scan |
| index | Index Scan      |
| range | 범위 검색       |
| ref   | Index lookup    |
| const | PK lookup       |

좋은 순서

```
const
ref
range
index
ALL
```

---

#### key

사용된 인덱스

```
idx_users_id
```

NULL이면

```
인덱스 사용 안함
```

---

#### rows

예상 스캔 row 수

```
rows = 1
```

→ 매우 좋음

```
rows = 1,000,000
```

→ 문제

---

### 실무 분석 방법

1️⃣ EXPLAIN 실행
2️⃣ Full Table Scan 확인
3️⃣ Index 생성 고려
4️⃣ Composite index 최적화
