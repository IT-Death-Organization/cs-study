## 1. 인덱스는 왜 필요하며 어떻게 조회 성능을 향상시키는가?
> 인덱스는 데이터베이스에서 **특정 열에 대해 색인을 생성하여 검색 속도를 높이는 데 필수적인 도구**입니다. 인덱스를 사용하면 데이터베이스는 전체 테이블을 스캔하는 대신 **인덱스를 활용해 인덱스 트리를 따라가며 필요한 데이터를 빠르게** 찾을 수 있습니다.

![title](https://velog.velcdn.com/images/seokhwan-an/post/35ba3cb4-ce8f-43f5-b634-3b6b26daa434/image.png)   

인덱스는 데이터 검색 속도를 획기적으로 높여주는 기능이지만, 무조건 성능이 좋아지는 것은 아닙니다. 특히 테이블의 레코드 수가 많고, **특정 조건으로 자주 조회되는 칼럼**에 인덱스를 설정하면 눈에 띄는 성능 향상을 경험할 수 있는 반면, **자주 갱신되거나 검색 조건에 거의 사용되지 않는** 칼럼에 인덱스를 추가할 경우 **오히려 성능 저하**가 발생할 수 있습니다.

### 1-1. 인덱스가 성능을 향상시키는 방식
일반적으로 데이터베이스에서 SELECT 쿼리를 실행할 때, 조건에 맞는 데이터를 찾기 위해 전체 테이블을 스캔합니다. (**Full Table Scan**) 이는 데이터가 많을수록 시간이 오래걸립니다.

하지만 인덱스를 이용하면 해당 조건에 맞는 데이터 위치를 바로 참조할 수 있어 **수 밀리초 단위로 응답 시간이 줄어드는 효과**를 얻을 수 있습니다.
> WHERE 조건에 자주 등장하는 컬럼이나 JOIN, ORDER BY, GROUP BY에 사용되는 컬럼은 인덱스 후보로 적합합니다.

> 조회 빈도가 높은 컬럼에 적절한 인덱스를 설정하면, 시스템 응답 시간이 비약적으로 향상되며 사용자 경험도 함께 개선됩니다.

## 2. 인덱스는 어떻게 동작하는가? B-Tree 구조를 설명하면?
> 인덱스는 주로 **B-트리(B-Tree)** 또는 **해시(Hash)** 구조를 사용하여 데이터를 저장합니다. 여기서 B-트리 구조는 가장 일반적으로 사용되는 인덱스 자료구조입니다.

### 2-1.  B-Tree의 특징
                    [50]
                   /    \
            [20, 30]    [70, 80]
            /  |  \      /  |  \
         [10][25][35] [60][75][90]
 
- **항상 정렬된 상태**를 유지합니다.
* **루트 노드에서 리프 노드까지의 거리가 동일**합니다. (Balanced)
* 데이터를 찾을 때 **O(log N)** 의 시간 복잡도를 가집니다.
> 100만 건의 데이터가 있다면
> - Full Table Scan : 최악의 경우 **100만 번** 확인
> - B-Tree Index : 약 **20번**의 비교로 찾음 (log₂ (1,000,000) ≒ 20)
### 2-2. B-Tree Index의 작동 방식
![title](https://velog.velcdn.com/images/calis_ws/post/c3411c4f-7425-4237-9777-4f605d1d9e36/image.png)   

- 데이터를 삽입할 때 자동으로 정렬된 트리 구조로 저장됩니다.
- 검색 시, 트리의 중간 노드를 통해 상위부터 하위 노드로 내려가며 데이터를 찾습니다.
- 이 과정에서 데이터의 크기에 관계없이 빠르게 데이터를 찾을 수 있습니다.
- 특히, **범위 검색(Range queries)** 이 필요한 경우, B-Tree는 매우 효과적입니다.

## 3. Hash Index와 B-Tree Index의 차이는? 각각 언제 사용하는가?

### 3-1. Hash Index
![title](https://velog.velcdn.com/images/calis_ws/post/2e6c0c73-4bf2-4e5f-8139-13d47df374b5/image.png)  
해시 테이블은 해시 함수를 통해 키를 주소(Index)로 변환하여 데이터를 관리합니다. 
### 구조 및 동작
- **해싱 메커니즘** : 키(Key)가 주어지면 알고리즘이 항목을 찾을 수 있는 위치(Bucket)를 계산합니다.
- **시간 복잡도** : 키 자체가 배열의 인덱스가 되므로 검색과 삽힙이 **O(1)** 로 매우 빠릅니다.
- **무작위성** : 해시 함수를 거치면 데이터가 무작위로 배치되므로 물리적인 정렬 순서가 없습니다.
### 적용 및 한계
- **정확한 일치(Exact Match)** : 고유 식별자 조회처럼 등호(=) 연산이 주를 이룰 때 최고의 성능을 냅니다.
- **범위 검색 불가** : 해시 값은 입력 값이 조금만 달라도 완전히 달라지기 때문에, 특정 범위 내의 데이터를 순차적으로 찾는 것이 불가능합니다.
### 3-2. B-Tree Index
![title](https://velog.velcdn.com/images/calis_ws/post/c3411c4f-7425-4237-9777-4f605d1d9e36/image.png) 
이진탐색트리를 확장한 **균형 트리(Balanced Tree)** 로, 노드 하나가 여러 개의 자식 노드를 가질 수 있는 m차 트리 구조입니다.
### 구조적 속성
- **노드 구성** : m차  B-Tree일 때, 각 노드는 최대 m-1개의 키를 가지며, 모든 키는 **오름차순**으로 정렬됩니다.
- **최소 조건** : 루트를 제외한 모든 노드는 최소 ⌈**m/2**⌉ 개의 자식을 가져야 합니다.
- **균형 유지** : 모든 리프 노드는 **동일한 레벨** 에 존재하여, 어떤 데이터를 찾더라도 탐색 시간이 O(log n)으로 일정합니다.
### 적용 및 장점
- **범위 검색의 효율성** : 데이터가 정렬되어 있어 `BETWEEN`, `>`, `<` 등의 범위 쿼리에서 뛰어난 성능을 보입니다.
- **범용성** : 정렬 (`ORDER BY`)이나 그룹핑(`GROUP BY`) 작업 시 별도의 연산 없이 인덱스를 그대로 활용할 수 있습니다.
### 3-3. Hash vs B-Tree 핵심 차이점 요약
| 구분 | Hash Index | B-Tree Index |
| --- | --- | --- |
| 검색 방식 |  해시 함수 계산 후 즉시 이동 (O(1)) | 루트부터 리프까지 트리 탐색(O(log n)) |
| 범위 검색 | **어려움**(전체 스캔 필요) | **매우 효과적**(정렬된 구조 활용) |
| 데이터 정렬 | 무작위 배치 (정렬되지 않음) | **키 값에 따라 정렬된 상태** 유지 |
| 인덱스 크기 | 상대적으로 작음 (충독 시 효율 저하) | Hash보다 클 수 있음 |
| 주요 연산 | 등호 비교 (`=`, `IN`) | 범위, 부등호, 정렬, 그룹핑 등 모든 연산 |

### 3-4. 언제 어떤 인덱스를 사용하는지?
### Hash Index를 쓰는 경우
-  **고유값 조회**: 세션 ID, 회원 토큰, 고유한 식별자 등 정확히 일치하는 값을 찾는 쿼리가 99%인 경우
- **In-Memory 캐시**: 메모리 기반 DB에서 아주 빠른 조회가 필요할 때 유용합니다.

### B-Tree Index를 쓰는 경우
- **일반적인 관계형 DB**: 대부분의 서비스 데이터(게시글, 주문 내역 등)는 날짜 범위 조회나 이름순 정렬이 필요하므로 B-Tree를 기본으로 사용합니다.
- **복합 조건 조회**: 여러 컬럼을 조합하여 필터링하거나 정렬해야 하는 상황에 최적입니다.

## 4. 클러스터드 인덱스와 논클러스터드 인덱스의 차이는?
> 클러스터드 인덱스는 테이블당 하나만 존재하며, 실제 데이터가 인덱스 순서대로 물리적으로 정렬됩니다. InnoDB에서는 PK가 클러스터드 인덱스입니다. 논클러스터드 인덱스는 여러 개 생성 가능하며, 인덱스에 키 값과 PK가 저장되어 있어 실제 데이터 조회 시 PK로 한 번 더 접근해야 합니다.
### 4-1. 클러스터드 인덱스 (Clustered Index)
![title](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Fblog.kakaocdn.net%2Fdna%2FdGsPxg%2FbtsQfyZn9lL%2FAAAAAAAAAAAAAAAAAAAAAMLjZLrnPH-qy4L-_l4zbP-IMIvRQyVUUXRM4ORdaDR0%2Fimg.png%3Fcredential%3DyqXZFxpELC7KVnFOS48ylbz2pIh7yKj8%26expires%3D1774969199%26allow_ip%3D%26allow_referer%3D%26signature%3DYDg%252FLd9elqK2FYaAn2%252Bcrif9MyE%253D)   
- 정의 : 데이터베이스 테이블에서 검색 성능을 향상시키기 위해 사용하는 보조 자료구조. 책의 목차와 유사한 역할 수행
- 목적 : 데이터 검색 속도 향상, 정렬/조인 성능 개선
- 단점 : 인덱스 생성/유지에 따른 추가 저장공간 필요, DML(INSERT/UPDATE/DELETE) 작업 시 성능 저하 가능
- 구조 : 일반적으로 B-Tree, B+Tree, Hash 등을 활용
### 4-2. 논클러스터드 인덱스 (Non-Clustered Index)
![title](https://img1.daumcdn.net/thumb/R1280x0/?scode=mtistory2&fname=https%3A%2F%2Fblog.kakaocdn.net%2Fdna%2FeXvvTn%2FbtsQdPuxPPn%2FAAAAAAAAAAAAAAAAAAAAAAGszBLpnx6FtwHkGMsIENfyjCGrHuELWaz5pOG2y6eW%2Fimg.png%3Fcredential%3DyqXZFxpELC7KVnFOS48ylbz2pIh7yKj8%26expires%3D1774969199%26allow_ip%3D%26allow_referer%3D%26signature%3DVvrHimEYh4fmwsnLNqngmeahskc%253D)   
- 정의 : 인덱스가 별도의 객체로 생성되고, 데이터는 원래 순서를 유지하면서 인덱스에서 실제 데이터의 위치 (Row Pointer)를 가리키는 구조
- 특징
1. 한 테이블에 여러개 생성 가능
2. 인덱스와 실제 데이터가 분리되어 존재 (실제 데이터는 정렬되어 있지 않음)
3. 임의 접근 (Random Access)에 유리
### 4-3. 클러스터드 인덱스 vs 논클러스터드 인덱스 비교

| 구분 | 클러스터드 인덱스 (Clustered) | 논클러스터드 인덱스 (Non-Clustered) |
| --- | --- | --- |
| 데이터 저장 방식 | **인덱스 순서**에 맞게 데이터 자체가 정렬되어 저장 | 인덱스와 데이터가 분리(데이터와 별도로 **인덱스 페이지**를 저), 인덱스는 데이터 위치를 가리킴 |
| 개수 제한 | 테이블당 1개 | 여러 개 생성 가능 |
| 성능 특성 | 범위 검색, 정렬, 순차 조회에 유리 | 임의 접근, 특정 값 조회에 유리 |
| 저장 구조 | 인덱스 = 데이터 | 인덱스 ≠ 데이터 (별도 구조 + 포인터) |
| 생성 예 | 기본 키 (Primary Key) | 보조 키 (Secondary Key), 자주 검색되는 칼럼 |
 > 인덱스는 데이터베이스 검색 성능 최적화에 필수적인 기술로, **사용 패턴에 맞는 인덱스 전략 수립**이 중요합니다.
> 클러스터드 인덱스는 **정렬 및 범위 조회 중심**, 논클러스터드 인덱스는 **다양한 조건 검색 중심**으로 활용됩니다.

## 5. 인덱스의 단점은 무엇인가? (저장 공간, CUD 성능 저하)
-  **추가 저장 공간 필요** : 인덱스는 데이터베이스의 특정열에 대한 위치 정보를 담고 있어 별도의 저장 공간을 차지합니다.
 - **CUD 성능(쓰기 성능) 저하** : 데이터베이스에서 데이터를 추가, 수정, 삭제할 때 테이블의 데이터뿐만 아니라 인덱스도 함께 갱신해야 하므로 쓰기 성능이 느려집니다.

## 6.  어떤 컬럼에 인덱스를 설정해야 하는가? (카디널리티, 선택도)
> - **카디널리티**(Cardinality) : 특정 컬럼에 포함된 **중복된 값의 정도**
> - **선택도**(Selectivity) : 전체 데이터 중 특정 조건에 의해 **데이터가 얼마나 잘 걸러지는지**에 대한 지표
> - **조회 활용도**(Query Utilization) : 특정 컬럼이 실제 SQL문의 **`WHERE`절, `JOIN`, `ORDER BY` 등에서 얼마나 자주 사용되는지**를 의미
> - **수정 빈도**(Update Frequency) : 인덱스로 지정된 컬럼의 데이터가 **얼마나 자주 변경(`INSERT`, `UPDATE`, `DELETE`)** 되는지를 의미

### 6-1. 카디널리티
한 컬럼의 중복도가 낮을수록 인덱스 설정에 유리합니다. **중복도가 낮을수록**(카디널리티가 높을수록) 인덱스를 통해 필터링되는 데이터 양이 적어져 **검색 속도가 크게 향상**됩니다.
- 높은 카디널리티 : 중복된 값이 적고 고유한 값이 많다는 뜻입니다. (ex. 주민등록번호, 회원 ID)
- 낮은 카디널리티 : 중복된 값이 많다는 뜻입니다. (ex. 성별, 결제 여부)
### 6-2. 선택도
한 컬럼의 값으로 여러 행이 찾아질 수 있는 경우, 선택도가 높아 인덱스 설정에 유리합니다. 선택도가 높은 컬럼일수록 인덱스 효율이 좋습니다.
- 계산 방식

`Selectivity = (Cardinality / Total Rows) x 100`
### 6-3. 조회 활용도
특정 컬럼이 실제 작업에 얼마나 활용되는지에 대한 지표로, 조회 활용도가 높으면 인덱스 설정에 유리합니다.
- 아무리 카디널리티가 높고 선택도가 좋아도, 실제 쿼리에서 필터 조건으로 쓰이지 않는다면 인덱스를 만들 필요가 없습니다.
- 자주 조회되는 조건에 인덱스를 설정해야 전체적인 시스템 성능 최적화가 가능합니다.

### 6-4. 수정 빈도
인덱스로 지정된 컬럼의 값이 자주 변경되면 인덱스 테이블도 새롭게 갱신되어야 하므로, 수정 빈도가 낮은 컬럼에 인덱스를 설정하는 것이 좋습니다. (∵데이터 수정이 너무 빈번한 컬럼에 인덱스를 걸면, 조회는 빨라질지 몰라도 쓰기 성능(CUD)이 크게 저하됨)

> 결론적으로, 인덱스는 **"자주 조회되지만(활용도↑), 데이터가 잘 변하지 않고(수정 빈도↓), 중복이 적어 데이터 구분이 확실한(카디널리티↑, 선택도↑)"** 컬럼에 설정하는 것이 가장 이상적입니다.

## 7. 복합 인덱스(Composite Index)는 언제 사용하며 컬럼 순서는 왜 중요한가?
> 복합 인덱스는 첫 번째 컬럼을 기준으로 정렬되고, 같은 값 내에서 두 번째 컬럼으로 정렬됩니다. 따라서 WHERE 절에서 첫 번째 컬럼이 조건에 포함되지 않으면 인덱스를 효율적으로 사용할 수 없습니다.

`CREATE INDEX idx_location_lat_lng_alt_ts
ON location(latitude, longitude, altitude, 'timestamp');`

- 복합 인덱스는 여러 개의 컬럼을 한 번에 묶어서 인덱스를 생성하는 방식입니다.
- 복합 인덱스는 데이터베이스에서 SELSCT문의 WHERE 절에서 자주 사용되는 쿼리에 유용합니다. 복합 인덱스는 여러 컬럼을 묶어 정렬해 데이터를 빠르게 검색할 수 있도록 도와줍니다.
### 7-1. 복합 인덱스의 장점
- 여러 개의 조건(latitude, longitude, altitude)을 **한 번에 필터링**이 가능합니다.
- **불필요한 인덱스 탐색을 줄여서 속도가 빠릅니다.**
- **한 번의 검색으로 데이터를 가져올 수 있습니다.**
### 7-2. 컬럼 순서는 왜 중요한가?
- 컬럼 순서는 인덱스의 성능에 큰 영향을 미치며, 쿼리에서 어떤 컬럼을 먼저 조건으로 사용하는지에 따라 인덱스가 어떻게 활용되는지 결정됩니다. 따라서 복합 인덱스를 설계할 때는 컬럼 순서를 잘 고려해야 합니다.
## 8. 실행 계획(Execution Plan)은 어떻게 보고 분석하는가?
> 실행 계획(Execution Plan)은 데이터베이스 옵티마이저가 SQL 쿼리를 처리하기 위해 수립한 **'최적의 경로 지도'** 입니다.

### 8-1. 분석 도구 및 확인 방법
각 DBMS에 맞는 키워드를 사용해 실행 계획을 확인할 수 있습니다.
- **MySQL / PostgreSQL**: EXPLAIN
- **Oracle**: EXPLAIN PLAN FOR
- **실측 데이터 포함 (MySQL 8.0.18+ / PostgreSQL)**: EXPLAIN ANALYZE (예상치가 아닌 실제 실행 시간과 행 수를 출력)

### 8-2. 주요 지표 분석 (MySQL 기준)
실행 계획 결과에서 가장 핵심적인 컬럼들의 의미는 다음과 같습니다.
### ① id : 실행순서
- 쿼리 내의 각 SELSCT 블록에 부여되는 번호입니다.
- 분석법 : 숫자가 클수록 하위 쿼리이거나 먼저 실행되며, 숫자가 같으면 위에서 아래 순서대로 조인이 실행됩니다.

### ② type : 테이블 접근 방식 (성능 핵심)
데이터를 어떻게 찾아가는지를 나타내며, 아래 순서대로 성능이 좋습니다.

`system > const > eq_ref > ref > range > index > ALL`


| 타입 | 설명 | 성 |
| --- | --- | --- |
| const/system | PK나 유니크 키로 1건만 조회 (상수 취급) | 최고 |
| eq_ref | 조인 시 다른 테이블에서 PK나 유니크 키로 정확히 1건 매칭 | 매우 좋음 |
| ref | 유니크하지 않은 인덱스로 동등 비교(`=`) 검색 | 좋음 |
| range | 인덱스를 활용한 범위 검색 (`<`, `>`, `BETWEEN`, `IN`) | 보통 |
| index | **인덱스 풀 스캔**. 테이블은 안 읽지만 인덱스 전체를 스캔 | 나쁨 |
| ALL | **테이블 풀 스캔**. 인덱스 없이 전체 테이블을 스캔 | 최악 (튜닝 필수) |

### ③ key & key_len : 인덱스 사용 실측
- key: 실제로 선택된 인덱스입니다. possible_keys에 후보가 많아도 key가 NULL이라면 인덱스를 타지 못한 것이므로, 성능 저하 시 가장 먼저 확인해야 합니다.
- key_len: 인덱스 사용 길이를 바이트(Byte) 단위로 나타냅니다. 복합 인덱스에서 몇 번째 컬럼까지 실제 조회에 사용되었는지 확인하여 인덱스 설계의 효율성을 판단할 수 있습니다.

### ④ rows & filtered
- rows: 옵티마이저가 쿼리 처리를 위해 조사를 예측한 행 수입니다. 수치가 클수록 성능 병목 가능성이 큽니다.
- filtered: 필터 조건을 만족하는 레코드 비율(%)입니다. `rows * (filtered / 100)`이 실제 처리 예상 행 수가 됩니다.

### ⑤ Extra: 추가 실행 정보
쿼리의 성능 저하 요인을 파악할 수 있는 중요한 힌트입니다.
- **Using index: 커버링 인덱스**. 테이블 접근 없이 인덱스만으로 결과 도출 (매우 효율적)
- **Using where**: 인덱스 추출 후 MySQL 엔진에서 다시 필터링을 수행함
- **Using filesort** : 인덱스로 정렬하지 못해 메모리나 디스크에서 별도 정렬 (성능 저하)
- **Using temporary** : 연산을 위해 임시 테이블을 생성 (매우 느려질 수 있음)

### 8-3. 실행 순서 시각화 이해하기
실행 계획의 `id`와` table `순서를 통해 쿼리가 어떻게 흐르는지 파악해야 합니다.
- **하위 쿼리 우선**: 서브쿼리(`id`가 큰 것)가 먼저 실행되어 결과를 반환합니다.

- **조인 순서**: 같은 `id` 내에서는 위에 표시된 테이블이 **드라이빙 테이블(먼저 읽는 테이블)**이 됩니다.

### 8-4. 실행 계획 분석 프로세스 (Step-by-Step)
1. `type` 확인: ALL이나 index가 보인다면 왜 인덱스를 타지 못했는지 분석합니다.
2. `key` 확인: 내가 의도한 인덱스가 실제로 쓰이고 있는지 확인합니다.
3. `rows` 확인: 읽어들이는 행의 수가 실제 데이터 규모에 비해 너무 많지는 않은지 체크합니다.
4. `Extra` 확인: `filesort`나 `temporary`가 있다면 인덱스 순서를 변경하거나 쿼리를 리팩토링하여 정렬 비용을 줄입니다.
