## Q1. 버블 정렬은 왜 비효율적인가? 어떤 경우에도 O(n²)인가?

[답변]
* 비교: 인접한 원소
* 이동: 한 칸씩
=> 원소를 제자리로 보내기 위해 불필요한 비교/교환이 많음

* 기본: 이미 정렬돼있어도 비교 수행 => 최선/평균/최악 모두 O(n^2)
* 조기 종료: 스왑이 한 번도 없으면 종료 => 이미 정렬된 경우 최선 O(n)
  - 한 패스 동안 교환이 없으면 이미 정렬됨 => 종료

---

## Q2. 선택 정렬은 왜 항상 O(n²)인가? 최선의 경우도 마찬가지인 이유는?

[답변]
* 매 단계마다 남은 구간에서 최소 원소를 찾는 과정이 필요 => 남은 원소들을 끝까지 전부 비교
* i번째 위치에 들어갈 원소 선택 => (n-1)개 확인
  => (n-1) + (n-2) + ... + 1 = n(n-1)/2
  : 입력 상태와 무관하게 O(n^2) 고정 (비교를 줄일 수 없음)
* 교환 적음, 비교 많음

---

## Q3. 삽입 정렬이 거의 정렬된 배열에서 O(n)에 가까운 이유는?

[답변]
* 비용: 현재 원소를 앞쪽의 정렬된 구간에 삽입하기 위해 '얼마나 뒤로 밀어야 하는가'
* 거의 정렬 -> 대부분의 원소가 제자리에 가까움: 비교만 몇 번, 이동 거의 X
* 시간 복잡도: 역전(inversion)의 개수에 비례 -> 역전이 적은 입력에선 총 작업량이 작아져 O(n+k)처럼 동작
  - n: 각 i에 대해 삽입 시작/종료 비교 + key 저장/대입
  - k: 원소 shift
  - 최선의 경우: k=0 -> O(n)
  - 최악의 경우: k=n(n-1)/2 ( n-1 + n-2 + ... 1 ) -> O(n^2)

## Q4. 안정 정렬(Stable Sort)이란 무엇이며 왜 중요한가?

[답변]
* 안정 정렬: 정렬 키가 같은 원소(값이 같음)들 사이에서 정렬 전의 상대적 순서가 정렬 후에도 유지되는 정렬
* 중요한 이유
  - 다중 키 정렬 구현 용이: ex. 이름 정럴 + 같은 이름일 시 가입일 순서 유지 (2차 키 보존)
  - 데이터가 레코드 형태: 사용자가 기대하는 '기존 순서' 유지 -> 예측 가능성 높음

---

## Q5. 세 정렬 중 어떤 것이 안정 정렬인가? 그 이유는?

[답변]
* 버블 정렬: 안정 정렬
  - 같은 값인 경우 교환 X
* 삽입 정렬: 안정 정렬
  - 같은 값인 경우 앞 값의 '뒤'에 삽입
* 선택 정렬: 안정 정렬 X
  - 최솟값을 찾아 i(현재값)와 교환 -> 최솟값과 i 사이에 같은 값의 원소 존재 가능
  - ex. [ 2(a), 2(b), 1 ] -> [ 1, 2(b), 2(a) ]

[보완]
* 구현 방식에 따라 달라질 수 있음

---

## Q6. 실무에서 이 세 정렬을 사용하는 경우가 있는가?

[답변]
* 버블 정렬: 드물음
* 선택 정렬: 거의 사용하지 않으나, 교환 횟수가 적어 쓰기가 매우 비싼 특수한 환경에서 고려 가능
* 삽입 정렬: 사용
  - 배열 크기가 작을 때, 거의 정렬된 데이터일 때
  - 하이브리드: 고급 정렬(퀵/병합/힙) -> 작은 구간으로 쪼개졌을 때 삽입 정렬로 마물

[보완]
* 실무 표준: 언어/라이브러리가 제공하는 정렬 - 내부적으로 하이브리드/최적화된 알고리즘 사용
* 하이브리드에서 삽입 정렬 선호 이유: 작은 n에서 빠름, 분기 예측/캐시 유리, 구현 단순 => 상수항 이점 큼
