## 1. TCP와 UDP의 차이는? 각각 어떤 상황에서 사용하는가?

**TCP (Transmission Control Protocol)**
- 연결 지향형으로 1:1 통신을 하며 신뢰성을 보장한다.
- 데이터의 순서가 중요하고 유실되면 안 되는 웹(HTTP), 파일 전송(FTP), 이메일(SMTP) 등에 사용된다.

**UDP (User Datagram Protocol)**
- 비연결형으로 정보를 주고받을 때 확인 절차가 없어 속도가 빠르다.
- 실시간성이 중요하여 약간의 데이터 손실을 감수하는 스트리밍, VoIP, 온라인 게임 등에 사용된다.

---

## 2. TCP는 어떻게 신뢰성을 보장하는가? (ACK, 재전송, 순서 보장)

신뢰성 있는 전송을 위해 다음 메커니즘을 사용한다.
- **ACK (Acknowledgement):** 수신 측이 데이터를 잘 받았다고 응답을 보낸다.
- **재전송 (Timeout):** 송신 후 일정 시간 내에 ACK가 오지 않으면 패킷이 유실된 것으로 간주하고 다시 보낸다.
- **순서 보장 (Sequence Number):** 각 패킷에 번호를 부여하여, 수신 측에서 섞인 패킷을 원래 순서대로 재조립한다.

---

## 3. 3-way handshake의 동작 과정과 존재 이유는?

- 연결을 설정하는 과정이다.
1. **Client → Server (SYN):** 연결 요청
2. **Server → Client (SYN + ACK):** 요청 수락 및 역연결 요청
3. **Client → Server (ACK):** 최종 연결 확인

**존재 이유**
- 양쪽 모두 데이터를 송신하고 수신할 준비가 되었음을 보장하기 위함이다.
- 통신 시작 전 초기 시퀀스 번호를 동기화하여 데이터 유실 및 중복을 방지한다.

---

## 4. 4-way handshake의 동작 과정은? TIME_WAIT 상태는 왜 필요한가?

연결을 해제하는 과정이다.
1. **Client → Server (FIN):** 연결 종료 선언
2. **Server → Client (ACK):** 확인 (남은 데이터 전송 대기)
3. **Server → Client (FIN):** 서버도 종료 준비 완료
4. **Client → Server (ACK):** 최종 확인

**TIME_WAIT 상태가 필요한 이유**
- 마지막 ACK가 서버에 전달되지 않았을 경우를 대비해 일정 시간 기다려 서버가 FIN을 다시 보낼 수 있게 한다.
- 이전 연결의 지연 패킷이 다음 연결에 영향을 주지 않도록 소멸 시간을 벌기 위함이다.

---

## 5. 흐름 제어와 혼잡 제어의 차이는? 각각 어떻게 동작하는가?

**흐름 제어 (Flow Control)**
- 송신측과 수신측의 속도 차이를 해결한다. 수신측 버퍼가 넘치지 않게 조절하는 것이 핵심이다. (Stop and Wait, Sliding Window 사용)

**혼잡 제어 (Congestion Control)**
- 네트워크 전체의 데이터 혼잡도를 조절한다. 네트워크 내 패킷 수가 과도하게 증가하여 전송 속도가 떨어지는 것을 방지한다. (Slow Start 등 사용)

---

## 6. Sliding Window는 어떻게 동작하는가?

- 수신 측에서 설정한 윈도우 크기(Window Size)만큼 **ACK 없이도 연속해서 데이터를 보낼 수 있게 하는 기법**이다.
- 수신 측으로부터 확인 응답(ACK)을 받으면, 윈도우를 오른쪽으로 밀어서(Slide) 다음 범위의 데이터를 전송한다.
- 매 패킷마다 ACK를 기다릴 필요가 없어 전송 효율이 대폭 향상된다.

---

## 7. UDP는 언제 사용하는가? (실시간 스트리밍, DNS, DHCP 등)

- 신뢰성보다 지연 시간(Latency) 최소화가 우선일 때 사용한다.
- **실시간 스트리밍 & VoIP:** 실시간 대화나 영상은 끊김 없는 재생이 중요하다.
- **DNS (Domain Name System):** 짧은 요청과 응답으로 끝나며 연결 설정 오버헤드를 줄여야 한다.
- **DHCP:** IP 주소 할당을 위해 네트워크에 브로드캐스트 통신을 해야 할 때 사용된다.
